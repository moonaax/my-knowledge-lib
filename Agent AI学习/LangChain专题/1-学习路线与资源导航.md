## LangChain 全景概览

### 生态组成

LangChain 已从一个单体框架演进为一个完整的 LLM 应用开发生态系统，包含以下核心组件：

| 组件 | 定位 | 说明 |
|------|------|------|
| **langchain-core** | 基础抽象层 | 定义核心接口（BaseLanguageModel、BaseTool、BaseRetriever 等）和 LCEL 表达式语言。所有其他包的基石，极少外部依赖 |
| **langchain** | 主框架 | 提供 Chains、Agents、Memory 等高级认知架构的实现。依赖 langchain-core |
| **langchain-community** | 社区集成 | 第三方集成的集合（各种 LLM Provider、VectorStore、DocumentLoader 等）。从 langchain 主包拆分而来 |
| **langchain-{provider}** | 官方集成包 | 如 langchain-openai、langchain-anthropic、langchain-google 等，由官方维护的高质量集成 |
| **LangGraph** | 有状态编排 | 基于图的 Agent 编排框架，支持循环、条件分支、持久化状态，是构建复杂 Agent 的推荐方式 |
| **LangServe** | 部署服务化 | 将 LangChain Runnable 一键部署为 REST API，基于 FastAPI，内置 Playground |
| **LangSmith** | 可观测平台 | 提供 Trace 追踪、评估（Evaluation）、数据集管理、Prompt 管理等生产化能力 |

### 版本演进历史

#### v0.1（2024 年初）— 统一与拆分

- **重大变化**：将原来的单体 `langchain` 包拆分为 `langchain-core` + `langchain` + `langchain-community`
- 引入 LCEL（LangChain Expression Language）作为核心编排范式
- 第三方集成开始迁移到独立的 `langchain-{provider}` 包
- 废弃了大量 legacy Chain 类（LLMChain、SequentialChain 等），推荐使用 LCEL

#### v0.2（2024 年中）— 稳定与增强

- **重大变化**：`langchain-community` 中的主流集成全部迁移到独立包（langchain-openai 等）
- 移除了对 Python 3.8 的支持
- `ChatModel.bind_tools()` 成为标准工具绑定方式
- Agent 推荐使用 `create_tool_calling_agent` + `AgentExecutor`，或迁移到 LangGraph
- Memory 模块开始标记为 legacy，推荐在 LangGraph 中用 State 管理对话历史

#### v0.3（2024 年底）— 精简与成熟

- **重大变化**：移除所有在 v0.2 中标记为 deprecated 的 API
- 内部全面使用 Pydantic v2（不再兼容 v1）
- `langchain-core` 中的 `BaseMessage` 体系进一步标准化
- LangGraph 成为构建 Agent 的官方推荐方式，`AgentExecutor` 标记为 legacy
- 流式输出（Streaming）和异步（Async）支持全面完善

> **建议**：新项目直接使用 v0.3+，老项目参考 [官方迁移指南](https://python.langchain.com/docs/versions/v0_3/) 逐步升级。

## 学习路线详解

> 总周期约 6-9 周，建议每天投入 1-2 小时。每个阶段都有明确的学习目标和动手项目，确保学以致用。

### 第一阶段：核心基础（1-2 周）

**学习目标**：理解 LangChain 的核心抽象，能够使用 LCEL 编排简单的 LLM 调用链，掌握多轮对话的记忆管理。

**核心知识点**：

1. **Model I/O（模型输入输出）**
   - ChatModel vs LLM：理解 `BaseChatModel` 接口，为什么 ChatModel 是主流
   - Prompt Template：`ChatPromptTemplate`、`MessagesPlaceholder`、Few-shot 模板
   - Output Parser：`StrOutputParser`、`JsonOutputParser`、`PydanticOutputParser`
   - 模型配置：temperature、max_tokens、stop sequences、模型切换

2. **LCEL 表达式语言**
   - 管道操作符 `|`：将 Prompt → Model → Parser 串联
   - `RunnablePassthrough`：透传输入数据
   - `RunnableParallel`：并行执行多个分支
   - `RunnableBranch`：条件路由
   - `RunnableLambda`：自定义函数包装
   - `.batch()`、`.stream()`、`.ainvoke()` 等调用方式

3. **Memory 记忆管理**
   - `ConversationBufferMemory`：完整历史
   - `ConversationSummaryMemory`：摘要记忆
   - `ConversationBufferWindowMemory`：滑动窗口
   - 在 LCEL 中手动管理消息历史（v0.3 推荐方式）
   - `RunnableWithMessageHistory` 的使用

**动手项目**：构建一个多轮对话 CLI 助手，支持上下文记忆、流式输出、可切换不同 LLM Provider。

- [[LCEL 表达式语言]] — 管道编排、RunnableParallel、RunnableBranch

**关联知识库文档**：

### 第二阶段：工具与 Agent（1-2 周）

**学习目标**：掌握 Tool 定义与绑定机制，理解 Agent 的运行原理（ReAct 循环），能够构建可调用外部工具的智能 Agent。

**核心知识点**：

1. **Tool 工具体系**
   - `@tool` 装饰器：快速定义工具
   - `StructuredTool`：复杂参数的工具定义
   - 工具描述的重要性：LLM 根据描述决定何时调用
   - `BaseTool` 自定义：异步工具、错误处理
   - `ChatModel.bind_tools()`：将工具绑定到模型

2. **Agent 构建**
   - Tool Calling Agent：`create_tool_calling_agent`（v0.2+ 推荐）
   - ReAct Agent：理解 Reasoning + Acting 循环
   - `AgentExecutor`：Agent 运行时，max_iterations、error_handling
   - Agent 与 LangGraph Agent 的区别与选择

3. **Callback 与流式输出**
   - `CallbackHandler`：自定义事件监听
   - `StreamingStdOutCallbackHandler`：流式打印
   - `astream_events()`：v0.2+ 推荐的流式事件 API
   - 与 LangSmith 的集成追踪

**动手项目**：构建一个能搜索网页、查询天气、执行计算的多工具 Agent，支持流式输出和错误重试。

### 第三阶段：RAG 实战（1-2 周）

**学习目标**：掌握完整的 RAG 流水线，从文档加载、分块、向量化到检索增强生成，能够构建生产级的知识库问答系统。

**核心知识点**：

1. **Document Loader（文档加载）**
   - `TextLoader`、`PyPDFLoader`、`CSVLoader`、`WebBaseLoader`
   - `DirectoryLoader`：批量加载目录
   - 自定义 Loader：继承 `BaseLoader`
   - `Document` 对象：`page_content` + `metadata`

2. **Text Splitter（文本分块）**
   - `RecursiveCharacterTextSplitter`：最常用，递归按分隔符切分
   - `TokenTextSplitter`：按 Token 数切分
   - `MarkdownHeaderTextSplitter`：按 Markdown 标题切分
   - chunk_size 与 chunk_overlap 的调优策略
   - 分块质量对 RAG 效果的关键影响

3. **VectorStore（向量存储）**
   - Embedding 模型选择：OpenAI、HuggingFace、本地模型
   - `Chroma`：轻量级本地向量库，适合开发测试
   - `FAISS`：高性能向量检索，适合大规模数据
   - `Pinecone`、`Weaviate`、`Milvus`：云端/分布式方案
   - 向量库的持久化与增量更新

4. **Retriever（检索器）**
   - `VectorStoreRetriever`：基础向量检索
   - `MultiQueryRetriever`：多查询扩展，提升召回率
   - `SelfQueryRetriever`：结构化元数据过滤
   - `EnsembleRetriever`：混合检索（向量 + BM25）
   - `ContextualCompressionRetriever`：检索后压缩

5. **RAG Chain 构建**
   - 基础 RAG：Retriever → Prompt → LLM → Output
   - 带来源引用的 RAG
   - 对话式 RAG：结合历史消息重写查询
   - RAG 评估：使用 LangSmith 评估检索和生成质量

**动手项目**：构建一个基于本地文档的知识库问答系统，支持 PDF/Markdown 导入、语义检索、来源引用、对话式追问。

### 第四阶段：LangGraph 进阶（1-2 周）

**学习目标**：掌握 LangGraph 的图编排模型，能够构建有状态、可循环、支持人机协作的复杂 Agent 系统，理解 Multi-Agent 协作模式。

**核心知识点**：

1. **StateGraph 基础**
   - `StateGraph`：定义状态 Schema（TypedDict / Pydantic）
   - 节点（Node）：普通函数或 Runnable，接收 State 返回更新
   - 边（Edge）：`add_edge`（固定）、`add_conditional_edges`（条件路由）
   - `START` / `END`：入口和出口节点
   - 状态更新机制：`Annotated[list, operator.add]` 等 reducer
   - `graph.compile()`：编译为可执行的 `CompiledGraph`

2. **循环与自纠错 Agent**
   - 循环结构：Agent 节点 → Tool 节点 → 条件判断 → 回到 Agent
   - 反思（Reflection）：Agent 自我评估输出质量
   - 重试机制：工具调用失败后的自动重试
   - 最大迭代限制：防止无限循环
   - `interrupt_before` / `interrupt_after`：人机协作断点

3. **持久化与检查点**
   - `MemorySaver`：内存检查点（开发用）
   - `SqliteSaver` / `PostgresSaver`：持久化检查点
   - `thread_id`：会话隔离
   - 状态回溯：从任意检查点恢复执行

4. **Multi-Agent 编排**
   - Supervisor 模式：一个主 Agent 调度多个子 Agent
   - 协作模式：Agent 之间直接传递消息
   - 层级模式：多层 Supervisor 嵌套
   - 共享状态 vs 独立状态的设计选择
   - `Command` 对象：跨节点通信

**动手项目**：构建一个研究助手 Multi-Agent 系统，包含搜索 Agent、分析 Agent、写作 Agent，由 Supervisor 协调，支持人工审核关键步骤。

### 第五阶段：生产化（1 周）

**学习目标**：掌握将 LangChain 应用部署为生产服务的完整流程，包括 API 服务化、可观测性、性能优化和容错降级。

**核心知识点**：

1. **LangServe 部署**
   - 将 Runnable 部署为 FastAPI 服务：`add_routes()`
   - 内置 Playground UI：交互式测试
   - 输入/输出 Schema 自动生成
   - 流式响应支持
   - 自定义中间件：认证、限流、日志
   - 与 Docker / Kubernetes 的部署集成

2. **LangSmith 深度使用**
   - Trace 追踪：可视化每一步的输入输出、耗时、Token 用量
   - 评估（Evaluation）：自定义评估器、批量评估
   - 数据集管理：构建测试集、回归测试
   - Prompt Hub：团队共享 Prompt 模板
   - 在线监控：生产环境的实时追踪与告警
   - 标注与反馈：收集用户反馈改进系统

3. **性能优化与降级**
   - 缓存策略：`InMemoryCache`、`SQLiteCache`、语义缓存
   - 并发控制：`RunnableParallel`、`batch()` 的并发参数
   - 流式输出：减少首 Token 延迟（TTFT）
   - Fallback 降级：`.with_fallbacks()` 配置备用模型
   - Token 优化：Prompt 压缩、上下文窗口管理
   - 异步调用：`ainvoke()`、`astream()` 提升吞吐量

**动手项目**：将前几个阶段的 RAG 问答系统部署为生产 API 服务，接入 LangSmith 监控，配置缓存和 Fallback 策略。

### 项目一：多轮对话翻译助手

| 属性 | 说明 |
|------|------|
| **难度** | ⭐⭐ 入门 |
| **预计耗时** | 3-5 天 |
| **涉及知识点** | ChatModel、Prompt Template、Output Parser、Memory、LCEL、Streaming |
| **对应学习阶段** | 第一阶段 |

**项目描述**：

构建一个支持多轮对话的智能翻译助手，用户可以指定源语言和目标语言，助手能记住上下文（如之前翻译的术语偏好），支持流式输出翻译结果。

**核心功能**：
- 支持多语言互译，自动检测源语言
- 多轮对话记忆：记住用户的翻译偏好和术语约定
- 流式输出：翻译结果逐字显示
- 可切换不同 LLM（OpenAI / 通义千问 / 本地模型）
- 输出格式化：支持 JSON 结构化输出（原文、译文、置信度）

**技术要点**：
```python
# 核心链路示例
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是专业翻译，目标语言：{target_lang}"),
    MessagesPlaceholder("history"),
    ("human", "{input}")
])
chain = prompt | model | StrOutputParser()
# 配合 RunnableWithMessageHistory 管理对话历史
```

## 常见问题 FAQ

### LangChain vs LlamaIndex 怎么选？

这两个框架定位不同，并非完全竞争关系：

| 维度 | LangChain | LlamaIndex |
|------|-----------|------------|
| **核心定位** | 通用 LLM 应用开发框架 | 数据索引与检索框架 |
| **擅长场景** | Agent、多步推理、工具调用、复杂编排 | RAG、知识库、结构化数据查询 |
| **编排能力** | 强（LCEL + LangGraph） | 较弱，专注数据管道 |
| **RAG 能力** | 完整但需要自己组装 | 开箱即用，内置多种索引策略 |
| **Agent 能力** | 强（LangGraph 支持复杂 Agent） | 基础 Agent 支持 |
| **生态集成** | 极广（数百个集成） | 聚焦数据源和向量库 |
| **可观测性** | LangSmith（成熟） | LlamaTrace（较新） |

**选择建议**：
- 如果你的核心需求是 **RAG / 知识库问答**，且希望快速上手 → 优先考虑 LlamaIndex
- 如果你需要构建 **复杂 Agent、多步推理、工具调用** → 优先选择 LangChain
- 如果两者都需要 → 可以组合使用，用 LlamaIndex 做数据索引，用 LangChain/LangGraph 做编排
- 两者都在快速演进，建议关注各自最新版本的能力变化

> 详细对比参见：[[../框架与工具/LlamaIndex框架详解]]

### LangChain 版本升级注意事项

**从 v0.1 升级到 v0.2**：
1. 将 `from langchain.chat_models import ChatOpenAI` 改为 `from langchain_openai import ChatOpenAI`
2. 安装独立的 provider 包：`pip install langchain-openai`
3. 检查是否使用了已废弃的 Chain 类（LLMChain、SequentialChain），迁移到 LCEL
4. Memory 相关代码开始标记 deprecated，考虑迁移到 LangGraph 的 State 管理

**从 v0.2 升级到 v0.3**：
1. 确保 Pydantic 版本 >= 2.0（v0.3 不再兼容 Pydantic v1）
2. 所有在 v0.2 中标记 deprecated 的 API 已被移除，需提前替换
3. `AgentExecutor` 标记为 legacy，新项目建议使用 LangGraph
4. 检查自定义的 `BaseModel` 子类是否兼容 Pydantic v2

**通用升级策略**：
```bash
# 1. 先在测试环境升级
pip install langchain==0.3.x langchain-core==0.3.x

# 2. 运行测试，关注 DeprecationWarning
python -W default::DeprecationWarning -m pytest

# 3. 逐步替换废弃 API
# 4. 参考官方迁移指南：https://python.langchain.com/docs/versions/
```

### 国内模型（通义千问、文心一言）如何接入？

LangChain 对国内主流大模型都有良好的支持：

**通义千问（Qwen）**：
```bash
pip install langchain-community dashscope
```
```python
from langchain_community.chat_models import ChatTongyi

model = ChatTongyi(
    model="qwen-plus",
    dashscope_api_key="your-api-key"
)
response = model.invoke("你好")
```

**文心一言（ERNIE）**：
```bash
pip install langchain-community qianfan
```
```python
from langchain_community.chat_models import QianfanChatEndpoint

model = QianfanChatEndpoint(
    model="ERNIE-4.0",
    qianfan_ak="your-ak",
    qianfan_sk="your-sk"
)
response = model.invoke("你好")
```

**智谱 AI（GLM）**：
```bash
pip install langchain-community zhipuai
```
```python
from langchain_community.chat_models import ChatZhipuAI

model = ChatZhipuAI(
    model="glm-4",
    api_key="your-api-key"
)
response = model.invoke("你好")
```

**通用接入方式（兼容 OpenAI API 的模型）**：

许多国内模型提供了 OpenAI 兼容的 API 接口，可以直接使用 `ChatOpenAI`：
```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(
    model="qwen-plus",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key="your-api-key"
)
```

> **提示**：国内模型的 Tool Calling / Function Calling 支持程度不一，使用前建议查阅对应模型的文档确认支持情况。

### 学 LangChain 需要什么前置知识？

**必备前置**：
- **Python 基础**：函数、类、装饰器、异步（async/await）、类型注解
- **LLM 基本概念**：Token、Prompt、Temperature、上下文窗口 → 参见 [[../基础概念/LLM大语言模型基础]]
- **Prompt Engineering 基础**：系统提示词、Few-shot、思维链 → 参见 [[../基础概念/Prompt Engineering]]
- **API 调用经验**：至少用过一次 OpenAI 或其他 LLM 的 API

**建议前置（非必须但有帮助）**：
- **FastAPI 基础**：理解 LangServe 的底层框架
- **向量数据库概念**：理解 Embedding 和相似度搜索（RAG 阶段需要）
- **图论基础概念**：节点、边、有向图（LangGraph 阶段需要）
- **Pydantic**：LangChain 大量使用 Pydantic 做数据校验

**不需要的前置**：
- 不需要深度学习 / 机器学习背景
- 不需要了解模型训练和微调
- 不需要数学基础（线性代数、概率论等）
