## 一、LangSmith 链路追踪

> 🔑 LangSmith 通过环境变量自动启用，不需要改代码。

### 1.1 配置方式

````bash
# .env
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=lsv2_pt_your-key
LANGCHAIN_PROJECT=langchain-agent
````

设置这三个环境变量后，LangChain 自动把所有 LLM 调用、Tool 调用、Graph 执行的 trace 发送到 LangSmith 控制台（smith.langchain.com）。

### 1.2 能看到什么

| 层级 | 内容 |
|------|------|
| **Graph 执行** | 每个节点的输入/输出、执行时间、条件路由走向 |
| **LLM 调用** | prompt 内容、模型响应、token 用量、延迟 |
| **Tool 调用** | 工具名、输入参数、返回结果、耗时 |
| **错误追踪** | 异常堆栈、重试次数、自纠错路径 |

### 1.3 面试怎么说

"我们用 LangSmith 做全链路追踪，每个请求的 LLM 调用、工具执行、条件路由都能在控制台看到。出了问题可以直接看 trace 定位是检索没命中、Prompt 不对、还是工具执行报错。"

## 二、Token 成本统计

> 🔑 从流式事件中提取 token 用量，每个请求记录 input/output tokens。

### 2.1 实现方案

用自定义 `TokenCounter` 类，从 `astream_events` 的 `on_llm_end` 事件中提取 token 用量：

````python
class TokenCounter:
    def __init__(self):
        self.input_tokens = 0
        self.output_tokens = 0

    def update(self, event: dict):
        """从 on_llm_end 事件中提取 token 用量"""
        usage = event.get("data", {}).get("usage_metadata") or {}
        if not usage:
            gen_info = event.get("data", {}).get("generation_info") or {}
            usage = gen_info.get("token_usage") or {}
            self.input_tokens += usage.get("prompt_tokens", 0)
            self.output_tokens += usage.get("completion_tokens", 0)
        else:
            self.input_tokens += usage.get("input_tokens", 0)
            self.output_tokens += usage.get("output_tokens", 0)

    def log(self, endpoint: str, session_id: str):
        if self.total > 0:
            print(f"[Token] {endpoint} session={session_id} "
                  f"input={self.input_tokens} output={self.output_tokens} total={self.total}")
````

### 2.2 为什么不用 get_openai_callback()

`get_openai_callback()` 适用于同步调用场景。我们的端点用 `astream_events` 流式推送，callback 在流式场景下统计不完整。改用从 `on_llm_end` 事件中提取更可靠。

### 2.3 集成方式

每个流式端点的 `generate()` 函数中：

````python
async def generate():
    counter = TokenCounter()
    async for event in graph_app.astream_events(...):
        kind = event["event"]
        if kind == "on_llm_end":
            counter.update(event)
        # ... 其他事件处理 ...
    counter.log("graph_chat", req.session_id)
    yield "data: [DONE]\n\n"
````

## 三、API 认证

> 🔑 环境变量 `API_KEY` 为空时跳过认证（开发模式），非空时校验 `X-API-Key` 请求头。

### 3.1 中间件实现

````python
API_KEY = os.getenv("API_KEY", "")

@app.middleware("http")
async def auth_middleware(request, call_next):
    # 跳过：健康检查、OpenAPI 文档、CORS 预检
    if request.url.path in ("/health", "/docs", "/openapi.json") or request.method == "OPTIONS":
        return await call_next(request)
    # 配置了 API_KEY 才校验
    if API_KEY:
        key = request.headers.get("x-api-key", "")
        if key != API_KEY:
            return JSONResponse(status_code=401, content={"detail": "Invalid API Key"})
    return await call_next(request)
````

### 3.2 前端适配

````typescript
const API_KEY = '';  // 与后端 .env 中的 API_KEY 一致

const apiHeaders = (extra: Record<string, string> = {}) => {
  const h: Record<string, string> = { ...extra };
  if (API_KEY) h['X-API-Key'] = API_KEY;
  return h;
};

// 使用
fetch(`${API}/graph_chat`, {
  method: 'POST',
  headers: apiHeaders({ 'Content-Type': 'application/json' }),
  body: JSON.stringify({ message: text, session_id: sid }),
});
````

### 3.3 设计考量

| 决策 | 理由 |
|------|------|
| 环境变量控制开关 | 开发时不填 API_KEY 就跳过认证，不用改代码 |
| `X-API-Key` 请求头 | 标准做法，比 query 参数安全 |
| 健康检查跳过 | Docker 健康检查不需要认证 |
| OPTIONS 跳过 | CORS 预检请求不能要求认证 |

## 四、Docker 容器化

> 🔑 只容器化后端（FastAPI），Electron 前端不容器化（桌面应用）。

### 4.1 Dockerfile

````dockerfile
FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY server.py tools.py graph_agent.py build_index.py eval_rag.py ./
COPY eval_set.json ./

ENV HF_HUB_OFFLINE=1
ENV TRANSFORMERS_CACHE=/app/hf_cache

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
    CMD python3 -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

CMD ["uvicorn", "server:app", "--host", "0.0.0.0", "--port", "8000"]
````

### 4.2 docker-compose.yml

````yaml
services:
  backend:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./faiss_index:/app/faiss_index
      - ./checkpoints.db:/app/checkpoints.db
      - hf_cache:/app/hf_cache
    restart: unless-stopped

volumes:
  hf_cache:
````

### 4.3 Volume 挂载策略

| 挂载项 | 目的 |
|--------|------|
| `faiss_index/` | FAISS 索引 + BM25 语料，避免每次重建 |
| `checkpoints.db` | SQLite 对话历史，重启不丢失 |
| `hf_cache` (named volume) | HuggingFace 模型缓存，避免重复下载 |

### 4.4 使用方式

````bash
# 首次构建并启动
docker compose up --build

# 后续启动
docker compose up

# 停止
docker compose down
````

## 五、健康检查端点

> 🔑 `/health` 端点供 Docker 和负载均衡器探活。

````python
@app.get("/health")
async def health():
    return {"status": "ok"}
````

Dockerfile 中的 `HEALTHCHECK` 指令每 30 秒检查一次，连续 3 次失败则容器标记为 unhealthy。

## 六、面试要点

**Q: 你的 Agent 项目怎么部署的？**

Docker 容器化后端，docker-compose 编排。FAISS 索引和 SQLite 数据库通过 volume 挂载持久化。有健康检查端点，支持自动重启。

**Q: 怎么监控 Agent 的执行过程？**

LangSmith 全链路追踪，环境变量启用，不需要改代码。能看到每个请求的 LLM 调用、工具执行、条件路由走向、token 用量和延迟。

**Q: 怎么控制 AI 调用成本？**

自定义 TokenCounter，从 astream_events 的 on_llm_end 事件中提取 token 用量，每个请求记录 input/output tokens。可以按 session 统计消耗，发现异常用量及时告警。

**Q: API 安全怎么做的？**

FastAPI 中间件校验 X-API-Key 请求头。环境变量控制开关，开发模式不填 key 就跳过认证。健康检查和 OPTIONS 预检请求跳过认证。
