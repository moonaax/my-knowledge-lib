# 端侧推理

> 把 AI 模型跑在手机/边缘设备上，不依赖云端 API。核心挑战：内存小、算力弱、功耗敏感。
> 相关主题：[[../部署与优化/Agent部署与性能优化]] | [[../基础概念/2-LLM大语言模型基础]]

---

## 一、为什么需要端侧推理

### 1.1 云端 vs 端侧

| 维度 | 云端推理 | 端侧推理 |
|------|---------|---------|
| 延迟 | 100-500ms（网络往返） | 10-50ms（本地计算） |
| 隐私 | 数据上传到服务器 | 数据不出设备 |
| 成本 | 按调用付费，用户越多越贵 | 一次性部署，无边际成本 |
| 离线 | 必须联网 | 完全离线可用 |
| 模型大小 | 无限制（A100/H100） | 受限（手机内存 2-8GB 可用） |
| 能力上限 | 最强模型（GPT-4 级别） | 受限（7B 以下为主） |

### 1.2 端侧 AI 的典型场景

- **输入法联想**：本地推理，不上云
- **相机场景识别**：拍照时实时识别场景
- **语音唤醒**：「小爱同学」「Hey Siri」本地检测
- **图片分类/搜索**：相册智能分类
- **智能摘要**：长文本地生成摘要
- **系统级 Agent**：跨应用操作、通知分类（小米 HyperOS 方向）

## 二、核心概念

### 2.1 模型量化

> 🔑 量化 = 把模型权重从高精度（FP32）压缩到低精度（INT8/INT4），牺牲少量精度换取 2-4 倍加速和内存节省。

| 精度 | 大小（7B 模型） | 速度 | 精度损失 |
|------|----------------|------|---------|
| FP32 | 28GB | 基准 | 无 |
| FP16 | 14GB | ~1.5x | 极小 |
| INT8 | 7GB | ~2x | 小 |
| INT4 | 3.5GB | ~3-4x | 可接受 |

**量化方法**：

```
训练后量化（PTQ）：直接对已训练好的模型量化，简单但精度损失略大
量化感知训练（QAT）：训练时模拟量化误差，精度更好但需要重新训练
GPTQ：专门为 LLM 设计的量化方法，INT4 下精度损失小
AWQ：Activation-aware Weight Quantization，保护重要权重通道
GGUF：llama.cpp 使用的格式，CPU 推理友好
```

### 2.2 模型剪枝

去掉模型中不重要的参数/通道/层，减小模型体积。

- **非结构化剪枝**：把小权重设为 0（稀疏矩阵，硬件加速困难）
- **结构化剪枝**：直接去掉整个通道/注意力头（硬件友好，推荐）
- **层剪枝**：直接去掉中间的某些层（对小模型效果差）

### 2.3 知识蒸馏

用大模型（Teacher）教小模型（Student），让小模型学到大模型的能力。

```
Teacher（7B）→ 软标签（概率分布）→ Student（1.5B）
Student 学习 Teacher 的"思维方式"，而不仅仅是正确答案
```

典型应用：DistilBERT（BERT 的 60% 大小，97% 性能）

## 三、主流端侧推理框架

### 3.1 TensorFlow Lite（Google）

```
PyTorch/TF 模型 → TensorFlow → TFLite 转换器 → .tflite 模型
```

- Android/iOS 原生支持
- 支持 INT8/FP16 量化
- GPU 加速（OpenGL/OpenCL/Metal）
- 适合 CV 模型（分类、检测、分割）

### 3.2 MNN（阿里巴巴）/ 小米也用

```
PyTorch/TF 模型 → MNN 转换器 → .mnn 模型
```

- 阿里开源，小米深度使用
- 支持 CPU/GPU/NPU 异构计算
- 模型压缩率高
- 支持 LLM 推理（MNN-LLM）
- 对国产芯片（骁龙/天玑/澎湃）优化好

### 3.3 ONNX Runtime（微软）

```
PyTorch 模型 → ONNX 导出 → ONNX Runtime 推理
```

- 跨平台（Android/iOS/Windows/Linux）
- 支持多种硬件后端
- NNAPI 加速（Android 神经网络 API）
- 生态最完善，模型兼容性最好

### 3.4 llama.cpp / MLX

```
GGUF 格式模型 → llama.cpp → CPU/GPU 推理
```

- 纯 C/C++ 实现，零依赖
- 专门优化 LLM 推理
- 支持 INT4/INT5/INT8 量化
- Mac 上用 MLX（Apple Silicon 优化）
- 适合在 Mac/Linux 上本地跑 LLM

## 四、Android 端侧 LLM

### 4.1 Google Gemini Nano

- 内置在 Pixel 8 Pro / Samsung S24 等旗舰机
- 1.8B 参数，INT8 量化
- 能力：摘要、改写、Smart Reply

## 五、端侧 Agent 架构

### 5.1 混合架构（推荐）

```
用户请求
    ↓
端侧小模型（意图识别、简单任务）
    ↓ 无法处理
云端大模型（复杂推理、知识检索）
    ↓
结果返回设备
```

- 简单任务走端侧（快、隐私好、免费）
- 复杂任务走云端（能力强）
- 路由策略：端侧模型判断任务复杂度

### 5.2 端侧 Agent 的挑战

| 挑战 | 说明 |
|------|------|
| 上下文窗口小 | 端侧模型通常 2K-4K tokens，对话历史受限 |
| 工具调用弱 | 小模型的 Function Calling 能力不稳定 |
| 内存限制 | Agent 状态 + 模型 + 工具结果，内存容易爆 |
| 功耗控制 | 持续推理会快速耗电 |
| 热管理 | 长时间推理导致设备发热降频 |

## 六、面试要点

**Q: 端侧推理和云端推理怎么选？**

看场景。隐私敏感（输入法、相册）必须端侧；实时性要求高（语音唤醒）必须端侧；复杂推理（多步骤 Agent）走云端。实际方案是混合架构——端侧处理简单任务，复杂的 fallback 到云端。

**Q: 模型量化会损失多少精度？**

INT8 量化通常精度损失 <1%，INT4 在 1-3%。GPTQ/AWQ 等专门为 LLM 设计的量化方法在 INT4 下精度损失更小。实际部署前需要在目标任务上评测，不能只看通用 benchmark。

**Q: 小米的端侧 AI 怎么做的？**

基于 MNN 推理引擎，支持 CPU/GPU/NPU 异构计算。HyperOS 集成了端侧大模型，结合小爱同学做系统级 Agent。用意图识别路由——简单任务端侧处理，复杂的走云端。
