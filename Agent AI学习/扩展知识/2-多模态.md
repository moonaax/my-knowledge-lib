# 多模态 AI

> 让 AI 不只处理文字，还能理解图片、语音、视频。多模态 = 多种输入/输出模态。
> 相关主题：[[../基础概念/2-LLM大语言模型基础]] | [[../RAG检索增强生成/1-RAG原理与架构]]

---

## 一、什么是模态

模态 = 信息的载体形式：

| 模态 | 例子 | 代表模型 |
|------|------|---------|
| **文本**（Text） | 文章、对话 | GPT-4、DeepSeek |
| **图像**（Vision） | 照片、截图、文档扫描 | GPT-4V、Gemini |
| **语音**（Audio） | 录音、实时语音 | Whisper、GPT-4o |
| **视频**（Video） | 监控、短视频 | Gemini 1.5、GPT-4o |
| **代码**（Code） | 程序源码 | Codex、DeepSeek-Coder |

多模态 AI = 模型能同时处理两种以上模态的输入/输出。

## 二、核心能力

### 2.1 图像理解（Vision）

模型能"看懂"图片，回答关于图片的问题。

```
输入：一张报错截图 + "这个错误怎么解决？"
输出：这是一个 NullPointerException，发生在第 42 行...
```

**典型场景**：
- 截图 OCR + 语义理解
- 文档/表格解析
- 商品图片识别
- UI 界面分析（Agent 操控手机的基础）

### 2.2 语音理解 + 语音合成

```
语音输入 → Whisper 转文字 → LLM 处理 → TTS 语音输出
```

- **ASR（语音识别）**：Whisper（OpenAI）、讯飞、百度
- **TTS（语音合成）**：OpenAI TTS、Edge TTS、VITS
- **端到端**：GPT-4o 直接语音输入/输出，不需要中间转文字

### 2.3 视频理解

模型能理解视频内容，回答关于视频的问题。

```
输入：一段监控视频 + "发生了什么？"
输出：视频中一个人走进房间，拿起了桌上的包，然后离开...
```

- Gemini 1.5 Pro：支持 1 小时视频输入
- 关键技术：视频抽帧 → 图像理解 → 时序推理

## 三、多模态 RAG

### 3.1 传统 RAG 的局限

传统 RAG 只处理纯文本。但企业数据很多是多模态的：
- PDF 里有图表、流程图
- 技术文档有架构图
- 会议记录有白板照片
- 产品有图片规格书

### 3.2 多模态 RAG 方案

**方案一：图片转文字（最简单）**

```
图片 → OCR/描述 → 文本 → 传统 RAG 流程
```

- 优点：复用现有 RAG 架构
- 缺点：OCR 丢失视觉信息（布局、颜色、图表关系）

**方案二：多模态 Embedding（推荐）**

```
图片 + 文本 → 多模态 Embedding 模型 → 统一向量空间 → FAISS 检索
```

- CLIP：OpenAI 的图文对齐模型，图片和文字映射到同一向量空间
- SigLIP：Google 的改进版，效果更好
- BGE-V：智谱的中文多模态 Embedding

**方案三：VLM 直接理解（最灵活）**

```
检索到相关图片 → 直接发给 GPT-4V/Gemini 理解 → 生成回答
```

- 优点：保留完整视觉信息
- 缺点：成本高（图片 token 消耗大）

### 3.3 实际落地建议

```
文档入库：
  PDF → 解析出文字和图片
  文字 → 文本 Embedding → FAISS
  图片 → 图片描述（VLM 生成）→ 文本 Embedding → FAISS
  图片 → CLIP Embedding → 单独的图片向量库

查询时：
  用户问题 → 文本检索 + CLIP 图片检索
  → 合并结果 → VLM 理解图片 + LLM 理解文字 → 生成回答
```

## 四、多模态 Agent

### 4.1 为什么 Agent 需要多模态

纯文本 Agent 的局限：
- 用户发截图描述问题 → Agent 看不懂，需要用户手动转文字
- Agent 操作手机 → 需要"看"屏幕才能知道当前状态
- 分析文档 → 图表、流程图无法用纯文本描述

### 4.2 多模态 Agent 架构

```
用户输入（文字/图片/语音）
    ↓
多模态理解层（Whisper 转文字、VLM 理解图片）
    ↓
Agent 决策层（LLM 推理 + 工具调用）
    ↓
执行层（调用工具、操作设备）
    ↓
多模态输出层（文字/图片/语音）
```

### 4.3 手机 Agent（Android 方向重点）

```
用户："帮我把刚才那张截图发给张三"
    ↓
Agent 看到截图（多模态输入）
    ↓
Agent 识别出截图内容
    ↓
Agent 调用工具：打开微信 → 搜索张三 → 发送图片
```

关键能力：
- **屏幕理解**：VLM 识别当前屏幕上的元素
- **操作规划**：LLM 规划操作步骤
- **UI 定位**：识别按钮/输入框的坐标
- **错误恢复**：操作失败时看屏幕重新规划

代表项目：Google Project Astra、小米 HyperOS Agent

## 六、LangChain 中使用多模态

### 6.1 图片输入

```python
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

llm = ChatOpenAI(model="gpt-4o")

message = HumanMessage(content=[
    {"type": "text", "text": "这张图片里有什么？"},
    {"type": "image_url", "image_url": {"url": "https://example.com/image.png"}},
])
response = llm.invoke([message])
```

### 6.2 多模态 RAG 工具

```python
from langchain.tools import tool

@tool
def analyze_image(image_url: str, question: str) -> str:
    """分析图片内容并回答问题"""
    llm = ChatOpenAI(model="gpt-4o")
    message = HumanMessage(content=[
        {"type": "text", "text": question},
        {"type": "image_url", "image_url": {"url": image_url}},
    ])
    return llm.invoke([message]).content
```

## 七、面试要点

**Q: 多模态 RAG 怎么实现？**

三种方案：①图片转文字后走传统 RAG（最简单但丢信息）②多模态 Embedding（CLIP）统一向量空间检索（推荐）③VLM 直接理解检索到的图片（最灵活但贵）。实际落地可以组合使用——文字用文本 Embedding，图片用 CLIP 检索 + VLM 理解。

**Q: 多模态 Agent 和纯文本 Agent 有什么区别？**

多模态 Agent 能接受图片/语音输入，核心区别在感知层。比如手机 Agent 需要"看"屏幕才能操作——VLM 识别屏幕元素，LLM 规划操作步骤，工具执行点击/滑动。纯文本 Agent 只能处理结构化数据。

**Q: 端侧多模态怎么做？**

端侧 VLM（如 MiniCPM-V、Qwen-VL-2B）可以在手机上运行，但能力有限。实际方案是混合架构——端侧做快速预处理（OCR、图片分类），复杂的理解走云端。
