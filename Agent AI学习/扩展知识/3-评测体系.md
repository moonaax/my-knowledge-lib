# 评测体系

> 量化评估 AI Agent / RAG / LLM 的效果，是上线前的必经之路。没有评测 = 不知道能不能上线。
> 相关主题：[[../RAG检索增强生成/1-RAG原理与架构]] | [[../基础概念/2-LLM大语言模型基础]]

---

## 一、为什么需要评测

### 1.1 没有评测的风险

```
用户问：LCEL 是什么？
Agent 回答：LCEL 是 LangChain Expression Language（正确）
→ 看起来没问题

但换个问法：
用户问：LCEL 怎么用？
Agent 回答：LCEL 是一种编程语言（错误）
→ 不评测根本发现不了
```

**核心问题**：LLM 的输出有随机性，同样的问题不同时间可能得到不同答案。靠人工测几个 case 不够，需要系统化评测。

### 1.2 评测的三个层次

| 层次 | 评测对象 | 方法 | 成本 |
|------|---------|------|------|
| **单元评测** | 单个组件（Prompt、Retriever、Tool） | 固定输入输出对比 | 低 |
| **集成评测** | 整条链路（RAG Pipeline） | QA 对匹配 | 中 |
| **端到端评测** | 完整产品（Agent 交互） | 人工评测 / LLM-as-Judge | 高 |

## 二、LLM 评测

### 2.1 通用基准

| 基准 | 评测内容 | 特点 |
|------|---------|------|
| **MMLU** | 57 个学科的选择题 | 最通用的 LLM 评测基准 |
| **HumanEval** | 代码生成 | 164 个编程题 |
| **GSM8K** | 数学推理 | 8500 道小学数学题 |
| **MT-Bench** | 多轮对话质量 | GPT-4 打分 |
| **AlpacaEval** | 指令遵循 | 与 GPT-4 对比胜率 |
| **C-Eval** | 中文综合 | 52 个中文科目 |
| **SuperCLUE** | 中文评测 | 国内最权威的中文 LLM 评测 |

### 2.2 MT-Bench 详解（最实用）

MT-Bench 用 GPT-4 作为裁判，给模型回答打 1-10 分：

```
评分流程：
1. 准备 80 个多轮对话题（涵盖写作、推理、数学、编程等）
2. 让被测模型回答
3. GPT-4 打分（1-10 分）
4. 计算平均分

评分标准：
- 9-10：优秀，专业水准
- 7-8：良好，基本满足需求
- 5-6：一般，有明显问题
- 3-4：较差，经常出错
- 1-2：很差，基本不可用
```

**为什么用 GPT-4 打分**：人工评测成本太高（一个回答需要多人标注），GPT-4 打分与人类评分相关性 > 0.85。

### 2.3 LLM-as-Judge 模式

用一个 LLM 评测另一个 LLM 的输出：

```python
# 评测 Prompt 模板
judge_prompt = """
请评判以下回答的质量，从 1-10 打分。

问题：{question}
参考答案：{reference}
模型回答：{answer}

评分标准：
- 准确性（0-4 分）：事实是否正确
- 完整性（0-3 分）：是否涵盖了关键点
- 清晰度（0-3 分）：表达是否清楚

输出格式：
分数：X
理由：...
"""
```

**注意事项**：
- 评委模型要比被测模型强（用 GPT-4 评测 7B 模型）
- 评分有 position bias（倾向给第一个回答高分）→ 打乱顺序测两次
- 不同评委模型打分标准不同 → 固定用同一个评委

## 三、RAG 评测

### 3.1 RAG 评测维度

RAG 系统需要从三个维度评测：

```
                    ┌─────────────┐
                    │  最终回答质量 │
                    └──────┬──────┘
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ 检索质量  │ │ 生成质量  │ │ 忠实度    │
        │ Retrieval│ │Generation│ │ Faithful │
        └──────────┘ └──────────┘ └──────────┘
```

| 维度 | 评测指标 | 含义 |
|------|---------|------|
| **检索质量** | Recall@K | 前 K 个结果中包含正确答案的比例 |
| **检索质量** | MRR | 第一个正确结果的排名倒数 |
| **检索质量** | NDCG | 考虑排序位置的检索质量 |
| **生成质量** | Answer Accuracy | 回答是否正确 |
| **生成质量** | Answer Completeness | 回答是否完整 |
| **忠实度** | Faithfulness | 回答是否基于检索到的内容（不幻觉） |
| **忠实度** | Relevance | 回答是否与问题相关 |

### 3.2 RAGAS 框架（最流行的 RAG 评测框架）

RAGAS（Retrieval Augmented Generation Assessment）提供四个自动评测指标：

```python
# 安装
pip install ragas

# 使用
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

# 准备评测数据
eval_data = {
    "question": ["LCEL 是什么？"],
    "answer": ["LCEL 是 LangChain Expression Language..."],
    "contexts": [["LCEL（LangChain Expression Language）是 LangChain 的核心..."]],
    "ground_truth": ["LCEL 是 LangChain 的表达式语言，用于链式调用"]
}

# 运行评测
result = evaluate(
    dataset=eval_data,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)
print(result)
```

**四个核心指标**：

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| **Faithfulness** | 回答是否忠实于检索内容 | 把回答拆成句子，检查每句是否有上下文支持 |
| **Answer Relevancy** | 回答是否与问题相关 | 用 LLM 从回答反推问题，与原问题比较相似度 |
| **Context Precision** | 检索内容的精确度 | 相关文档是否排在前面 |
| **Context Recall** | 检索内容的召回率 | 参考答案中的信息是否被检索到 |

### 3.3 构建 RAG 评测集

**方法一：人工构建（最准确）**

```json
[

  }

```

**方法二：LLM 自动生成（规模化）**

```python
# 用 LLM 从文档自动生成 QA 对
generate_prompt = """

文档内容：{context}

```

## 四、Agent 评测

### 4.1 Agent 评测的难点

Agent 评测比 RAG 更复杂：

| 挑战 | 说明 |
|------|------|
| **多步骤** | Agent 可能调用多个工具，中间步骤也需要评测 |
| **非确定性** | 同一个问题可能有多种正确的工具调用路径 |
| **工具依赖** | 外部 API 不稳定，影响评测结果 |
| **状态管理** | 多轮对话中的状态变化需要追踪 |

### 4.2 Agent 评测维度

| 维度 | 指标 | 说明 |
|------|------|------|
| **任务完成率** | Success Rate | 最终是否完成了用户任务 |
| **工具选择准确率** | Tool Accuracy | 是否选对了工具 |
| **参数提取准确率** | Argument Accuracy | 工具参数是否正确 |
| **步骤效率** | Step Efficiency | 完成任务用了多少步（越少越好） |
| **错误恢复率** | Recovery Rate | 出错后能否自动修复 |
| **延迟** | Latency | 端到端响应时间 |

### 4.3 实战：我们的评测脚本

项目中的 `eval_rag.py` 实现了简单的 RAG 评测：

```python
# eval_rag.py 的评测逻辑
def evaluate():
    results = []
    for qa in eval_set:
        # 调用 RAG 链
        answer = rag_chain.invoke(qa["question"])
        # 关键词命中率
        keywords = qa["keywords"]
        hits = sum(1 for kw in keywords if kw in answer)
        keyword_score = hits / len(keywords)
        # 来源准确率（检查是否检索到了正确文档）
        source_score = check_sources(answer, qa["expected_source"])
        results.append({
            "question": qa["question"],
            "keyword_score": keyword_score,
            "source_score": source_score
        })
    # 汇总
    avg_keyword = sum(r["keyword_score"] for r in results) / len(results)
    avg_source = sum(r["source_score"] for r in results) / len(results)
    print(f"关键词命中率: {avg_keyword:.1%}")
    print(f"来源准确率: {avg_source:.1%}")
```

**25 个 QA 对覆盖的知识领域**：
- LangChain 基础（LCEL、Chain、Memory）
- Agent 概念（ReAct、Tool、Function Calling）
- RAG（向量检索、Embedding、Chunk）
- LangGraph（StateGraph、条件路由）
- 生产化（Docker、LangSmith、量化）

## 五、评测最佳实践

### 5.1 评测流程

```
1. 构建评测集（至少 20-50 个 QA 对）
2. 跑基线（当前版本的分数）
3. 改动 Prompt / 检索策略 / 模型
4. 重新评测，对比基线
5. 回归测试（确保没引入新问题）
```

### 5.2 自动化评测 Pipeline

```python
# 评测 Pipeline
def eval_pipeline(version: str):
    # 1. 跑评测集
    results = run_eval(eval_set)
    # 2. 计算指标
    metrics = compute_metrics(results)
    # 3. 与基线对比
    baseline = load_baseline()
    diff = compare(metrics, baseline)
    # 4. 生成报告
    report = generate_report(version, metrics, diff)
    # 5. 发送到 LangSmith
    log_to_langsmith(version, metrics)
    return report
```

### 5.3 评测陷阱

| 陷阱 | 说明 | 解决方案 |
|------|------|---------|
| **评测集太小** | 5 个 case 不能代表真实场景 | 至少 20 个，覆盖不同类型 |
| **评测集泄露** | 用训练数据做评测 | 评测集必须独立于训练数据 |
| **只看平均分** | 平均分高但某个类型很差 | 分类统计，找到薄弱点 |
| **忽略边界情况** | 只测正常输入 | 加入模糊/错误/对抗性输入 |
| **评测环境不一致** | 模型版本/API 变化影响结果 | 锁定版本，记录环境信息 |

## 七、面试要点

**Q: RAG 系统怎么评测？**

从三个维度：①检索质量（Recall@K、MRR）②生成质量（Answer Accuracy）③忠实度（Faithfulness，是否幻觉）。推荐用 RAGAS 框架自动化评测，核心指标是 Faithfulness、Answer Relevancy、Context Precision、Context Recall。评测集至少 20 个 QA 对，覆盖不同类型的问题。

**Q: LLM-as-Judge 是什么？有什么问题？**

用强 LLM（如 GPT-4）给弱 LLM 的输出打分。优点是成本低、可规模化；问题是存在 bias：position bias（倾向第一个回答）、verbosity bias（倾向长回答）、self-enhancement bias（给自己打高分）。解决方法：打乱顺序测两次、用多个评委、固定评委模型。

**Q: Agent 评测和 RAG 评测有什么区别？**

RAG 评测关注检索+生成，指标相对固定。Agent 评测更复杂——需要评测多步骤的工具选择、参数提取、错误恢复能力。核心指标是任务完成率和步骤效率。Agent 评测需要模拟工具调用（mock 外部 API），否则评测结果不稳定。

**Q: 你们项目的评测是怎么做的？**

用 25 个 QA 对覆盖 LangChain/RAG/Agent 核心知识点，评测指标是关键词命中率和来源准确率。通过 eval_rag.py 自动化评测，每次改动 Prompt 或检索策略后跑一遍对比基线。生产环境接入 LangSmith 做 trace + 评测，可以追踪每次请求的完整链路。
