# 最小化依赖与体积控制

> SDK 的体积直接影响宿主 App 的包大小，是 SDK 设计中最容易被忽视却最影响接入意愿的指标。本文系统梳理 SDK 体积组成、依赖管理策略与实战优化手段。

相关主题：[[AAR与APK打包]] · [[混淆与ProGuard规则]] · [[Gradle依赖管理]]

---

## 一、核心概念

### 1.1 SDK 体积的四大组成

一个 AAR/SDK 最终对宿主 APK 体积的贡献来自以下四个维度：

| 维度 | 说明 | 典型占比 |
|------|------|----------|
| **Java/Kotlin 代码** | 编译后的 .class → DEX 字节码 | 20%-40% |
| **资源文件** | drawable / layout / raw / assets 等 | 10%-30% |
| **Native SO** | JNI 库，按 ABI 分发（armeabi-v7a / arm64-v8a / x86_64） | 30%-60%（若有） |
| **依赖传递** | SDK 声明的第三方依赖会被传递到宿主 | 不可控，可能翻倍 |

```
AAR 文件结构：
├── classes.jar          ← 代码（DEX 来源）
├── res/                 ← 资源
├── jni/                 ← native so
│   ├── armeabi-v7a/
│   └── arm64-v8a/
├── assets/              ← 原始资源
├── R.txt                ← 资源 ID 映射
├── AndroidManifest.xml
└── proguard.txt         ← 混淆规则
```

### 1.2 依赖传递的危害

当 SDK 使用 `api` 或默认的 `implementation` 声明依赖时，这些依赖会通过 POM 文件传递给宿主：

```kotlin
// SDK 的 build.gradle.kts
dependencies {
    // ❌ 危害一：api 会把 Gson 暴露并传递给所有消费者
    api("com.google.code.gson:gson:2.10.1")

    // ⚠️ implementation 仍会写入 POM，宿主仍会拉取
    implementation("com.squareup.okhttp3:okhttp:4.12.0")
}
```

依赖传递带来的问题：**体积膨胀**（宿主被迫引入不需要的库）、**版本冲突**（不同 SDK 传递同一库的不同版本导致 `NoSuchMethodError`）、**依赖地狱**（多级传递难以追踪）、**合规风险**（传递依赖可能包含 GPL 等不合规协议）。

典型冲突场景：宿主 App 同时接入 SDK-A（依赖 Gson 2.8）和 SDK-B（依赖 Gson 2.10），Gradle 会选择高版本 2.10，但 SDK-A 可能不兼容。

### 1.3 api vs implementation 的本质区别

```kotlin
dependencies {
    // api: 编译期 + 运行期都可见，会传递给消费者
    // 适用场景：SDK 公开 API 的返回值/参数类型来自该依赖
    api("androidx.annotation:annotation:1.7.0")

    // implementation: 仅运行期可见，不传递编译期依赖
    // 适用场景：SDK 内部实现使用，不暴露给外部
    implementation("com.squareup.okhttp3:okhttp:4.12.0")

    // compileOnly: 仅编译期，不打包、不传递
    // 适用场景：宿主一定会提供的依赖（如 Gson）
    compileOnly("com.google.code.gson:gson:2.10.1")
}
```

对比表：

| 声明方式 | 编译期可见 | 运行期打包 | 传递给消费者 | SDK 推荐场景 |
|----------|-----------|-----------|-------------|-------------|
| `api` | ✅ | ✅ | ✅ | 公开 API 类型依赖 |
| `implementation` | ✅ | ✅ | ❌（编译期） | 内部实现依赖 |
| `compileOnly` | ✅ | ❌ | ❌ | 可选功能 / 宿主提供 |
| `runtimeOnly` | ❌ | ✅ | ❌ | SPI / 反射加载 |

> **SDK 设计黄金法则**：能用 `implementation` 就不用 `api`，能用 `compileOnly` 就不用 `implementation`。

## 二、原理与实践

### 2.1 R8/ProGuard 代码缩减

R8 是 Android 默认的代码缩减器（替代 ProGuard），通过 Tree Shaking 移除未使用的类、方法和字段。参考 [[混淆与ProGuard规则]]。

```kotlin
// SDK 模块的 build.gradle.kts
android {
    buildTypes {
        release {
            // SDK 库模块通常不在自身开启 minify
            // 而是提供 consumerProguardFiles 让宿主缩减
            isMinifyEnabled = false
            consumerProguardFiles("consumer-rules.pro")
        }
    }
}
```

```proguard
# consumer-rules.pro — 随 AAR 发布，宿主打包时自动应用
# 保留 SDK 公开 API
-keep public class com.example.sdk.** {
    public protected *;
}

# 保留通过反射调用的内部类
-keep class com.example.sdk.internal.ReflectHelper { *; }

# 移除 SDK 内部日志（代码缩减优化）
-assumenosideeffects class com.example.sdk.internal.Logger {
    public static void d(...);
    public static void v(...);
}
```

SDK 自身代码优化策略：

```kotlin
// ✅ 避免使用 kotlin-reflect（增加约 2.5MB），用 Java 反射替代
val name = MyClass::class.java.name

// ✅ 内部类不需要 data class 的 copy()/componentN() 开销
internal class InternalConfig(var timeout: Long = 30_000L, var retryCount: Int = 3)

// ✅ @JvmStatic 减少 Companion 额外方法生成
class SdkClient {
    companion object {
        @JvmStatic
        fun getInstance(): SdkClient = Holder.INSTANCE
    }
    private object Holder { val INSTANCE = SdkClient() }
}
```

### 2.2 资源缩减（shrinkResources）

```kotlin
// 宿主 App 的 build.gradle.kts（SDK 自身无法开启 shrinkResources）
android {
    buildTypes {
        release {
            isMinifyEnabled = true
            isShrinkResources = true  // 必须配合 minify 使用
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}
```

SDK 侧的资源优化策略：

```kotlin
// SDK build.gradle.kts — 资源前缀避免冲突
android {
    resourcePrefix = "hcsdk_"  // 所有资源必须以此为前缀
}
```

使用 VectorDrawable 替代 PNG（体积减少 80%+），所有资源加 `resourcePrefix` 前缀避免与宿主冲突。

SDK 中可通过 `res/raw/keep.xml` 标记可安全移除的资源（`tools:discard` 属性）。

### 2.3 SO 的 ABI 过滤

Native SO 通常是 SDK 体积的最大来源。合理的 ABI 过滤至关重要：

```kotlin
// SDK 的 build.gradle.kts
android {
    defaultConfig {
        ndk {
            // 2024+ 推荐仅支持 arm64-v8a（Google Play 要求 64 位）
            abiFilters += listOf("armeabi-v7a", "arm64-v8a")
            // ❌ 不要包含 x86/x86_64（仅模拟器需要，体积翻倍）
        }
    }
}
```

SO 体积优化：在 CMakeLists.txt 中使用 `-Oz`（最小体积优化）和 `-fvisibility=hidden`（隐藏非导出符号），并确保 release 构建自动 strip 调试信息。

### 2.4 依赖分析工具（dependency-analysis-gradle-plugin）

```kotlin
// 根 build.gradle.kts
plugins {
    id("com.autonomousapps.dependency-analysis") version "1.28.0"
}

dependencyAnalysis {
    issues {
        all {
            onUsedTransitiveDependencies { severity("fail") }
            onUnusedDependencies { severity("warn") }
            onIncorrectConfiguration { severity("warn") }
        }
    }
}
```

```bash
# 运行分析
./gradlew buildHealth
# 会报告：未使用的依赖、直接使用了传递依赖、api/implementation 配置错误
```

```bash
# Gradle 内置依赖树查看
./gradlew :sdk-core:dependencies --configuration releaseRuntimeClasspath
```

### 2.5 可选依赖 compileOnly

`compileOnly` 是 SDK 减少依赖传递的核心武器：

```kotlin
// SDK build.gradle.kts
dependencies {
    // 核心功能：无外部依赖
    implementation("androidx.core:core-ktx:1.12.0")

    // 可选功能：宿主有 Gson 才启用 JSON 序列化
    compileOnly("com.google.code.gson:gson:2.10.1")

    // 可选功能：宿主有 OkHttp 才使用 OkHttp 引擎
    compileOnly("com.squareup.okhttp3:okhttp:4.12.0")
}
```

配合运行时检测实现优雅降级：

```java
/**
 * 运行时检测某个类是否存在（判断宿主是否引入了对应依赖）
 */
class DependencyChecker {
    private DependencyChecker() {}

    private static volatile Boolean gsonAvailable;
    private static volatile Boolean okHttpAvailable;

    public static boolean isGsonAvailable() {
        if (gsonAvailable == null) {
            try {
                Class.forName("com.google.gson.Gson");
                gsonAvailable = true;
            } catch (ClassNotFoundException e) {
                gsonAvailable = false;
            }
        }
        return gsonAvailable;
    }

    public static boolean isOkHttpAvailable() {
        if (okHttpAvailable == null) {
            try {
                Class.forName("okhttp3.OkHttpClient");
                okHttpAvailable = true;
            } catch (ClassNotFoundException e) {
                okHttpAvailable = false;
            }
        }
        return okHttpAvailable;
    }
}

/**
 * 根据宿主环境自动选择实现
 */
public class SerializerFactory {
    private SerializerFactory() {}

    public static Serializer create() {
        if (DependencyChecker.isGsonAvailable()) {
            return new GsonSerializer();
        }
        return new BuiltInSerializer(); // SDK 内置的轻量实现
    }
}

// Gson 实现（仅在宿主有 Gson 时才会被加载）
class GsonSerializer implements Serializer {
    private final com.google.gson.Gson gson = new com.google.gson.Gson();

    @Override
    public <T> String toJson(T obj) { return gson.toJson(obj); }

    @Override
    public <T> T fromJson(String json, Class<T> clazz) { return gson.fromJson(json, clazz); }
}

// 内置实现（零依赖，基于 Android 内置 JSONObject）
class BuiltInSerializer implements Serializer {
    @Override
    public <T> String toJson(T obj) { return new org.json.JSONObject().toString(); }

    @Override
    public <T> T fromJson(String json, Class<T> clazz) {
        throw new UnsupportedOperationException("请引入 Gson 以获得完整序列化支持");
    }
}
```

## 三、面试题

### Q1：SDK 中 api 和 implementation 的区别是什么？为什么 SDK 应该尽量用 implementation？

**A**：`api` 声明的依赖会传递给消费者，消费者在编译期和运行期都能访问。`implementation` 声明的依赖不会传递编译期可见性（运行期仍存在于 classpath）。

SDK 应尽量用 `implementation`：减少依赖泄漏、降低版本冲突概率、加速编译、更好的封装性。只有当公开 API 签名中包含某依赖的类型时才应使用 `api`。

### Q2：compileOnly 在 SDK 开发中有什么作用？使用时需要注意什么？

**A**：`compileOnly` 声明的依赖仅编译期存在，不打包进 AAR，不传递给消费者。适用于宿主一定会提供的依赖或可选功能依赖。

注意：必须做运行时 `Class.forName()` 检测、文档中说明可选依赖、编译版本需与宿主兼容范围一致、不能在初始化路径中直接引用 compileOnly 的类。

### Q3：如何分析一个 AAR 对最终 APK 体积的影响？

**A**：分层分析：

1. **AAR 自身**：解压 AAR 查看 classes.jar / res / jni 各部分大小
2. **依赖传递**：`./gradlew :app:dependencies` 查看传递依赖
3. **APK 增量对比**：分别构建有/无 SDK 的 APK，用 APK Analyzer 对比
4. **DEX 方法数**：使用 `dexcount-gradle-plugin` 统计

关键工具：APK Analyzer（Android Studio 内置）、`dexcount-gradle-plugin`、`dependency-analysis-gradle-plugin`。

### Q4：SDK 中的 native SO 体积很大，有哪些优化手段？

**A**：主要手段包括：ABI 过滤（仅保留 arm64-v8a）、编译优化（`-Oz` + LTO）、符号裁剪（`-fvisibility=hidden` + strip）、按需动态下载 SO、分 ABI 发布多个 AAR 变体。

### Q5：什么是依赖冲突？Gradle 默认如何解决？SDK 开发者应该怎么处理？

**A**：依赖冲突指同一库被不同模块引入了不同版本。Gradle 默认使用**最高版本策略**，但高版本可能移除了低版本 API，导致 `NoSuchMethodError`。

SDK 开发者应：使用 `implementation` 减少传递、使用 `compileOnly` 让宿主决定版本、文档声明兼容版本范围、使用 `constraints` 声明版本约束。

```kotlin
dependencies {
    implementation("com.squareup.okhttp3:okhttp:4.12.0")
    // 约束传递依赖的版本下限
    constraints {
        implementation("com.squareup.okio:okio:3.4.0") {
            because("修复了内存泄漏问题")
        }
    }
}
```

### Q6：shrinkResources 为什么必须配合 minifyEnabled 使用？SDK 模块能否直接开启？

**A**：`shrinkResources` 依赖 R8 的代码可达性分析来判断哪些资源未被引用，因此必须配合 `minifyEnabled`。

SDK（Library 模块）**不能直接开启**，因为 Library 不生成最终 APK，无法确定资源的最终使用情况。SDK 的替代方案：使用 `tools:discard` 标记可丢弃资源、使用 VectorDrawable 替代 PNG、避免打包 debug 专用资源到 release AAR。

## 四、实战与踩坑

### 4.1 AAR 体积分析实战

参考 [[AAR与APK打包]] 了解 AAR 的完整结构。

#### 步骤一：解压 AAR 分析各部分体积

```bash
# AAR 本质是 ZIP 文件
cp sdk-core-release.aar sdk-core-release.zip
unzip sdk-core-release.zip -d sdk-core-contents/

# 查看各部分大小
du -sh sdk-core-contents/*
# 输出示例：
# 156K    classes.jar
# 24K     res/
# 4.2M    jni/
# 8K      assets/
# 4K      AndroidManifest.xml
# 4K      R.txt
```

#### 步骤二：APK 增量对比（最准确的方式）

```kotlin
// 创建两个 flavor 对比
// app/build.gradle.kts
android {
    flavorDimensions += "sdk"
    productFlavors {
        create("withSdk") {
            dimension = "sdk"
        }
        create("withoutSdk") {
            dimension = "sdk"
        }
    }
}

dependencies {
    "withSdkImplementation"(project(":sdk-core"))
    // withoutSdk 不引入 SDK
}
```

```bash
# 构建两个变体并对比
./gradlew assembleWithSdkRelease assembleWithoutSdkRelease
apkanalyzer apk compare app-withSdk-release.apk app-withoutSdk-release.apk
```

#### 步骤三：使用 Gradle 任务自动化体积监控

```kotlin
// buildSrc 或 convention plugin 中
tasks.register("checkAarSize") {
    dependsOn("assembleRelease")
    doLast {
        val aar = file("build/outputs/aar/sdk-core-release.aar")
        val sizeKB = aar.length() / 1024
        val limitKB = 500 // 体积上限 500KB

        println("AAR size: ${sizeKB}KB (limit: ${limitKB}KB)")

        if (sizeKB > limitKB) {
            throw GradleException(
                "AAR 体积超标！当前 ${sizeKB}KB，上限 ${limitKB}KB"
            )
        }
    }
}
```

### 4.2 依赖冲突排查

#### 场景一：运行时 NoSuchMethodError

```
java.lang.NoSuchMethodError: No virtual method 
  newBuilder()Lokhttp3/Request$Builder; 
  in class Lokhttp3/Request;
```

排查步骤：

```bash
# 查看最终解析的 OkHttp 版本及冲突来源
./gradlew :app:dependencyInsight --dependency okhttp --configuration releaseRuntimeClasspath
```

```kotlin
// 解决方案一：宿主侧强制版本
configurations.all {
    resolutionStrategy { force("com.squareup.okhttp3:okhttp:4.12.0") }
}

// 解决方案二：SDK 侧用 compileOnly 避免传递
dependencies {
    compileOnly("com.squareup.okhttp3:okhttp:4.12.0")
}
```

#### 场景二：重复类冲突

```
Duplicate class kotlin.collections.jdk8.CollectionsJDK8Kt found in modules:
  kotlin-stdlib-1.8.22 and kotlin-stdlib-jdk8-1.6.21
```

```kotlin
// 解决：统一 kotlin-stdlib 版本（1.8+ 已合并 jdk7/jdk8）
configurations.all {
    resolutionStrategy {
        force("org.jetbrains.kotlin:kotlin-stdlib:1.9.22")
    }
}
```

### 4.3 exclude transitive 依赖

#### 方式一：单个依赖排除

```kotlin
// SDK 的 build.gradle.kts
dependencies {
    implementation("com.example:network-lib:2.0.0") {
        // 排除传递的 Gson（SDK 自己用 Moshi）
        exclude(group = "com.google.code.gson", module = "gson")
        // 排除传递的旧版 support 库
        exclude(group = "com.android.support")
    }
}
```

#### 方式二：全局排除

```kotlin
// 对所有 configuration 生效
configurations.all {
    exclude(group = "com.android.support", module = "support-v4")
    exclude(group = "org.json", module = "json")  // Android 内置，无需额外引入
}
```

#### 方式三：在发布 POM 中控制传递依赖

```kotlin
publishing {
    publications {
        create<MavenPublication>("release") {
            afterEvaluate { from(components["release"]) }
            pom.withXml {
                // 遍历 POM 中的 dependencies，移除内部依赖
                (asNode()["dependencies"] as? groovy.util.NodeList)?.forEach { deps ->
                    ((deps as groovy.util.Node).children() as List<*>).toList().forEach { dep ->
                        val node = dep as groovy.util.Node
                        val groupId = (node["groupId"] as groovy.util.NodeList).text()
                        if (groupId.startsWith("com.example.internal")) {
                            node.parent().remove(node)
                        }
                    }
                }
            }
        }
    }
}
```

#### 实战踩坑：exclude 过度导致运行时崩溃

```kotlin
// ❌ 排除了 OkHttp 的核心依赖 Okio → NoClassDefFoundError: okio.Buffer
implementation("com.squareup.okhttp3:okhttp:4.12.0") {
    exclude(group = "com.squareup.okio")  // Okio 是必要依赖！
}

// ✅ 只排除确实不需要的传递依赖
implementation("com.squareup.okhttp3:okhttp:4.12.0") {
    exclude(group = "org.jetbrains.kotlin", module = "kotlin-stdlib-jdk8")
}
```

#### 实战踩坑：SDK 内部模块误用 api 导致泄漏

```kotlin
// ❌ 内部模块不应该用 api，会导致宿主能访问内部工具类
dependencies { api(project(":internal-utils")) }

// ✅ 改为 implementation，公开 API 通过接口隔离
dependencies { implementation(project(":internal-utils")) }
```

#### SDK 发布前体积检查清单

- [ ] AAR 体积在预算范围内（建议 < 500KB，不含 SO）
- [ ] 内部依赖使用 implementation，可选依赖使用 compileOnly
- [ ] 提供了 consumer-rules.pro（参考 [[混淆与ProGuard规则]]）
- [ ] Native SO 仅包含必要 ABI（arm64-v8a）
- [ ] 移除 debug 专用资源，使用 VectorDrawable 替代 PNG
- [ ] 运行 dependency-analysis-gradle-plugin 检查
- [ ] CI 中加入体积监控卡点

---

> **总结**：SDK 体积控制需从代码、资源、Native SO、依赖传递四个维度综合优化。核心原则：**最小暴露、最少依赖、按需加载**。
