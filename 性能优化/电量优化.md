# 电量优化

> Android 性能优化系列：深入理解电量消耗机制与优化策略。
> 关联：[[Service全解析]] · [[网络优化]] · [[内存优化]]

## 一、核心概念

### 1.1 Android 电量消耗来源

Android 设备的电量消耗主要来自以下五大模块：

| 消耗源 | 说明 | 典型场景 |
|--------|------|----------|
| CPU | 高频计算、长时间占用 CPU | 音视频编解码、复杂算法、死循环 |
| 网络 | 蜂窝网络（Radio）唤醒与数据传输 | 频繁短连接、轮询请求、大文件下载 |
| GPS | 持续定位、高精度定位 | 导航、运动轨迹记录、后台定位上报 |
| 屏幕 | 高亮度、高刷新率、持续亮屏 | 视频播放、游戏、WakeLock 保持亮屏 |
| WakeLock | 阻止 CPU 或屏幕休眠 | 后台任务未释放锁、音乐播放 |

其中，**网络（蜂窝 Radio）** 和 **GPS** 是后台耗电的两大元凶。蜂窝网络的状态机模型如下：

```
空闲(Idle) --请求触发--> 高功率(Active) --无数据 5s--> 低功率(Tail) --无数据 12s--> 空闲(Idle)
```

每次从 Idle 唤醒到 Active 都会产生约 **2 秒的额外高功耗尾部时间**，因此频繁的小请求比一次批量请求耗电得多。

### 1.2 Doze 模式与 App Standby

**Doze 模式（Android 6.0+）**

当设备满足以下条件时进入 Doze：
- 屏幕关闭
- 未充电
- 设备静止（Android 7.0+ 去掉了静止要求，引入 Light Doze）

Doze 模式下系统的限制：

```
┌─────────────────────────────────────────────┐
│              Doze 模式限制                    │
├─────────────────────────────────────────────┤
│ ✗ 暂停网络访问                               │
│ ✗ 推迟 AlarmManager（非精确闹钟）             │
│ ✗ 暂停 WiFi 扫描                             │
│ ✗ 暂停 SyncAdapter                           │
│ ✗ 暂停 JobScheduler                          │
│ ✓ 维护窗口(Maintenance Window)内短暂恢复      │
└─────────────────────────────────────────────┘
```

系统会周期性地打开**维护窗口**，让应用集中处理挂起的任务，窗口间隔会逐渐拉长（从几分钟到数小时）。

**App Standby（应用待机）**

当应用满足以下条件时被标记为 Standby：
- 用户未主动启动该应用
- 应用没有前台 [[Service全解析|Service]] 或可见 Activity
- 应用未产生用户可见的通知

被标记为 Standby 的应用，网络访问和任务调度会受到严格限制。

Android 9.0 引入了 **App Standby Buckets**，将应用分为 5 个桶：

| Bucket | 说明 | 限制程度 |
|--------|------|----------|
| Active | 正在使用 | 无限制 |
| Working Set | 经常使用 | 轻度限制 |
| Frequent | 常用但非每天 | 限制 JobScheduler、Alarm |
| Rare | 很少使用 | 严格限制网络、任务 |
| Restricted (12+) | 极少使用 | 几乎完全限制 |

### 1.3 后台限制（Android 8.0+）

Android 8.0（API 26）对后台执行引入了重大限制：

**后台服务限制**

应用进入后台后，系统会在几分钟内停止其后台 [[Service全解析|Service]]。替代方案：

```java
// ✗ 不推荐：直接启动后台 Service（8.0+ 会抛出 IllegalStateException）
context.startService(new Intent(context, BackgroundService.class));

// ✓ 推荐方案一：使用前台 Service
ContextCompat.startForegroundService(context,
    new Intent(context, ForegroundService.class));

// ✓ 推荐方案二：使用 WorkManager（首选）
WorkManager.getInstance(context)
    .enqueue(new OneTimeWorkRequest.Builder(MyWorker.class).build());
```

**广播限制**

大部分隐式广播无法通过 Manifest 静态注册接收，必须使用动态注册或 JobScheduler 替代：

```java
// ✗ AndroidManifest.xml 中静态注册隐式广播（8.0+ 大部分无效）
// <receiver android:name=".NetworkChangeReceiver">
//     <intent-filter>
//         <action android:name="android.net.conn.CONNECTIVITY_CHANGE"/>
//     </intent-filter>
// </receiver>

// ✓ 动态注册
IntentFilter filter = new IntentFilter(ConnectivityManager.CONNECTIVITY_ACTION);
registerReceiver(networkReceiver, filter);
```

**后台位置限制（Android 8.0+）**

后台应用获取位置的频率被限制为**每小时仅几次**。Android 10+ 需要额外申请 `ACCESS_BACKGROUND_LOCATION` 权限。

## 二、优化方案

### 2.1 WakeLock 正确使用

WakeLock 是最常见的电量泄漏源。核心原则：**能不用就不用，用了必须释放**。

```java
PowerManager pm = (PowerManager) getSystemService(Context.POWER_SERVICE);
PowerManager.WakeLock wakeLock = pm.newWakeLock(
    PowerManager.PARTIAL_WAKE_LOCK, "myapp:my_task_tag");

// ✓ 推荐：使用超时自动释放
wakeLock.acquire(10 * 60 * 1000L); // 最多持有 10 分钟

try {
    doBackgroundWork();
} finally {
    // ✓ 必须在 finally 中释放
    if (wakeLock.isHeld()) {
        wakeLock.release();
    }
}
```

**WakeLock 类型对比：**

| 类型 | CPU | 屏幕 | 键盘灯 | 使用场景 |
|------|-----|------|--------|----------|
| PARTIAL_WAKE_LOCK | ✓ | ✗ | ✗ | 后台计算任务 |
| SCREEN_DIM_WAKE_LOCK | ✓ | 暗 | ✗ | 已废弃，用 FLAG_KEEP_SCREEN_ON |
| SCREEN_BRIGHT_WAKE_LOCK | ✓ | 亮 | ✗ | 已废弃 |
| FULL_WAKE_LOCK | ✓ | 亮 | ✓ | 已废弃 |

**最佳实践：**
- 优先使用 `FLAG_KEEP_SCREEN_ON` 替代屏幕相关 WakeLock
- 始终设置超时时间 `acquire(timeout)`
- 使用 `try-finally` 确保释放
- Tag 命名规范：`包名:功能描述`

```java
// ✓ 保持屏幕常亮的正确方式（Activity 中）
getWindow().addFlags(WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON);

// 不需要时移除
getWindow().clearFlags(WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON);
```

### 2.2 JobScheduler / WorkManager 批量任务

将零散的后台任务合并为批量执行，利用系统调度在最佳时机运行，是电量优化的核心策略。

**WorkManager（推荐首选）**

WorkManager 是 [[Service全解析|后台任务]] 的统一解决方案，内部会根据 API 级别自动选择 JobScheduler 或 AlarmManager：

```java
// 定义 Worker
public class SyncWorker extends Worker {
    public SyncWorker(@NonNull Context context, @NonNull WorkerParameters params) {
        super(context, params);
    }

    @NonNull
    @Override
    public Result doWork() {
        // 执行同步逻辑
        syncDataToServer();
        return Result.success();
    }
}

// 构建约束条件 —— 仅在充电 + WiFi 下执行
Constraints constraints = new Constraints.Builder()
    .setRequiresCharging(true)
    .setRequiredNetworkType(NetworkType.UNMETERED)
    .build();

// 周期性任务：每 6 小时同步一次
PeriodicWorkRequest syncRequest = new PeriodicWorkRequest.Builder(
        SyncWorker.class, 6, TimeUnit.HOURS)
    .setConstraints(constraints)
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL,
        WorkRequest.MIN_BACKOFF_MILLIS, TimeUnit.MILLISECONDS)
    .build();

// 入队（KEEP 策略避免重复）
WorkManager.getInstance(context)
    .enqueueUniquePeriodicWork("data_sync",
        ExistingPeriodicWorkPolicy.KEEP, syncRequest);
```

**JobScheduler（API 21+）**

```java
JobScheduler scheduler = (JobScheduler) getSystemService(JOB_SCHEDULER_SERVICE);
ComponentName serviceName = new ComponentName(this, MyJobService.class);

JobInfo jobInfo = new JobInfo.Builder(JOB_ID, serviceName)
    .setRequiresCharging(true)
    .setRequiredNetworkType(JobInfo.NETWORK_TYPE_UNMETERED)
    .setPeriodic(6 * 60 * 60 * 1000L) // 6 小时
    .setPersisted(true) // 重启后保留
    .build();

scheduler.schedule(jobInfo);
```

**批量任务的核心思想：**
- 非紧急任务延迟到充电 + WiFi 时执行
- 多个小任务合并为一个批量任务
- 利用约束条件让系统选择最佳执行时机
- 避免使用精确 Alarm 做周期性任务

### 2.3 位置服务优化

GPS 是高耗电组件，优化策略：

```java
// ✓ 根据场景选择合适的精度和频率
LocationRequest request = LocationRequest.create()
    .setPriority(LocationRequest.PRIORITY_BALANCED_POWER_ACCURACY) // 平衡精度
    .setInterval(60 * 1000L)        // 期望间隔 60 秒
    .setFastestInterval(30 * 1000L) // 最快间隔 30 秒
    .setSmallestDisplacement(50f);  // 最小位移 50 米才回调

FusedLocationProviderClient client =
    LocationServices.getFusedLocationProviderClient(this);
client.requestLocationUpdates(request, locationCallback, Looper.getMainLooper());
```

**位置优化策略表：**

| 场景 | Priority | Interval | 说明 |
|------|----------|----------|------|
| 导航 | HIGH_ACCURACY | 1-5s | GPS + 网络，最耗电 |
| 城市级定位 | BALANCED_POWER | 30-60s | WiFi + 基站 |
| 粗略定位 | LOW_POWER | 5-10min | 仅基站 |
| 被动接收 | NO_POWER | - | 借用其他应用的定位结果 |

**关键优化点：**
- 不需要定位时**立即移除监听** `removeLocationUpdates()`
- 后台场景使用 `PRIORITY_NO_POWER` 或 `PRIORITY_LOW_POWER`
- 设置 `setSmallestDisplacement()` 避免原地不动时频繁回调
- Android 12+ 使用**粗略位置** `ACCESS_COARSE_LOCATION` 即可满足大部分场景

### 2.4 网络请求合并

频繁的 [[网络优化|网络请求]] 会反复唤醒蜂窝 Radio，造成严重耗电。

**优化策略：**

```java
// ✗ 反面示例：多个独立请求
void badApproach() {
    api.fetchUserProfile();   // 唤醒 Radio → Active → Tail → Idle
    api.fetchNotifications(); // 再次唤醒 Radio → Active → Tail → Idle
    api.fetchFeed();          // 再次唤醒 Radio → Active → Tail → Idle
}

// ✓ 正面示例：合并为一个批量请求
void goodApproach() {
    // 方案一：服务端提供聚合接口
    api.fetchDashboard(); // 一次请求返回 profile + notifications + feed

    // 方案二：客户端并发合并
    Observable.zip(
        api.fetchUserProfile(),
        api.fetchNotifications(),
        api.fetchFeed(),
        (profile, notifications, feed) -> new DashboardData(profile, notifications, feed)
    ).subscribe(this::updateUI);
}
```

**网络电量优化清单：**
- 合并多个 API 为聚合接口，减少请求次数
- 使用 HTTP/2 多路复用，复用连接
- 预取数据：在 WiFi 下预加载可能需要的数据
- 压缩传输：启用 Gzip，使用 Protocol Buffers 替代 JSON
- 缓存策略：合理使用 OkHttp Cache，减少重复请求
- 避免轮询：使用 [[网络优化|WebSocket]] 或 Push 替代定时轮询

### 2.5 Battery Historian 分析

Battery Historian 是 Google 提供的电量分析工具，可以可视化分析 bugreport 中的电量数据。

**使用步骤：**

```bash
# 1. 重置电量统计
adb shell dumpsys batterystats --reset

# 2. 断开 USB，使用 WiFi adb 或让用户正常使用一段时间

# 3. 重新连接，导出 bugreport
adb bugreport bugreport.zip

# 4. 启动 Battery Historian（Docker 方式）
docker run -p 9999:9999 gcr.io/android-battery-historian/stable:3.1 \
    --port 9999

# 5. 浏览器打开 http://localhost:9999 上传 bugreport.zip
```

**关键分析指标：**

| 指标 | 含义 | 关注点 |
|------|------|--------|
| Wakelock | CPU 唤醒锁持有时间 | 是否有长时间未释放的锁 |
| Mobile Radio | 蜂窝网络活跃时间 | 是否频繁短时间唤醒 |
| GPS | 定位使用时间 | 后台是否持续定位 |
| JobScheduler | 任务调度情况 | 任务是否过于频繁 |
| Alarm | 闹钟触发频率 | 是否滥用精确闹钟 |

**代码中获取电量信息（调试用）：**

```java
IntentFilter filter = new IntentFilter(Intent.ACTION_BATTERY_CHANGED);
Intent batteryStatus = context.registerReceiver(null, filter);

int level = batteryStatus.getIntExtra(BatteryManager.EXTRA_LEVEL, -1);
int scale = batteryStatus.getIntExtra(BatteryManager.EXTRA_SCALE, -1);
float batteryPct = level * 100 / (float) scale;

boolean isCharging = batteryStatus.getIntExtra(
    BatteryManager.EXTRA_STATUS, -1) == BatteryManager.BATTERY_STATUS_CHARGING;

Log.d("Battery", "电量: " + batteryPct + "%, 充电中: " + isCharging);
```

## 三、面试题

### Q1：Android 中有哪些主要的耗电来源？如何定位耗电问题？

**A：** 主要耗电来源包括五大类：
1. **CPU**：长时间高负载计算、WakeLock 阻止休眠
2. **网络**：频繁唤醒蜂窝 Radio，每次唤醒都有 Tail 尾部功耗
3. **GPS**：持续高精度定位
4. **屏幕**：高亮度、常亮
5. **WakeLock**：未正确释放导致设备无法休眠

定位耗电问题的方法：
- 使用 `adb shell dumpsys batterystats` 查看电量统计
- 使用 Battery Historian 可视化分析 bugreport
- Android Studio Profiler 的 Energy Profiler 实时监控
- 检查代码中的 WakeLock、定位监听、Alarm 使用情况

### Q2：Doze 模式是什么？对应用有什么影响？如何适配？

**A：** Doze 是 Android 6.0 引入的省电机制。当设备未充电、屏幕关闭且静止一段时间后，系统进入 Doze 模式，限制应用的网络访问、Alarm、JobScheduler、SyncAdapter 等。

系统会周期性打开维护窗口，间隔逐渐拉长。Android 7.0 引入 Light Doze，去掉了静止要求。

适配方式：
- 使用 `WorkManager` 或 `JobScheduler` 替代 `AlarmManager`
- 如果必须在 Doze 中执行，使用 `setAndAllowWhileIdle()` 或 `setExactAndAllowWhileIdle()`（有频率限制）
- 高优先级 FCM 推送可以临时唤醒设备
- **不要** 引导用户关闭电池优化（`REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`），除非是即时通讯等核心场景

### Q3：WakeLock 使用不当会导致什么问题？如何正确使用？

**A：** WakeLock 使用不当是后台耗电的首要原因。常见问题：
- `acquire()` 后未调用 `release()`，导致 CPU 永远无法休眠
- 在异常路径中未释放锁
- 未设置超时时间，异常情况下锁永远不释放

正确使用方式：

```java
PowerManager.WakeLock wl = pm.newWakeLock(
    PowerManager.PARTIAL_WAKE_LOCK, "myapp:sync_task");
wl.acquire(5 * 60 * 1000L); // 必须设置超时
try {
    doWork();
} finally {
    if (wl.isHeld()) wl.release(); // finally 中释放
}
```

更好的方案是使用 `WorkManager`，它内部会自动管理 WakeLock 的获取和释放。

### Q4：JobScheduler 和 WorkManager 有什么区别？该如何选择？

**A：**

| 对比项 | JobScheduler | WorkManager |
|--------|-------------|-------------|
| 最低 API | 21 | 14（通过兼容层） |
| 实现方式 | 系统原生 API | Jetpack 库，内部适配多种实现 |
| 持久化 | 需要 `setPersisted(true)` | 默认持久化到数据库 |
| 链式任务 | 不支持 | 支持 `then()` 链式调用 |
| 约束条件 | 支持 | 支持，且更丰富 |
| 进程安全 | 单进程 | 支持多进程 |

**选择建议：** 绝大多数场景使用 WorkManager。它是 Google 推荐的统一后台任务方案，内部在 API 23+ 使用 JobScheduler，低版本回退到 AlarmManager + BroadcastReceiver。

### Q5：如何优化应用的网络耗电？

**A：** 网络耗电优化的核心是**减少 Radio 唤醒次数**：

1. **请求合并**：将多个小请求合并为一个批量请求，或使用聚合 API
2. **预取策略**：在 WiFi 下预加载用户可能需要的数据
3. **避免轮询**：使用 Push（FCM）或 [[网络优化|WebSocket]] 替代定时轮询
4. **延迟非紧急请求**：日志上报、数据同步等延迟到充电 + WiFi 时执行
5. **压缩数据**：Gzip 压缩、Protocol Buffers、WebP 图片
6. **缓存**：合理设置 HTTP 缓存，避免重复请求相同数据

### Q6：Android 8.0 的后台限制有哪些？如何适配？

**A：** Android 8.0 引入了三大后台限制：

1. **后台服务限制**：应用进入后台后，系统会在几分钟内停止后台 [[Service全解析|Service]]。调用 `startService()` 会抛出 `IllegalStateException`。
   - 适配：使用 `startForegroundService()` + 5 秒内调用 `startForeground()`，或使用 WorkManager

2. **广播限制**：大部分隐式广播无法通过 Manifest 静态注册。
   - 适配：改为动态注册，或使用 JobScheduler 监听条件变化

3. **后台位置限制**：后台应用获取位置频率降低为每小时几次。
   - 适配：使用前台 Service + 定位，或降低定位频率

## 四、实战与踩坑

### 4.1 踩坑：WakeLock 泄漏导致用户投诉耗电

**场景：** 应用有一个后台同步功能，使用 WakeLock 保持 CPU 唤醒。线上出现大量用户投诉耗电异常。

**问题代码：**

```java
public class SyncService extends IntentService {
    private PowerManager.WakeLock wakeLock;

    @Override
    protected void onHandleIntent(Intent intent) {
        PowerManager pm = (PowerManager) getSystemService(POWER_SERVICE);
        wakeLock = pm.newWakeLock(PowerManager.PARTIAL_WAKE_LOCK, "myapp:sync");
        wakeLock.acquire(); // ✗ 没有超时！

        try {
            syncData(); // 如果这里抛异常或网络超时...
        } catch (Exception e) {
            Log.e("Sync", "同步失败", e);
            // ✗ 异常路径没有释放 WakeLock！
        }
        wakeLock.release();
    }
}
```

**修复方案：**

```java
public class SyncWorker extends Worker {
    // WorkManager 自动管理 WakeLock，无需手动处理
    @NonNull
    @Override
    public Result doWork() {
        try {
            syncData();
            return Result.success();
        } catch (Exception e) {
            return Result.retry();
        }
    }
}
```

**教训：** 能用 WorkManager 就不要手动管理 WakeLock。如果必须使用，一定要设置超时并在 `finally` 中释放。

### 4.2 踩坑：AlarmManager 在 Doze 模式下失效

**场景：** 使用 `AlarmManager.setRepeating()` 做定时数据上报，用户反馈数据经常丢失。

**原因：** Doze 模式下，非精确闹钟会被推迟到维护窗口执行。`setRepeating()` 在 API 19+ 已经变为非精确。

```java
// ✗ Doze 下会被推迟
AlarmManager am = (AlarmManager) getSystemService(ALARM_SERVICE);
am.setRepeating(AlarmManager.RTC_WAKEUP,
    System.currentTimeMillis(), 15 * 60 * 1000, pendingIntent);

// ✓ 使用 WorkManager 替代
PeriodicWorkRequest reportWork = new PeriodicWorkRequest.Builder(
        ReportWorker.class, 15, TimeUnit.MINUTES)
    .setConstraints(new Constraints.Builder()
        .setRequiredNetworkType(NetworkType.CONNECTED)
        .build())
    .build();

WorkManager.getInstance(context)
    .enqueueUniquePeriodicWork("report",
        ExistingPeriodicWorkPolicy.KEEP, reportWork);
```

### 4.3 踩坑：后台定位持续耗电

**场景：** 运动类 App 在用户结束运动后，忘记移除定位监听，导致 GPS 持续工作。

```java
// ✗ 只注册了监听，没有在合适时机移除
public class TrackingActivity extends AppCompatActivity {
    private LocationCallback callback;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        callback = new LocationCallback() {
            @Override
            public void onLocationResult(LocationResult result) {
                recordLocation(result.getLastLocation());
            }
        };
        startTracking();
    }

    void startTracking() {
        LocationRequest request = LocationRequest.create()
            .setPriority(LocationRequest.PRIORITY_HIGH_ACCURACY)
            .setInterval(1000); // 每秒定位！
        fusedClient.requestLocationUpdates(request, callback, getMainLooper());
    }
    // ✗ 没有 removeLocationUpdates！
}
```

**修复：**

```java
@Override
protected void onDestroy() {
    super.onDestroy();
    if (callback != null) {
        fusedClient.removeLocationUpdates(callback); // ✓ 必须移除
    }
}

// 更好的做法：运动结束时立即停止，而非等 onDestroy
void stopTracking() {
    fusedClient.removeLocationUpdates(callback);
    callback = null;
}
```

### 4.4 实战：电量优化检查清单

在项目中落地电量优化，建议按以下清单逐项检查：

```
□ WakeLock
  ├── 所有 acquire() 都设置了超时时间
  ├── 所有 release() 都在 finally 块中
  └── 优先使用 WorkManager 替代手动 WakeLock

□ 后台任务
  ├── 使用 WorkManager 替代 IntentService / 后台 Service
  ├── 非紧急任务设置 requiresCharging + UNMETERED 约束
  └── 避免使用 AlarmManager 做周期性任务

□ 定位
  ├── 根据场景选择合适的 Priority
  ├── 设置合理的 interval 和 smallestDisplacement
  ├── 不需要时立即 removeLocationUpdates
  └── 后台场景使用 LOW_POWER 或 NO_POWER

□ 网络
  ├── 合并多个小请求为批量请求
  ├── 使用 Push 替代轮询
  ├── 启用 HTTP 缓存和数据压缩
  └── 大文件传输延迟到 WiFi 环境

□ 监控
  ├── 集成 Battery Historian 到 CI 流程
  ├── 使用 Energy Profiler 做开发期检测
  └── 线上监控异常耗电用户占比
```

---

> 📌 **总结：** 电量优化的核心思想是「**减少唤醒、批量执行、及时释放**」。优先使用系统提供的调度框架（WorkManager），让系统在最佳时机批量执行任务，避免手动管理 WakeLock 和 Alarm。结合 Battery Historian 做数据驱动的优化分析。
