# 卡顿优化

> Android 性能优化核心专题。卡顿直接影响用户体验，是线上质量的关键指标。
> 相关：[[启动优化]] · [[Handler消息机制]] · [[View绘制流程]] · [[内存优化]]

---

## 一、核心概念

### 1.1 16ms 帧率原理

人眼感知流畅动画的最低帧率约 60fps，每帧渲染预算：`1000ms / 60 ≈ 16.67ms`。

每个 VSYNC 周期内，应用需完成：
1. **Input** 事件处理（触摸、按键）
2. **Animation** 动画计算
3. **Traversal** 视图遍历（measure → layout → draw，参考 [[View绘制流程]]）

某帧处理超过 16ms → **掉帧（Jank）** → 用户感知卡顿。

### 1.2 VSYNC 与 Choreographer

Android 4.1（Project Butter）引入 VSYNC + Choreographer 协调渲染：

```
VSYNC 信号 → Choreographer 收到回调
→ 依次执行 CALLBACK_INPUT → CALLBACK_ANIMATION → CALLBACK_TRAVERSAL
→ ViewRootImpl.performTraversals() → measure → layout → draw
```

Choreographer 回调类型（优先级从高到低）：

```java
public static final int CALLBACK_INPUT = 0;
public static final int CALLBACK_ANIMATION = 1;
public static final int CALLBACK_INSETS_ANIMATION = 2;
public static final int CALLBACK_TRAVERSAL = 3;
public static final int CALLBACK_COMMIT = 4;
```

### 1.3 主线程消息队列与卡顿的关系

主线程本质是消息驱动模型（参考 [[Handler消息机制]]）：

```java
// ActivityThread.main()
public static void main(String[] args) {
    Looper.prepareMainLooper();
    ActivityThread thread = new ActivityThread();
    thread.attach(false, startSeq);
    Looper.loop(); // 进入消息循环
}
```

所有 UI 操作（含 VSYNC 回调）都通过 Handler 发送到 MessageQueue 排队执行。**卡顿本质：某个 Message 处理时间过长，阻塞了后续 VSYNC 回调。**

常见阻塞源：主线程 I/O、复杂布局 inflate/measure、主线程 Binder 调用、锁竞争、频繁 GC。

### 1.4 掉帧的定义与分级

| 级别 | 掉帧数 | 耗时 | 用户感知 |
|------|--------|------|----------|
| 轻微 | 1-2 帧 | 16-48ms | 几乎无感 |
| 中等 | 3-5 帧 | 48-96ms | 轻微卡顿 |
| 严重 | 6-15 帧 | 96-250ms | 明显卡顿 |
| 冻帧 | >15 帧 | >250ms | 界面冻结 |
| ANR | - | >5s | 系统弹窗 |

通过 `FrameMetrics`（API 24+）精确获取每帧耗时：

```java
window.addOnFrameMetricsAvailableListener((w, metrics, dropCount) -> {
    long total = metrics.getMetric(FrameMetrics.TOTAL_DURATION);
    if (total > 16_000_000) { // 纳秒
        Log.w("Jank", "掉帧! total=" + total / 1_000_000 + "ms");
    }
}, new Handler(Looper.getMainLooper()));
```

## 二、原理与源码

### 2.1 Choreographer.doFrame 源码

```java
// Choreographer.doFrame() 简化
void doFrame(long frameTimeNanos, int frame) {
    synchronized (mLock) {
        long startNanos = System.nanoTime();
        final long jitterNanos = startNanos - frameTimeNanos;
        if (jitterNanos >= mFrameIntervalNanos) {
            long skippedFrames = jitterNanos / mFrameIntervalNanos;
            if (skippedFrames >= SKIPPED_FRAME_WARNING_LIMIT) {
                // "Skipped XX frames! The application may be doing
                //  too much work on its main thread."
                Log.i(TAG, "Skipped " + skippedFrames + " frames!");
            }
        }
    }
    // 按优先级依次执行回调
    doCallbacks(CALLBACK_INPUT, frameTimeNanos);
    doCallbacks(CALLBACK_ANIMATION, frameTimeNanos);
    doCallbacks(CALLBACK_INSETS_ANIMATION, frameTimeNanos);
    doCallbacks(CALLBACK_TRAVERSAL, frameTimeNanos);
    doCallbacks(CALLBACK_COMMIT, frameTimeNanos);
}
```

`jitterNanos` 反映主线程被阻塞的时间。`SKIPPED_FRAME_WARNING_LIMIT` 默认 30，掉 30 帧以上才打印警告。

### 2.2 Looper.loop 中的消息处理耗时

```java
// Looper.loop() 核心
public static void loop() {
    final Looper me = myLooper();
    for (;;) {
        Message msg = me.mQueue.next();
        if (msg == null) return;

        // ★ 消息处理前后的 Printer 回调——卡顿监控的切入点
        final Printer logging = me.mLogging;
        if (logging != null) {
            logging.println(">>>>> Dispatching to " + msg.target
                    + " " + msg.callback + ": " + msg.what);
        }

        msg.target.dispatchMessage(msg);

        if (logging != null) {
            logging.println("<<<<< Finished to " + msg.target
                    + " " + msg.callback);
        }
        msg.recycleUnchecked();
    }
}
```

### 2.3 BlockCanary 原理

通过设置主线程 Looper 的 Printer 监控每条消息耗时：

```java
public class BlockCanary {
    public static void install(Context ctx, long thresholdMs) {
        Looper.getMainLooper().setMessageLogging(new Printer() {
            private long mStartTime;

            @Override
            public void println(String x) {
                if (x.startsWith(">>>>> Dispatching")) {
                    mStartTime = SystemClock.uptimeMillis();
                    startStackSampling(); // 启动采样线程
                } else if (x.startsWith("<<<<< Finished")) {
                    long cost = SystemClock.uptimeMillis() - mStartTime;
                    stopStackSampling();
                    if (cost > thresholdMs) {
                        reportBlock(cost, getStackTraces());
                    }
                }
            }
        });
    }
}
```

局限性：字符串拼接有性能开销；只能监控单条消息耗时；采样间隔影响堆栈精度。

### 2.4 ANR-WatchDog 原理

子线程定时检测主线程是否响应：

```java
public class ANRWatchDog extends Thread {
    private static final int TIMEOUT = 5000;
    private volatile long mTick = 0;
    private final Handler mHandler = new Handler(Looper.getMainLooper());
    private final Runnable mTicker = () -> { mTick = 0; };

    @Override
    public void run() {
        while (!isInterrupted()) {
            mTick += TIMEOUT;
            mHandler.post(mTicker);
            try { Thread.sleep(TIMEOUT); } catch (InterruptedException e) { return; }
            if (mTick != 0) {
                // 主线程 5 秒未响应，抓取堆栈上报
                StackTraceElement[] stack = Looper.getMainLooper()
                        .getThread().getStackTrace();
                onAnrDetected(stack);
            }
        }
    }
}
```

### 3.4 自定义卡顿监控方案

#### 帧率监控

```java
public class FPSMonitor {
    private long mLastFrameNanos;
    private int mFrameCount;

    public void start() {
        Choreographer.getInstance().postFrameCallback(new Choreographer.FrameCallback() {
            @Override
            public void doFrame(long frameTimeNanos) {
                if (mLastFrameNanos == 0) mLastFrameNanos = frameTimeNanos;
                mFrameCount++;
                long elapsed = frameTimeNanos - mLastFrameNanos;
                if (elapsed >= 1_000_000_000L) {
                    int fps = (int) (mFrameCount * 1_000_000_000L / elapsed);
                    reportFPS(fps);
                    mFrameCount = 0;
                    mLastFrameNanos = frameTimeNanos;
                }
                Choreographer.getInstance().postFrameCallback(this);
            }
        });
    }
}
```

#### 综合监控

```java
public class JankMonitor implements Printer {
    private static final long BLOCK_THRESHOLD = 100;
    private long mStartTime;
    private final List<String> mSamples = new ArrayList<>();
    private volatile boolean mMonitoring;
    private final Handler mSampleHandler; // 采样线程 Handler

    @Override
    public void println(String x) {
        if (x.charAt(0) == '>') {
            mStartTime = SystemClock.uptimeMillis();
            mSamples.clear();
            mMonitoring = true;
            scheduleSampling();
        } else {
            mMonitoring = false;
            long cost = SystemClock.uptimeMillis() - mStartTime;
            if (cost > BLOCK_THRESHOLD) {
                reportJank(cost, mSamples, getCurrentScene());
            }
        }
    }

    private void scheduleSampling() {
        mSampleHandler.postDelayed(() -> {
            if (mMonitoring) {
                Thread main = Looper.getMainLooper().getThread();
                mSamples.add(stackToString(main.getStackTrace()));
                scheduleSampling();
            }
        }, 50);
    }
}
```

方案对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| Looper Printer | 精度高，可定位原因 | 字符串拼接开销 |
| Choreographer 帧回调 | 直接反映帧率 | 无法定位原因 |
| ANR-WatchDog | 可检测 ANR | 精度低（秒级） |
| FrameMetrics | 系统 API，分阶段数据 | API 24+ |

## 四、面试题

### Q1：卡顿和 ANR 有什么区别？

**A：** 卡顿是单帧耗时超 16ms 导致掉帧，用户感知不流畅。ANR 是卡顿的极端情况——前台 Activity 输入事件 5s 未处理、BroadcastReceiver 10s 超时、前台 Service 20s 超时，系统会弹出 ANR 对话框。两者本质都是主线程阻塞，程度不同。

### Q2：Choreographer 工作原理？如何监控帧率？

**A：** Choreographer 是渲染调度器。`ViewRootImpl.scheduleTraversals()` 向 Choreographer 注册回调 → Choreographer 请求 VSYNC → VSYNC 到达时通过 [[Handler消息机制]] 发消息到主线程 → `doFrame()` 依次执行 Input/Animation/Traversal/Commit 回调。通过 `postFrameCallback()` 在每帧记录时间戳，计算相邻帧间隔即可得到实时 FPS。

### Q3：BlockCanary 原理及局限性？

**A：** 利用 `Looper.loop()` 中消息分发前后的 `Printer` 回调计时。`>>>>> Dispatching` 时记录开始时间并启动堆栈采样，`<<<<< Finished` 时计算耗时，超阈值则上报。局限：Printer 字符串拼接有开销；只能监控单条 Message；采样间隔影响精度；无法区分消息类型。

### Q4：RecyclerView 有哪些优化手段？

**A：**
- **缓存**：增大 `itemViewCacheSize`、共享 `RecycledViewPool`、合理设置 ViewType
- **绑定**：`onBindViewHolder` 避免耗时操作、使用 payloads 局部绑定、图片异步加载
- **布局**：`setHasFixedSize(true)`、item 布局扁平化、嵌套 RV 设置 `setInitialPrefetchItemCount`
- **数据**：`DiffUtil` / `AsyncListDiffer` 替代全量刷新、Paging 分页加载

### Q5：如何设计线上卡顿监控方案？

**A：** 四层架构：
1. **采集层**：Looper Printer 监控消息耗时 + Choreographer 监控帧率 + 子线程采样主线程堆栈
2. **策略层**：卡顿阈值分级（100ms/5s）、采样率控制、堆栈聚合去重
3. **上报层**：耗时 + 堆栈 + 页面 + 设备信息 + CPU/内存状态
4. **分析层**：Top 方法聚合、按页面/版本/机型维度分析、卡顿率趋势告警

### Q6：为什么不能在主线程做耗时操作？

**A：** 主线程是单线程消息循环（[[Handler消息机制]]），承担 UI 渲染（每 16ms）、事件分发、生命周期回调等。所有任务串行执行，某个任务耗时过长会阻塞后续 VSYNC 回调导致掉帧，阻塞输入事件导致无响应，超过系统阈值触发 ANR。

## 五、实战与踩坑

### 5.1 案例：列表滑动卡顿

**现象：** 商品列表快速滑动卡顿，Systrace 显示 `RV OnBindView` 耗时 30-50ms。

**根因：** `onBindViewHolder` 中同步读取 SharedPreferences。

```java
// ❌ 问题代码
@Override
public void onBindViewHolder(ViewHolder holder, int pos) {
    boolean isFav = PreferenceManager.getDefaultSharedPreferences(context)
            .getBoolean("fav_" + items.get(pos).getId(), false); // 可能触发磁盘 I/O
    holder.bindData(items.get(pos), isFav);
}

// ✅ 修复：预加载到内存
private final Set<String> mFavIds = new HashSet<>();

public void preloadFavorites() {
    Executors.newSingleThreadExecutor().execute(() -> {
        Map<String, ?> all = sp.getAll();
        for (Map.Entry<String, ?> e : all.entrySet()) {
            if (e.getKey().startsWith("fav_") && Boolean.TRUE.equals(e.getValue())) {
                mFavIds.add(e.getKey().substring(4));
            }
        }
    });
}

@Override
public void onBindViewHolder(ViewHolder holder, int pos) {
    boolean isFav = mFavIds.contains(items.get(pos).getId());
    holder.bindData(items.get(pos), isFav);
}
```

**效果：** bindView 从 30-50ms 降至 2-5ms。

### 5.2 踩坑记录

**坑 1：SP.apply() 导致 ANR**

`apply()` 虽异步写入，但 `Activity.onPause()` 时 `QueuedWork.waitToFinish()` 会同步等待所有 pending apply。解决：使用 MMKV 替代。

**坑 2：WebView 首次初始化卡顿（200-500ms）**

```java
// 利用 IdleHandler 在空闲时预创建
Looper.myQueue().addIdleHandler(() -> {
    new WebView(appContext).loadUrl("about:blank");
    return false;
});
```

**坑 3：Gson 反序列化大对象卡主线程**

传递 ID 按需查询，或使用 Parcelable 替代 JSON 序列化。

**坑 4：过度绘制**

多层背景叠加导致 GPU 渲染耗时。移除多余背景：`getWindow().setBackgroundDrawable(null);`
