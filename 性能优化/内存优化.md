# 内存优化（内存泄漏、Bitmap）

> 本文聚焦 Android 内存优化的核心知识体系，涵盖 JVM 内存模型、常见泄漏场景、Bitmap 内存管理、检测工具及面试高频题。

## 一、核心概念

### 1.1 JVM 内存模型

Android 运行时（ART / Dalvik）基于 JVM 规范，内存区域主要分为以下几块：

| 区域 | 线程共享 | 存储内容 | 异常 |
|------|---------|---------|------|
| **堆（Heap）** | 是 | 对象实例、数组 | OutOfMemoryError |
| **虚拟机栈（VM Stack）** | 否 | 局部变量表、操作数栈、方法出口 | StackOverflowError |
| **方法区（Method Area）** | 是 | 类信息、常量池、静态变量 | OutOfMemoryError |
| **程序计数器（PC Register）** | 否 | 当前线程执行的字节码行号 | 无 |
| **本地方法栈（Native Stack）** | 否 | Native 方法调用信息 | StackOverflowError |

在 Android 中，堆内存是我们优化的重点。每个进程的堆大小受系统限制：

```java
// 获取当前应用的堆内存上限（单位：MB）
ActivityManager am = (ActivityManager) context.getSystemService(Context.ACTIVITY_SERVICE);
int memoryClass = am.getMemoryClass();        // 普通上限，如 256MB
int largeMemoryClass = am.getLargeMemoryClass(); // 大堆上限，如 512MB
```

### 1.2 GC 机制

Android ART 采用**分代收集**策略：

- **Young Generation（新生代）**：新创建的对象分配在此，使用 **Copying GC**，速度快但会暂停线程（STW）。
- **Old Generation（老年代）**：经过多次 GC 存活的对象晋升到此，使用 **Concurrent Mark-Sweep（CMS）**。
- **Large Object Space**：大对象（如 Bitmap）直接分配在此区域。

GC 触发时机：
1. 分配对象时堆空间不足
2. 显式调用 `System.gc()`（不推荐，仅建议）
3. 系统内存紧张时的 `onTrimMemory()` 回调

```java
// 监听系统内存紧张回调
public class MyApplication extends Application {
    @Override
    public void onTrimMemory(int level) {
        super.onTrimMemory(level);
        if (level >= TRIM_MEMORY_MODERATE) {
            // 清理图片缓存、关闭数据库连接等
            ImageLoader.getInstance().clearMemoryCache();
        }
    }
}
```

### 1.3 内存泄漏 vs 内存溢出

| 对比项 | 内存泄漏（Memory Leak） | 内存溢出（OOM） |
|-------|------------------------|-----------------|
| 定义 | 对象不再使用但无法被 GC 回收 | 申请内存超过系统可分配的最大值 |
| 表现 | 内存持续增长，最终可能导致 OOM | 直接抛出 `OutOfMemoryError` |
| 原因 | GC Root 仍持有对象引用 | 堆内存不足、Bitmap 过大、泄漏累积 |
| 关系 | 泄漏是溢出的常见**诱因** | 溢出是泄漏的最终**结果** |

### 1.4 Android 内存限制

Android 系统通过 `dalvik.vm.heapsize` 等属性限制每个进程的堆大小：

```bash
# 查看设备内存限制（adb shell）
adb shell getprop dalvik.vm.heapsize       # 堆上限，如 512m
adb shell getprop dalvik.vm.heapgrowthlimit # 普通应用上限，如 256m
adb shell getprop dalvik.vm.heapstartsize  # 堆初始大小，如 8m
```

- `heapgrowthlimit`：普通应用的堆上限，超过即 OOM。
- `heapsize`：声明 `android:largeHeap="true"` 后的堆上限。
- 不同设备差异巨大：低端机可能仅 128MB，旗舰机可达 512MB+。

> ⚠️ 不要依赖 `largeHeap`，它会导致 GC 时间变长，应从根本上优化内存使用。

## 二、原理与源码

### 2.1 常见内存泄漏场景

#### 场景一：静态引用持有 Activity

静态变量的生命周期与进程一致，如果持有 Activity 引用，Activity 销毁后无法被回收。

```java
// ❌ 错误示范：静态变量持有 Activity
public class MemoryLeakDemo {
    private static Activity sActivity;

    public static void setActivity(Activity activity) {
        sActivity = activity; // Activity 永远无法被回收
    }
}

// ✅ 正确做法：使用 WeakReference
public class SafeDemo {
    private static WeakReference<Activity> sActivityRef;

    public static void setActivity(Activity activity) {
        sActivityRef = new WeakReference<>(activity);
    }

    public static Activity getActivity() {
        return sActivityRef != null ? sActivityRef.get() : null;
    }
}
```

#### 场景二：Handler 泄漏

非静态内部类的 Handler 隐式持有外部 Activity 的引用。当 Handler 的 MessageQueue 中还有未处理的消息时，Activity 无法被回收。

```java
// ❌ 错误示范：非静态 Handler
public class LeakActivity extends AppCompatActivity {
    private Handler mHandler = new Handler() {
        @Override
        public void handleMessage(Message msg) {
            // 隐式持有 LeakActivity.this
            updateUI();
        }
    };

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        // 延迟 10 分钟发送消息，期间 Activity 无法回收
        mHandler.sendEmptyMessageDelayed(0, 10 * 60 * 1000);
    }
}

// ✅ 正确做法：静态内部类 + WeakReference + onDestroy 清理
public class SafeActivity extends AppCompatActivity {

    private static class SafeHandler extends Handler {
        private final WeakReference<SafeActivity> mActivityRef;

        SafeHandler(SafeActivity activity) {
            mActivityRef = new WeakReference<>(activity);
        }

        @Override
        public void handleMessage(Message msg) {
            SafeActivity activity = mActivityRef.get();
            if (activity != null && !activity.isFinishing()) {
                activity.updateUI();
            }
        }
    }

    private final SafeHandler mHandler = new SafeHandler(this);

    @Override
    protected void onDestroy() {
        super.onDestroy();
        mHandler.removeCallbacksAndMessages(null); // 清除所有消息
    }

    private void updateUI() {
        // 更新界面
    }
}
```

#### 场景三：匿名内部类 / 非静态内部类

匿名内部类会隐式持有外部类引用，常见于 `Runnable`、`Listener`、`AsyncTask` 等。

```java
// ❌ 错误示范：匿名 Runnable 持有 Activity
public class LeakActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        new Thread(new Runnable() {
            @Override
            public void run() {
                // 隐式持有 LeakActivity.this
                SystemClock.sleep(60_000);
            }
        }).start();
    }
}

// ✅ 正确做法：静态内部类
public class SafeActivity extends AppCompatActivity {

    private static class MyRunnable implements Runnable {
        @Override
        public void run() {
            SystemClock.sleep(60_000);
        }
    }

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        new Thread(new MyRunnable()).start();
    }
}
```

#### 场景四：未注销的监听器 / 广播接收器

注册了监听但忘记在生命周期结束时注销，导致系统持有 Activity 引用。

```java
// ❌ 错误示范：注册了 BroadcastReceiver 但未注销
public class LeakActivity extends AppCompatActivity {
    private BroadcastReceiver mReceiver = new BroadcastReceiver() {
        @Override
        public void onReceive(Context context, Intent intent) {
            // 处理广播
        }
    };

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        registerReceiver(mReceiver, new IntentFilter("ACTION_TEST"));
    }
    // 忘记在 onDestroy 中 unregisterReceiver(mReceiver)
}

// ✅ 正确做法：配对注册与注销
public class SafeActivity extends AppCompatActivity {
    private BroadcastReceiver mReceiver = new BroadcastReceiver() {
        @Override
        public void onReceive(Context context, Intent intent) { }
    };

    @Override
    protected void onStart() {
        super.onStart();
        registerReceiver(mReceiver, new IntentFilter("ACTION_TEST"));
    }

    @Override
    protected void onStop() {
        super.onStop();
        unregisterReceiver(mReceiver); // 配对注销
    }
}
```

#### 场景五：单例持有 Context

```java
// ❌ 错误示范：单例持有 Activity Context
public class AppManager {
    private static AppManager sInstance;
    private Context mContext;

    private AppManager(Context context) {
        mContext = context; // 如果传入 Activity，则泄漏
    }

    public static AppManager getInstance(Context context) {
        if (sInstance == null) {
            sInstance = new AppManager(context);
        }
        return sInstance;
    }
}

// ✅ 正确做法：使用 Application Context
public class AppManager {
    private static AppManager sInstance;
    private Context mContext;

    private AppManager(Context context) {
        mContext = context.getApplicationContext(); // 使用 Application Context
    }

    public static AppManager getInstance(Context context) {
        if (sInstance == null) {
            sInstance = new AppManager(context);
        }
        return sInstance;
    }
}
```


### 2.2 Bitmap 内存计算

[[Bitmap]] 是 Android 中最大的内存消耗者之一。其内存占用公式：

```
内存大小 = 宽（px）× 高（px）× 每像素字节数
```

每像素字节数取决于 `Bitmap.Config`：

| Config | 每像素字节数 | 说明 |
|--------|------------|------|
| ARGB_8888 | 4 字节 | 默认，最高质量 |
| RGB_565 | 2 字节 | 无透明通道，适合不需要透明度的图片 |
| ARGB_4444 | 2 字节 | 已废弃，质量差 |
| ALPHA_8 | 1 字节 | 仅透明通道 |

**实际计算还需考虑资源目录的缩放因子**：

```java
// 一张 1920x1080 的图片放在 drawable-xxhdpi（480dpi）目录
// 在 420dpi 的设备上加载：
// 实际宽 = 1920 * (420 / 480) = 1680
// 实际高 = 1080 * (420 / 480) = 945
// 内存 = 1680 * 945 * 4 = 6,350,400 字节 ≈ 6.05 MB
```

```java
// 获取 Bitmap 实际内存占用
public static long getBitmapMemorySize(Bitmap bitmap) {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.KITKAT) {
        return bitmap.getAllocationByteCount(); // API 19+，考虑 inBitmap 复用
    }
    return bitmap.getByteCount();
}
```

### 2.3 BitmapFactory.Options 采样压缩

加载大图时，通过 `inSampleSize` 降采样可大幅减少内存：

```java
/**
 * 高效加载大图：先读取尺寸，再按目标大小采样加载
 */
public static Bitmap decodeSampledBitmap(Resources res, int resId,
                                          int reqWidth, int reqHeight) {
    // 第一步：仅读取图片尺寸，不加载像素到内存
    BitmapFactory.Options options = new BitmapFactory.Options();
    options.inJustDecodeBounds = true;
    BitmapFactory.decodeResource(res, resId, options);

    // 第二步：计算合适的采样率
    options.inSampleSize = calculateInSampleSize(options, reqWidth, reqHeight);

    // 第三步：以采样率加载图片
    options.inJustDecodeBounds = false;
    options.inPreferredConfig = Bitmap.Config.RGB_565; // 不需要透明度时用 565 省一半内存
    return BitmapFactory.decodeResource(res, resId, options);
}

/**
 * 计算 inSampleSize（必须是 2 的幂次）
 */
private static int calculateInSampleSize(BitmapFactory.Options options,
                                          int reqWidth, int reqHeight) {
    int width = options.outWidth;
    int height = options.outHeight;
    int inSampleSize = 1;

    if (height > reqHeight || width > reqWidth) {
        int halfHeight = height / 2;
        int halfWidth = width / 2;
        while ((halfHeight / inSampleSize) >= reqHeight
                && (halfWidth / inSampleSize) >= reqWidth) {
            inSampleSize *= 2;
        }
    }
    return inSampleSize;
}
```

### 2.4 inBitmap 复用机制

`inBitmap` 允许新解码的 Bitmap 复用已有 Bitmap 的内存区域，避免频繁分配和回收大块内存。

```java
/**
 * 使用 inBitmap 复用已有 Bitmap 内存
 * API 19+ 支持不同尺寸复用（新 Bitmap 字节数 <= 旧 Bitmap 的 allocationByteCount）
 * API 11-18 要求尺寸完全一致且 inSampleSize=1
 */
public class BitmapPool {
    // 使用 LRU 缓存管理可复用的 Bitmap
    private final Set<SoftReference<Bitmap>> mReusablePool =
            Collections.synchronizedSet(new HashSet<SoftReference<Bitmap>>());

    /**
     * 将不再显示但可复用的 Bitmap 放入池中
     */
    public void putReusableBitmap(Bitmap bitmap) {
        if (bitmap != null && bitmap.isMutable()) {
            mReusablePool.add(new SoftReference<>(bitmap));
        }
    }

    /**
     * 从池中查找可复用的 Bitmap，设置到 Options.inBitmap
     */
    public Bitmap getReusableBitmap(BitmapFactory.Options options) {
        Bitmap targetBitmap = null;
        Iterator<SoftReference<Bitmap>> iterator = mReusablePool.iterator();
        while (iterator.hasNext()) {
            Bitmap candidate = iterator.next().get();
            if (candidate == null) {
                iterator.remove();
                continue;
            }
            if (canUseForInBitmap(candidate, options)) {
                targetBitmap = candidate;
                iterator.remove();
                break;
            }
        }
        return targetBitmap;
    }

    private boolean canUseForInBitmap(Bitmap candidate, BitmapFactory.Options options) {
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.KITKAT) {
            int newByteCount = options.outWidth * options.outHeight
                    * getBytesPerPixel(candidate.getConfig());
            return newByteCount <= candidate.getAllocationByteCount();
        }
        return candidate.getWidth() == options.outWidth
                && candidate.getHeight() == options.outHeight
                && options.inSampleSize == 1;
    }

    private int getBytesPerPixel(Bitmap.Config config) {
        switch (config) {
            case ARGB_8888: return 4;
            case RGB_565:   return 2;
            case ALPHA_8:   return 1;
            default:        return 4;
        }
    }
}
```

> 💡 主流图片库如 [[Glide]] 内部已实现了完善的 BitmapPool，生产环境建议直接使用 Glide 而非手动管理。

## 三、工具与实践

### 3.1 LeakCanary 原理

[[LeakCanary]] 是 Square 开源的内存泄漏检测库，核心原理：

1. **监听 Activity/Fragment 销毁**：通过 `Application.registerActivityLifecycleCallbacks()` 监听 `onActivityDestroyed`。
2. **弱引用 + ReferenceQueue**：将销毁的 Activity 包装为 `WeakReference`，关联一个 `ReferenceQueue`。
3. **触发 GC 后检测**：等待 5 秒后手动触发 GC，检查 `ReferenceQueue` 中是否收到该弱引用。
4. **Dump Heap**：如果 GC 后对象仍未被回收，说明泄漏，dump hprof 文件。
5. **分析引用链**：使用 Shark 库解析 hprof，找到从 GC Root 到泄漏对象的最短引用链。

```java
// LeakCanary 核心检测逻辑简化版
public class LeakDetector {
    private final ReferenceQueue<Object> mQueue = new ReferenceQueue<>();
    private final Set<String> mRetainedKeys = new CopyOnWriteArraySet<>();

    public void watch(Object watchedObject, String description) {
        String key = UUID.randomUUID().toString();
        mRetainedKeys.add(key);
        KeyedWeakReference ref = new KeyedWeakReference(watchedObject, key, mQueue);

        // 5 秒后检查是否被回收
        ScheduledExecutorService executor = Executors.newSingleThreadScheduledExecutor();
        executor.schedule(() -> {
            // 先清理已回收的引用
            removeWeaklyReachableReferences();
            if (mRetainedKeys.contains(key)) {
                // 手动触发 GC
                Runtime.getRuntime().gc();
                SystemClock.sleep(100);
                removeWeaklyReachableReferences();
                if (mRetainedKeys.contains(key)) {
                    // 确认泄漏，dump heap 并分析
                    dumpHeapAndAnalyze();
                }
            }
        }, 5, TimeUnit.SECONDS);
    }

    private void removeWeaklyReachableReferences() {
        KeyedWeakReference ref;
        while ((ref = (KeyedWeakReference) mQueue.poll()) != null) {
            mRetainedKeys.remove(ref.key);
        }
    }
}
```

集成方式（仅 debug 包生效）：

```groovy
// build.gradle
dependencies {
    debugImplementation 'com.squareup.leakcanary:leakcanary-android:2.14'
    // LeakCanary 2.x 无需手动初始化，通过 ContentProvider 自动安装
}
```

### 3.2 Android Profiler

Android Studio 内置的 **Memory Profiler** 提供实时内存监控：

- **实时曲线**：观察堆内存、Native 内存、Graphics 内存的变化趋势。
- **Dump Heap**：捕获当前堆快照，查看对象分配情况。
- **Record Allocations**：记录一段时间内的对象分配，定位频繁分配的热点。
- **Force GC**：手动触发 GC，观察内存是否回落。

使用步骤：
1. 运行 App（Debug 模式）
2. 打开 Android Studio → View → Tool Windows → Profiler
3. 选择目标进程 → 点击 Memory 区域
4. 操作 App 复现问题场景
5. 点击 "Dump Java Heap" 分析对象

关键观察指标：
- **反复进出某页面后内存是否持续增长**（泄漏信号）
- **某类对象的实例数是否异常偏多**
- **Bitmap 对象的数量和总大小**

### 3.3 MAT（Memory Analyzer Tool）分析

MAT 是 Eclipse 基金会的堆分析工具，适合深度分析 hprof 文件：

```bash
# 1. 从 Android Studio 导出 hprof
# 2. 转换为标准 hprof 格式
hprof-conv android.hprof standard.hprof
# 3. 用 MAT 打开 standard.hprof
```

MAT 核心功能：
- **Dominator Tree**：按对象支配关系排序，快速找到占用内存最大的对象。
- **Histogram**：按类统计对象数量和内存占用。
- **Leak Suspects Report**：自动分析可疑泄漏点。
- **OQL（Object Query Language）**：类 SQL 查询对象。

```sql
-- MAT OQL 示例：查找所有 Activity 实例
SELECT * FROM instanceof android.app.Activity

-- 查找大于 1MB 的 Bitmap
SELECT * FROM android.graphics.Bitmap b WHERE b.@retainedHeapSize > 1048576
```

### 3.4 StrictMode

[[StrictMode]] 可在开发阶段检测潜在的内存和性能问题：

```java
// 在 Application.onCreate() 中启用（仅 Debug）
public class MyApplication extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        if (BuildConfig.DEBUG) {
            // 检测线程策略：主线程磁盘读写、网络访问等
            StrictMode.setThreadPolicy(new StrictMode.ThreadPolicy.Builder()
                    .detectAll()
                    .penaltyLog()      // 输出到 Logcat
                    .penaltyFlashScreen() // 屏幕闪烁提醒
                    .build());

            // 检测 VM 策略：Activity 泄漏、未关闭的 Closeable 等
            StrictMode.setVmPolicy(new StrictMode.VmPolicy.Builder()
                    .detectActivityLeaks()       // 检测 Activity 泄漏
                    .detectLeakedClosableObjects() // 检测未关闭的 IO
                    .detectLeakedSqlLiteObjects()  // 检测未关闭的 SQLite
                    .setClassInstanceLimit(MainActivity.class, 1) // 限制实例数
                    .penaltyLog()
                    .build());
        }
    }
}
```

## 四、面试题

### Q1：内存泄漏和内存溢出的区别？如何排查内存泄漏？

**A：**
- **内存泄漏（Memory Leak）**：对象已不再使用，但因被 GC Root 直接或间接引用而无法回收。常见原因包括静态引用持有 Activity、Handler 消息未清理、未注销的监听器等。
- **内存溢出（OOM）**：应用申请的内存超过系统分配的堆上限，抛出 `OutOfMemoryError`。泄漏累积是 OOM 的常见诱因。
- **排查方式**：
  1. 开发阶段集成 [[LeakCanary]]，自动检测并报告泄漏引用链。
  2. 使用 Android Profiler 观察内存曲线，反复进出页面看内存是否持续增长。
  3. Dump hprof 文件，用 MAT 的 Dominator Tree 和 Leak Suspects 分析。
  4. 启用 StrictMode 的 `detectActivityLeaks()` 辅助检测。

### Q2：Handler 为什么会导致内存泄漏？怎么解决？

**A：**
非静态内部类的 Handler 隐式持有外部 Activity 的引用。当 Handler 的 `MessageQueue` 中存在延迟消息或正在处理的消息时，引用链为：`主线程 Looper → MessageQueue → Message → Handler → Activity`，导致 Activity 无法被 GC 回收。

解决方案：
1. 将 Handler 声明为**静态内部类**，通过 `WeakReference` 持有 Activity。
2. 在 `onDestroy()` 中调用 `handler.removeCallbacksAndMessages(null)` 清除所有消息。
3. 使用 `Lifecycle` 感知组件，在生命周期结束时自动清理。

### Q3：一张 1920×1080 的 PNG 图片加载到内存占多少？如何优化？

**A：**
- 默认 ARGB_8888 配置：`1920 × 1080 × 4 = 8,294,400 字节 ≈ 7.9 MB`。
- 如果图片放在低密度资源目录（如 drawable-hdpi），在高密度设备上会被放大，内存更大。
- 优化手段：
  1. **降采样**：通过 `BitmapFactory.Options.inSampleSize` 按比例缩小。
  2. **选择合适的 Config**：不需要透明度时用 `RGB_565`，内存减半。
  3. **inBitmap 复用**：复用已有 Bitmap 的内存块，减少 GC 压力。
  4. **使用图片库**：[[Glide]] / Fresco 内置了完善的缓存和复用机制。
  5. **按需加载**：根据 ImageView 的实际尺寸加载，而非原图尺寸。

### Q4：LeakCanary 的检测原理是什么？

**A：**
1. 通过 `ActivityLifecycleCallbacks` 监听 Activity 的 `onDestroy` 回调。
2. 将销毁的 Activity 封装为 `WeakReference`，关联 `ReferenceQueue`。
3. 延迟 5 秒后触发 GC，检查 `ReferenceQueue` 是否收到该引用。
4. 如果 GC 后对象仍未进入 `ReferenceQueue`，说明存在泄漏。
5. Dump hprof 堆快照，使用 Shark 库解析出从 GC Root 到泄漏对象的最短引用链。
6. 通过通知栏展示泄漏信息，开发者可直接定位问题代码。

### Q5：Android 中有哪些 GC Root？为什么理解 GC Root 对排查泄漏很重要？

**A：**
Android/JVM 中的 GC Root 包括：
- **虚拟机栈中的局部变量**（正在执行的方法中的引用）
- **静态变量**（类的 static 字段）
- **JNI 引用**（Native 代码持有的全局引用）
- **活跃线程**（Thread 对象本身）
- **系统类加载器加载的类**

理解 GC Root 的重要性：内存泄漏的本质是**从 GC Root 到目标对象存在一条不应存在的引用链**。排查泄漏就是找到这条链并切断它。MAT 的 "Path to GC Roots" 功能正是基于此原理。

### Q6：onTrimMemory 回调的作用？不同 level 代表什么？

**A：**
`onTrimMemory(int level)` 是系统在内存紧张时通知应用释放资源的回调。

| Level | 含义 | 建议操作 |
|-------|------|---------|
| `TRIM_MEMORY_UI_HIDDEN` | App UI 不可见 | 释放 UI 相关资源（图片缓存） |
| `TRIM_MEMORY_RUNNING_MODERATE` | 系统内存偏低，App 在前台 | 释放非必要缓存 |
| `TRIM_MEMORY_RUNNING_LOW` | 系统内存低，App 在前台 | 积极释放缓存 |
| `TRIM_MEMORY_RUNNING_CRITICAL` | 系统内存极低 | 尽可能释放所有缓存 |
| `TRIM_MEMORY_BACKGROUND` | App 在后台，可能被杀 | 释放所有可恢复的资源 |
| `TRIM_MEMORY_MODERATE` | App 在后台 LRU 中间 | 释放更多资源 |
| `TRIM_MEMORY_COMPLETE` | App 在后台 LRU 末尾，即将被杀 | 释放一切可释放的资源 |

## 五、实战与踩坑

### 踩坑 1：WebView 内存泄漏——Android 最顽固的泄漏源

WebView 内部持有大量 Native 资源，且会通过多条路径引用 Activity，是 Android 中最难处理的泄漏场景之一。

```java
// ✅ 解决方案：独立进程 + 动态添加 + 手动销毁
// AndroidManifest.xml 中声明独立进程
// <activity android:name=".WebViewActivity" android:process=":web" />

public class WebViewActivity extends AppCompatActivity {
    private WebView mWebView;
    private ViewGroup mContainer;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_webview);
        mContainer = findViewById(R.id.web_container);

        // 使用 Application Context 创建 WebView，避免持有 Activity
        mWebView = new WebView(getApplicationContext());
        mContainer.addView(mWebView, new ViewGroup.LayoutParams(
                ViewGroup.LayoutParams.MATCH_PARENT,
                ViewGroup.LayoutParams.MATCH_PARENT));

        mWebView.loadUrl("https://example.com");
    }

    @Override
    protected void onDestroy() {
        if (mWebView != null) {
            mWebView.stopLoading();
            mWebView.setWebViewClient(null);
            mWebView.setWebChromeClient(null);
            mContainer.removeView(mWebView); // 先从父容器移除
            mWebView.removeAllViews();
            mWebView.destroy();              // 再销毁
            mWebView = null;
        }
        super.onDestroy();
    }
}
```

> ⚠️ 踩坑点：直接在 XML 中声明 `<WebView>` 会导致 WebView 持有 Activity Context，即使调用 `destroy()` 也可能泄漏。务必用代码动态创建并传入 Application Context。

### 踩坑 2：Bitmap 加载 OOM——图片目录放错导致内存暴涨

```
// 真实案例：一张 200KB 的 PNG（1024x768）放在 drawable 目录（默认 160dpi）
// 在 480dpi 的设备上加载：
// 缩放比 = 480 / 160 = 3
// 实际尺寸 = (1024*3) x (768*3) = 3072 x 2304
// 内存 = 3072 * 2304 * 4 = 28,311,552 字节 ≈ 27 MB！
// 一张图就占了 27MB，几张图就 OOM 了
```

**解决方案**：
1. 将图片放在正确的密度目录（drawable-xxhdpi 对应 480dpi）。
2. 使用 `BitmapFactory.Options.inSampleSize` 降采样。
3. 大图使用 `BitmapRegionDecoder` 分区域加载。

```java
// 使用 BitmapRegionDecoder 加载超大图的局部区域
public static Bitmap loadRegion(InputStream is, Rect region) throws IOException {
    BitmapRegionDecoder decoder = BitmapRegionDecoder.newInstance(is, false);
    BitmapFactory.Options options = new BitmapFactory.Options();
    options.inPreferredConfig = Bitmap.Config.RGB_565;
    Bitmap bitmap = decoder.decodeRegion(region, options);
    decoder.recycle();
    return bitmap;
}
```

### 踩坑 3：集合对象未清理——Map/List 持有对象导致泄漏

```java
// ❌ 错误示范：静态 Map 缓存 Activity 相关对象
public class DataCache {
    private static final Map<String, Object> sCache = new HashMap<>();

    public static void put(String key, Object value) {
        sCache.put(key, value); // 如果 value 持有 Activity 引用，永远不会释放
    }
}

// ✅ 正确做法：使用 WeakHashMap 或 LruCache，并在适当时机清理
public class DataCache {
    private static final LruCache<String, Object> sCache = new LruCache<String, Object>(50) {
        @Override
        protected int sizeOf(String key, Object value) {
            return 1; // 按条目数限制
        }
    };

    public static void put(String key, Object value) {
        sCache.put(key, value);
    }

    public static void clear() {
        sCache.evictAll();
    }
}
```

### 踩坑 4：Cursor / Stream 未关闭

```java
// ❌ 错误示范：查询数据库后忘记关闭 Cursor
public List<String> queryNames(SQLiteDatabase db) {
    Cursor cursor = db.rawQuery("SELECT name FROM users", null);
    List<String> names = new ArrayList<>();
    while (cursor.moveToNext()) {
        names.add(cursor.getString(0));
    }
    // 忘记 cursor.close()，导致内存泄漏和数据库连接泄漏
    return names;
}

// ✅ 正确做法：try-finally 确保关闭
public List<String> queryNames(SQLiteDatabase db) {
    Cursor cursor = null;
    try {
        cursor = db.rawQuery("SELECT name FROM users", null);
        List<String> names = new ArrayList<>();
        while (cursor.moveToNext()) {
            names.add(cursor.getString(0));
        }
        return names;
    } finally {
        if (cursor != null) {
            cursor.close();
        }
    }
}
```

### 踩坑 5：线上内存监控方案

仅靠开发阶段的 [[LeakCanary]] 不够，线上需要建立内存监控体系：

```java
/**
 * 线上内存监控：定期采集内存指标上报
 */
public class MemoryMonitor {

    private static final long THRESHOLD_RATIO = 85; // 堆使用率超过 85% 告警

    public static void startMonitor(final Context context) {
        ScheduledExecutorService executor = Executors.newSingleThreadScheduledExecutor();
        executor.scheduleAtFixedRate(new Runnable() {
            @Override
            public void run() {
                Runtime runtime = Runtime.getRuntime();
                long maxMemory = runtime.maxMemory();       // 堆上限
                long totalMemory = runtime.totalMemory();   // 当前已分配
                long freeMemory = runtime.freeMemory();     // 已分配中空闲的
                long usedMemory = totalMemory - freeMemory; // 实际使用

                long usagePercent = usedMemory * 100 / maxMemory;

                if (usagePercent > THRESHOLD_RATIO) {
                    // 上报告警：内存使用率过高
                    reportMemoryWarning(usedMemory, maxMemory, usagePercent);
                }

                // 定期上报内存快照
                reportMemorySnapshot(usedMemory, maxMemory, totalMemory, freeMemory);
            }
        }, 0, 30, TimeUnit.SECONDS);
    }

    private static void reportMemoryWarning(long used, long max, long percent) {
        // 上报到 APM 平台
        Log.w("MemoryMonitor", "内存告警: " + percent + "%, used="
                + (used / 1024 / 1024) + "MB, max=" + (max / 1024 / 1024) + "MB");
    }

    private static void reportMemorySnapshot(long used, long max, long total, long free) {
        // 上报内存快照数据
    }
}
```
