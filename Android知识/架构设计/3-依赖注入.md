# 依赖注入（Hilt / Dagger / Koin）

> 依赖注入是 Android 架构中解耦组件、提升可测试性的核心手段。本文系统梳理 DI 的概念、三大主流框架的原理与实战，以及高频面试题。
>
> 相关主题：[[2-Clean-Architecture]] · [[1-MVC-MVP-MVVM-MVI]] · [[Android-组件化]]

---

## 一、核心概念

### 1.1 DI 是什么

**依赖注入（Dependency Injection）** 是一种设计模式：对象不自行创建其依赖，而是由外部将依赖"注入"进来。

````java
// ❌ 硬编码依赖 —— 紧耦合
public class OrderService {
    private final OrderRepository repo = new OrderRepositoryImpl();
}

// ✅ 依赖注入 —— 松耦合
public class OrderService {
    private final OrderRepository repo;

    @Inject
    public OrderService(OrderRepository repo) {
        this.repo = repo;
    }
}
````
核心思想：**调用方不关心依赖的创建细节，只声明"我需要什么"**。

### 1.2 为什么需要 DI

| 痛点 | DI 如何解决 |
|------|------------|
| 类之间紧耦合，改一处牵一发 | 面向接口编程，实现可替换 |
| 单元测试困难，无法 Mock | 注入 Mock 实现即可 |
| 对象生命周期管理混乱 | 框架统一管理 Scope |
| 重复的样板代码 | 框架自动生成/反射创建 |

### 1.3 控制反转 IoC

**IoC（Inversion of Control）** 是 DI 的上层原则——将"对象创建与装配"的控制权从调用方转移到框架/容器。

````
传统流程：A 主动 new B() → A 控制 B 的创建
IoC 流程：容器创建 B，注入给 A → 控制权反转到容器
````
DI 是 IoC 最常见的实现方式；另一种实现是**服务定位器**。

### 1.4 服务定位器 vs DI

| 维度 | 服务定位器（Service Locator） | 依赖注入（DI） |
|------|------|------|
| 获取方式 | 主动调用 `ServiceLocator.get(Foo.class)` | 被动接收，由容器注入 |
| 依赖可见性 | 隐式——看不出类需要哪些依赖 | 显式——构造函数/字段声明 |
| 可测试性 | 需要全局替换定位器 | 直接传入 Mock |
| 编译期安全 | ❌ 运行时才发现缺失 | ✅ Dagger/Hilt 编译期校验 |
| 典型代表 | Koin（本质是 SL + DSL） | Dagger2、Hilt |

> Google 官方推荐 DI 而非 Service Locator，但 Koin 因其简洁性在中小项目中仍广泛使用。

## 二、原理与源码

### 2.1 Dagger2：编译时代码生成

Dagger2 基于 **JSR 330** 注解 + **APT（Annotation Processing Tool）** 在编译期生成依赖图和工厂代码，零反射、零运行时开销。

**三大核心注解：**

#### @Inject —— 声明依赖

````java
public class UserRepository {
    private final ApiService api;
    private final UserDao dao;

    @Inject
    public UserRepository(ApiService api, UserDao dao) {
        this.api = api;
        this.dao = dao;
    }
}
````
编译后生成 `UserRepository_Factory.java`，包含 `newInstance()` 方法。

#### @Module + @Provides —— 提供无法 @Inject 的依赖

````java
@Module
public class NetworkModule {

    @Provides
    @Singleton
    public OkHttpClient provideOkHttp() {
        return new OkHttpClient.Builder()
                .connectTimeout(15, TimeUnit.SECONDS)
                .build();
    }

    @Provides
    @Singleton
    public Retrofit provideRetrofit(OkHttpClient client) {
        return new Retrofit.Builder()
                .baseUrl("https://api.example.com/")
                .client(client)
                .addConverterFactory(GsonConverterFactory.create())
                .build();
    }

    @Provides
    public ApiService provideApiService(Retrofit retrofit) {
        return retrofit.create(ApiService.class);
    }
}
````
#### @Component —— 连接依赖图

````java
@Singleton
@Component(modules = {NetworkModule.class, DatabaseModule.class})
public interface AppComponent {
    void inject(MainActivity activity);
    UserRepository userRepository();
}
````
编译后生成 `DaggerAppComponent.java`，内部持有所有 Provider，按拓扑顺序创建依赖。

编译后生成 `DaggerAppComponent.java`，内部用 `DoubleCheck`（双重检查锁）包装 Provider 实现 Scope 单例缓存。

### 2.2 Hilt：对 Dagger 的简化

Hilt 是 Google 在 Dagger 之上的封装层，核心简化点：

1. **预定义 Component 层级**——不再手写 Component

````
SingletonComponent          ← Application
 ├── ActivityRetainedComponent ← ViewModel
 │    └── ActivityComponent     ← Activity
 │         ├── FragmentComponent    ← Fragment
 │         └── ViewComponent        ← View
 └── ServiceComponent          ← Service
````
2. **@AndroidEntryPoint** —— 自动注入 Android 组件

````java
@AndroidEntryPoint
public class MainActivity extends AppCompatActivity {

    @Inject
    UserRepository userRepository;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        // userRepository 已自动注入，可直接使用
    }
}
````
3. **@HiltAndroidApp** —— 替代手动 Component 初始化

````java
@HiltAndroidApp
public class MyApp extends Application {
    // Hilt 自动生成 Component 并完成初始化
}
````
**Hilt 背后原理：** Hilt 的注解处理器在编译期为每个 `@AndroidEntryPoint` 类生成基类，在 `onCreate` 中自动调用 inject，并将 `@InstallIn` 的 Module 自动挂载到对应 Component。

### 2.3 Koin：运行时服务定位

Koin 不使用代码生成，而是基于 **Kotlin DSL + HashMap** 在运行时注册和查找依赖。

````kotlin
// 定义 Module（Kotlin DSL，无法用 Java）
val appModule = module {
    single { OkHttpClient.Builder().build() }
    single { Retrofit.Builder().client(get()).build() }
    factory { UserRepository(get(), get()) }
    viewModel { MainViewModel(get()) }
}

// 启动
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        startKoin {
            androidContext(this@MyApp)
            modules(appModule)
        }
    }
}
````
**核心原理（简化）：**

````kotlin
// Koin 内部本质是一个 Map
class BeanRegistry {
    // key = 类型 + qualifier, value = 工厂 lambda
    private val definitions = HashMap<BeanDefinition, InstanceFactory>()

    fun resolve(clazz: KClass<*>): Any {
        val factory = definitions[clazz]
            ?: throw NoBeanDefFoundException(clazz)
        return factory.create()  // 运行时调用 lambda 创建实例
    }
}
````
使用时通过 `by inject()` 或 `get()` 从容器查找——这正是**服务定位器**模式。

### 2.4 三者对比

| 维度 | Dagger2 | Hilt | Koin |
|------|---------|------|------|
| 实现方式 | 编译时 APT 代码生成 | 编译时（基于 Dagger） | 运行时 HashMap + Lambda |
| 语言要求 | Java / Kotlin | Java / Kotlin | Kotlin only |
| 编译期校验 | ✅ 依赖缺失编译报错 | ✅ 同 Dagger | ❌ 运行时才报错 |
| 学习曲线 | 陡峭（Component/Subcomponent） | 中等（预定义 Component） | 平缓（DSL 直觉） |
| 运行时性能 | 极高（无反射） | 极高（同 Dagger） | 较低（反射 + Map 查找） |
| 编译速度 | 较慢（APT 生成大量代码） | 较慢（同 Dagger） | 快（无代码生成） |
| 多模块支持 | 手动配置 Component 依赖 | `@InstallIn` 自动聚合 | `loadModules` 手动加载 |
| Google 推荐 | ✅ | ✅✅（首选） | ❌ |
| 适用场景 | 大型项目、极致性能 | 中大型项目（首选） | 中小项目、快速原型 |

### 3.1 Hilt 完整示例

#### 第一步：Gradle 配置

````groovy
// project build.gradle
plugins {
    id 'com.google.dagger.hilt.android' version '2.51' apply false

// app build.gradle
plugins {
    id 'com.android.application'
    id 'com.google.dagger.hilt.android'
}

dependencies {
    implementation 'com.google.dagger:hilt-android:2.51'
    annotationProcessor 'com.google.dagger:hilt-compiler:2.51'
}
````
#### 第二步：Application

````java
@HiltAndroidApp
public class MyApp extends Application { }
````
#### 第三步：Module 提供依赖

````java
@Module
@InstallIn(SingletonComponent.class)
public class NetworkModule {

    @Provides
    @Singleton
    public OkHttpClient provideOkHttp() {
        return new OkHttpClient.Builder().build();
    }

    @Provides
    @Singleton
    public ApiService provideApiService(OkHttpClient client) {
        Retrofit retrofit = new Retrofit.Builder()
                .baseUrl("https://api.example.com/")

                .addConverterFactory(GsonConverterFactory.create())
                .build();
        return retrofit.create(ApiService.class);
    }
}

@Module
@InstallIn(SingletonComponent.class)
public abstract class RepositoryModule {

    @Binds
    @Singleton
    public abstract UserRepository bindUserRepository(UserRepositoryImpl impl);
}
````
#### 第四步：Repository + ViewModel

````java
public class UserRepositoryImpl implements UserRepository {

    @Override

    @Inject

````

### 3.2 Scope 管理

Hilt 预定义的 Scope 与 Component 对应关系：

| 注解 | Component | 生命周期 |
|------|-----------|---------|
| `@Singleton` | SingletonComponent | Application |
| `@ActivityRetainedScoped` | ActivityRetainedComponent | ViewModel（配置变更存活） |
| `@ActivityScoped` | ActivityComponent | Activity |
| `@FragmentScoped` | FragmentComponent | Fragment |
| `@ViewScoped` | ViewComponent | View |
| `@ServiceScoped` | ServiceComponent | Service |

**原则：Scope 尽量小。** 不需要全局共享的对象不要标 `@Singleton`，否则内存无法释放。

````java
// ✅ 数据库、网络客户端 → @Singleton
@Provides @Singleton
public AppDatabase provideDb(@ApplicationContext Context ctx) {
    return Room.databaseBuilder(ctx, AppDatabase.class, "app.db").build();
}

// ✅ 页面级缓存 → @ActivityScoped
@Provides @ActivityScoped
public PageCache providePageCache() {
    return new PageCache();
}

// ✅ 无状态工具 → 不加 Scope（每次新建）
@Provides
public Gson provideGson() {
    return new Gson();
}
````
### 3.3 多绑定与 @Qualifier

当同一接口有多个实现时，用 `@Qualifier` 区分：

````java
// 定义 Qualifier
@Qualifier
@Retention(RetentionPolicy.RUNTIME)
public @interface AuthInterceptor { }

@Qualifier
@Retention(RetentionPolicy.RUNTIME)
public @interface LoggingInterceptor { }

// Module 中提供不同实现
@Module
@InstallIn(SingletonComponent.class)
public class InterceptorModule {

    @Provides
    @AuthInterceptor
    public Interceptor provideAuthInterceptor(TokenManager tokenManager) {
        return chain -> {
            Request request = chain.request().newBuilder()
                    .addHeader("Authorization", "Bearer " + tokenManager.getToken())
                    .build();
            return chain.proceed(request);
        };
    }

    @Provides
    @LoggingInterceptor
    public Interceptor provideLoggingInterceptor() {
        return new HttpLoggingInterceptor()
                .setLevel(HttpLoggingInterceptor.Level.BODY);
    }
}

// 注入时指定 Qualifier
@Module
@InstallIn(SingletonComponent.class)
public class NetworkModule {

    @Provides
    @Singleton
    public OkHttpClient provideOkHttp(
            @AuthInterceptor Interceptor auth,
            @LoggingInterceptor Interceptor logging) {
        return new OkHttpClient.Builder()
                .addInterceptor(auth)
                .addInterceptor(logging)
                .build();
    }
}
````
Hilt 还内置了 `@ApplicationContext` 和 `@ActivityContext` 两个 Qualifier，无需自定义。

## 四、面试题

### Q1：Dagger2 的 @Inject、@Module、@Component 分别起什么作用？

**A：** 三者构成 Dagger 的依赖图：
- `@Inject`：标记在构造函数上，告诉 Dagger "这个类可以被自动创建"；标记在字段上表示"请注入这个依赖"
- `@Module` + `@Provides`：为无法 `@Inject` 的类（第三方库、接口）提供创建方式
- `@Component`：连接 Module 和注入目标，是依赖图的入口。编译后生成 `DaggerXxxComponent`，内部按拓扑排序组装所有 Provider

### Q2：Hilt 相比 Dagger2 简化了什么？

**A：**
1. 不需要手写 Component 和 Subcomponent——Hilt 预定义了 `SingletonComponent`、`ActivityComponent` 等层级
2. 不需要手动调用 `component.inject(this)`——`@AndroidEntryPoint` 自动完成
3. 不需要在 Application 中手动构建 Component——`@HiltAndroidApp` 自动处理
4. ViewModel 注入只需 `@HiltViewModel` + `@Inject` 构造函数
5. Module 通过 `@InstallIn` 声明归属，多模块项目自动聚合

### Q3：Koin 和 Dagger 的本质区别是什么？

**A：**
- **Dagger** 是真正的 DI 框架，编译期生成代码，依赖关系在编译时校验，运行时无反射开销
- **Koin** 本质是**服务定位器**，运行时通过 HashMap 存储和查找依赖，使用 `get()` 主动获取。优点是简单无代码生成；缺点是依赖缺失只能在运行时发现，且有反射和 Map 查找的性能开销

### Q4：什么是 Scope？不加 Scope 注解会怎样？

**A：** Scope 控制对象的生命周期和复用范围。
- 加了 `@Singleton`：整个 App 生命周期内只创建一个实例（Component 持有缓存）
- 加了 `@ActivityScoped`：同一个 Activity 内共享同一实例，Activity 销毁后释放
- **不加 Scope**：每次注入都会创建新实例（unscoped），适合无状态的轻量对象

底层实现：Dagger 用 `DoubleCheck`（双重检查锁）包装 Provider 来实现 Scope 缓存。

### Q5：@Binds 和 @Provides 有什么区别？

**A：**
- `@Provides`：写在具体方法中，方法体包含创建逻辑，可以调用构造函数、Builder 等
- `@Binds`：写在抽象方法中，仅用于"接口绑定实现类"的场景，参数是实现类，返回值是接口。Dagger 不会生成额外工厂类，性能更优

````java
// @Binds —— 接口绑定（推荐）
@Binds abstract UserRepository bind(UserRepositoryImpl impl);

// @Provides —— 等价写法（多一层包装）
@Provides UserRepository provide(UserRepositoryImpl impl) { return impl; }
````
### Q6：如何在非 Android 类（如 Worker、ContentProvider）中使用 Hilt？

**A：**
- **WorkManager**：使用 `@HiltWorker` + `@AssistedInject`，并配置 `HiltWorkerFactory`
- **ContentProvider**：Hilt 不直接支持，因为 ContentProvider 在 Application.onCreate 之前初始化。解决方案是通过 `EntryPointAccessors` 手动获取依赖：

````java
public class MyContentProvider extends ContentProvider {
    @Override
    public boolean onCreate() {
        MyEntryPoint entryPoint = EntryPointAccessors.fromApplication(
                getContext(), MyEntryPoint.class);
        UserRepository repo = entryPoint.userRepository();
        return true;
    }
}

@EntryPoint
@InstallIn(SingletonComponent.class)
public interface MyEntryPoint {
    UserRepository userRepository();
}
````

## 五、实战与踩坑

### 5.1 Hilt 多模块配置

在组件化/多模块项目中，每个模块可以独立定义 Module：

````
app/                  ← @HiltAndroidApp
feature-user/         ← 包含 UserModule
feature-order/        ← 包含 OrderModule
core-network/         ← 包含 NetworkModule
````
**关键配置：**

````groovy
// 每个模块都需要添加 Hilt 依赖
// feature-user/build.gradle
plugins {
    id 'com.google.dagger.hilt.android'
}
dependencies {
    implementation 'com.google.dagger:hilt-android:2.51'
    annotationProcessor 'com.google.dagger:hilt-compiler:2.51'
}
````
````java
// feature-user 模块中的 Module
@Module
@InstallIn(SingletonComponent.class)
public class UserModule {

    @Provides
    public UserRepository provideUserRepo(ApiService api, UserDao dao) {
        return new UserRepositoryImpl(api, dao);
    }
}
````
**踩坑点：**

1. **每个包含 `@Module` 或 `@AndroidEntryPoint` 的模块都必须配置 `hilt-compiler`**，否则编译报 `MissingBinding`
2. **模块间依赖方向**：`feature` 模块依赖 `core` 模块，`core` 中的 `@Provides` 对 `feature` 可见；反之不行
3. **避免循环依赖**：Module A 提供的类型依赖 Module B 提供的类型，而 B 又依赖 A → 编译报错。解决方案是抽取公共接口到 `core` 层
4. **`@InstallIn` 必须一致**：如果 NetworkModule 装在 `SingletonComponent`，那么依赖它的 UserModule 不能装在 `ActivityComponent`（子 Component 可以访问父 Component 的绑定，反之不行）

### 5.2 Dagger 迁移 Hilt

对于已有 Dagger2 项目，推荐**渐进式迁移**：

**第一阶段：共存**

````java
// 保留原有 Dagger Component，同时新增 @HiltAndroidApp
@HiltAndroidApp
public class MyApp extends Application {
    private LegacyComponent legacyComponent;

    @Override
    public void onCreate() {
        super.onCreate();
        legacyComponent = DaggerLegacyComponent.create();
    }

    public LegacyComponent getLegacyComponent() {
        return legacyComponent;
    }
}
````
**第二阶段：逐步迁移 Module**

````java
// 将旧 Module 改为 @InstallIn
@Module
@InstallIn(SingletonComponent.class)  // 添加这一行
public class LegacyNetworkModule {
    // @Provides 方法不变
}
````
**第三阶段：迁移注入点**

````java
// 旧代码
public class OldActivity extends AppCompatActivity {
    @Inject UserRepository repo;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        ((MyApp) getApplication()).getLegacyComponent().inject(this);
        super.onCreate(savedInstanceState);
    }
}

// 迁移后
@AndroidEntryPoint
public class OldActivity extends AppCompatActivity {
    @Inject UserRepository repo;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        // 自动注入，删除手动 inject 调用
    }
}
````
**迁移注意事项：**
- `@AndroidEntryPoint` 的 Activity 必须继承 `ComponentActivity`（或 `AppCompatActivity`）
- 自定义 Scope 需要映射到 Hilt 预定义 Scope，或使用 `@DefineComponent` 自定义
- `@Component.Builder` / `@Component.Factory` 在 Hilt 中不再需要
- 迁移期间可以通过 `@EntryPoint` 让 Hilt 管理的依赖暴露给旧 Dagger 代码

### 5.3 Koin 性能问题与优化

Koin 的运行时特性带来几个性能隐患：

**问题一：启动时间**

````kotlin
// ❌ 所有 Module 在 startKoin 时注册，模块多时启动变慢
startKoin {
    modules(networkModule, dbModule, userModule, orderModule,
            paymentModule, chatModule, analyticsModule /* ... */)
}

// ✅ 使用 lazyModules 延迟加载（Koin 3.2+）
startKoin {
    lazyModules(featureUserModule, featureOrderModule)
}
````
**问题二：运行时崩溃**

Koin 不做编译期校验，依赖缺失只在运行时 `get()` 调用时抛出 `NoBeanDefFoundException`。

````kotlin
// ✅ 使用 checkModules 在测试中提前发现问题
class KoinModuleTest : KoinTest {
    @Test
    fun verifyKoinModules() {
        koinApplication {
            modules(allModules)
            checkModules()
        }
    }
}
````
**问题三：混淆与反射**

Koin 依赖 Kotlin 反射，ProGuard/R8 混淆可能导致运行时找不到类，需添加 keep 规则。

### 5.4 常见踩坑汇总

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `UninitializedPropertyAccessException` | 在 `super.onCreate()` 之前访问 `@Inject` 字段 | 确保在 `super.onCreate()` 之后使用 |
| Hilt `ActivityComponent` 中注入 `Application Context` | 用了 `@ActivityContext` 但需要 App 级 Context | 改用 `@ApplicationContext` |
| 多模块编译报 `DuplicateBindings` | 两个 Module 提供了相同类型 | 使用 `@Qualifier` 区分 |
| Koin `ClosedScopeException` | Scope 已关闭后仍尝试 `get()` | 检查生命周期，在 `onDestroy` 前关闭 |
| Hilt 测试中注入失败 | 测试类未加 `@HiltAndroidTest` | 添加注解并使用 `HiltAndroidRule` |
| Dagger 编译时间过长 | Component 过大，依赖图复杂 | 拆分 Component，使用 `@Subcomponent` |
