# 响应式架构（Flow / RxJava）

> 响应式编程是现代 Android 开发的核心范式。本文系统梳理 [[3-RxJava]] 与 Kotlin Flow 的原理、实战与迁移策略。
>
> 相关：[[2-Jetpack-LiveData与Flow]] · [[Kotlin协程]]

## 一、核心概念

### 1.1 响应式编程的定义

响应式编程（Reactive Programming）是面向**数据流**和**变化传播**的声明式编程范式。数据源变化时，下游**自动**收到通知并更新，无需主动轮询。

核心三要素：

| 要素 | 说明 |
|------|------|
| **数据流（Stream）** | 随时间推移产生的一系列事件/数据 |
| **操作符（Operator）** | 对流进行变换、过滤、合并等中间操作 |
| **订阅者（Subscriber）** | 消费最终数据，驱动副作用（如 UI 更新） |

### 1.2 观察者模式

底层设计模式是**观察者模式**：RxJava 中为 `Observable`/`Observer`，Flow 中为 `emit()`/`collect()` 配对。

**RxJava（Java）：**

```java
Observable.create(emitter -> {
    emitter.onNext("Hello");
    emitter.onComplete();
}).subscribe(
    s -> Log.d("Rx", s),
    e -> Log.e("Rx", e.getMessage()),
    () -> Log.d("Rx", "Done")
);
```

**Flow（Kotlin）：**

```kotlin
flow {
    emit("Hello")
}.collect { value -> Log.d("Flow", value) }
```

### 1.3 背压（Backpressure）

生产者发射速度远快于消费者处理速度时，需要流量控制机制。

| 框架 | 背压支持 | 说明 |
|------|---------|------|
| RxJava `Observable` | ❌ | 数据量小或可控场景 |
| RxJava `Flowable` | ✅ | 支持 `BUFFER`/`DROP`/`LATEST` 策略 |
| Kotlin `Flow` | ✅ | 基于协程挂起，消费者未就绪时生产者自动挂起 |

```java
// RxJava Flowable 背压（Java）
Flowable.create(emitter -> {
    for (int i = 0; i < 10000; i++) emitter.onNext(i);
    emitter.onComplete();
}, BackpressureStrategy.DROP)
    .observeOn(Schedulers.io())
    .subscribe(item -> Log.d("Rx", "Received: " + item));
```

```kotlin
// Flow 天然背压（Kotlin）
flow {
    for (i in 0..10000) emit(i)  // 消费者慢时自动挂起
}.buffer(64).collect { delay(10); Log.d("Flow", "$it") }
```

### 1.4 冷流 vs 热流

| 特性 | 冷流（Cold） | 热流（Hot） |
|------|-------------|-------------|
| **触发** | 有订阅者才发射 | 无论是否有订阅者 |
| **数据共享** | 每个订阅者独立获得完整数据 | 多订阅者共享同一流 |
| **RxJava** | `Observable.create()` | `PublishSubject` / `BehaviorSubject` |
| **Flow** | `flow { }` | `StateFlow` / `SharedFlow` |
| **场景** | 网络请求、DB 查询 | UI 状态、事件总线 |

## 二、原理与对比

### 2.1 RxJava 调度器与操作符链

**调度器：**

| 调度器 | 用途 |
|--------|------|
| `Schedulers.io()` | IO 密集型（网络/磁盘），缓存线程池 |
| `Schedulers.computation()` | CPU 密集型，固定 = CPU 核心数 |
| `AndroidSchedulers.mainThread()` | Android 主线程（需 rxandroid） |

**操作符分类：**

| 类别 | 操作符 |
|------|--------|
| 创建 | `just` / `fromIterable` / `create` / `interval` |
| 变换 | `map` / `flatMap` / `switchMap` / `concatMap` |
| 过滤 | `filter` / `debounce` / `distinctUntilChanged` / `take` |
| 合并 | `merge` / `zip` / `combineLatest` / `concat` |
| 错误 | `onErrorReturn` / `onErrorResumeNext` / `retry` |

### 2.2 Kotlin Flow 的协程基础

Flow 构建在协程之上：`emit()` 和 `collect()` 都是挂起函数，天然异步；生命周期绑定协程作用域，自动取消。

```kotlin
fun fetchUsers(): Flow<List<User>> = flow {
    emit(apiService.getUsers())
}.flowOn(Dispatchers.IO)

// ViewModel 中收集
viewModelScope.launch {
    fetchUsers()
        .catch { e -> _uiState.value = UiState.Error(e.message) }
        .collect { users -> _uiState.value = UiState.Success(users) }
}
```

### 2.3 StateFlow 与 SharedFlow

参见 [[2-Jetpack-LiveData与Flow]]。

| 特性 | StateFlow | SharedFlow |
|------|-----------|------------|
| **初始值** | ✅ 必须 | ❌ 无需 |
| **重放** | 最新 1 个值 | 可配置 `replay` |
| **去重** | ✅ 内置 `distinctUntilChanged` | ❌ |
| **用途** | UI 状态（替代 LiveData） | 一次性事件（Toast/导航） |

```kotlin
// StateFlow：UI 状态
private val _uiState = MutableStateFlow<UiState>(UiState.Loading)
val uiState: StateFlow<UiState> = _uiState.asStateFlow()

// SharedFlow：一次性事件
private val _events = MutableSharedFlow<UiEvent>()
val events: SharedFlow<UiEvent> = _events.asSharedFlow()
```

### 2.4 RxJava vs Flow 对比

| 维度 | RxJava 2/3 | Kotlin Flow |
|------|-----------|-------------|
| **语言** | Java / Kotlin | Kotlin only |
| **依赖** | 第三方库 | kotlinx-coroutines |
| **学习曲线** | 陡峭 | 较平缓 |
| **背压** | `Flowable` 显式 | 协程挂起天然支持 |
| **线程切换** | `subscribeOn`/`observeOn` | `flowOn`（仅上游） |
| **生命周期** | 手动 `dispose` | 协程作用域自动管理 |
| **冷流/热流** | `Observable`/`Subject` | `flow{}`/`StateFlow` |
| **错误处理** | `onErrorReturn` | `catch { }` |
| **Google 推荐** | ❌ | ✅ 官方推荐 |
| **包体积** | ~2MB | ~800KB |

## 三、实战

### 3.1 RxJava 经典场景

#### 网络请求 + 重试（Java）

```java
apiService.getUsers()
    .subscribeOn(Schedulers.io())
    .retryWhen(errors -> errors
        .zipWith(Observable.range(1, 3), (err, count) -> count)
        .flatMap(count -> Observable.timer((long) Math.pow(2, count), TimeUnit.SECONDS))
    )
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe(users -> adapter.submitList(users), error -> showError(error));
```

#### 搜索防抖（Java）

```java
RxTextView.textChanges(searchEditText)
    .debounce(400, TimeUnit.MILLISECONDS)
    .map(CharSequence::toString)
    .filter(text -> text.length() >= 2)
    .distinctUntilChanged()
    .switchMap(keyword ->
        apiService.search(keyword)
            .subscribeOn(Schedulers.io())
            .onErrorReturnItem(Collections.emptyList())
    )
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe(results -> showResults(results));
```

#### 多接口并行合并（Java）

```java
Observable.zip(
    apiService.getUser(userId).subscribeOn(Schedulers.io()),
    apiService.getOrders(userId).subscribeOn(Schedulers.io()),
    (user, orders) -> new UserDetailVO(user, orders)
)
.observeOn(AndroidSchedulers.mainThread())
.subscribe(vo -> renderDetail(vo), error -> showError(error));
```

### 3.2 Flow 在 MVVM 中的应用

**Repository → ViewModel → UI 完整链路（Kotlin）：**

```kotlin
// Repository
class UserRepository(private val api: ApiService, private val dao: UserDao) {
    fun observeUsers(): Flow<List<User>> = dao.observeAll()
    suspend fun refreshUsers() { dao.insertAll(api.getUsers()) }
}

// ViewModel
class UserListViewModel(private val repo: UserRepository) : ViewModel() {
    val uiState: StateFlow<UserListUiState> = repo.observeUsers()
        .map<List<User>, UserListUiState> { users ->
            if (users.isEmpty()) UserListUiState.Empty else UserListUiState.Success(users)
        }
        .catch { emit(UserListUiState.Error(it.message ?: "")) }
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), UserListUiState.Loading)

    fun refresh() { viewModelScope.launch { repo.refreshUsers() } }
}

// UI：安全收集
viewLifecycleOwner.lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { state -> renderState(state) }
    }
}
```

**Flow 搜索防抖（Kotlin）：**

```kotlin
class SearchViewModel : ViewModel() {
    private val queryFlow = MutableStateFlow("")

    val results: StateFlow<List<SearchResult>> = queryFlow
        .debounce(400).filter { it.length >= 2 }.distinctUntilChanged()
        .flatMapLatest { keyword ->
            flow { emit(apiService.search(keyword)) }.catch { emit(emptyList()) }
        }
        .flowOn(Dispatchers.IO)
        .stateIn(viewModelScope, SharingStarted.Lazily, emptyList())

    fun onQueryChanged(q: String) { queryFlow.value = q }
}
```

### 3.3 RxJava 迁移 Flow 策略

推荐**渐进式迁移**，使用 `kotlinx-coroutines-rx2/rx3` 桥接库：

```kotlin
// 桥接：RxJava ↔ Flow
val flow: Flow<Data> = rxObservable.asFlow()
val observable: Observable<Data> = flowOf(1, 2, 3).asObservable()
```

**迁移优先级：** ViewModel（最高）→ Repository（中）→ 数据源（最低）

**迁移对照表：**

| RxJava | Flow |
|--------|------|
| `Observable.create` | `flow { }` |
| `Observable.just(x)` | `flowOf(x)` |
| `subscribeOn(io())` | `flowOn(Dispatchers.IO)` |
| `observeOn(main())` | 在主线程协程中 collect |
| `flatMap` | `flatMapConcat` |
| `switchMap` | `flatMapLatest` |
| `CompositeDisposable` | `CoroutineScope` |
| `BehaviorSubject` | `MutableStateFlow` |
| `PublishSubject` | `MutableSharedFlow` |
| `dispose()` | `Job.cancel()` |

## 四、面试题

### Q1：flatMap、switchMap、concatMap 的区别？

**A：**

| 操作符 | 行为 | 顺序 | 场景 |
|--------|------|------|------|
| `flatMap` | 并发执行，交错发射 | ❌ 无序 | 并行请求不关心顺序 |
| `concatMap` | 串行执行 | ✅ 有序 | 保证顺序的批量操作 |
| `switchMap` | 新元素来时取消上一个 | ✅ 最新 | 搜索防抖 |

Flow 对应：`flatMapMerge` / `flatMapConcat` / `flatMapLatest`。

### Q2：flowOn 和 subscribeOn/observeOn 的区别？

**A：** RxJava 的 `subscribeOn` 指定上游线程，`observeOn` 指定下游线程，可多次调用 `observeOn`。Flow 的 `flowOn` **仅影响上游**，`collect` 始终在调用者协程上下文执行。

```kotlin
flow { emit(data) }
    .map { transform(it) }     // IO 线程
    .flowOn(Dispatchers.IO)     // 仅影响上面
    .collect { updateUI(it) }   // Main 线程（collect 所在协程）
```

### Q3：StateFlow 和 LiveData 的区别？

**A：** 参见 [[2-Jetpack-LiveData与Flow]]。

| 维度 | LiveData | StateFlow |
|------|----------|-----------|
| 初始值 | 可选 | 必须 |
| 生命周期感知 | ✅ 内置 | 需 `repeatOnLifecycle` |
| 操作符 | 有限 | Flow 全套 |
| 多平台 | Android only | KMP 支持 |
| 线程安全 | `postValue` 可能丢值 | `value` 线程安全 |

### Q4：Flow 如何解决背压？

**A：** 基于协程挂起机制：消费者处理中时，`emit()` 自动挂起等待。额外策略：`buffer()` 添加缓冲、`conflate()` 只保留最新、`collectLatest {}` 新值来时取消上次处理。

### Q5：RxJava 如何避免内存泄漏？

**A：** 订阅持有 Observer 引用 → Observer 持有 Activity → 泄漏。方案：
1. `CompositeDisposable` + `onDestroy` 中 `clear()`
2. ViewModel 中在 `onCleared()` 清理
3. Flow 配合协程作用域自动取消，无需手动管理

### Q6：stateIn 中 WhileSubscribed(5000) 的 5000 是什么？

**A：** 最后一个订阅者消失后等待 **5 秒**再停止上游 Flow。用于应对屏幕旋转等配置变更——旋转时 Fragment 短暂销毁再重建，5 秒内重新订阅不会重新触发数据加载。

## 五、实战与踩坑

### 5.1 RxJava 内存泄漏

**典型泄漏（Java）：**

```java
// ❌ Activity 销毁后订阅仍执行，lambda 捕获 this 导致泄漏
apiService.getUsers()
    .subscribeOn(Schedulers.io())
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe(users -> recyclerView.setAdapter(new UserAdapter(users)));
```

**正确做法（Java）：**

```java
private final CompositeDisposable disposables = new CompositeDisposable();

disposables.add(
    apiService.getUsers()
        .subscribeOn(Schedulers.io())
        .observeOn(AndroidSchedulers.mainThread())
        .subscribe(users -> render(users), error -> showError(error))
);

@Override protected void onDestroy() {
    super.onDestroy();
    disposables.clear();  // 取消所有订阅
}
```

### 5.2 Flow 的 repeatOnLifecycle

```kotlin
// ❌ 后台仍在收集，浪费资源
lifecycleScope.launch {
    viewModel.uiState.collect { updateUI(it) }
}

// ✅ 仅 STARTED 以上状态收集
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { updateUI(it) }
    }
}

// ✅ 多个 Flow 并行收集：必须用 launch 包裹
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        launch { viewModel.uiState.collect { updateUI(it) } }
        launch { viewModel.events.collect { handleEvent(it) } }
    }
}
```

> ⚠️ 串行 collect 多个 Flow 时，第二个永远不会执行，必须用 `launch` 包裹。

### 5.3 线程切换的坑

**RxJava（Java）：**

```java
// ❌ 多次 subscribeOn 只有第一次生效
Observable.create(emitter -> { })
    .subscribeOn(Schedulers.io())          // ✅ 生效
    .subscribeOn(Schedulers.computation()) // ❌ 无效

// ❌ observeOn 后做耗时操作会卡主线程
apiService.getData()
    .subscribeOn(Schedulers.io())
    .observeOn(AndroidSchedulers.mainThread())
    .map(data -> heavyTransform(data))  // ❌ 主线程耗时！
    .subscribe(result -> render(result));
// ✅ 先 map 再 observeOn
```

**Flow（Kotlin）：**

```kotlin
// ❌ flow 构建器中不能用 withContext（Context Preservation 规则）
flow {
    withContext(Dispatchers.IO) { emit(fetchData()) }  // 崩溃！
}
// ✅ 用 flowOn
flow { emit(fetchData()) }.flowOn(Dispatchers.IO)

// 注意：flowOn 只影响上游
flow { emit(data) }
    .map { transform(it) }     // IO 线程
    .flowOn(Dispatchers.IO)     // 仅影响上面
    .filter { it.isValid() }   // Main 线程
    .collect { updateUI(it) }  // Main 线程
```

### 5.4 踩坑速查表

| 坑 | 框架 | 解决方案 |
|----|------|---------|
| 忘记 dispose | RxJava | `CompositeDisposable` + `onDestroy` |
| 后台收集 | Flow | `repeatOnLifecycle` |
| 多次 subscribeOn | RxJava | 只写一次 |
| observeOn 后耗时操作 | RxJava | 耗时放 `observeOn` 之前 |
| flow 内 withContext | Flow | 用 `flowOn` |
| 串行 collect 多 Flow | Flow | 每个 `launch` 包裹 |
| SharedFlow 事件丢失 | Flow | 先订阅再发射或设 `replay` |
| StateFlow 相同值不触发 | Flow | 用 `data class` 或 `SharedFlow` |
