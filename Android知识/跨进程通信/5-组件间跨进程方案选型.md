# 组件间跨进程方案选型

> Android 多进程架构下，组件间通信方案的全面对比与实战选型指南。
> 前置知识：[[1-Binder机制原理]] [[2-AIDL使用与实践]]

---

## 一、核心概念：Android 中所有 IPC 方案总览

Android 基于 Linux 内核，进程间天然隔离，无法直接共享内存。系统提供了多种 IPC（Inter-Process Communication）机制，所有方案最终都依赖 [[1-Binder机制原理]] 或 Linux 原生机制实现。

### 1.1 Bundle

最简单的跨进程数据传递方式，依附于 `Intent`，通过 Binder 传输序列化数据。

````java
// 发送端
Intent intent = new Intent(this, RemoteActivity.class);
intent.putExtra("user_id", 1001);
intent.putExtra("user_name", "test_user");
startActivity(intent);

// 接收端
int userId = intent.getIntExtra("user_id", -1);
````
**限制**：Bundle 通过 Binder 传输，受 Binder 事务缓冲区限制（通常 **1MB**，所有进程共享），传递大数据会抛出 `TransactionTooLargeException`。

### 1.2 文件共享

多个进程通过读写同一个文件来交换数据，适合对实时性要求不高的场景。

````java
// 进程 A：写入数据
File file = new File(context.getFilesDir(), "shared_data.json");
FileWriter writer = new FileWriter(file);
writer.write(new Gson().toJson(config));
writer.close();

// 进程 B：读取数据
File file2 = new File(context.getFilesDir(), "shared_data.json");
String json = new String(Files.readAllBytes(file2.toPath()));
AppConfig config = new Gson().fromJson(json, AppConfig.class);
````
**问题**：并发读写无锁保护，可能读到不完整数据。Android 的 `SharedPreferences` 本质也是文件共享，**多进程下不安全**。

### 1.3 Messenger

基于 [[1-Binder机制原理]] 的轻量级 IPC 方案，底层使用 AIDL 实现，以 `Message` 为载体，串行处理请求。详见 [[3-Messenger]]。

````java
// 服务端 Service
public class MessengerService extends Service {
    private final Handler handler = new Handler(Looper.getMainLooper()) {
        @Override
        public void handleMessage(Message msg) {
            switch (msg.what) {
                case MSG_SAY_HELLO:
                    String data = msg.getData().getString("content");
                    Log.d("Server", "收到: " + data);
                    // 回复客户端
                    if (msg.replyTo != null) {
                        Message reply = Message.obtain(null, MSG_REPLY);
                        Bundle bundle = new Bundle();
                        bundle.putString("result", "已收到");
                        reply.setData(bundle);
                        try {
                            msg.replyTo.send(reply);
                        } catch (RemoteException e) {
                            e.printStackTrace();
                        }
                    }
                    break;
            }
        }
    };

    @Override
    public IBinder onBind(Intent intent) {
        return new Messenger(handler).getBinder();
    }
}

// 客户端绑定
private Messenger serverMessenger;
private final Messenger clientMessenger = new Messenger(new Handler(Looper.getMainLooper()) {
    @Override
    public void handleMessage(Message msg) {
        Log.d("Client", "服务端回复: " + msg.getData().getString("result"));
    }
});

private final ServiceConnection connection = new ServiceConnection() {
    @Override
    public void onServiceConnected(ComponentName name, IBinder service) {
        serverMessenger = new Messenger(service);
    }

    @Override
    public void onServiceDisconnected(ComponentName name) {
        serverMessenger = null;
    }
};

// 发送消息
public void sendMessage(String content) {
    Message msg = Message.obtain(null, MSG_SAY_HELLO);
    Bundle bundle = new Bundle();
    bundle.putString("content", content);
    msg.setData(bundle);
    msg.replyTo = clientMessenger;
    try {
        if (serverMessenger != null) {
            serverMessenger.send(msg);
        }
    } catch (RemoteException e) {
        e.printStackTrace();
    }
}
````
**特点**：串行处理，无法并发；只能传递 `Message` 支持的数据类型。

### 1.4 AIDL

Android 接口定义语言，支持自定义接口、并发调用，是最灵活的 IPC 方案。详见 [[2-AIDL使用与实践]]。

````java
// IBookManager.aidl
interface IBookManager {
    List<Book> getBookList();
    void addBook(in Book book);
    void registerCallback(IOnBookAddedListener listener);
}

// 服务端实现
public class BookService extends Service {
    private final CopyOnWriteArrayList<Book> bookList = new CopyOnWriteArrayList<>();
    private final RemoteCallbackList<IOnBookAddedListener> callbacks = new RemoteCallbackList<>();

    private final IBookManager.Stub binder = new IBookManager.Stub() {
        @Override
        public List<Book> getBookList() {
            return new ArrayList<>(bookList);
        }

        @Override
        public void addBook(Book book) {
            bookList.add(book);
            notifyBookAdded(book);
        }

        @Override
        public void registerCallback(IOnBookAddedListener listener) {
            callbacks.register(listener);
        }
    };

    private void notifyBookAdded(Book book) {
        int count = callbacks.beginBroadcast();
        for (int i = 0; i < count; i++) {
            try {
                callbacks.getBroadcastItem(i).onBookAdded(book);
            } catch (RemoteException e) {
                e.printStackTrace();
            }
        }
        callbacks.finishBroadcast();
    }

    @Override
    public IBinder onBind(Intent intent) {
        return binder;
    }
}
````
**注意**：AIDL 调用默认在 Binder 线程池执行，服务端需要处理线程安全；客户端调用是同步阻塞的，耗时操作不要在主线程调用。

### 1.5 ContentProvider

四大组件之一，专为跨进程数据共享设计，底层基于 [[1-Binder机制原理]]，提供标准 CRUD 接口。详见 [[4-ContentProvider跨进程]]。

````java
// 自定义 Provider
public class UserProvider extends ContentProvider {
    private SQLiteDatabase db;

    @Override
    public boolean onCreate() {
        db = new UserDbHelper(getContext()).getWritableDatabase();
        return true;
    }

    @Override
    public Cursor query(Uri uri, String[] projection,
                        String selection, String[] selectionArgs,
                        String sortOrder) {
        switch (uriMatcher.match(uri)) {
            case CODE_USER:
                return db.query("user", projection, selection, selectionArgs, null, null, sortOrder);
            default:
                return null;
        }
    }

    @Override
    public Uri insert(Uri uri, ContentValues values) {
        long id = db.insert("user", null, values);
        if (getContext() != null) {
            getContext().getContentResolver().notifyChange(uri, null);
        }
        return ContentUris.withAppendedId(uri, id);
    }

    // update / delete 省略，模式相同
    @Override
    public int update(Uri uri, ContentValues values, String s, String[] sa) { return 0; }
    @Override
    public int delete(Uri uri, String s, String[] sa) { return 0; }
    @Override
    public String getType(Uri uri) { return null; }

    private static final int CODE_USER = 1;
    private static final UriMatcher uriMatcher = new UriMatcher(UriMatcher.NO_MATCH);
    static {
        uriMatcher.addURI("com.example.user.provider", "user", CODE_USER);
    }
}
````
**特点**：天然支持跨 App 访问（需声明权限），适合结构化数据共享。

### 1.6 Socket

基于 TCP/UDP 的通用网络通信方案，不依赖 Binder，可跨设备通信。

````java
// 服务端（在子线程运行）
public void startServer() throws IOException {
    LocalServerSocket serverSocket = new LocalServerSocket("my_local_socket");
    while (true) {
        LocalSocket client = serverSocket.accept();
        BufferedReader reader = new BufferedReader(new InputStreamReader(client.getInputStream()));
        String input = reader.readLine();
        client.getOutputStream().write(("ACK:" + input + "\n").getBytes());
        client.close();
    }
}

// 客户端
public String sendToServer(String message) throws IOException {
    LocalSocket socket = new LocalSocket();
    socket.connect(new LocalSocketAddress("my_local_socket"));
    socket.getOutputStream().write((message + "\n").getBytes());
    BufferedReader reader = new BufferedReader(new InputStreamReader(socket.getInputStream()));
    String response = reader.readLine();
    socket.close();
    return response;
}
````
**特点**：灵活度最高，但开发成本大，需自行处理协议、粘包、断线重连等问题。

### 1.7 Binder 直传（Transact）

绕过 AIDL 封装，直接操作 `IBinder.transact()` 进行通信，适合极致性能优化场景。

````java
// 直接通过 Binder transact 通信
public String queryViaTransact(IBinder binder, String key) throws RemoteException {
    Parcel data = Parcel.obtain();
    Parcel reply = Parcel.obtain();
    try {
        data.writeInterfaceToken("com.example.IDirectService");
        data.writeString(key);
        binder.transact(TRANSACTION_QUERY, data, reply, 0);
        reply.readException();
        return reply.readString();
    } finally {
        data.recycle();
        reply.recycle();
    }
}
````
**特点**：性能最优，但可读性差、维护成本高，一般框架内部使用。

## 二、方案对比

| 方案 | 底层机制 | 性能 | 易用性 | 安全性 | 数据大小限制 | 是否支持并发 | 适用场景 |
|------|---------|------|--------|--------|-------------|-------------|---------|
| **Bundle** | Binder | ★★★★ | ★★★★★ | ★★★ | ~1MB | — | 四大组件间简单数据传递 |
| **文件共享** | 文件系统 | ★★ | ★★★★ | ★★ | 无限制 | ✗（需加锁） | 低频、大数据、非实时场景 |
| **Messenger** | Binder (AIDL) | ★★★ | ★★★★ | ★★★ | ~1MB | ✗（串行） | 低频、简单的跨进程消息 |
| **AIDL** | Binder | ★★★★ | ★★★ | ★★★★ | ~1MB | ✓ | 高频、复杂接口、需要回调 |
| **ContentProvider** | Binder | ★★★ | ★★★★ | ★★★★★ | 支持大数据流 | ✓ | 结构化数据共享、跨 App |
| **Socket** | TCP/UDP | ★★ | ★★ | ★★ | 无限制 | ✓ | 跨设备、自定义协议 |
| **Binder 直传** | Binder | ★★★★★ | ★ | ★★★ | ~1MB | ✓ | 框架内部、极致性能 |

**选型决策树**：

````
需要跨进程通信？
├── 只是传递简单数据 → Bundle
├── 需要共享结构化数据
│   ├── 同 App 内 → AIDL / Messenger
│   └── 跨 App → ContentProvider
├── 需要复杂接口 + 回调 → AIDL
├── 需要传输大文件 → 文件共享 / ContentProvider（openFile）
├── 需要跨设备 → Socket
└── 框架底层极致优化 → Binder 直传
````

## 三、实战选型：不同业务场景的推荐方案

### 3.1 推送服务（独立进程保活）

推送 SDK 通常运行在独立进程中，需要与主进程双向通信。

**推荐方案**：AIDL + DeathRecipient

````java
// 主进程绑定推送服务，监听进程死亡
private final IBinder.DeathRecipient deathRecipient = () -> {
    Log.w("Push", "推送进程死亡，尝试重连");
    reconnectPushService();
};

private final ServiceConnection connection = new ServiceConnection() {
    @Override
    public void onServiceConnected(ComponentName name, IBinder service) {
        pushService = IPushService.Stub.asInterface(service);
        try {
            service.linkToDeath(deathRecipient, 0);
        } catch (RemoteException e) {
            e.printStackTrace();
        }
    }

    @Override
    public void onServiceDisconnected(ComponentName name) {
        pushService = null;
    }
};

// 推送进程收到消息后通知主进程
// AIDL 回调接口
interface IOnMessageListener {
    void onMessageReceived(in PushMessage message);
}
````
**理由**：推送服务需要高频双向通信 + 回调通知 + 进程死亡监听，AIDL 是最合适的方案。

### 3.2 多进程 WebView

WebView 内存泄漏严重，常放在独立进程中隔离风险。

**推荐方案**：AIDL 或 Messenger

````java
// WebView 进程中的 Service
public class WebViewService extends Service {
    private final IWebViewBridge.Stub binder = new IWebViewBridge.Stub() {
        @Override
        public String evaluateJs(String script) {
            // 在 WebView 进程执行 JS
            return webView.evaluateJavascriptSync(script);
        }

        @Override
        public void loadUrl(String url) {
            handler.post(() -> webView.loadUrl(url));
        }
    };

    @Override
    public IBinder onBind(Intent intent) {
        return binder;
    }
}
````
**理由**：主进程需要控制 WebView 行为（加载 URL、执行 JS），WebView 进程需要回调事件（页面加载完成、JS 调用 Native），双向通信用 AIDL 最清晰。

### 3.3 插件化框架

插件运行在宿主进程或独立进程中，需要访问宿主的服务和数据。

**推荐方案**：Binder 直传 + 自定义 ServiceManager

````java
// 宿主维护一个 ServiceManager，插件通过它获取服务 Binder
public class PluginServiceManager {
    private static final ConcurrentHashMap<String, IBinder> services = new ConcurrentHashMap<>();

    public static void registerService(String name, IBinder binder) {
        services.put(name, binder);
    }

    public static IBinder getService(String name) {
        return services.get(name);
    }
}

// 插件获取宿主服务
IUserService userService = IUserService.Stub.asInterface(
    PluginServiceManager.getService("user_service")
);
````
**理由**：插件化需要灵活的服务发现机制，自定义 ServiceManager 模式（类似系统 `ServiceManager`）最适合。

### 3.4 跨 App 数据共享

不同 App 之间共享用户数据、配置信息等。

**推荐方案**：ContentProvider，详见 [[4-ContentProvider跨进程]]

````xml
<!-- 提供方 AndroidManifest.xml -->
<provider
    android:name=".UserProvider"
    android:authorities="com.example.user.provider"
    android:exported="true"
    android:permission="com.example.permission.READ_USER"
    android:readPermission="com.example.permission.READ_USER"
    android:writePermission="com.example.permission.WRITE_USER" />
````
````java
// 消费方查询
Cursor cursor = contentResolver.query(
    Uri.parse("content://com.example.user.provider/user"),
    new String[]{"id", "name"}, null, null, null
);
if (cursor != null) {
    try {
        while (cursor.moveToNext()) {
            String name = cursor.getString(cursor.getColumnIndexOrThrow("name"));
        }
    } finally {
        cursor.close();
    }
}
````
**理由**：ContentProvider 是 Android 官方推荐的跨 App 数据共享方案，有完善的权限控制机制。

## 四、面试题

### Q1：Android 中有哪些 IPC 方式？各自的优缺点是什么？

**A**：Android 提供了 7 种主要 IPC 方式：

1. **Bundle**：最简单，依附 Intent，受 1MB 限制，适合简单数据传递
2. **文件共享**：无大小限制，但并发不安全，实时性差
3. **Messenger**：基于 AIDL 的轻量封装，串行处理，适合低频简单消息
4. **AIDL**：最灵活，支持并发和自定义接口，但开发复杂度较高
5. **ContentProvider**：标准 CRUD 接口，天然支持跨 App，权限控制完善
6. **Socket**：最灵活，可跨设备，但开发成本最高
7. **Binder 直传**：性能最优，但可读性差

所有基于 Binder 的方案（Bundle/Messenger/AIDL/ContentProvider）都受 Binder 事务缓冲区限制，底层原理参考 [[1-Binder机制原理]]。

### Q2：Messenger 和 AIDL 有什么区别？什么时候用 Messenger？

**A**：

| 对比项 | Messenger | AIDL |
|--------|-----------|------|
| 并发处理 | 串行（Handler 队列） | 并发（Binder 线程池） |
| 接口定义 | 固定 Message 格式 | 自定义接口方法 |
| 数据类型 | Bundle 支持的类型 | Parcelable 自定义类型 |
| 开发复杂度 | 低 | 中高 |
| 回调机制 | replyTo | 回调接口 |

**选择 Messenger 的场景**：不需要并发、接口简单、消息频率低。例如简单的进程间状态同步。详见 [[3-Messenger]]。

### Q3：为什么 SharedPreferences 在多进程下不安全？如何解决？

**A**：SharedPreferences 在内存中维护了一份缓存，每个进程各自持有独立的缓存副本。当进程 A 写入后，进程 B 的内存缓存不会更新，导致读到过期数据。即使使用 `MODE_MULTI_PROCESS`（已废弃），也只是在 `getSharedPreferences()` 时重新读取文件，无法保证实时一致性。

**解决方案**：
- **MMKV**（推荐）：基于 mmap，天然支持多进程
- **ContentProvider** 封装：通过 Binder 保证跨进程一致性
- **DataStore**：Jetpack 方案，但多进程支持需要额外配置

### Q4：Binder 传输数据的大小限制是多少？超过了怎么办？

**A**：Binder 事务缓冲区大小为 **1MB**（`/proc/sys/kernel/binder/transaction_size`），且是所有正在进行的事务共享的，实际可用通常不到 500KB。

超过限制的解决方案：
1. **大文件**：使用文件共享 + 传递文件路径
2. **大数据集**：ContentProvider 的 `query()` 返回 `Cursor`（底层使用匿名共享内存 Ashmem）
3. **大图片/流数据**：ContentProvider 的 `openFile()` 返回 `ParcelFileDescriptor`
4. **分片传输**：将数据拆分为多次 Binder 调用

### Q5：多进程下 Application 会创建几次？如何区分进程？

**A**：每个进程启动时都会创建一个新的 `Application` 实例。如果 App 有 N 个进程，`Application.onCreate()` 会执行 N 次。

````java
public class MyApp extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        String processName = getProcessName();
        if (processName.equals(getPackageName())) {
            initMainProcess();
        } else if (processName.endsWith(":push")) {
            initPushProcess();
        } else if (processName.endsWith(":web")) {
            initWebProcess();
        } else {
            /* 其他进程最小化初始化 */
        }
    }

    private String getProcessName() {
        if (Build.VERSION.SDK_INT >= 28) {
            return Application.getProcessName();
        } else {
            int pid = android.os.Process.myPid();
            ActivityManager am = (ActivityManager) getSystemService(ACTIVITY_SERVICE);
            List<ActivityManager.RunningAppProcessInfo> processes = am.getRunningAppProcesses();
            if (processes != null) {
                for (ActivityManager.RunningAppProcessInfo info : processes) {
                    if (info.pid == pid) return info.processName;
                }
            }
            return "";
        }
    }
}
````
### Q6：AIDL 中的 `in`、`out`、`inout` 关键字分别是什么含义？

**A**：这三个关键字定义了数据的传输方向，影响 Parcel 的序列化行为：

- **`in`**（默认）：数据从客户端传到服务端，服务端对参数的修改不会回传
- **`out`**：客户端传空对象，服务端填充后回传给客户端
- **`inout`**：双向传输，客户端传入数据，服务端修改后回传

````
// AIDL 示例
void updateBook(inout Book book);  // 客户端传入，服务端修改后客户端能看到变化
void fillBook(out Book book);      // 服务端填充数据返回给客户端
void addBook(in Book book);        // 客户端传入，服务端只读
````
**性能影响**：`in` 只序列化一次，`out` 只反序列化一次，`inout` 两次都做。优先使用 `in` 以减少开销。详见 [[2-AIDL使用与实践]]。

## 五、实战与踩坑

### 5.1 多进程导致的 Application 多次创建

**问题**：在 `AndroidManifest.xml` 中声明 `android:process` 后，每个进程都会重新创建 `Application`，导致：
- 第三方 SDK 重复初始化（崩溃或资源浪费）
- 单例对象在不同进程中是不同实例
- 静态变量在不同进程中互不影响

**解决方案**：按进程名分发初始化逻辑。

````java
public class MyApp extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        String processName = getProcessName();
        // 所有进程都需要的最小初始化
        initCrashReport();

        if (processName.equals(getPackageName())) {
            // 仅主进程初始化
            initNetworkSdk();
            initImageLoader();
            initRouter();
        }
        // 其他进程不做多余初始化，加快启动速度
    }
}
````
**关键原则**：非主进程只做最小化初始化，避免不必要的内存和时间开销。

### 5.2 SharedPreferences 多进程不安全

**问题复现**：

````java
// 进程 A 写入
getSharedPreferences("config", MODE_PRIVATE).edit()
    .putString("token", "abc123")
    .apply();

// 进程 B 立即读取 —— 大概率读到旧值或空值！
String token = getSharedPreferences("config", MODE_PRIVATE)
    .getString("token", null);
````
**根因分析**：
1. 每个进程在内存中维护独立的 SP 缓存 Map
2. `apply()` 是异步写磁盘，进程 B 可能在写入完成前就读取
3. 即使用 `commit()` 同步写入，进程 B 的内存缓存也不会自动刷新
4. `MODE_MULTI_PROCESS` 已在 API 23 废弃，且只在 `getSharedPreferences()` 时重新加载文件

### 5.3 MMKV 替代方案

[MMKV](https://github.com/Tencent/MMKV) 是腾讯开源的高性能 KV 存储框架，基于 `mmap` 内存映射，天然支持多进程。

**接入**：

````java
// build.gradle
dependencies {
    implementation("com.tencent:mmkv:1.3.4");
}

// Application 初始化
MMKV.initialize(this);
````
**多进程使用**：

````java
// 进程 A 写入
MMKV kv = MMKV.mmkvWithID("app_config", MMKV.MULTI_PROCESS_MODE);
kv.encode("token", "abc123");
kv.encode("login_time", System.currentTimeMillis());

// 进程 B 读取 —— 实时可见！
MMKV kv2 = MMKV.mmkvWithID("app_config", MMKV.MULTI_PROCESS_MODE);
String token = kv2.decodeString("token");  // "abc123"
````
**MMKV vs SharedPreferences 对比**：

| 对比项 | SharedPreferences | MMKV |
|--------|------------------|------|
| 多进程安全 | ✗ | ✓（mmap） |
| 写入性能 | 慢（文件 I/O） | 快 100x（内存映射） |
| 数据格式 | XML | protobuf |
| 增量更新 | ✗（全量写入） | ✓（append） |
| ANR 风险 | 有（apply 在 Activity stop 时同步等待） | 无 |

**SP 迁移到 MMKV**：

````java
// 一行代码迁移
SharedPreferences oldSp = getSharedPreferences("config", MODE_PRIVATE);
MMKV kv = MMKV.mmkvWithID("config", MMKV.MULTI_PROCESS_MODE);
kv.importFromSharedPreferences(oldSp);
oldSp.edit().clear().apply();  // 清理旧数据
````
### 5.4 其他常见坑

**坑 1：跨进程回调用普通接口导致 ClassCastException**

````java
// ❌ 错误：直接传递普通接口对象
service.setCallback(new MyCallback() {
    @Override
    public void onResult(String data) { }
});

// ✓ 正确：使用 AIDL 定义回调接口 + RemoteCallbackList 管理
RemoteCallbackList<IMyCallback> callbacks = new RemoteCallbackList<>();
````
**坑 2：忘记处理 Binder 死亡导致调用崩溃**

````java
// 必须设置 DeathRecipient 或 try-catch RemoteException
try {
    if (remoteService != null) {
        remoteService.doSomething();
    }
} catch (RemoteException e) {
    Log.e("IPC", "远程服务已死亡", e);
    reconnect();
}
````
**坑 3：AIDL 方法在 Binder 线程调用，直接操作 UI 崩溃**

````java
// 服务端 AIDL 实现中，回到主线程操作 UI
@Override
public void updateUI(String text) {
    new Handler(Looper.getMainLooper()).post(() -> {
        textView.setText(text);
    });
}
````
