# 插件化与热修复

## 一、ClassLoader 机制（基础前置知识）

Android 的类加载是理解插件化和热修复的基础。

### 1.1 Java 的 ClassLoader

````
Bootstrap ClassLoader（加载 JDK 核心类）
    ↓
Extension ClassLoader（加载扩展类）
    ↓
Application ClassLoader（加载应用类）
````
**双亲委派模型：** 加载一个类时，先委托父 ClassLoader 加载，父加载不了再自己加载。目的是防止核心类被篡改。

### 1.2 Android 的 ClassLoader

````
BootClassLoader（加载 Android 框架类）
    ↓
PathClassLoader（加载已安装的 APK，dex 路径固定）
    ↓
DexClassLoader（可以加载任意路径的 dex/apk/jar）
````
| ClassLoader | 用途 |
|-------------|------|
| `PathClassLoader` | 加载已安装的 App 的 dex，系统默认使用 |
| `DexClassLoader` | 可以加载外部的 dex/apk 文件，插件化的核心 |
| `InMemoryDexClassLoader` | 从内存中加载 dex（API 26+） |

````java
// 加载外部 dex 文件
DexClassLoader classLoader = new DexClassLoader(
    "/sdcard/plugin.apk",          // dex 路径
    context.getCacheDir().getPath(), // 优化后的 dex 输出目录
    null,                           // native 库路径
    context.getClassLoader()        // 父 ClassLoader
);

// 反射调用插件中的类
Class<?> clazz = classLoader.loadClass("com.plugin.MyClass");
Method method = clazz.getMethod("doSomething");
Object instance = clazz.newInstance();
method.invoke(instance);
````
### 1.3 dexElements

PathClassLoader 和 DexClassLoader 都继承自 `BaseDexClassLoader`，内部有一个 `DexPathList`，其中的 `dexElements` 数组存储了所有 dex 文件。

````
BaseDexClassLoader
    └── DexPathList pathList
            └── Element[] dexElements
                    ├── dex1 (classes.dex)
                    ├── dex2 (classes2.dex)
                    └── dex3 (plugin.dex)  ← 可以动态插入
````
**类查找过程：** 遍历 dexElements 数组，在每个 dex 中查找目标类，找到就返回。**先找到的优先**——这就是热修复的原理。

## 二、插件化

### 2.1 是什么？

插件化是把 App 的部分功能模块打包成独立的 APK（插件），在运行时动态加载和执行，不需要安装。

**为什么需要插件化？**
- 减小主包体积
- 动态下发功能模块
- 多团队并行开发
- 绕过 65536 方法数限制（现在有 MultiDex，这个理由不太成立了）

### 2.2 核心问题

插件化需要解决三个核心问题：

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| 类加载 | 怎么加载插件中的类？ | DexClassLoader 加载插件 APK |
| 资源加载 | 怎么加载插件中的资源？ | 创建新的 AssetManager，addAssetPath 加载插件资源 |
| 组件生命周期 | 插件中的 Activity 没有在 Manifest 注册，怎么启动？ | Hook AMS / 占坑 Activity / 动态代理 |

### 2.3 类加载方案

````java
// 方案一：创建独立的 DexClassLoader
DexClassLoader pluginClassLoader = new DexClassLoader(
    pluginApkPath, optimizedDir, null, hostClassLoader);
Class<?> clazz = pluginClassLoader.loadClass("com.plugin.PluginActivity");

// 方案二：合并 dexElements（把插件的 dex 插入宿主的 dexElements）
public static void loadPlugin(Context context, String pluginPath) throws Exception {
    ClassLoader hostClassLoader = context.getClassLoader();
    Object hostPathList = getField(hostClassLoader, "pathList");
    Object[] hostDexElements = (Object[]) getField(hostPathList, "dexElements");

    // 创建插件的 ClassLoader
    DexClassLoader pluginClassLoader = new DexClassLoader(
        pluginPath, context.getCacheDir().getPath(), null, hostClassLoader);
    Object pluginPathList = getField(pluginClassLoader, "pathList");
    Object[] pluginDexElements = (Object[]) getField(pluginPathList, "dexElements");

    // 合并数组
    Object[] newElements = (Object[]) Array.newInstance(
        hostDexElements.getClass().getComponentType(),
        hostDexElements.length + pluginDexElements.length);
    System.arraycopy(hostDexElements, 0, newElements, 0, hostDexElements.length);
    System.arraycopy(pluginDexElements, 0, newElements, hostDexElements.length, pluginDexElements.length);

    // 替换宿主的 dexElements
    setField(hostPathList, "dexElements", newElements);
}
````

### 2.4 资源加载方案

````java
public static Resources loadPluginResources(Context context, String pluginPath) throws Exception {
    AssetManager assetManager = AssetManager.class.newInstance();
    // 反射调用 addAssetPath 加载插件资源
    Method method = AssetManager.class.getMethod("addAssetPath", String.class);
    method.invoke(assetManager, pluginPath);

    return new Resources(assetManager, context.getResources().getDisplayMetrics(),
        context.getResources().getConfiguration());
}
````
### 2.5 Activity 启动方案（Hook AMS）

插件中的 Activity 没有在 Manifest 中注册，直接启动会报 ActivityNotFoundException。

**占坑方案：**
1. 在宿主 Manifest 中预先注册几个"占坑" Activity
2. 启动插件 Activity 时，先把 Intent 中的目标替换为占坑 Activity（骗过 AMS 检查）
3. AMS 检查通过后，在 ActivityThread 创建 Activity 之前，把目标换回插件 Activity

````
startActivity(PluginActivity)
    ↓
Hook: 替换为 StubActivity（已注册）
    ↓
AMS 检查通过
    ↓
Hook: 替换回 PluginActivity
    ↓
ActivityThread 创建 PluginActivity 实例
````
### 2.6 主流插件化框架

| 框架 | 公司 | 特点 |
|------|------|------|
| RePlugin | 360 | 全面 Hook，兼容性好 |
| VirtualApk | 滴滴 | 支持四大组件，侵入性低 |
| Shadow | 腾讯 | 零反射，通过代理实现，兼容性最好 |
| DroidPlugin | 360 | 免安装运行第三方 APK |

## 三、热修复

### 3.1 是什么？

热修复是在不重新发版的情况下，动态修复线上 Bug。用户不需要重新下载安装 APK。

### 3.2 三种技术路线

| 路线 | 原理 | 代表框架 | 即时生效 | 修复范围 |
|------|------|---------|---------|---------|
| 类替换 | 修改 dexElements 顺序，让修复的类优先加载 | Tinker、QZone | ❌ 需要重启 | 类级别 |
| 底层替换 | 修改 ArtMethod 结构体，替换方法实现 | AndFix、Sophix | ✅ 即时生效 | 方法级别 |
| Instant Run | 在每个方法前插入判断逻辑，运行时替换 | Robust | ✅ 即时生效 | 方法级别 |

### 3.3 类替换方案（Tinker 原理）

利用 ClassLoader 的 dexElements 查找顺序：**先找到的优先**。

````
修复前的 dexElements：
[classes.dex(有Bug的类)] → [classes2.dex]

修复后的 dexElements：
[patch.dex(修复后的类)] → [classes.dex(有Bug的类)] → [classes2.dex]
                ↑
        插入到最前面，优先被加载
````
````java
// 简化版热修复原理
public static void applyPatch(Context context, String patchDexPath) throws Exception {
    ClassLoader classLoader = context.getClassLoader();
    Object pathList = getField(classLoader, "pathList");
    Object[] dexElements = (Object[]) getField(pathList, "dexElements");

    // 加载补丁 dex
    DexClassLoader patchClassLoader = new DexClassLoader(
        patchDexPath, context.getCacheDir().getPath(), null, classLoader);
    Object patchPathList = getField(patchClassLoader, "pathList");
    Object[] patchElements = (Object[]) getField(patchPathList, "dexElements");

    // 把补丁 dex 插入到最前面
    Object[] newElements = (Object[]) Array.newInstance(
        dexElements.getClass().getComponentType(),
        patchElements.length + dexElements.length);
    System.arraycopy(patchElements, 0, newElements, 0, patchElements.length);  // 补丁在前
    System.arraycopy(dexElements, 0, newElements, patchElements.length, dexElements.length);

    setField(pathList, "dexElements", newElements);
}
````
**为什么需要重启？** 因为类一旦被加载到内存中就不会再次加载（ClassLoader 的缓存机制）。只有重启 App 后，类才会重新从 dexElements 中查找，这时补丁 dex 在前面，修复后的类就会被优先加载。

### 3.4 底层替换方案（AndFix 原理）

直接在 Native 层修改方法的 ArtMethod 结构体，把有 Bug 的方法指向修复后的方法。

````c
// Native 层（简化）
void replaceMethod(JNIEnv* env, jobject bugMethod, jobject fixMethod) {
    // 获取 ArtMethod 指针
    ArtMethod* bug = getArtMethod(env, bugMethod);
    ArtMethod* fix = getArtMethod(env, fixMethod);

    // 直接替换方法的入口点
    bug->entry_point_ = fix->entry_point_;
    bug->dex_code_item_offset_ = fix->dex_code_item_offset_;
    // ... 替换其他字段
}
````
**优点：** 即时生效，不需要重启。
**缺点：** ArtMethod 结构体在不同 Android 版本中不同，兼容性差。不能增减字段、不能修改类结构。

### 3.5 Instant Run 方案（Robust 原理）

在编译时给每个方法插入一段判断逻辑：

````java
// 编译时插桩后的代码
public class BugClass {
    public static ChangeQuickRedirect changeQuickRedirect;

    public String bugMethod() {
        if (changeQuickRedirect != null) {
            // 有补丁，执行补丁逻辑
            return (String) changeQuickRedirect.accessDispatch("bugMethod", new Object[]{});
        }
        // 原始逻辑
        return "bug result";
    }
}

// 下发补丁后，设置 changeQuickRedirect
BugClass.changeQuickRedirect = new PatchProxy();
````
**优点：** 即时生效，兼容性好（不依赖底层实现）。
**缺点：** 每个方法都插入了判断逻辑，增加包体积和方法数。

### 3.6 资源修复

资源修复的核心是替换 AssetManager 中的资源路径。

````java
// 简化原理
public static void patchResources(Context context, String patchResPath) throws Exception {
    AssetManager assetManager = AssetManager.class.newInstance();
    Method addAssetPath = AssetManager.class.getMethod("addAssetPath", String.class);
    addAssetPath.invoke(assetManager, patchResPath);

    // 替换所有 Resources 对象中的 AssetManager
    Resources resources = context.getResources();
    Field field = Resources.class.getDeclaredField("mAssets");
    field.setAccessible(true);
    field.set(resources, assetManager);
}
````
### 3.7 主流热修复框架对比

| 框架 | 公司 | 类修复 | 资源修复 | SO 修复 | 即时生效 | 原理 |
|------|------|--------|---------|---------|---------|------|
| Tinker | 微信 | ✅ | ✅ | ✅ | ❌ 重启 | dex 差量合并 |
| Sophix | 阿里 | ✅ | ✅ | ✅ | ✅ 部分 | 底层替换 + 类替换 |
| Robust | 美团 | ✅ | ❌ | ❌ | ✅ | Instant Run |
| AndFix | 阿里（已停维） | ✅ | ❌ | ❌ | ✅ | ArtMethod 替换 |

## 四、面试题库

### Q1：Android 的 ClassLoader 有哪些？和 Java 的有什么区别？

> Android 主要有 PathClassLoader 和 DexClassLoader，都继承自 BaseDexClassLoader。PathClassLoader 用于加载已安装 App 的 dex，DexClassLoader 可以加载任意路径的 dex/apk/jar 文件。和 Java 的区别是 Android 加载的是 dex 格式而不是 class 格式，底层通过 DexPathList 的 dexElements 数组管理多个 dex 文件。类查找时遍历 dexElements，在每个 dex 中查找，先找到的优先返回。这个机制是热修复和插件化的基础。

### Q2：插件化的核心原理是什么？需要解决哪些问题？

> 插件化的核心是在运行时动态加载未安装的 APK 中的代码和资源。需要解决三个问题：类加载——用 DexClassLoader 加载插件 APK 中的 dex；资源加载——反射创建 AssetManager 并 addAssetPath 加载插件资源；组件生命周期——插件中的四大组件没有在宿主 Manifest 中注册，需要通过 Hook AMS 或占坑 Activity 的方式绕过系统检查。其中组件生命周期是最复杂的，不同框架的实现方式差异很大，腾讯的 Shadow 通过代理模式避免了反射和 Hook，兼容性最好。

### Q3：热修复的几种方案？各有什么优缺点？

> 三种主流方案：一是类替换（Tinker），把补丁 dex 插入到 dexElements 数组最前面，利用类加载的优先级覆盖有 Bug 的类，优点是修复范围广可以修改类结构，缺点是需要重启才能生效。二是底层替换（AndFix/Sophix），在 Native 层直接修改 ArtMethod 结构体替换方法实现，优点是即时生效，缺点是不同 Android 版本的 ArtMethod 结构不同兼容性差，且不能增减字段。三是 Instant Run 方案（Robust），编译时在每个方法前插入判断逻辑，运行时通过设置代理对象切换到补丁方法，优点是即时生效且兼容性好，缺点是增加包体积。

### Q4：Tinker 的原理？为什么需要重启？

> Tinker 采用 dex 差量合并方案。服务端对比新旧 APK 生成差量补丁，客户端下载后在后台将补丁和原 dex 合并生成新的 dex 文件，然后把新 dex 插入到 dexElements 数组最前面。需要重启的原因是 Java 的 ClassLoader 有缓存机制，一个类一旦被加载到内存中就不会再次加载。只有重启 App 后类才会重新从 dexElements 中查找，这时修复后的类在前面会被优先加载。Tinker 还支持资源修复和 SO 修复，是目前功能最全面的热修复框架。

### Q5：双亲委派模型是什么？Android 中有什么变化？

> 双亲委派是指 ClassLoader 加载类时先委托父 ClassLoader 加载，父加载不了再自己加载。目的是保证核心类（如 java.lang.String）不会被篡改。Android 中基本遵循双亲委派，但 DexClassLoader 可以打破它——通过创建独立的 DexClassLoader 加载插件，插件中的类不会委托给宿主的 ClassLoader，实现了类的隔离。不过在热修复场景中，我们通常是合并 dexElements 而不是创建独立的 ClassLoader，这样补丁类和宿主类在同一个 ClassLoader 中，通过加载顺序实现替换。

### Q6：插件化中 Activity 的生命周期怎么管理？

> 插件 Activity 没有在 Manifest 中注册，直接启动会被 AMS 拒绝。主流方案是占坑：在宿主 Manifest 中预注册几个 StubActivity，启动插件 Activity 时先 Hook startActivity 把 Intent 中的目标替换为 StubActivity 骗过 AMS 检查，然后在 ActivityThread 的 Handler 中拦截 LAUNCH_ACTIVITY 消息，把目标换回插件 Activity。腾讯的 Shadow 用了不同的思路——插件 Activity 不继承系统 Activity，而是继承 Shadow 定义的 ShadowActivity，由宿主的代理 Activity 转发所有生命周期回调，完全避免了 Hook 系统 API。

### Q7：热修复能修复哪些问题？有什么限制？

> 类替换方案可以修复 Java/Kotlin 代码的 Bug、修改类结构（加字段加方法）、修复资源文件、替换 SO 库。限制是不能修改 Manifest（因为 Manifest 在安装时就解析了）、不能新增四大组件、不能修改 Application 类（它在补丁加载之前就已经初始化了）。底层替换方案只能修复方法级别的 Bug，不能增减字段和方法。另外 Google Play 禁止热修复（违反开发者政策），国内应用市场允许。

### Q8：为什么现在插件化用得越来越少了？

> 几个原因：一是 Android 系统对非公开 API 的限制越来越严格（Android 9 开始限制反射调用隐藏 API），插件化大量依赖反射和 Hook，兼容性维护成本很高。二是 App Bundle 和 Dynamic Feature Module 提供了官方的动态下发方案。三是组件化架构已经能满足大部分多团队协作的需求。四是 MultiDex 解决了方法数限制问题。现在插件化主要在超级 App（如微信、支付宝）中使用，普通 App 更推荐用组件化 + 动态下发的方案。

### Q9：资源修复的原理？

> 资源修复的核心是替换 AssetManager。Android 的资源加载通过 AssetManager 实现，它内部维护了资源文件的路径。修复时创建一个新的 AssetManager，通过反射调用 addAssetPath 加载补丁资源包，然后替换所有 Resources 对象中的 AssetManager 引用。Tinker 的做法更彻底——把补丁资源和原始资源合并生成新的资源包，替换整个资源路径。需要注意的是资源 ID 的一致性问题，补丁中的资源 ID 必须和原始 APK 中的一致，否则会找不到资源。

### Q10：如果让你设计一个热修复方案，你会怎么做？

> 我会采用类替换 + Robust 的混合方案。对于紧急 Bug 用 Robust 方案即时修复——编译时通过 Gradle Plugin + ASM 在每个方法前插入判断逻辑，下发补丁后立即生效。对于需要修改类结构或资源的场景，用类替换方案——下发差量补丁，后台合并 dex，下次启动时生效。补丁的安全性方面，补丁包需要签名校验防止被篡改，传输过程加密。灰度策略方面，先小比例下发观察崩溃率，确认无问题后全量推送。回滚机制方面，服务端可以下发空补丁清除之前的修复。
