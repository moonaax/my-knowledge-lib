# 组件生命周期管理

> 在 [[组件化架构设计与拆分策略]] 中，各业务模块被拆分为独立组件。每个组件都有自己的初始化逻辑、资源依赖和生命周期诉求。如何统一管理这些组件的生命周期——何时初始化、按什么顺序初始化、如何感知 Application 生命周期事件——是组件化工程落地的关键一环。

---

## 一、核心概念

### 1.1 组件初始化时机

组件初始化时机通常分为三类：

| 时机 | 说明 | 典型场景 |
|------|------|----------|
| Application.onCreate | 最早，进程启动即执行 | 网络库、日志库、埋点 SDK |
| 首次路由到达时（懒加载） | 用户首次访问该组件时触发 | 低频业务模块 |
| 特定生命周期回调 | 如首个 Activity 创建后 | 需要 Context/Activity 的 SDK |

核心原则：**非必要不提前初始化**。将组件初始化推迟到真正需要的时刻，是 [[启动优化]] 的重要手段。

### 1.2 Application 生命周期分发

在单体应用中，所有初始化逻辑堆积在 `Application.onCreate()` 里。组件化后，每个组件需要独立感知 Application 的生命周期事件（onCreate、onLowMemory、onTrimMemory、onTerminate 等），但又不能让壳工程直接依赖各业务组件。

解决思路：**定义统一的生命周期接口，由框架层负责收集和分发。**

```
Application.onCreate()
       │
       ▼
  ┌─────────────────────┐
  │  生命周期分发器       │
  │  (LifecycleDispatcher)│
  └─────────┬───────────┘
            │ 遍历已注册组件
    ┌───────┼───────┐
    ▼       ▼       ▼
 组件A    组件B    组件C
 .onCreate .onCreate .onCreate
```

### 1.3 组件按需加载

并非所有组件都需要在 App 启动时初始化。按需加载策略：

- **立即加载**：基础组件（网络、日志、存储），启动时必须就绪
- **延迟加载**：业务组件在首次被路由访问时触发初始化
- **条件加载**：根据用户身份、AB 实验、设备能力等决定是否加载

```java
public enum LoadPolicy {
    EAGER,      // 立即加载
    LAZY,       // 延迟加载（首次访问时）
    CONDITIONAL // 条件加载
}
```

## 二、原理与实践

### 2.1 IAppLifecycle 接口设计

定义一个统一的组件生命周期接口，所有需要感知 Application 生命周期的组件都实现该接口：

```java
/**
 * 组件生命周期接口
 * 每个组件模块提供一个实现类，由框架统一调度
 */
public interface IAppLifecycle {

    /** 组件优先级，数值越小越先执行 */
    default int priority() { return 100; }

    /** 依赖的其他组件（用于拓扑排序） */
    default List<Class<? extends IAppLifecycle>> dependencies() {
        return Collections.emptyList();
    }

    /** 加载策略 */
    default LoadPolicy loadPolicy() { return LoadPolicy.EAGER; }

    /** Application.onCreate 时回调 */
    void onCreate(Application application);

    /** Application.onTerminate 时回调 */
    default void onTerminate() {}

    /** Application.onLowMemory 时回调 */
    default void onLowMemory() {}

    /** Application.onTrimMemory 时回调 */
    default void onTrimMemory(int level) {}
}
```

各组件模块的实现示例：

```java
// network 模块
public class NetworkLifecycle implements IAppLifecycle {
    @Override
    public int priority() { return 10; }

    @Override
    public void onCreate(Application application) {
        OkHttpManager.init(application);
        RetrofitManager.init();
    }
}

// login 模块，依赖 network
public class LoginLifecycle implements IAppLifecycle {
    @Override
    public int priority() { return 50; }

    @Override
    public List<Class<? extends IAppLifecycle>> dependencies() {
        return Collections.singletonList(NetworkLifecycle.class);
    }

    @Override
    public void onCreate(Application application) {
        LoginSdk.init(application);
    }
}
```

### 2.2 APT 自动注册 vs 反射扫描 vs SPI 发现

框架如何发现各组件的 `IAppLifecycle` 实现？三种主流方案对比：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| APT 自动注册 | 编译期注解处理器生成注册代码 | 零反射，性能最优 | 需要自定义 Gradle 插件或 APT |
| 反射扫描 | 运行时扫描 dex 查找实现类 | 实现简单 | 启动耗时，Android P+ 限制 |
| SPI (ServiceLoader) | META-INF/services 配置文件 | Java 标准机制 | 需手动维护配置文件 |

#### 方案一：APT + ASM 字节码插桩（推荐）

这是 ARouter、TheRouter 等框架采用的方式。编译期通过注解处理器收集信息，再通过 Gradle Transform / ASM 插桩将注册代码写入指定类。

```java
// 1. 定义注解
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.CLASS)
public @interface AppLifecycle {}

// 2. 在组件中使用
@AppLifecycle
public class NetworkLifecycle implements IAppLifecycle {
    @Override
    public void onCreate(Application application) {
        OkHttpManager.init(application);
    }
}
```

APT 处理器在编译期扫描 `@AppLifecycle` 注解，生成中间注册类：

```java
// APT 自动生成（build/generated/...）
public class NetworkLifecycle$$Registry implements ILifecycleRegistry {
    @Override
    public void register(LifecycleManager manager) {
        manager.add(NetworkLifecycle.class);
    }
}
```

Gradle 插件在打包阶段通过 ASM 将所有 `$$Registry` 类的调用插入到 `LifecycleManager.init()` 方法中，实现零反射的自动注册。

#### 方案二：SPI（ServiceLoader）

利用 Java 标准 SPI 机制，配合 Google 的 `auto-service` 简化配置：

```java
// 添加依赖: annotationProcessor("com.google.auto.service:auto-service:1.1.1")

@AutoService(IAppLifecycle.class)
public class NetworkLifecycle implements IAppLifecycle {
    @Override
    public void onCreate(Application application) {
        OkHttpManager.init(application);
    }
}
```

框架侧加载：

```java
public class LifecycleManager {
    private static final List<IAppLifecycle> lifecycles = new ArrayList<>();

    public static void init(Application app) {
        // SPI 发现所有实现
        for (IAppLifecycle lifecycle : ServiceLoader.load(IAppLifecycle.class)) {
            lifecycles.add(lifecycle);
        }
        // 拓扑排序后依次初始化
        for (IAppLifecycle lifecycle : topologicalSort(lifecycles)) {
            lifecycle.onCreate(app);
        }
    }
}
```

### 2.3 App Startup 在组件化中的应用

Jetpack App Startup 提供了官方的组件初始化方案，天然支持依赖声明和懒加载：

```java
// network 模块的 Initializer
public class NetworkInitializer implements Initializer<OkHttpManager> {

    @NonNull
    @Override
    public OkHttpManager create(@NonNull Context context) {
        OkHttpManager.init(context);
        return OkHttpManager.getInstance();
    }

    // 声明无依赖
    @NonNull
    @Override
    public List<Class<? extends Initializer<?>>> dependencies() {
        return Collections.emptyList();
    }
}

// login 模块的 Initializer，依赖 network
public class LoginInitializer implements Initializer<LoginSdk> {

    @NonNull
    @Override
    public LoginSdk create(@NonNull Context context) {
        LoginSdk.init(context);
        return LoginSdk.getInstance();
    }

    @NonNull
    @Override
    public List<Class<? extends Initializer<?>>> dependencies() {
        return Collections.singletonList(NetworkInitializer.class);
    }
}
```

在各组件的 `AndroidManifest.xml` 中声明（合并时自动汇总）：

```xml
<provider
    android:name="androidx.startup.InitializationProvider"
    android:authorities="${applicationId}.androidx-startup"
    android:exported="false"
    tools:node="merge">

    <meta-data
        android:name="com.example.network.NetworkInitializer"
        android:value="androidx.startup" />
</provider>
```

App Startup 的优势与局限：

- ✅ 官方方案，依赖声明清晰，自动拓扑排序
- ✅ 支持懒加载：`AppInitializer.getInstance(context).initializeComponent(LoginInitializer::class.java)`
- ❌ 只覆盖 `onCreate` 时机，不支持 `onLowMemory` 等其他生命周期
- ❌ 不支持优先级微调，只能通过依赖关系控制顺序

因此实际项目中常将 App Startup 与自定义 `IAppLifecycle` 结合使用。

### 2.4 组件初始化依赖排序（DAG 拓扑排序）

当组件之间存在初始化依赖时（如 login 依赖 network，push 依赖 login 和 network），需要用 **有向无环图（DAG）拓扑排序** 来确定正确的初始化顺序。

```
network ──→ login ──→ push
   │                    ↑
   └────────────────────┘
```

Kahn 算法实现：

```java
public class TopologicalSorter {

    public static List<IAppLifecycle> sort(List<IAppLifecycle> lifecycles) {
        Map<Class<?>, IAppLifecycle> classToInstance = new HashMap<>();
        for (IAppLifecycle lc : lifecycles) {
            classToInstance.put(lc.getClass(), lc);
        }
        // 入度表
        Map<Class<?>, Integer> inDegree = new HashMap<>();
        // 邻接表
        Map<Class<?>, List<Class<?>>> graph = new HashMap<>();

        // 初始化
        for (IAppLifecycle node : lifecycles) {
            Class<?> cls = node.getClass();
            inDegree.putIfAbsent(cls, 0);
            graph.putIfAbsent(cls, new ArrayList<>());
        }

        // 构建图：dependency → current（dependency 完成后才能执行 current）
        for (IAppLifecycle node : lifecycles) {
            Class<?> cls = node.getClass();
            for (Class<?> dep : node.dependencies()) {
                graph.computeIfAbsent(dep, k -> new ArrayList<>()).add(cls);
                inDegree.merge(cls, 1, Integer::sum);
            }
        }

        // BFS
        Queue<Class<?>> queue = new LinkedList<>();
        for (Map.Entry<Class<?>, Integer> entry : inDegree.entrySet()) {
            if (entry.getValue() == 0) queue.add(entry.getKey());
        }

        List<IAppLifecycle> result = new ArrayList<>();
        while (!queue.isEmpty()) {
            Class<?> current = queue.poll();
            IAppLifecycle instance = classToInstance.get(current);
            if (instance != null) result.add(instance);
            List<Class<?>> neighbors = graph.get(current);
            if (neighbors != null) {
                for (Class<?> neighbor : neighbors) {
                    int newDegree = inDegree.getOrDefault(neighbor, 1) - 1;
                    inDegree.put(neighbor, newDegree);
                    if (newDegree == 0) queue.add(neighbor);
                }
            }
        }

        // 环检测
        if (result.size() != lifecycles.size()) {
            throw new IllegalStateException(
                String.format("组件依赖存在循环！已排序 %d，总计 %d",
                    result.size(), lifecycles.size())
            );
        }
        return result;
    }
}
```

完整的生命周期分发器：

```java
public class LifecycleDispatcher implements Application.ActivityLifecycleCallbacks {

    private static Application app;
    private static final List<IAppLifecycle> lifecycles = new ArrayList<>();
    private static final LifecycleDispatcher INSTANCE = new LifecycleDispatcher();

    public static void init(Application application) {
        app = application;
        // 1. 收集所有 IAppLifecycle 实现（SPI / APT 均可）
        for (IAppLifecycle lifecycle : ServiceLoader.load(IAppLifecycle.class)) {
            lifecycles.add(lifecycle);
        }
        // 2. 拓扑排序
        List<IAppLifecycle> eagerList = new ArrayList<>();
        for (IAppLifecycle lc : lifecycles) {
            if (lc.loadPolicy() == LoadPolicy.EAGER) eagerList.add(lc);
        }
        List<IAppLifecycle> sorted = TopologicalSorter.sort(eagerList);
        // 3. 按序初始化
        for (IAppLifecycle lifecycle : sorted) {
            long start = System.currentTimeMillis();
            lifecycle.onCreate(application);
            long cost = System.currentTimeMillis() - start;
            Log.d("Lifecycle", lifecycle.getClass().getSimpleName() + " init cost " + cost + "ms");
        }
        // 4. 注册 Activity 生命周期监听
        application.registerActivityLifecycleCallbacks(INSTANCE);
    }

    public static void dispatchLowMemory() {
        for (IAppLifecycle lc : lifecycles) { lc.onLowMemory(); }
    }

    public static void dispatchTrimMemory(int level) {
        for (IAppLifecycle lc : lifecycles) { lc.onTrimMemory(level); }
    }

    // ActivityLifecycleCallbacks 省略...
    @Override public void onActivityCreated(Activity activity, Bundle bundle) {}
    @Override public void onActivityStarted(Activity activity) {}
    @Override public void onActivityResumed(Activity activity) {}
    @Override public void onActivityPaused(Activity activity) {}
    @Override public void onActivityStopped(Activity activity) {}
    @Override public void onActivitySaveInstanceState(Activity activity, Bundle bundle) {}
    @Override public void onActivityDestroyed(Activity activity) {}
}
```

壳工程 Application 中调用：

```java
public class MyApp extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        LifecycleDispatcher.init(this);
    }

    @Override
    public void onLowMemory() {
        super.onLowMemory();
        LifecycleDispatcher.dispatchLowMemory();
    }

    @Override
    public void onTrimMemory(int level) {
        super.onTrimMemory(level);
        LifecycleDispatcher.dispatchTrimMemory(level);
    }
}
```

## 三、面试题

### Q1：组件化项目中，各模块的初始化逻辑如何统一管理？

**A**：定义统一的 `IAppLifecycle` 接口，各组件模块提供实现类。框架层通过 APT 自动注册、SPI 或 App Startup 等机制自动发现所有实现，在 `Application.onCreate()` 中统一收集、排序、分发。这样壳工程无需直接依赖业务组件，实现了生命周期管理的解耦。

### Q2：APT 自动注册和 SPI 发现各有什么优缺点？实际项目中如何选择？

**A**：
- **APT + ASM 插桩**：编译期完成注册，运行时零反射开销，性能最优；但需要自定义 Gradle 插件，维护成本较高。适合对启动性能要求极高的大型项目。
- **SPI (ServiceLoader)**：Java 标准机制，配合 `@AutoService` 使用简单；但 ServiceLoader 内部使用反射创建实例，且首次加载需要读取 JAR 中的配置文件。适合中小型项目或组件数量不多的场景。
- 实际项目中，如果已经引入了 ARouter / TheRouter 等路由框架，可以复用其 APT + Transform 基础设施；否则 SPI 是性价比最高的选择。

### Q3：如何解决组件之间的初始化顺序依赖问题？

**A**：让每个组件在 `IAppLifecycle` 中声明自己依赖的其他组件（`dependencies()` 方法），框架层使用 **DAG 拓扑排序**（Kahn 算法或 DFS）计算出正确的初始化顺序。如果检测到循环依赖，应在编译期或启动时抛出异常，提前暴露架构问题。参考 [[组件化架构设计与拆分策略]] 中的分层原则，基础层组件不应依赖业务层组件。

### Q4：Jetpack App Startup 和自定义 IAppLifecycle 方案有什么区别？如何取舍？

**A**：
- **App Startup**：官方方案，通过 ContentProvider 触发，自动合并多个 ContentProvider 为一个以减少启动开销，内置依赖排序和懒加载支持。但只覆盖初始化场景，不支持 `onLowMemory`、`onTrimMemory` 等生命周期回调。
- **自定义 IAppLifecycle**：灵活度更高，可以覆盖完整的 Application 生命周期，支持优先级、加载策略等自定义扩展。但需要自行实现发现机制和排序逻辑。
- 推荐做法：基础 SDK 初始化用 App Startup（利用其依赖排序），完整生命周期管理用自定义方案，两者结合使用。

### Q5：多进程场景下，组件初始化需要注意什么？

**A**：Android 中每个进程都会创建独立的 Application 实例并执行 `onCreate()`。需要注意：
1. **按进程过滤**：并非所有组件都需要在所有进程中初始化。通过 `ProcessUtils.getCurrentProcessName()` 判断当前进程，只初始化该进程需要的组件。
2. **避免重复初始化**：某些 SDK（如推送）只需在主进程初始化，在子进程中重复初始化可能导致异常。
3. **进程间通信**：如果组件状态需要跨进程共享，需要借助 ContentProvider、AIDL 或 [[组件化架构设计与拆分策略]] 中提到的跨进程通信方案。

### Q6：组件按需加载（懒加载）如何实现？对启动速度有多大影响？

**A**：懒加载的核心是将组件初始化推迟到首次使用时。实现方式：
1. 在 `IAppLifecycle` 中标记 `loadPolicy = LAZY`，框架启动时跳过这些组件。
2. 当路由首次导航到该组件时，触发其初始化逻辑。
3. 使用 `by lazy` 或双重检查锁保证线程安全和单次初始化。

对 [[启动优化]] 的影响非常显著——将非核心组件从 EAGER 改为 LAZY，可以直接减少 `Application.onCreate()` 的耗时。实测中，一个包含 20+ 组件的项目，将 60% 的业务组件改为懒加载后，冷启动时间减少了 300-500ms。

## 四、实战与踩坑

### 4.1 初始化顺序依赖问题

**问题**：组件 A 在 `onCreate` 中调用了组件 B 提供的服务，但组件 B 尚未初始化，导致 NPE 或空数据。

**典型场景**：埋点组件依赖网络组件，但两者初始化顺序不确定。

**解决方案**：

```java
// ❌ 错误：隐式依赖，顺序不可控
public class TrackLifecycle implements IAppLifecycle {
    @Override
    public void onCreate(Application application) {
        // 此时 NetworkManager 可能还没初始化！
        TrackSdk.init(NetworkManager.getClient());
    }
}

// ✅ 正确：显式声明依赖
public class TrackLifecycle implements IAppLifecycle {
    @Override
    public List<Class<? extends IAppLifecycle>> dependencies() {
        return Collections.singletonList(NetworkLifecycle.class);
    }

    @Override
    public void onCreate(Application application) {
        // 拓扑排序保证 NetworkLifecycle 已经执行完毕
        TrackSdk.init(NetworkManager.getClient());
    }
}
```

**进阶：运行时依赖检查**

```java
public class InitStateTracker {
    private static final ConcurrentHashMap<String, Boolean> initialized = new ConcurrentHashMap<>();

    public static void markInitialized(String name) {
        initialized.put(name, true);
    }

    public static void ensureInitialized(String name) {
        if (!Boolean.TRUE.equals(initialized.get(name))) {
            throw new IllegalStateException(
                "组件 [" + name + "] 尚未初始化！请检查依赖声明或初始化顺序。"
            );
        }
    }
}

// 在组件初始化完成后标记
public class NetworkLifecycle implements IAppLifecycle {
    @Override
    public void onCreate(Application application) {
        OkHttpManager.init(application);
        InitStateTracker.markInitialized("network");
    }
}

// 在依赖方使用前检查
public class TrackLifecycle implements IAppLifecycle {
    @Override
    public void onCreate(Application application) {
        InitStateTracker.ensureInitialized("network");
        TrackSdk.init(NetworkManager.getClient());
    }
}
```

### 4.2 组件卸载与资源释放

**问题**：组件化项目中，组件初始化容易被关注，但资源释放往往被忽视，导致内存泄漏或状态残留。

**场景**：用户退出登录后，登录模块的长连接、缓存、监听器需要清理。

```java
public interface IAppLifecycle {
    // ... 其他方法

    /** 组件卸载/清理，用于退出登录、切换账号等场景 */
    default void onDestroy() {}
}

// login 模块
public class LoginLifecycle implements IAppLifecycle {
    @Override
    public void onCreate(Application application) {
        LoginSdk.init(application);
        TokenManager.startRefreshLoop();
    }

    @Override
    public void onDestroy() {
        TokenManager.stopRefreshLoop();
        LoginSdk.clearCache();
        EventBus.getDefault().unregister(this);
    }
}
```

分发器中增加卸载支持：

```java
public class LifecycleDispatcher {
    // ...

    /**
     * 卸载指定组件（如退出登录时清理业务组件）
     * 按拓扑排序的逆序执行，确保被依赖方最后清理
     */
    public static void destroy(Predicate<IAppLifecycle> filter) {
        List<IAppLifecycle> filtered = new ArrayList<>();
        for (IAppLifecycle lc : lifecycles) {
            if (filter.test(lc)) filtered.add(lc);
        }
        Collections.reverse(filtered);
        for (IAppLifecycle lifecycle : filtered) {
            try {
                lifecycle.onDestroy();
            } catch (Exception e) {
                Log.e("Lifecycle", "destroy failed: " + lifecycle.getClass().getSimpleName(), e);
            }
        }
    }
}
```

**踩坑点**：
- `onDestroy` 中要用 try-catch 包裹，避免一个组件清理失败影响其他组件
- 卸载顺序应与初始化顺序相反（逆拓扑序）
- 注意清理静态变量和单例持有的 Context 引用

### 4.3 多进程下的组件初始化

**问题**：App 存在多个进程（主进程、推送进程、WebView 进程等），每个进程都会执行 `Application.onCreate()`，但大部分组件只需要在主进程初始化。

```java
// ❌ 错误：所有进程都执行全部初始化
public class MyApp extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        LifecycleDispatcher.init(this); // 推送进程也会执行所有组件初始化
    }
}
```

**解决方案：为 IAppLifecycle 增加进程过滤**

```java
public interface IAppLifecycle {
    /** 该组件需要在哪些进程中初始化，空列表表示所有进程 */
    default List<String> targetProcesses() { return Collections.emptyList(); }

    // ... 其他方法
}
```

分发器中增加进程过滤：

```java
public class LifecycleDispatcher {

    public static void init(Application application) {
        String currentProcess = getProcessName(application);
        boolean isMainProcess = currentProcess.equals(application.getPackageName());

        for (IAppLifecycle lifecycle : ServiceLoader.load(IAppLifecycle.class)) {
            lifecycles.add(lifecycle);
        }

        List<IAppLifecycle> filtered = new ArrayList<>();
        for (IAppLifecycle lifecycle : lifecycles) {
            List<String> targets = lifecycle.targetProcesses();
            if (targets.isEmpty()) {
                filtered.add(lifecycle);
                continue;
            }
            for (String target : targets) {
                if ("main".equals(target) && isMainProcess) {
                    filtered.add(lifecycle);
                    break;
                } else if (currentProcess.endsWith(target)) {
                    filtered.add(lifecycle);
                    break;
                }
            }
        }

        for (IAppLifecycle lc : TopologicalSorter.sort(filtered)) {
            lc.onCreate(application);
        }
    }

    private static String getProcessName(Context context) {
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.P) {
            return Application.getProcessName();
        }
        int pid = android.os.Process.myPid();
        ActivityManager am = (ActivityManager) context.getSystemService(Context.ACTIVITY_SERVICE);
        List<ActivityManager.RunningAppProcessInfo> processes = am.getRunningAppProcesses();
        if (processes != null) {
            for (ActivityManager.RunningAppProcessInfo info : processes) {
                if (info.pid == pid) return info.processName;
            }
        }
        return "";
    }
}
```

**踩坑点**：
- `getProcessName()` 在 Android P 以下需要通过 ActivityManager 获取，有一定耗时
- 子进程中不要初始化 UI 相关的组件（如路由表、Activity 栈管理）
- ContentProvider 的 `onCreate()` 在 `Application.onCreate()` 之前执行，如果用 App Startup，注意多进程下 ContentProvider 也会在每个进程创建
- 某些三方 SDK（如地图、推送）内部会启动子进程，要确认其文档中关于多进程初始化的说明

---

## 总结

| 关注点 | 推荐方案 |
|--------|----------|
| 组件发现 | APT + ASM（大型项目）/ SPI（中小项目） |
| 依赖排序 | DAG 拓扑排序（Kahn 算法） |
| 官方初始化 | Jetpack App Startup |
| 生命周期分发 | 自定义 IAppLifecycle + LifecycleDispatcher |
| 按需加载 | LoadPolicy + 路由触发懒初始化 |
| 多进程 | targetProcesses 过滤 |
| 资源释放 | 逆拓扑序 onDestroy |

组件生命周期管理的核心目标：**让每个组件在正确的时机、正确的进程中完成初始化，同时保持组件间的解耦**。结合 [[组件化架构设计与拆分策略]] 的分层原则和 [[启动优化]] 的性能要求，才能构建出高效、可维护的组件化架构。
