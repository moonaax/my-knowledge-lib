# 资源隔离与冲突处理

> 组件化开发中，多模块资源合并是最容易被忽视却最频繁踩坑的环节。本文系统梳理 Android 资源合并机制、冲突类型、隔离策略与实战经验。
>
> 关联：[[1-组件化架构设计与拆分策略]] · [[5-AAR与APK打包]] · [[1-Gradle生命周期与Task]]

## 一、核心概念

### 1.1 Android 资源合并机制

Android 构建系统在打包 APK 时，会将 **App 模块、Library 模块、AAR 依赖** 三层资源进行合并。合并优先级为：

````
App 模块资源 > Library 模块资源 > 外部 AAR 依赖资源
````
具体规则：
- **同名同类型资源**：高优先级覆盖低优先级，不会报错（静默覆盖）
- **同名不同类型资源**：可以共存（如 `color/primary` 和 `drawable/primary`）
- **values 类资源**（strings.xml / colors.xml / dimens.xml）：按条目级别合并，同 name 属性的条目被高优先级覆盖
- **文件类资源**（layout / drawable / mipmap）：按文件名整体覆盖

合并发生在 `mergeResources` Task 中，由 AAPT2 执行：

````
:app:mergeDebugResources
  ├── 收集 app/src/main/res
  ├── 收集 library-a 的 res（来自 AAR 解压或本地模块）
  ├── 收集 library-b 的 res
  └── AAPT2 compile + link → 合并产物写入 build/intermediates/res/merged/
````
### 1.2 资源冲突的类型

#### 类型一：同名资源静默覆盖

这是最危险的冲突类型——**编译不报错，运行时表现异常**。

````
// module-login 的 res/values/strings.xml
<string name="btn_confirm">登录</string>

// module-pay 的 res/values/strings.xml
<string name="btn_confirm">支付</string>
````
最终 APK 中 `btn_confirm` 的值取决于模块合并顺序，结果不可预测。

#### 类型二：R 文件字段冲突

在传统（transitive R）模式下，每个模块的 R 文件包含 **自身 + 所有依赖** 的资源 ID：

````java
// module-login 的 R.java（transitive 模式）
public final class R {
    public static final class string {
        public static final int btn_confirm = 0x7f0a0001; // 自己的
        public static final int app_name = 0x7f0a0002;    // 来自 app 或其他模块
    }
}
````
这导致：
- R 文件体积膨胀（大型项目 R 文件可达数万行）
- 模块间通过 R 类「意外」访问到不属于自己的资源
- 编译速度下降（每个模块都要生成完整 R）

#### 类型三：Manifest 合并冲突

多模块的 `AndroidManifest.xml` 合并时，若出现同一属性的不同值，构建直接失败：

````
Manifest merger failed: Attribute application@theme value=(@style/ThemeA)
  from AndroidManifest.xml is also present at [library-b] AndroidManifest.xml
  with value=(@style/ThemeB).
````
### 1.3 resourcePrefix 的作用与局限

`resourcePrefix` 是 AGP 提供的资源前缀检查机制：

````kotlin
// module-login/build.gradle.kts
android {
    resourcePrefix = "login_"
}
````
**作用**：
- 在 Lint 检查时，对不符合前缀的资源名发出警告
- 帮助团队建立命名规范，从源头避免同名冲突

**局限**：
- ⚠️ **仅是 Lint 警告，不会阻止编译**——开发者可以忽略
- ⚠️ **不检查 drawable / mipmap 文件名**——只检查 values 类资源的 name 属性
- ⚠️ **不检查第三方 AAR 的资源**——只对当前模块源码生效
- ⚠️ **不支持自动重命名**——需要开发者手动遵守

````
// ✅ 符合 login_ 前缀
res/values/strings.xml → <string name="login_title">登录</string>
res/layout/login_activity_main.xml

// ❌ Lint 会警告但不阻止编译
res/values/strings.xml → <string name="title">登录</string>

// ❌ drawable 文件名不受检查！
res/drawable/ic_back.png  ← 不会警告，这是已知的坑
````

## 二、原理与源码

### 2.1 AAPT2 资源合并流程

AAPT2（Android Asset Packaging Tool 2）是资源编译与链接的核心工具，分为两个阶段：

````
┌─────────────────────────────────────────────────────┐
│                   AAPT2 Compile 阶段                  │
│  将每个模块的资源文件编译为 .flat 二进制中间格式          │
│  aapt2 compile -o output.flat res/layout/main.xml    │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│                   AAPT2 Link 阶段                     │
│  合并所有 .flat 文件 + AndroidManifest.xml             │
│  生成 resources.arsc / R.java / 最终资源文件            │
│  aapt2 link -o output.apk --manifest Manifest.xml    │
│       -R moduleA.flat -R moduleB.flat ...             │
└─────────────────────────────────────────────────────┘
````
在 AGP 中，对应的 Task 链路（参考 [[1-Gradle生命周期与Task]]）：

````
:app:mergeDebugResources          // 收集并合并所有模块资源
  → ResourceMerger.mergeData()    // 核心合并逻辑
    → 遍历 ResourceSet 列表（按优先级排序）
    → 同名资源：高优先级 ResourceItem 替换低优先级
    → 写入 merged-res 目录
:app:processDebugResources        // 调用 AAPT2 link
  → 生成 resources.arsc
  → 生成 R.java / R.jar
````
`ResourceMerger` 源码关键逻辑（简化）：

````java
// com.android.ide.common.resources.ResourceMerger
public void mergeData(MergeConsumer<ResourceMergerItem> consumer) {
    // dataSetList 按优先级排序：app > library > aar
    for (DataSet dataSet : dataSetList) {
        for (Map.Entry<String, ResourceMergerItem> entry : dataSet.getDataMap().entrySet()) {
            String key = entry.getKey();
            ResourceMergerItem item = entry.getValue();
            if (mergedMap.containsKey(key)) {
                // 同名资源 → 高优先级覆盖，标记被覆盖项为 REMOVED
                ResourceMergerItem existing = mergedMap.get(key);
                existing.setRemoved(true);
                consumer.removeItem(existing);
            }
            mergedMap.put(key, item);
            consumer.addItem(item);
        }
    }
}
````
资源的 key 由 **类型 + 限定符 + 名称** 组成：

````
ResourceType:layout / qualifier:land / name:activity_main
→ key = "layout-land/activity_main"
````
### 2.2 Manifest 合并规则

Manifest 合并由 `ManifestMerger2` 处理，规则如下：

| 场景 | 默认行为 | 解决方案 |
|------|---------|---------|
| 同一属性不同值 | 编译失败 | `tools:replace` |
| Library 声明了不需要的权限 | 合并进 App | `tools:remove` |
| Library 声明了不需要的组件 | 合并进 App | `tools:node="remove"` |
| minSdk 冲突 | 编译失败 | `tools:overrideLibrary` |

**tools:replace** —— 用 App 的值替换 Library 的值：

````xml
<!-- app/AndroidManifest.xml -->
<application
    android:name=".MyApp"
    android:theme="@style/AppTheme"
    tools:replace="android:theme">
    <!-- 强制使用 App 的 theme，忽略所有 Library 的 theme -->
</application>
````
**tools:remove** —— 移除 Library 引入的属性或权限：

````xml
<!-- 移除某个 Library 引入的不需要的权限 -->
<uses-permission
    android:name="android.permission.CAMERA"
    tools:node="remove" />

<!-- 移除属性 -->
<application
    tools:remove="android:allowBackup" />
````
**tools:overrideLibrary** —— 解决 minSdk 冲突：

````xml
<!-- App 的 minSdk=21，但某个 Library 声明 minSdk=23 -->
<uses-sdk tools:overrideLibrary="com.example.library.high.minsdk" />
````
合并产物位于：

````
app/build/intermediates/merged_manifests/debug/AndroidManifest.xml
````
可通过 `merged-manifest` 报告查看合并来源（AS → Merged Manifest 视图）。

### 2.3 R 文件的生成：R.jar vs 非 transitive R

#### 传统模式（Transitive R）

每个模块的 R.java 包含自身 + 所有传递依赖的资源 ID：

````
module-base 定义了 R.string.app_name
module-login 依赖 module-base
→ module-login 的 R.java 也包含 R.string.app_name
→ module-login 的代码可以直接使用 R.string.app_name（来自 base）
````
问题：
- R 文件巨大，编译慢
- 模块边界模糊，任何模块都能访问任何资源
- 不利于组件化隔离

#### 非 transitive R（AGP 4.1+ / Gradle 属性）

开启方式（参考 [[1-Gradle生命周期与Task]]）：

````properties
# gradle.properties
android.nonTransitiveRClass=true
````
开启后：
- 每个模块的 R 类 **只包含自身定义的资源**
- 要使用依赖模块的资源，必须显式 import 对应模块的 R 类
- R 文件体积大幅缩小，编译速度提升

````kotlin
// module-login 中使用 module-base 的资源
// ❌ 非 transitive R 模式下，直接用 R.string.app_name 编译报错
// ✅ 必须显式引用
import com.example.base.R as BaseR

class LoginActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // 使用自己模块的资源
        setContentView(R.layout.login_activity_main)
        // 使用 base 模块的资源 → 显式引用
        title = getString(BaseR.string.app_name)
    }
}
````
#### R.jar 优化

AGP 7.0+ 默认将 Library 模块的 R 类编译为 `R.jar`（而非 R.java），进一步提升编译速度：

````
传统：R.java → javac 编译 → R.class（每次资源变更都要重新编译）
优化：直接生成 R.jar（跳过 javac，由 AGP 直接生成字节码）
````
在 `build/intermediates/compile_and_runtime_not_namespaced_r_class_jar/` 可以找到生成的 R.jar。

## 三、面试题

### Q1：多模块项目中，两个 Library 定义了同名资源会怎样？

**A**：取决于资源类型和合并优先级：

- **values 类资源**（string / color / dimen）：按条目级别合并，同 `name` 的条目被更高优先级模块覆盖，**不会报错**。优先级由依赖顺序决定——在 `dependencies {}` 中先声明的模块优先级更高。
- **文件类资源**（layout / drawable）：同名文件被高优先级模块整体覆盖，同样不报错。
- App 模块的资源优先级最高，始终覆盖 Library。

这种静默覆盖是组件化中最隐蔽的 Bug 来源，解决方案是使用 `resourcePrefix` + 团队命名规范。

### Q2：resourcePrefix 能完全防止资源冲突吗？

**A**：不能。`resourcePrefix` 有三个关键局限：

1. **仅 Lint 警告**，不阻止编译，开发者可以忽略
2. **不检查 drawable / mipmap 的文件名**，只检查 values 资源的 `name` 属性和 layout 文件名
3. **不检查第三方 AAR** 的资源

要真正防止冲突，需要结合 CI 流程中的自定义 Lint 规则或资源名检查脚本。

### Q3：什么是非 transitive R？开启后有什么影响？

**A**：非 transitive R（`android.nonTransitiveRClass=true`）让每个模块的 R 类只包含自身定义的资源，不再传递依赖模块的资源 ID。

影响：
- ✅ R 文件体积大幅缩小，编译速度提升（大型项目可提升 10-20%）
- ✅ 模块边界更清晰，强制显式引用依赖模块的资源
- ❌ 迁移成本高——已有代码中直接使用其他模块资源的地方都会编译报错，需要逐一修改为显式 import

迁移建议：使用 AS 的 `Migrate to Non-transitive R Classes` 重构工具，或 Google 开源的 `not-transitive-r` 迁移脚本。

### Q4：Manifest 合并冲突如何排查和解决？

**A**：排查步骤：

1. 查看编译错误日志，定位冲突属性和来源模块
2. 打开 AS 的 **Merged Manifest** 视图（`AndroidManifest.xml` 底部 tab），查看每个属性的合并来源
3. 查看合并报告：`app/build/outputs/logs/manifest-merger-debug-report.txt`

解决方案：
- `tools:replace="attr"` —— 用 App 的值覆盖
- `tools:node="remove"` —— 移除整个节点
- `tools:remove="attr"` —— 移除特定属性
- `tools:overrideLibrary="pkg"` —— 覆盖 minSdk 限制

### Q5：AAPT2 的 compile 和 link 阶段分别做什么？

**A**：

- **Compile 阶段**：将单个资源文件编译为 `.flat` 二进制中间格式。这一步是增量的——只有变更的资源文件需要重新编译。对应命令：`aapt2 compile -o output.flat input.xml`
- **Link 阶段**：将所有 `.flat` 文件 + `AndroidManifest.xml` 合并，生成 `resources.arsc`（资源索引表）、R.java、以及最终的资源文件。对应命令：`aapt2 link -o output.apk --manifest Manifest.xml -R *.flat`

分两阶段的好处是支持增量编译——只有 compile 阶段需要处理变更文件，link 阶段使用缓存的 `.flat` 文件。

### Q6：如何在 CI 中自动检测资源命名冲突？

**A**：几种方案：

1. **自定义 Gradle Task**：在 `mergeResources` 之前扫描所有模块的 `res` 目录，检测同名资源并报错

````kotlin
// buildSrc 或 convention plugin 中
tasks.register("checkResourceConflict") {
    doLast {
        val resMap = mutableMapOf<String, MutableList<String>>()
        // 遍历所有子模块的 res 目录
        rootProject.subprojects.forEach { sub ->
            val resDir = sub.file("src/main/res")
            if (resDir.exists()) {
                resDir.walkTopDown()
                    .filter { it.isFile }
                    .forEach { file ->
                        val key = "${file.parentFile.name}/${file.name}"
                        resMap.getOrPut(key) { mutableListOf() }.add(sub.name)
                    }
            }
        }
        val conflicts = resMap.filter { it.value.size > 1 }
        if (conflicts.isNotEmpty()) {
            conflicts.forEach { (res, modules) ->
                logger.error("资源冲突: $res → 存在于 $modules")
            }
            throw GradleException("检测到 ${conflicts.size} 个资源冲突")
        }
    }
}
````
2. **自定义 Lint 规则**：编写 `ResourceDetector` 检查 `resourcePrefix` 合规性
3. **AGP API**：利用 `AndroidComponentsExtension` 的 `onVariants` 回调，在资源合并后检查产物

## 四、实战与踩坑

### 4.1 resourcePrefix 不支持 drawable/mipmap 的坑

这是一个被广泛忽视的问题。`resourcePrefix` 的检查范围：

| 资源类型 | 是否检查 | 说明 |
|---------|---------|------|
| values (string/color/dimen) | ✅ name 属性 | `<string name="login_xxx">` |
| layout | ✅ 文件名 | `login_activity_main.xml` |
| drawable | ❌ 文件名不检查 | `ic_back.png` 不会警告 |
| mipmap | ❌ 文件名不检查 | `ic_launcher.png` 不会警告 |
| anim / animator | ✅ 文件名 | `login_fade_in.xml` |
| menu | ✅ 文件名 | `login_menu_main.xml` |

**实际踩坑场景**：

````
module-login/res/drawable/ic_arrow_back.png   (24x24, 白色)
module-pay/res/drawable/ic_arrow_back.png     (24x24, 黑色)
````
编译不报错，最终 APK 中只保留一个 `ic_arrow_back.png`，另一个模块的 UI 显示异常。排查时很难定位，因为资源名完全合法。

**解决方案**：

````kotlin
// 方案一：团队规范 + CI 检查脚本
// 所有 drawable 文件必须以模块前缀开头
// module-login → login_ic_arrow_back.png
// module-pay   → pay_ic_arrow_back.png

// 方案二：自定义 Lint 规则强制检查 drawable 前缀
class DrawablePrefixDetector : ResourceXmlDetector() {
    companion object {
        val ISSUE = Issue.create(
            id = "DrawablePrefixMissing",
            briefDescription = "Drawable 文件缺少模块前缀",
            explanation = "组件化项目中，drawable 文件必须以模块前缀命名",
            category = Category.CORRECTNESS,
            priority = 8,
            severity = Severity.ERROR,
            implementation = Implementation(
                DrawablePrefixDetector::class.java,
                Scope.RESOURCE_FILE_SCOPE
            )
        )
    }
}

// 方案三：Gradle Task 在 CI 中检查
tasks.register("checkDrawablePrefix") {
    doLast {
        val prefix = android.resourcePrefix ?: return@doLast
        val resDir = file("src/main/res")
        listOf("drawable", "mipmap").forEach { type ->
            resDir.resolve(type).listFiles()?.forEach { file ->
                if (!file.name.startsWith(prefix)) {
                    throw GradleException(
                        "文件 ${file.name} 缺少前缀 '$prefix'（位于 $type）"
                    )
                }
            }
        }
    }
}
````
### 4.2 非 transitive R 迁移实战

迁移步骤（参考 [[5-AAR与APK打包]]）：

#### Step 1：评估影响范围

````bash
# 在项目根目录执行，统计跨模块 R 引用数量
grep -rn "import.*\.R$" --include="*.kt" --include="*.java" | \
  grep -v "import ${APP_PACKAGE}.R" | wc -l
````
#### Step 2：开启非 transitive R

````properties
# gradle.properties
android.nonTransitiveRClass=true
````
#### Step 3：修复编译错误

开启后会出现大量编译错误，典型模式：

````kotlin
// ❌ 编译报错：Unresolved reference: app_name
// module-login 中直接使用了 module-base 的资源
class LoginFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        binding.title.text = getString(R.string.app_name) // 报错
    }
}

// ✅ 修复：显式引用 base 模块的 R 类
import com.example.base.R as BaseR

class LoginFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        binding.title.text = getString(BaseR.string.app_name)
    }
}
````
#### Step 4：处理常见的迁移难点

````java
// 难点一：XML 中引用其他模块的资源
// XML 不支持 import，需要将资源复制到当前模块或抽取到公共模块

// 难点二：自定义 View 的 styleable
// 如果 module-base 定义了 <declare-styleable name="CustomView">
// module-login 中使用该 View 时，styleable 属性需要通过 base 的 R 访问
TypedArray a = context.obtainStyledAttributes(attrs, com.example.base.R.styleable.CustomView);
int color = a.getColor(com.example.base.R.styleable.CustomView_customColor, Color.BLACK);
a.recycle();

// 难点三：Navigation / DataBinding 中的资源引用
// Navigation XML 中的 @id/xxx 引用如果跨模块，需要确保 id 定义在正确的模块
// DataBinding 的 @{@string/xxx} 只能引用当前模块的资源
````
#### Step 5：渐进式迁移策略

对于大型项目，建议分批迁移：

````kotlin
// settings.gradle.kts 中可以按模块逐步开启
// AGP 8.0+ 支持模块级别的 nonTransitiveRClass 配置
// 但推荐全局开启，逐模块修复

// 迁移顺序建议：
// 1. 先迁移叶子模块（无其他模块依赖它）
// 2. 再迁移中间层模块
// 3. 最后迁移 base / common 模块
// 4. App 模块最后处理
````
### 4.3 资源命名规范

推荐的组件化资源命名规范（参考 [[1-组件化架构设计与拆分策略]]）：

````
格式：{模块前缀}_{资源类型缩写}_{描述}

模块前缀：
  login_    → module-login
  pay_      → module-pay
  base_     → module-base
  common_   → module-common

资源类型缩写：
  layout    → {prefix}_activity_{name} / {prefix}_fragment_{name} / {prefix}_item_{name}
  drawable  → {prefix}_ic_{name} / {prefix}_bg_{name} / {prefix}_shape_{name}
  string    → {prefix}_{描述}
  color     → {prefix}_color_{name}
  dimen     → {prefix}_dp_{name} / {prefix}_sp_{name}
  style     → {Prefix}.{Name}（大驼峰，用点分隔）
  id        → {prefix}_{view类型缩写}_{name}
````
完整示例：

````
module-login/
  res/
    layout/
      login_activity_main.xml
      login_fragment_verify.xml
      login_item_account.xml
    drawable/
      login_ic_logo.xml
      login_bg_button.xml
      login_shape_input_border.xml
    values/
      login_strings.xml      → <string name="login_title">登录</string>
      login_colors.xml       → <color name="login_color_primary">#FF6200</color>
````

### 4.4 资源隔离的最佳实践总结

````
┌─────────────────────────────────────────────────────────┐
│                  资源隔离策略金字塔                        │
├─────────────────────────────────────────────────────────┤
│  Level 4: CI 自动检测（Lint + 自定义 Task）    ← 最可靠   │
│  Level 3: 非 transitive R（编译期隔离）                   │
│  Level 2: resourcePrefix（Lint 警告）                    │
│  Level 1: 团队命名规范（人工约束）             ← 最基础   │
└─────────────────────────────────────────────────────────┘
````

````kotlin

    resourcePrefix = "login_"

// ./gradlew checkResourceConflict checkDrawablePrefix lint
````

> 📌 **小结**：资源隔离是组件化工程质量的基石。`resourcePrefix` 提供基础约束，非 transitive R 提供编译期隔离，CI 自动检测提供最终保障。三者结合才能在大型组件化项目中有效避免资源冲突。
