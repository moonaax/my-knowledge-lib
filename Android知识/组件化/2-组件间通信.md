## 一、核心概念

### 1.1 组件间通信的挑战

在 [[1-组件化架构设计与拆分策略]] 中，我们将单体工程拆分为多个独立的业务模块（module）。拆分后最核心的约束是：

> **业务模块之间不能存在直接编译依赖。**

如果 `module-order` 直接 `implementation project(':module-pay')`，就会产生以下问题：

| 问题 | 说明 |
|------|------|
| 循环依赖 | A 依赖 B、B 又依赖 A，Gradle 直接报错 |
| 编译耦合 | 任何模块改动都会触发依赖链上所有模块重新编译 |
| 独立运行失败 | 单模块无法脱离其他模块独立编译运行 |
| 代码边界模糊 | 开发者可以随意调用其他模块内部类，架构形同虚设 |

因此，我们需要一种 **间接通信机制**，让模块之间在 **编译期隔离、运行期连通**。

### 1.2 三种主流方案概述

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  路由方案     │     │  SPI 方案    │     │ 接口下沉方案  │
│  (ARouter)   │     │(ServiceLoader)│    │(Interface in │
│              │     │              │     │   base)      │
├─────────────┤     ├─────────────┤     ├─────────────┤
│ APT 生成路由表│     │ META-INF 注册│     │ 接口定义在公共│
│ 运行时按 path │     │ 运行时反射加载│     │ 模块，实现在各│
│ 查找目标      │     │ 实现类       │     │ 业务模块      │
└─────────────┘     └─────────────┘     └─────────────┘
```

| 维度 | ARouter | SPI | 接口下沉 |
|------|---------|-----|----------|
| 通信方式 | URL 路由 + 服务发现 | ServiceLoader 加载 | 依赖注入 / 手动注册 |
| 编译隔离 | ✅ 完全隔离 | ✅ 完全隔离 | ⚠️ 接口在 base，实现隔离 |
| 类型安全 | ⚠️ 页面跳转靠 path 字符串 | ✅ 接口约束 | ✅ 接口约束 |
| 学习成本 | 中 | 低 | 低 |
| 适用场景 | 页面跳转 + 服务调用 | 插件化扩展点 | 模块间服务调用 |

## 二、原理与源码

### 2.1 ARouter 的 APT 注解处理 + 路由表生成

[[5-ARouter]] 的核心流程分为三个阶段：**编译期注解处理 → 路由表生成 → 运行时路由分发**。

#### 2.1.1 编译期：APT 注解处理器

ARouter 定义了 `@Route` 注解：

```java
// arouter-annotation 模块
@Target({ElementType.TYPE})
@Retention(RetentionPolicy.CLASS)
public @interface Route {
    String path();        // 路由路径，如 "/order/detail"
    String group() default ""; // 分组，默认取 path 第一段
    int extras() default Integer.MIN_VALUE;
}
```

编译期由 `RouteProcessor`（继承 `AbstractProcessor`）扫描所有 `@Route` 注解，核心逻辑：

```java
// 简化后的 RouteProcessor 核心逻辑
@Override
public boolean process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv) {
    Set<? extends Element> routeElements = roundEnv.getElementsAnnotatedWith(Route.class);
    // 1. 按 group 分组收集
    HashMap<String, List<RouteMeta>> groupMap = new HashMap<>();
    for (Element element : routeElements) {
        Route route = element.getAnnotation(Route.class);
        String group = route.group().isEmpty() ? route.path().split("/")[1] : route.group();
        RouteType type;
        if (types.isSubtype(element.asType(), activityType)) {
            type = RouteType.ACTIVITY;
        } else if (types.isSubtype(element.asType(), iProviderType)) {
            type = RouteType.PROVIDER;
        } else {
            type = RouteType.UNKNOWN;
        }
        RouteMeta routeMeta = new RouteMeta(type, route.path(), group, element);
        groupMap.computeIfAbsent(group, k -> new ArrayList<>()).add(routeMeta);
    }
    // 2. 为每个 group 生成一个 Java 类
    for (Map.Entry<String, List<RouteMeta>> entry : groupMap.entrySet()) {
        generateGroupClass(entry.getKey(), entry.getValue());
    }
    // 3. 生成 Root 类，记录 group -> GroupClass 的映射
    generateRootClass(groupMap.keySet());
    return true;
}
```

#### 2.1.2 生成的路由表类

APT 会在每个模块的 `build/generated/source/kapt` 下生成类似如下代码：

```java
// 自动生成：ARouter$$Group$$order
public class ARouter$$Group$$order implements IRouteGroup {
    @Override
    public void loadInto(Map<String, RouteMeta> atlas) {
        atlas.put("/order/detail", RouteMeta.build(

        atlas.put("/order/list", RouteMeta.build(

}

// 自动生成：ARouter$$Root$$app

    @Override
    public void loadInto(Map<String, Class<? extends IRouteGroup>> routes) {
        routes.put("order", ARouter$$Group$$order.class);
        routes.put("pay",   ARouter$$Group$$pay.class);
    }
}
```

#### 2.1.3 运行时：路由查找与分发

```java
// LogisticsCenter.completion() 简化逻辑
public void completion(Postcard postcard) {
    // 1. 先从 Warehouse.routes 查找 RouteMeta
    RouteMeta routeMeta = Warehouse.routes.get(postcard.getPath());
    if (routeMeta == null) {
        // 2. 按需加载 group（懒加载设计，避免启动时全量加载）
        Class<? extends IRouteGroup> groupClass = Warehouse.groupsIndex.get(postcard.getGroup());
        if (groupClass != null) {
            IRouteGroup group = groupClass.newInstance();
            group.loadInto(Warehouse.routes);
        }
        routeMeta = Warehouse.routes.get(postcard.getPath());
    }
    // 3. 填充 Postcard 信息
    postcard.setDestination(routeMeta.getDestination());
    postcard.setType(routeMeta.getType());
}
```

关键设计：**分组懒加载**。启动时只加载 Root 表（group → class 映射），具体路由信息在首次访问该 group 时才加载，减少启动耗时。

#### 2.1.4 拦截器链源码分析

ARouter 的拦截器采用 **责任链模式**，与 OkHttp Interceptor 类似：

```java
// 拦截器接口
public interface IInterceptor extends IProvider {
    void process(Postcard postcard, InterceptorCallback callback);
}

// InterceptorServiceImpl 中的链式调用（简化）
public class InterceptorServiceImpl implements InterceptorService {
    @Override
    public void doInterceptions(Postcard postcard, InterceptorCallback callback) {
        // 在子线程中执行拦截器链
        LogisticsCenter.executor.execute(() -> {
            List<IInterceptor> interceptors = Warehouse.interceptors; // 按 priority 排序
            _execute(0, new CancelableCountDownLatch(1), postcard, interceptors, callback);
        });
    }

    private void _execute(int index, CancelableCountDownLatch latch,
                          Postcard postcard, List<IInterceptor> interceptors,
                          InterceptorCallback callback) {
        if (index >= interceptors.size()) {
            callback.onContinue(postcard); // 全部通过
            return;
        }
        interceptors.get(index).process(postcard, new InterceptorCallback() {
            @Override
            public void onContinue(Postcard postcard) {
                latch.countDown();
                // 递归执行下一个拦截器
                _execute(index + 1, latch, postcard, interceptors, callback);
            }
            @Override
            public void onInterrupt(Throwable exception) {
                postcard.setTag(exception.getMessage());
                callback.onInterrupt(exception); // 中断路由
            }
        });
    }
}
```

拦截器典型应用：**登录检查**。

```java
@Interceptor(priority = 8, name = "登录拦截器")
public class LoginInterceptor implements IInterceptor {
    @Override
    public void process(Postcard postcard, InterceptorCallback callback) {
        if (postcard.getExtra() == NeedLogin.FLAG && !UserManager.isLoggedIn()) {
            // 中断当前路由，跳转登录页
            ARouter.getInstance().build("/account/login").navigation();
            callback.onInterrupt(new RuntimeException("需要登录"));
        } else {
            callback.onContinue(postcard);
        }
    }
    @Override
    public void init(Context context) {}
}
```



### 2.2 Java SPI 与 ServiceLoader

SPI（Service Provider Interface）是 JDK 内置的服务发现机制，核心类为 `java.util.ServiceLoader`。

#### 工作原理

```
META-INF/services/
  └── com.example.base.IPayService    ← 文件名 = 接口全限定名
        内容：com.example.pay.PayServiceImpl  ← 实现类全限定名
```

`ServiceLoader.load(IPayService::class.java)` 的内部流程：

1. 读取 `META-INF/services/接口全限定名` 文件
2. 逐行解析实现类的全限定名
3. 通过 `Class.forName()` + `newInstance()` 反射创建实例
4. 返回 `Iterator<S>` 供调用方遍历

```java
// ServiceLoader 核心源码简化
private class LazyIterator implements Iterator<S> {
    private Class<S> service;
    private Enumeration<URL> configs = null;
    private Iterator<String> pending = null;

    LazyIterator(Class<S> service) {
        this.service = service;
    }

    @Override
    public boolean hasNext() {
        if (configs == null) {
            String fullName = "META-INF/services/" + service.getName();
            configs = classLoader.getResources(fullName);
        }
        while (pending == null || !pending.hasNext()) {
            if (!configs.hasMoreElements()) return false;
            pending = parse(configs.nextElement()); // 解析文件内容
        }
        return true;
    }

    @Override
    public S next() {
        String className = pending.next();
        Class<?> clazz = Class.forName(className, false, classLoader);
        return service.cast(clazz.newInstance()); // 反射创建
    }
}
```

#### Android 上的注意事项

- Android 的 ClassLoader 与标准 JVM 不同，但 `ServiceLoader` 在 Android 上可正常工作
- 每次 `ServiceLoader.load()` 都会重新读取文件 + 反射，**性能较差**
- Google 的 [[AutoService]] 注解处理器可自动生成 `META-INF/services` 文件

```java
// 使用 AutoService 自动注册
@AutoService(IPayService.class)
public class PayServiceImpl implements IPayService {
    @Override
    public boolean pay(String orderId, long amount) {
        // 支付逻辑
        return true;
    }
}
```

### 2.3 接口下沉到 base 模块的设计

接口下沉是最朴素但非常实用的方案。核心思想：

```
┌──────────────────────────────────────────────┐
│                   app (壳)                    │
├──────────┬──────────┬──────────┬─────────────┤
│ module-  │ module-  │ module-  │ module-     │
│ order    │ pay      │ account  │ logistics   │
├──────────┴──────────┴──────────┴─────────────┤
│              lib-base / lib-common            │
│  ┌─────────────────────────────────────────┐  │
│  │ IPayService / IOrderService / ...       │  │
│  │ ServiceManager (注册表)                  │  │
│  └─────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

所有业务模块都依赖 `lib-base`，接口定义在 `lib-base` 中，实现在各业务模块中。通过一个 **ServiceManager** 做运行时注册与查找：

```kotlin
// lib-base 中定义
object ServiceManager {
    private val services = ConcurrentHashMap<Class<*>, Any>()

    fun <T> register(clazz: Class<T>, impl: T) {
        services[clazz] = impl as Any
    }

    @Suppress("UNCHECKED_CAST")
    fun <T> get(clazz: Class<T>): T? {
        return services[clazz] as? T
    }

    // 带默认值的安全获取
    inline fun <reified T> getOrThrow(): T {
        return get(T::class.java) ?: throw IllegalStateException(
            "${T::class.java.simpleName} 未注册，请检查模块初始化"
        )
    }
}
```

### 3.2 SPI 方案

```java
// lib-base 中定义接口
public interface IAccountService {
    boolean isLoggedIn();
    String getUserId();
    void logout();
}

// module-account 中实现
@AutoService(IAccountService.class)
public class AccountServiceImpl implements IAccountService {
    @Override
    public boolean isLoggedIn() { return TokenStore.token != null; }
    @Override
    public String getUserId() { return TokenStore.userId; }
    @Override
    public void logout() {
        TokenStore.clear();
    }
}

// module-order 中调用（无需依赖 module-account）
public class AccountServiceFinder {
    private static volatile IAccountService service;

    public static IAccountService getService() {
        if (service == null) {
            synchronized (AccountServiceFinder.class) {
                if (service == null) {
                    Iterator<IAccountService> iterator =
                        ServiceLoader.load(IAccountService.class).iterator();
                    if (iterator.hasNext()) {
                        service = iterator.next();
                    } else {
                        throw new IllegalStateException("IAccountService 实现未找到");
                    }
                }
            }
        }
        return service;
    }
}

// 使用
if (AccountServiceFinder.getService().isLoggedIn()) {
    // 已登录，展示订单
}
```

### 3.4 三种方案对比表格

| 对比维度 | ARouter | SPI (ServiceLoader) | 接口下沉 |
|---------|---------|---------------------|----------|
| 页面跳转 | ✅ 原生支持，参数自动注入 | ❌ 不支持 | ❌ 需自行封装 |
| 服务调用 | ✅ IProvider 机制 | ✅ 接口 + 实现 | ✅ 接口 + 实现 |
| 编译期检查 | ⚠️ path 字符串无编译检查 | ✅ 接口类型安全 | ✅ 接口类型安全 |
| 性能 | 首次反射，后续缓存 | 每次 load 都反射（需缓存） | 直接调用，最优 |
| 初始化时机 | Application.onCreate | 首次调用时 | Application.onCreate |
| 拦截器 | ✅ 内置拦截器链 | ❌ 无 | ❌ 需自行实现 |
| 降级处理 | ✅ DegradeService | ❌ 无 | ⚠️ 需自行处理 null |
| 依赖 | 三方库 ~120KB | JDK 内置 | 无额外依赖 |
| Gradle 兼容 | ⚠️ 8.x 需适配 | ✅ 无问题 | ✅ 无问题 |
| 推荐场景 | 页面路由 + 复杂服务发现 | 插件化扩展点 | 模块间简单服务调用 |

> **实践建议**：大多数项目采用 **ARouter（页面跳转）+ 接口下沉（服务调用）** 的组合方案，兼顾灵活性与类型安全。

## 四、面试题

### Q1：ARouter 是如何实现编译期隔离、运行期通信的？

**A**：ARouter 利用 APT（注解处理器）在每个模块的编译期扫描 `@Route` 注解，为每个模块独立生成路由表类（`ARouter$$Group$$xxx`、`ARouter$$Root$$xxx`）。模块之间不需要互相依赖，只需要共同依赖 `arouter-api`。运行时，`ARouter.init()` 通过扫描 dex 文件（或 Gradle 插件注入）收集所有模块的 Root 类，合并成全局路由表。调用 `navigation()` 时，根据 path 查表找到目标 Activity/IProvider 的 Class，通过反射或 Intent 完成跳转/调用。核心设计是 **分组懒加载**——启动时只加载 group 索引，具体路由信息在首次访问时才加载。

### Q2：ARouter 的 IProvider 和 SPI 的 ServiceLoader 有什么区别？

**A**：

| 维度 | IProvider | ServiceLoader |
|------|-----------|---------------|
| 注册方式 | `@Route` 注解，APT 生成路由表 | `META-INF/services` 文件 |
| 发现机制 | 路由表查找 | 文件扫描 + 反射 |
| 生命周期 | 单例，首次获取时 `init()` | 每次 `load()` 创建新实例 |
| 缓存 | 框架内置缓存 | 需自行缓存 |
| 拦截能力 | 支持拦截器链 | 不支持 |

IProvider 本质上是 ARouter 对 SPI 思想的增强实现，增加了缓存、拦截、降级等能力。

### Q3：接口下沉方案中，如何避免 base 模块过度膨胀？

**A**：这是接口下沉最大的痛点。解决策略：

1. **分层抽取**：将接口从 `lib-base` 拆分到独立的 `lib-export` 模块，每个业务模块可以有自己的 export 模块（如 `module-order-export`），只暴露接口和数据类
2. **最小接口原则**：接口只定义跨模块必须的方法，模块内部逻辑不暴露
3. **数据类独立**：跨模块传输的 DTO 放在 export 模块，而非 base
4. **定期审查**：CI 中检测 base 模块的类数量和方法数，超过阈值告警

```
module-order-export/   ← 只有接口和 DTO
  ├── IOrderService.kt
  └── OrderDTO.kt

module-order/          ← 实现，依赖自己的 export
  └── OrderServiceImpl.kt
```

### Q4：ARouter 的拦截器是在什么线程执行的？能做耗时操作吗？

**A**：ARouter 的拦截器默认在 **子线程** 中执行（通过内部线程池 `LogisticsCenter.executor`）。因此可以做耗时操作（如网络请求判断登录态），但需要注意：

- 拦截器有 **超时机制**（默认 300 秒），超时会触发 `onInterrupt`
- 如果需要弹 Dialog 等 UI 操作，必须切回主线程
- 拦截器按 `priority` 值从小到大排序执行，值越小优先级越高
- `callback.onContinue()` 和 `callback.onInterrupt()` 必须且只能调用一次

### Q5：为什么 ARouter 要设计分组懒加载？直接全量加载不行吗？

**A**：全量加载意味着 App 启动时就要反射创建所有路由表类并填充 HashMap。假设有 200 个页面分布在 20 个 group 中：

- 全量加载：启动时创建 20 个 Group 类实例，填充 200 条路由记录
- 分组懒加载：启动时只加载 Root 表（20 条 group → class 映射），用户实际访问某个 group 时才加载该 group 的路由

在大型 App 中（如淘宝有上千个页面），分组懒加载可以显著减少启动时间。这是一种典型的 **空间换时间 + 延迟初始化** 策略。

### Q6：如果不用 ARouter，如何自己实现一个简易路由框架？

**A**：核心三步：

```java
// 1. 定义注解
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME) // 简易版用 RUNTIME，正式版用 APT
public @interface MyRoute {
    String path();
}

// 2. 路由表（手动注册或反射扫描）
public class SimpleRouter {
    private static final HashMap<String, Class<? extends Activity>> table = new HashMap<>();

    public static void register(String path, Class<? extends Activity> clazz) {
        table.put(path, clazz);
    }

    public static void navigate(Context context, String path, Bundle extras) {
        Class<? extends Activity> clazz = table.get(path);
        if (clazz == null) {
            Toast.makeText(context, "页面未找到: " + path, Toast.LENGTH_SHORT).show();
            return;
        }
        Intent intent = new Intent(context, clazz);
        if (extras != null) {
            intent.putExtras(extras);
        }
        context.startActivity(intent);
    }
}

// 3. 使用
SimpleRouter.register("/order/detail", OrderDetailActivity.class);
Bundle bundle = new Bundle();
bundle.putString("orderId", "123");
SimpleRouter.navigate(this, "/order/detail", bundle);
```

这个简易版缺少 APT 自动注册、拦截器、降级、参数自动注入等能力，但展示了路由框架的核心思想：**字符串 path → Class 映射 → Intent 跳转**。

## 五、实战与踩坑

### 5.1 ARouter 的 Gradle 8.x 兼容问题

Gradle 8.x 默认禁用了 `Transform API`，而 ARouter 的 `arouter-register` 插件依赖 Transform 在编译期自动注册路由表。升级后会遇到：

```
> Could not find method transform() for arguments
> [com.alibaba.android.arouter.register.core.RegisterTransform]
```

**解决方案**：

```kotlin
// 方案一：使用社区适配版本（推荐）
// build.gradle.kts (project)
plugins {
    // 使用支持 AGP 8.x 的 fork 版本
    id("com.github.ARouter-register") version "1.0.0-agp8" apply false
}

// 方案二：放弃 arouter-register 插件，改用运行时扫描 dex
// Application 中初始化时会自动 fallback 到 dex 扫描
// 缺点：首次启动慢 200-500ms
ARouter.init(application) // 无插件时自动扫描 dex

// 方案三：迁移到 KSP + 手动初始化
// 在 Application 中手动加载各模块的路由表
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        ARouter.init(this)
        // 如果插件失效，手动注册（不推荐，模块多时维护成本高）
        // LogisticsCenter.register("com.example.module_order")
    }
}
```

**kapt → KSP 迁移注意事项**：

```kotlin
// ❌ kapt 方式（Gradle 8.x 中 kapt 仍可用但已 deprecated）
kapt("com.alibaba:arouter-compiler:1.5.8")
kapt {
    arguments { arg("AROUTER_MODULE_NAME", project.name) }
}

// ✅ KSP 方式（需要 ARouter 支持 KSP 的版本或社区 fork）
ksp("com.alibaba:arouter-compiler:1.5.8") // 需确认版本支持 KSP
ksp {
    arg("AROUTER_MODULE_NAME", project.name)
}
```

> ⚠️ 截至目前，ARouter 官方对 KSP 的支持仍不完善。大型项目可考虑迁移到 **TheRouter**（美团开源）或 **WMRouter**（美团外卖），它们对 AGP 8.x 和 KSP 的支持更好。参考 [[TheRouter]] [[WMRouter]]。

### 5.2 SPI 的性能问题

`ServiceLoader` 的性能陷阱在于 **每次调用 `load()` 都会重新扫描文件和反射创建实例**：

```java
// ❌ 错误用法：每次调用都 load
public IPayService getPayService() {
    return ServiceLoader.load(IPayService.class).iterator().next();
    // 每次都：读文件 → Class.forName → newInstance
}

// ✅ 正确用法：缓存实例
public class SPIServiceManager {
    private static final ConcurrentHashMap<Class<?>, Object> cache = new ConcurrentHashMap<>();

    @SuppressWarnings("unchecked")
    public static <T> T get(Class<T> clazz) {
        return (T) cache.computeIfAbsent(clazz, key -> {
            Iterator<T> iterator = ServiceLoader.load(clazz).iterator();
            if (iterator.hasNext()) {
                return iterator.next();
            }
            throw new IllegalStateException(clazz.getSimpleName() + " 的 SPI 实现未找到");
        });
    }

    // 预加载（可在 Application.onCreate 中调用）
    public static void preload(Class<?>... classes) {
        for (Class<?> clazz : classes) {
            get(clazz);
        }
    }
}
```

**性能对比测试**（Pixel 6, Android 14）：

| 操作 | 耗时 |
|------|------|
| ServiceLoader.load() 首次 | ~3ms |
| ServiceLoader.load() 重复调用 | ~1.5ms |
| 缓存后直接获取 | ~0.001ms |
| ARouter.navigation(IProvider) 首次 | ~5ms |
| ARouter.navigation(IProvider) 缓存后 | ~0.01ms |
| 接口下沉 ServiceManager.get() | ~0.001ms |

> 结论：SPI 必须配合缓存使用。如果对启动性能敏感，接口下沉方案的直接 HashMap 查找是最快的。

### 5.3 接口下沉导致 base 膨胀

随着业务增长，`lib-base` 容易变成"垃圾场"：

```
lib-base/
  ├── IOrderService.kt
  ├── IPayService.kt
  ├── IAccountService.kt
  ├── ILogisticsService.kt
  ├── ICartService.kt
  ├── ICouponService.kt
  ├── OrderDTO.kt
  ├── PayDTO.kt
  ├── UserDTO.kt
  ├── ... (几十个接口和 DTO)
  ├── NetworkUtils.kt      ← 工具类也塞进来了
  ├── ImageLoader.kt       ← 图片加载也塞进来了
  └── Constants.kt         ← 全局常量也塞进来了
```

**治理方案：export 模块拆分**

```kotlin
// settings.gradle.kts
include(":lib-base")              // 真正的基础库：网络、图片、工具
include(":module-order-export")   // 订单模块对外暴露的接口和 DTO
include(":module-pay-export")     // 支付模块对外暴露的接口和 DTO

include(":module-pay")            // 支付模块实现

// module-order/build.gradle.kts
dependencies {
    implementation(project(":lib-base"))
    implementation(project(":module-order-export"))
    // 需要调用支付服务时，只依赖 export
    implementation(project(":module-pay-export"))
}

// module-pay-export/build.gradle.kts
dependencies {
    implementation(project(":lib-base")) // export 模块只依赖 base
}
```

// buildSrc 中的自定义 Gradle Task
tasks.register("checkBaseModuleSize") {

        val ktFiles = baseDir.walkTopDown().filter { it.extension == "kt" }.toList()
        val interfaceCount = ktFiles.count { it.readText().contains("interface ") }

            )
        }
    }
}

### 5.4 综合推荐方案

```kotlin
// 大型项目推荐组合
// 1. 页面跳转 → ARouter（或 TheRouter）
ARouter.getInstance().build("/order/detail")
    .withString("orderId", id)
    .navigation()

// 2. 跨模块服务调用 → 接口下沉 + export 模块
val payService = ServiceManager.getOrThrow<IPayService>()
payService.pay(orderId, amount)

// 3. 插件化扩展点 → SPI（如日志、埋点策略）
val logStrategies = ServiceLoader.load(ILogStrategy::class.java).toList()
logStrategies.forEach { it.log(level, tag, message) }
```
