# 公共组件抽取与复用

> 在 [[1-组件化架构设计与拆分策略]] 中，公共组件是整个组件化体系的基石。合理地抽取和复用公共组件，能够显著降低代码冗余、提升开发效率、保证技术栈一致性。本文聚焦公共组件的分类、抽取原则、分层设计、版本管理与 API 设计等核心话题。

---

## 一、核心概念

### 1.1 公共组件的分类

在 Android 组件化工程中，公共组件通常可划分为以下几类：

| 分类 | 职责 | 典型示例 |
|------|------|----------|
| UI 组件 | 提供统一的视觉控件与交互规范 | TitleBar、Dialog、EmptyView、刷新控件 |
| 工具库 | 通用工具方法，与平台/业务无关 | StringUtils、DateUtils、加解密、JSON 解析 |
| 网络层 | 统一的网络请求框架与拦截器 | OkHttp 封装、Retrofit ServiceFactory、Token 刷新 |
| 图片加载 | 图片加载与缓存策略的统一封装 | Glide/Coil 封装、圆角/占位图策略 |
| 埋点组件 | 统一的数据上报与事件追踪 | 页面曝光、点击事件、性能埋点 |

````
┌─────────────────────────────────────────┐
│            业务模块 A / B / C            │
├─────────────────────────────────────────┤
│  lib-ui  │ lib-net │ lib-image │ lib-track │
├─────────────────────────────────────────┤
│              lib-common (工具库)          │
└─────────────────────────────────────────┘
````
### 1.2 抽取原则

公共组件的抽取需要遵循三大原则：

**原则一：通用性**
组件必须被 2 个及以上业务模块使用，才有抽取价值。单一模块使用的代码应留在模块内部。

**原则二：稳定性**
公共组件的 API 变更频率应远低于业务模块。频繁变动的接口不适合下沉到公共层。

**原则三：无业务逻辑**
公共组件不应包含任何业务判断。业务差异通过接口回调、策略模式或配置项注入。

````java
// ✅ 正确：纯粹的工具方法，无业务逻辑
public class DeviceUtils {
    public static int getScreenWidth(Context context) {
        DisplayMetrics dm = context.getResources().getDisplayMetrics();
        return dm.widthPixels;
    }

    public static String getDeviceId(Context context) {
        String id = Settings.Secure.getString(
            context.getContentResolver(),
            Settings.Secure.ANDROID_ID
        );
        return id != null ? id : "";
    }
}

// ❌ 错误：工具类中混入了业务逻辑
public class DeviceUtils {
    public static String getDeviceId(Context context) {
        String id = Settings.Secure.getString(
            context.getContentResolver(), Settings.Secure.ANDROID_ID
        );
        if (id == null) id = "";
        // 业务逻辑不应出现在公共组件中
        if (UserManager.isVipUser()) {
            reportToVipServer(id);
        }
        return id;
    }
}
````
### 1.3 判断是否应该抽取的决策流程

````
代码片段 → 是否被多模块使用？
              ├─ 否 → 保留在业务模块
              └─ 是 → 是否包含业务逻辑？
                        ├─ 是 → 剥离业务逻辑后再抽取
                        └─ 否 → API 是否稳定？
                                  ├─ 否 → 先稳定 API 再下沉
                                  └─ 是 → ✅ 抽取为公共组件
````

## 二、原理与实践

### 2.1 公共组件的分层设计

参考 [[1-组件化架构设计与拆分策略]]，公共组件自身也需要分层，避免形成一个巨大的 `lib-common`：

````
┌───────────────────────────────────┐
│     lib-ui（依赖 lib-common）      │  ← 上层公共组件
│     lib-net（依赖 lib-common）     │
│     lib-image（依赖 lib-common）   │
│     lib-track（依赖 lib-common）   │
├───────────────────────────────────┤
│     lib-common（纯工具，零依赖）    │  ← 基础公共组件
└───────────────────────────────────┘
````
分层规则：
- `lib-common`：最底层，只包含纯 Kotlin/Java 工具，不依赖 Android Framework 以外的三方库
- 上层公共组件：可依赖 `lib-common`，但彼此之间不应产生横向依赖
- 业务模块按需依赖所需的公共组件，而非依赖一个全量包

````kotlin
// lib-common/build.gradle.kts — 零三方依赖
plugins {
    id("java-library")
    id("org.jetbrains.kotlin.jvm")
}

dependencies {
    // 仅依赖 Kotlin 标准库
    implementation("org.jetbrains.kotlin:kotlin-stdlib")
}
````
````kotlin
// lib-net/build.gradle.kts — 依赖 lib-common + 网络库
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
}

dependencies {
    implementation(project(":lib-common"))
    implementation("com.squareup.okhttp3:okhttp:4.12.0")
    implementation("com.squareup.retrofit2:retrofit:2.9.0")
}
````
### 2.2 版本管理：独立版本 vs 统一版本

参考 [[6-版本管理与发布]]，公共组件的版本管理有两种主流策略：

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 独立版本 | 各组件独立迭代，影响范围小 | 版本碎片化，兼容性维护成本高 | 大型团队、组件成熟度差异大 |
| 统一版本（BOM） | 版本一致，集成简单 | 一个组件变更触发全量发布 | 中小团队、组件耦合度较高 |

**BOM（Bill of Materials）统一版本示例：**

````kotlin
// bom/build.gradle.kts
plugins {
    id("java-platform")
}

dependencies {
    constraints {
        api("com.example:lib-common:1.3.0")
        api("com.example:lib-net:1.3.0")
        api("com.example:lib-ui:1.3.0")
        api("com.example:lib-image:1.3.0")
        api("com.example:lib-track:1.3.0")
    }
}
````
````kotlin
// 业务模块使用 BOM
dependencies {
    implementation(platform("com.example:lib-bom:1.3.0"))
    implementation("com.example:lib-net")   // 无需指定版本
    implementation("com.example:lib-ui")
}
````
### 2.3 API 设计：面向接口与扩展点

参考 [[1-API设计原则]]，公共组件的 API 设计直接决定了组件的可复用性和可维护性。

**核心原则：面向接口编程，预留扩展点。**

````kotlin
// ========== 图片加载组件：面向接口设计 ==========

// 1. 定义接口（在 lib-common 或独立 api 模块中）
interface ImageLoader {
    fun load(context: Context, url: String, target: ImageView)
    fun load(context: Context, request: ImageRequest, target: ImageView)
    fun preload(context: Context, url: String)
    fun clearCache(context: Context)
}

data class ImageRequest(
    val url: String,
    val placeholder: Int = 0,
    val error: Int = 0,
    val cornerRadius: Float = 0f,
    val isCircle: Boolean = false,
    val cacheStrategy: CacheStrategy = CacheStrategy.ALL
)

enum class CacheStrategy { ALL, NONE, MEMORY_ONLY, DISK_ONLY }

// 2. 提供默认实现（在 lib-image 中）
class CoilImageLoader : ImageLoader {
    override fun load(context: Context, url: String, target: ImageView) {
        target.load(url) {
            crossfade(true)
        }
    }

    override fun load(context: Context, request: ImageRequest, target: ImageView) {
        target.load(request.url) {
            crossfade(true)
            if (request.placeholder != 0) placeholder(request.placeholder)
            if (request.error != 0) error(request.error)
            if (request.isCircle) transformations(CircleCropTransformation())
            else if (request.cornerRadius > 0) {
                transformations(RoundedCornersTransformation(request.cornerRadius))
            }
        }
    }

    override fun preload(context: Context, url: String) {
        val request = coil.request.ImageRequest.Builder(context)
            .data(url).build()
        context.imageLoader.enqueue(request)
    }

    override fun clearCache(context: Context) {
        context.imageLoader.memoryCache?.clear()
    }
}

// 3. 通过 ServiceLoader 或手动注册提供实例
object ImageLoaderFactory {
    private var loader: ImageLoader = CoilImageLoader()

    fun init(loader: ImageLoader) {
        this.loader = loader
    }

    fun get(): ImageLoader = loader
}
````
**网络层的扩展点设计：**

````java
// 网络层提供拦截器扩展点，业务模块可注入自定义行为
public interface NetworkInitializer {
    String baseUrl();
    default List<Interceptor> interceptors() { return Collections.emptyList(); }
    default List<Interceptor> networkInterceptors() { return Collections.emptyList(); }
    default long connectTimeout() { return 15L; }
    default long readTimeout() { return 15L; }
}

public class RetrofitManager {
    private static Retrofit retrofit;

    public static void init(NetworkInitializer initializer) {
        OkHttpClient.Builder builder = new OkHttpClient.Builder()
            .connectTimeout(initializer.connectTimeout(), TimeUnit.SECONDS)
            .readTimeout(initializer.readTimeout(), TimeUnit.SECONDS);
        for (Interceptor i : initializer.interceptors()) {
            builder.addInterceptor(i);
        }
        for (Interceptor i : initializer.networkInterceptors()) {
            builder.addNetworkInterceptor(i);
        }
        OkHttpClient client = builder.build();

        retrofit = new Retrofit.Builder()
            .baseUrl(initializer.baseUrl())
            .client(client)
            .addConverterFactory(GsonConverterFactory.create())
            .build();
    }

    public static <T> T create(Class<T> service) {
        return retrofit.create(service);
    }
}
````

## 三、面试题

### Q1：公共组件抽取的标准是什么？如何判断一段代码是否应该下沉到公共层？

**A：** 需要同时满足三个条件：
1. **通用性**：被 2 个及以上业务模块使用
2. **稳定性**：API 变更频率低，不会因业务迭代频繁修改
3. **无业务逻辑**：不包含任何业务判断，业务差异通过接口/回调/配置注入

实际操作中，可以先在业务模块中开发，当第二个模块需要相同功能时，再通过 "Rule of Three" 原则决定是否抽取。过早抽取可能导致抽象不准确。

### Q2：公共组件之间可以互相依赖吗？如何处理？

**A：** 应尽量避免公共组件之间的横向依赖。如果确实存在依赖关系，需要遵循分层原则：

````
lib-ui 可以依赖 lib-image（UI 组件需要加载图片）
lib-net 不应依赖 lib-ui（网络层不应感知 UI）
````
处理方式：
- 提取共同依赖到更底层的 `lib-common`
- 通过接口隔离，在运行时注入实现
- 使用依赖倒置原则，上层定义接口，下层提供实现

### Q3：独立版本和统一版本（BOM）各自的优缺点？你们项目用的哪种？

**A：**
- **独立版本**：每个组件有自己的版本号（如 lib-net:2.1.0、lib-ui:1.5.0）。优点是各组件独立迭代，缺点是版本碎片化严重，可能出现兼容性问题。
- **统一版本（BOM）**：所有组件统一版本号。优点是集成简单、兼容性有保障，缺点是一个组件变更可能触发全量发布。

实践建议：中小团队优先使用 BOM 统一版本；大型团队中成熟稳定的组件可以独立版本，活跃迭代的组件用统一版本。参考 [[6-版本管理与发布]]。

### Q4：如何避免公共组件变成 God Module（上帝模块）？

**A：** God Module 是指一个公共模块承担了过多职责，导致体积膨胀、编译缓慢、修改风险高。避免方法：

1. **单一职责拆分**：按功能域拆分为 lib-net、lib-ui、lib-image 等，而非一个 lib-common 包含所有
2. **定期审查**：检查模块的类数量和依赖数量，超过阈值时考虑拆分
3. **依赖分析**：使用 `./gradlew :lib-common:dependencies` 检查是否引入了不必要的三方库
4. **代码行数监控**：对公共模块设置代码行数告警阈值

### Q5：公共组件的 API 发生 Breaking Change 时如何处理？

**A：** 分三步走：

````kotlin
// Step 1: 新增新 API，标记旧 API 为 Deprecated
@Deprecated(
    message = "Use load(context, ImageRequest, target) instead",
    replaceWith = ReplaceWith("load(context, ImageRequest(url), target)")
)
fun loadImage(url: String, target: ImageView)

// Step 2: 给业务方 1-2 个版本的迁移窗口期
// 在 CHANGELOG 和群通知中明确标注

// Step 3: 在约定版本中移除旧 API（遵循语义化版本，Major 版本升级）
````
关键点：永远不要在 Minor 版本中删除已有 API，遵循语义化版本规范。

### Q6：埋点组件如何设计才能做到业务无侵入？

**A：** 埋点组件应采用 AOP + 接口抽象的方式：

````java
// 1. 定义埋点接口
public interface TrackReporter {
    void reportEvent(TrackEvent event);
    void reportPageView(String pageName, Map<String, String> params);
}

public class TrackEvent {
    public final String eventId;
    public final Map<String, String> params;
    public final long timestamp;

    public TrackEvent(String eventId, Map<String, String> params) {
        this.eventId = eventId;
        this.params = params != null ? params : Collections.emptyMap();
        this.timestamp = System.currentTimeMillis();
    }
}

// 2. 页面曝光通过 Lifecycle 自动采集
public class AutoTrackLifecycleObserver implements DefaultLifecycleObserver {
    private final TrackReporter reporter;

    public AutoTrackLifecycleObserver(TrackReporter reporter) {
        this.reporter = reporter;
    }

    @Override
    public void onResume(@NonNull LifecycleOwner owner) {
        String pageName = owner.getClass().getSimpleName();
        reporter.reportPageView(pageName, Collections.emptyMap());
    }
}

// 3. 业务模块零代码接入页面埋点
// 在 Application 中注册
ProcessLifecycleOwner.get().getLifecycle().addObserver(
    new AutoTrackLifecycleObserver(new TrackReporterImpl())
);
````
核心思路：通过 Lifecycle 感知页面生命周期实现自动化，通过接口抽象隔离具体的上报 SDK（友盟、Firebase 等），业务模块无需手动调用埋点代码。

## 四、实战与踩坑

### 4.1 公共组件的版本升级策略

版本升级是公共组件维护中最容易出问题的环节。推荐采用以下策略：

**语义化版本（SemVer）：**
- `MAJOR.MINOR.PATCH`（如 2.1.3）
- MAJOR：不兼容的 API 变更
- MINOR：向后兼容的功能新增
- PATCH：向后兼容的 Bug 修复

**灰度升级流程：**

````
1. 发布 RC 版本（如 2.0.0-rc01）
2. 选择 1-2 个业务模块先行接入验证
3. 验证通过后发布正式版本
4. 其余业务模块在下一个迭代周期内完成升级
````
**强制升级 vs 柔性升级：**

````kotlin
// 在 CI 中检查公共组件版本，超过 N 个版本未升级则构建告警
// build.gradle.kts 中通过自定义 Plugin 实现
class VersionCheckPlugin : Plugin<Project> {
    override fun apply(project: Project) {
        project.afterEvaluate {
            configurations.all {
                resolutionStrategy {
                    eachDependency {
                        if (requested.group == "com.example" &&
                            requested.name.startsWith("lib-")) {
                            // 检查版本是否在允许范围内
                            val minVersion = getMinAllowedVersion(requested.name)
                            if (requested.version != null &&
                                compareVersions(requested.version!!, minVersion) < 0) {
                                throw GradleException(
                                    "${requested.name} 版本 ${requested.version} " +
                                    "低于最低要求 $minVersion，请升级"
                                )
                            }
                        }
                    }
                }
            }
        }
    }
}
````
### 4.2 避免 God Module

God Module 是组件化中最常见的反模式。以下是真实踩坑案例与解决方案：

**踩坑场景：** 项目初期将所有公共代码放入 `lib-common`，随着业务增长，该模块膨胀到 500+ 类、引入 20+ 三方依赖，编译耗时占整体 30%。

**拆分策略：**

````
拆分前：
lib-common/
├── network/        → 抽取为 lib-net
├── image/          → 抽取为 lib-image
├── ui/widget/      → 抽取为 lib-ui
├── track/          → 抽取为 lib-track
├── utils/          → 保留在 lib-common
└── extensions/     → 保留在 lib-common

拆分后：
lib-common   (50 类, 0 三方依赖)   ← 纯工具
lib-net      (30 类, OkHttp/Retrofit)
lib-ui       (40 类, Material/RecyclerView)
lib-image    (15 类, Coil)
lib-track    (20 类, 无三方依赖)
````
**拆分时的注意事项：**

````kotlin
// 问题：拆分后业务模块的 import 路径变化，需要批量修改
// 解决：使用 typealias 做过渡，给业务方迁移时间

// lib-common 中保留 typealias（过渡期）
@Deprecated("已迁移至 lib-net，请修改 import")
typealias HttpClient = com.example.lib.net.HttpClient

@Deprecated("已迁移至 lib-image，请修改 import")
typealias ImageLoader = com.example.lib.image.ImageLoader
````
### 4.3 组件间的依赖传递控制

依赖传递是组件化中的隐形杀手。如果不加控制，业务模块可能间接依赖了不需要的三方库。

**问题示例：**

````kotlin
// lib-net 依赖了 Gson
// 业务模块依赖 lib-net 后，自动获得了 Gson 的传递依赖
// 当 lib-net 将 Gson 替换为 Moshi 时，业务模块中直接使用 Gson 的代码全部编译失败
````
**解决方案：使用 `implementation` 而非 `api` 控制依赖可见性**

````kotlin
// lib-net/build.gradle.kts
dependencies {
    // ✅ implementation：Gson 不会传递给依赖 lib-net 的模块
    implementation("com.google.code.gson:gson:2.10.1")

    // ❌ api：Gson 会传递给所有依赖 lib-net 的模块
    // api("com.google.code.gson:gson:2.10.1")

    // 如果确实需要暴露某个依赖（如 OkHttp 的 Interceptor 接口），用 api
    api("com.squareup.okhttp3:okhttp:4.12.0")
}
````
**依赖冲突排查：**

````bash
# 查看某个模块的完整依赖树
./gradlew :feature-home:dependencies --configuration runtimeClasspath

# 查找特定库的依赖来源
./gradlew :feature-home:dependencyInsight --dependency gson
````
**使用 Lint 规则防止公共组件引入不当依赖：**

````kotlin
// 自定义 Gradle 校验：确保 lib-common 不依赖 Android Framework
// settings.gradle.kts 或 buildSrc 中
fun validateCommonDependencies(project: Project) {
    project.afterEvaluate {
        val forbidden = listOf("com.android.support", "androidx.")
        configurations.getByName("implementation").dependencies.forEach { dep ->
            if (forbidden.any { dep.group?.startsWith(it) == true }) {
                throw GradleException(
                    "lib-common 不允许依赖 Android 库: ${dep.group}:${dep.name}"
                )
            }
        }
    }
}
````
### 4.4 其他常见踩坑

**踩坑 1：资源冲突**

多个公共组件定义了同名资源（如 `R.color.primary`），导致运行时取到错误的值。

````kotlin
// 解决：为每个公共组件设置资源前缀
// lib-ui/build.gradle.kts
android {
    resourcePrefix = "lib_ui_"
}
// 所有资源必须以 lib_ui_ 开头，否则编译告警
````
**踩坑 2：初始化顺序**

多个公共组件都需要在 Application.onCreate 中初始化，顺序不当导致崩溃。

````java
// 解决：使用 App Startup 管理初始化顺序与依赖
public class NetworkInitializer implements Initializer<RetrofitManager> {
    @NonNull
    @Override
    public RetrofitManager create(@NonNull Context context) {
        RetrofitManager.init(new DefaultNetworkConfig());
        return RetrofitManager.getInstance();
    }

    @NonNull
    @Override
    public List<Class<? extends Initializer<?>>> dependencies() {
        return Collections.emptyList(); // 无依赖，最先初始化
    }
}

public class ImageInitializer implements Initializer<ImageLoaderFactory> {
    @NonNull
    @Override
    public ImageLoaderFactory create(@NonNull Context context) {
        ImageLoaderFactory.init(new CoilImageLoader());
        return ImageLoaderFactory.getInstance();
    }

    @NonNull
    @Override
    public List<Class<? extends Initializer<?>>> dependencies() {
        // 依赖网络层先初始化（图片加载可能需要自定义 OkHttpClient）
        return Collections.singletonList(NetworkInitializer.class);
    }
}
````
**踩坑 3：ProGuard/R8 混淆规则分散**

每个公共组件都有自己的混淆规则，容易遗漏。

````kotlin
// 解决：在公共组件的 build.gradle.kts 中通过 consumerProguardFiles 自动携带
android {
    defaultConfig {
        consumerProguardFiles("proguard-rules.pro")
    }
}
// 业务模块依赖该组件时，混淆规则自动合并，无需手动配置
````
