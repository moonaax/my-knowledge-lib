# 组件化架构设计与拆分策略

> 本文系统梳理 Android 组件化架构的核心概念、工程实践、面试高频题与实战踩坑经验。
> 相关主题：[[2-组件间通信]] · [[6-组件独立运行与调试]] · [[ARouter 路由框架]]

## 一、核心概念

### 1.1 组件化 vs 模块化

| 维度 | 模块化（Modularization） | 组件化（Componentization） |
|------|------------------------|--------------------------|
| 关注点 | 按**功能职责**拆分代码 | 按**业务边界**拆分为可独立运行的单元 |
| 粒度 | 较细，如 `utils`、`network`、`database` | 较粗，如 `module-login`、`module-shop`、`module-im` |
| 独立运行 | 通常不可独立运行 | 可切换为 `application` 独立编译运行 |
| 依赖方向 | 上层依赖下层，单向 | 业务组件之间**禁止直接依赖**，通过路由/接口下沉通信 |
| 目标 | 代码复用、职责清晰 | 团队并行开发、编译加速、业务隔离 |

> 一句话总结：**模块化是手段，组件化是目标**。组件化是模块化在大型项目中的进一步演进。

### 1.2 分层架构

组件化项目通常采用四层架构：

````
┌─────────────────────────────────────────────┐
│                  app 壳工程                   │
│         (组装所有业务组件，无业务代码)           │
├──────────┬──────────┬──────────┬─────────────┤
│ module-  │ module-  │ module-  │  module-    │
│  login   │  shop    │   im     │  profile    │
│ (业务组件)│ (业务组件)│ (业务组件) │ (业务组件)   │
├──────────┴──────────┴──────────┴─────────────┤
│              基础业务组件层                     │
│   lib-common-ui / lib-share / lib-push       │
├──────────────────────────────────────────────┤
│              基础功能组件层（公共库）             │
│   lib-network / lib-image / lib-utils        │
│   lib-log / lib-storage / lib-router         │
└──────────────────────────────────────────────┘
````
各层职责说明：

| 层级 | 说明 | 示例 |
|------|------|------|
| **app 壳工程** | 仅负责组装，不包含业务逻辑；管理 `Application` 初始化与组件注册 | `app` |
| **业务组件层** | 按业务域拆分，可独立编译运行，组件间**不直接依赖** | `module-login`、`module-shop` |
| **基础业务组件层** | 与业务相关但被多个业务组件复用的能力 | `lib-share`（分享）、`lib-push`（推送） |
| **基础功能组件层** | 纯技术能力，与业务无关，可跨项目复用 | `lib-network`、`lib-image`、`lib-utils` |

> 依赖规则：**上层可以依赖下层，同层之间禁止直接依赖**。业务组件间通信参见 [[2-组件间通信]]。

### 1.3 组件拆分原则

#### 原则一：高内聚、低耦合

- **高内聚**：一个组件内部的代码围绕同一个业务域，所有相关的 Activity、Fragment、ViewModel、Repository、资源文件都放在同一个模块内。
- **低耦合**：组件对外暴露最小接口，内部实现细节不泄露。组件间通过 **路由**（如 [[ARouter 路由框架]]）或 **接口下沉** 通信。

````
✅ 正确：module-login 包含登录页、注册页、找回密码页、登录相关的网络请求
❌ 错误：module-login 直接引用 module-shop 中的商品详情页
````
#### 原则二：单一职责（SRP）

每个组件只负责一个业务域或一个技术能力：

````
✅ lib-network   → 只负责网络请求封装
✅ lib-image     → 只负责图片加载
✅ module-login  → 只负责登录注册业务
❌ lib-common    → 什么都往里塞的"万能模块"（应进一步拆分）
````
#### 原则三：接口隔离

通过 `api` 模块暴露组件能力，隐藏实现细节：

````
module-login/
├── module-login-api/    ← 对外暴露的接口与数据模型
│   ├── ILoginService.kt
│   └── LoginUserInfo.kt
└── module-login-impl/   ← 具体实现，外部不可直接依赖
    ├── LoginActivity.kt
    ├── LoginViewModel.kt
    └── LoginRepository.kt
````
#### 原则四：依赖方向单一

````
app → module-login-impl → module-login-api → lib-common
                        → lib-network
````
禁止出现反向依赖或循环依赖。

## 二、原理与实践

### 2.1 Gradle 多模块配置

#### settings.gradle.kts — 统一注册所有模块

````kotlin
// settings.gradle.kts
rootProject.name = "MyApp"

include(":app")

// 业务组件
include(":module-login:module-login-api")
include(":module-login:module-login-impl")
include(":module-shop:module-shop-api")
include(":module-shop:module-shop-impl")
include(":module-im:module-im-api")
include(":module-im:module-im-impl")

// 基础业务组件
include(":lib-share")
include(":lib-push")
include(":lib-common-ui")

// 基础功能组件
include(":lib-network")
include(":lib-image")
include(":lib-utils")
include(":lib-router")
````
#### 根 build.gradle.kts — 统一版本管理

````kotlin
// build.gradle.kts (root)
plugins {
    alias(libs.plugins.android.application) apply false
    alias(libs.plugins.android.library) apply false
    alias(libs.plugins.kotlin.android) apply false
}

// 通过 gradle.properties 控制组件独立运行
// gradle.properties:
// isModuleLogin=false
// isModuleShop=false
````
### 2.2 application / library 动态切换

这是组件化的核心技巧之一：让业务组件在开发阶段可以作为独立 App 运行调试，在集成阶段作为 Library 被 app 壳依赖。详见 [[6-组件独立运行与调试]]。

#### 组件级 build.gradle.kts

````kotlin
// module-login/module-login-impl/build.gradle.kts

// 从 gradle.properties 读取开关
val isModuleLogin: String by project

// 动态切换插件
if (isModuleLogin.toBoolean()) {
    plugins {
        id("com.android.application")
        id("org.jetbrains.kotlin.android")
    }
} else {
    plugins {
        id("com.android.library")
        id("org.jetbrains.kotlin.android")
    }
}

android {
    namespace = "com.example.module.login"
    compileSdk = 34

    defaultConfig {
        minSdk = 24
        // 仅在独立运行时需要 applicationId
        if (isModuleLogin.toBoolean()) {
            applicationId = "com.example.module.login"
        }
    }

    sourceSets {
        getByName("main") {
            if (isModuleLogin.toBoolean()) {
                // 独立运行时使用独立的 AndroidManifest（含 launcher Activity）
                manifest.srcFile("src/main/debug/AndroidManifest.xml")
            } else {
                manifest.srcFile("src/main/release/AndroidManifest.xml")
            }
        }
    }
}

dependencies {
    implementation(project(":module-login:module-login-api"))
    implementation(project(":lib-network"))
    implementation(project(":lib-common-ui"))
    implementation(project(":lib-router"))
    implementation(libs.androidx.core.ktx)
    implementation(libs.androidx.lifecycle.viewmodel)
}
````
> 更优雅的方式：使用自定义 Gradle 插件统一处理动态切换逻辑，避免每个组件重复配置。

#### 抽取公共配置插件（Convention Plugin）

````kotlin
// build-logic/convention/src/main/kotlin/ModuleConventionPlugin.kt
import org.gradle.api.Plugin
import org.gradle.api.Project

class ModuleConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            val moduleName = path.replace(":", ".")
                .removePrefix(".").replace("-impl", "")
            val propKey = "isModule${
                name.replaceFirstChar { it.uppercase() }
                    .replace("-impl", "")
            }"
            val isRunAlone = findProperty(propKey)?.toString()?.toBoolean() ?: false

            if (isRunAlone) {
                pluginManager.apply("com.android.application")
            } else {
                pluginManager.apply("com.android.library")
            }
            pluginManager.apply("org.jetbrains.kotlin.android")
        }
    }
}
````

> 关键规则：
> - `impl` 模块依赖自己的 `api` 模块
> - 业务组件间若需通信，只依赖对方的 `api` 模块（或通过路由）
> - 所有箭头**只能向下**，不可反向

### 2.4 api 模块暴露接口的设计

api 模块是组件化的精髓——它定义了组件对外的"契约"，实现了**接口与实现分离**。

#### 定义服务接口

````java
// module-login/module-login-api/src/.../ILoginService.java
public interface ILoginService {
    /** 是否已登录 */
    boolean isLoggedIn();

    /** 获取当前用户信息，未登录返回 null */
    LoginUserInfo getCurrentUser();

    /** 跳转到登录页 */
    void navigateToLogin(Context context);
}
````
#### 定义数据模型

````java
// module-login/module-login-api/src/.../LoginUserInfo.java
public class LoginUserInfo {
    public final long uid;
    public final String nickname;
    public final String avatar;
    public final String token;

    public LoginUserInfo(long uid, String nickname, String avatar, String token) {
        this.uid = uid;
        this.nickname = nickname;
        this.avatar = avatar;
        this.token = token;
    }
}
````
#### impl 模块提供实现

````java
// module-login/module-login-impl/src/.../LoginServiceImpl.java
public class LoginServiceImpl implements ILoginService {
    @Override
    public boolean isLoggedIn() {
        return !TokenManager.getToken().isEmpty();
    }

    @Override
    public LoginUserInfo getCurrentUser() {
        return UserCache.get();
    }

    @Override
    public void navigateToLogin(Context context) {
        context.startActivity(
            new Intent(context, LoginActivity.class)
        );
    }
}
````
#### 注册与发现（配合 ServiceLoader 或 DI 框架）

````java
// 使用简单的 ServiceManager 手动注册
public class ServiceManager {
    private static final Map<Class<?>, Object> services = new HashMap<>();

    public static <T> void register(Class<T> clazz, T impl) {
        services.put(clazz, impl);
    }

    @SuppressWarnings("unchecked")
    public static <T> T get(Class<T> clazz) {
        return (T) services.get(clazz);
    }
}

// 在 module-login-impl 的初始化中注册
ServiceManager.register(ILoginService.class, new LoginServiceImpl());

// 在 module-shop-impl 中使用（只依赖 module-login-api）
ILoginService loginService = ServiceManager.get(ILoginService.class);
if (loginService != null && loginService.isLoggedIn()) {
    LoginUserInfo user = loginService.getCurrentUser();
    // 展示用户信息
}
````
> 更完善的方案可结合 [[2-组件间通信]] 中的 ARouter `IProvider` 或 Koin/Hilt 依赖注入。

## 三、面试题

### Q1：组件化和模块化有什么区别？为什么要做组件化？

**A：**

- **模块化**侧重于代码层面的职责拆分（如 network、utils），目标是代码复用和职责清晰。
- **组件化**侧重于业务层面的隔离，每个业务组件可以独立编译运行，目标是：
  1. **团队并行开发**：不同团队负责不同组件，互不干扰
  2. **编译加速**：只编译修改的组件，增量编译更快
  3. **业务隔离**：组件间通过接口通信，降低耦合，方便业务的插拔和替换
  4. **独立测试**：组件可单独运行和调试，参见 [[6-组件独立运行与调试]]

### Q2：组件化项目中，业务组件之间如何通信？

**A：**

业务组件之间**禁止直接依赖**，常见通信方式：

1. **路由跳转**：通过 ARouter 等路由框架实现页面跳转，传递基本参数
2. **接口下沉 + ServiceLoader**：将接口定义在 api 模块中，impl 模块提供实现，通过 `ServiceManager` 或 DI 框架发现服务
3. **EventBus / LiveDataBus**：适合松耦合的事件通知，但不宜滥用

推荐组合：**路由（页面跳转）+ 接口下沉（能力调用）+ EventBus（事件通知）**。详见 [[2-组件间通信]]。

### Q3：如何实现组件的 application / library 动态切换？

**A：**

核心思路：通过 `gradle.properties` 中的布尔开关控制。

````kotlin
// gradle.properties
isModuleLogin=false  // false=library模式, true=独立app模式

// 组件 build.gradle.kts 中读取
val isModuleLogin: String by project
if (isModuleLogin.toBoolean()) {
    apply(plugin = "com.android.application")
} else {
    apply(plugin = "com.android.library")
}
````
同时需要准备两套 `AndroidManifest.xml`：
- `src/main/debug/AndroidManifest.xml`：包含 `launcher` Activity，独立运行用
- `src/main/release/AndroidManifest.xml`：仅声明组件内的 Activity，集成用

### Q4：组件化中如何解决资源名冲突？

**A：**

1. **resourcePrefix 约束**：在每个组件的 `build.gradle.kts` 中设置前缀

````kotlin
android {
    resourcePrefix = "login_"  // 所有资源必须以 login_ 开头
}
````
2. 这是一个 Lint 级别的约束，不会导致编译失败，但会在 IDE 中给出警告
3. 团队需要配合 Code Review 确保命名规范
4. 对于 `AndroidManifest.xml` 中的冲突，可通过 `tools:replace` 解决

### Q5：组件化项目中 Application 的初始化如何处理？

**A：**

问题：每个组件可能都需要在 `Application.onCreate()` 中做初始化，但组件不能直接访问 app 壳的 `Application`。

解决方案——**组件生命周期分发**：

````java
// lib-common 中定义接口
public interface IModuleInit {
    /** 高优先级初始化（主线程，阻塞） */
    void onInitHighPriority(Application app);
    /** 低优先级初始化（可延迟/异步） */
    void onInitLowPriority(Application app);
}

// 各组件实现
public class LoginModuleInit implements IModuleInit {
    @Override
    public void onInitHighPriority(Application app) {
        // 初始化登录相关 SDK
    }
    @Override
    public void onInitLowPriority(Application app) {
        // 预加载用户信息缓存
    }
}

// app 壳中统一调度
public class MyApp extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        List<IModuleInit> modules = Arrays.asList(
            new LoginModuleInit(),
            new ShopModuleInit(),
            new IMModuleInit()
        );
        for (IModuleInit module : modules) {
            module.onInitHighPriority(this);
        }
        // 低优先级可放到子线程或 IdleHandler
        Looper.myQueue().addIdleHandler(() -> {
            for (IModuleInit module : modules) {
                module.onInitLowPriority(MyApp.this);
            }
            return false;
        });
    }
}
````
也可以通过 APT/KSP 自动收集各组件的初始化类，避免手动注册。

### Q6：组件化拆分的粒度如何把握？拆得太细或太粗会有什么问题？

**A：**

| 问题 | 拆分太粗 | 拆分太细 |
|------|---------|---------|
| 耦合 | 组件内部耦合严重，难以维护 | 组件间交互频繁，接口爆炸 |
| 编译 | 单个组件编译慢 | 模块数量过多，Gradle sync 慢 |
| 团队 | 多人改同一组件，冲突多 | 一个小需求要改多个组件 |

**实践建议**：
1. 以**业务域**为边界拆分，一个业务域对应一个组件
2. 初期可以粗一些，随着业务增长再逐步细化
3. 如果一个组件超过 3 个人同时开发，考虑拆分
4. 如果一个需求需要同时修改超过 3 个组件，考虑合并

## 四、实战与踩坑

### 4.1 拆分粒度的权衡

#### 踩坑：一开始就过度拆分

某项目初期就拆出了 30+ 个模块，结果：
- Gradle sync 时间从 10s 涨到 60s+
- 每个小需求都要跨 4-5 个模块修改
- api 模块数量爆炸，维护成本极高

#### 推荐策略：渐进式拆分

````
阶段一（初期）：
app + lib-common + lib-network + lib-image
→ 先把基础能力抽出来

阶段二（业务增长）：
app + module-login + module-main + 基础库
→ 按核心业务域拆分

阶段三（团队扩大）：
app + N个业务组件 + M个基础组件
→ 按团队边界进一步细化

阶段四（成熟期）：
引入 api/impl 分离、Convention Plugin、CI 组件级构建
→ 工程化治理
````
#### 判断是否需要拆分的信号

````
✅ 需要拆分的信号：
- 模块代码量超过 5 万行
- 超过 3 个开发者同时修改同一模块
- 模块内存在明显的业务边界
- 需要在多个项目中复用某部分能力

❌ 不需要拆分的信号：
- 模块代码量不到 1 万行
- 只有 1-2 个人维护
- 拆分后组件间交互比组件内部逻辑还复杂
````
### 4.2 循环依赖的解决

#### 问题场景

````
module-login 需要调用 module-shop 的商品详情页（跳转）
module-shop 需要调用 module-login 的登录状态检查（能力）

→ 形成循环依赖：login ↔ shop
````
#### 解决方案一：接口下沉（推荐）

将双方需要的接口下沉到公共层：

````java
// lib-common-service/src/.../ILoginService.java
public interface ILoginService {
    boolean isLoggedIn();
    UserInfo getCurrentUser();
}

// lib-common-service/src/.../IShopService.java
public interface IShopService {
    void navigateToProductDetail(Context context, long productId);
}
````
依赖关系变为：

````
module-login-impl ──→ lib-common-service ←── module-shop-impl
       │                                            │
       └──── 实现 ILoginService      实现 IShopService ────┘
````
循环依赖消除，双方都只依赖公共的接口层。

#### 解决方案二：路由解耦

通过路由框架实现页面跳转，完全不需要依赖对方模块：

````java
// module-shop 中跳转到登录页（不依赖 module-login）
ARouter.getInstance()
    .build("/login/main")
    .navigation();

// module-login 中跳转到商品详情（不依赖 module-shop）
ARouter.getInstance()
    .build("/shop/detail")
    .withLong("productId", 12345L)
    .navigation();
````
#### 解决方案三：EventBus 事件通知

适合松耦合的通知场景：

````java
// 定义事件（放在 lib-common-service 中）
public class LoginSuccessEvent {
    public final long userId;
    public LoginSuccessEvent(long userId) { this.userId = userId; }
}

// module-login 发送事件
EventBus.getDefault().post(new LoginSuccessEvent(userId));

// module-shop 接收事件
@Subscribe(threadMode = ThreadMode.MAIN)
public void onLoginSuccess(LoginSuccessEvent event) {
    refreshCartInfo();
}
````
#### Gradle 层面检测循环依赖

````kotlin
// 在根 build.gradle.kts 中添加检测任务
tasks.register("checkCircularDependency") {
    doLast {
        val graph = mutableMapOf<String, MutableSet<String>>()
        subprojects.forEach { project ->
            val deps = project.configurations
                .filter { it.name == "implementation" }
                .flatMap { it.dependencies }
                .filterIsInstance<ProjectDependency>()
                .map { it.dependencyProject.path }
                .toMutableSet()
            graph[project.path] = deps
        }
        // DFS 检测环
        val visited = mutableSetOf<String>()
        val stack = mutableSetOf<String>()
        fun dfs(node: String): Boolean {
            visited.add(node)
            stack.add(node)
            graph[node]?.forEach { dep ->
                if (dep in stack) {
                    logger.error("循环依赖: $node → $dep")
                    return true
                }
                if (dep !in visited && dfs(dep)) return true
            }
            stack.remove(node)
            return false
        }
        graph.keys.forEach { if (it !in visited) dfs(it) }
    }
}
````
### 4.3 编译速度优化

组件化的一大目标就是加速编译，但如果配置不当反而会更慢。

#### 策略一：组件级增量编译

````kotlin
// gradle.properties — 开启并行编译与构建缓存
org.gradle.parallel=true
org.gradle.caching=true
org.gradle.configureondemand=true
org.gradle.daemon=true
org.gradle.jvmargs=-Xmx4g -XX:+UseParallelGC
````
#### 策略二：使用 api / implementation 正确声明依赖

````kotlin
dependencies {
    // ✅ implementation：依赖不传递，修改 lib-network 不会触发 module-shop 重编译
    implementation(project(":lib-network"))

    // ❌ api：依赖传递，lib-network 的任何改动都会导致依赖链上所有模块重编译
    api(project(":lib-network"))
}
````
> 原则：**默认用 `implementation`，只有需要暴露给上层的依赖才用 `api`**。

#### 策略三：组件 AAR 化

对于稳定的基础组件，发布为 AAR 到 Maven 仓库，避免每次都参与源码编译：

````kotlin
// lib-network/build.gradle.kts
plugins {
    id("com.android.library")
    id("maven-publish")
}

afterEvaluate {
    publishing {
        publications {
            create<MavenPublication>("release") {
                from(components["release"])
                groupId = "com.example.lib"
                artifactId = "network"
                version = "1.2.0"
            }
        }
        repositories {
            maven {
                url = uri("https://maven.example.com/releases")
            }
        }
    }
}

// 使用方切换为远程依赖
dependencies {
    // 开发阶段用源码依赖
    // implementation(project(":lib-network"))

    // 稳定后切换为 AAR 依赖
    implementation("com.example.lib:network:1.2.0")
}
````
#### 策略四：按需编译（只编译修改的组件）

````bash
# 只编译 module-login 组件（独立运行模式）
./gradlew :module-login:module-login-impl:assembleDebug

# 配合 CI，检测 Git diff 决定编译范围
CHANGED_MODULES=$(git diff --name-only HEAD~1 | \
  grep -oP '(module-\w+|lib-\w+)' | sort -u)
for mod in $CHANGED_MODULES; do
  ./gradlew :$mod:assembleDebug
done
````
#### 编译速度优化效果参考

| 优化手段 | 优化前 | 优化后 | 提升 |
|---------|--------|--------|------|
| 开启并行编译 | 180s | 120s | 33% |
| implementation 替代 api | 120s | 90s | 25% |
| 基础库 AAR 化 | 90s | 55s | 39% |
| 组件独立编译 | 55s | 15s | 73% |
