# 组件独立运行与调试

> 在 [[组件化架构设计与拆分策略]] 中我们完成了模块拆分，但如果每次调试一个小功能都要编译整个宿主 App，开发体验会急剧下降。本文聚焦于**如何让每个业务组件像独立 App 一样运行和调试**，这是组件化工程落地的关键一环。

---

## 一、核心概念

### 1.1 为什么需要组件独立运行

| 痛点 | 独立运行后的改善 |
|------|-----------------|
| **编译速度慢** — 全量编译动辄 3-5 分钟 | 单模块编译通常 < 30s，增量编译更快 |
| **调试困难** — 需要从宿主 App 层层跳转才能到达目标页面 | 组件自带入口 Activity，启动即达目标页 |
| **测试耦合** — 单元测试 / UI 测试依赖宿主环境 | 组件内可独立运行 instrumented test |
| **多人协作冲突** — 所有人改同一个 App 模块 | 各组件独立开发，CI 只编译变更模块 |

核心思路：让每个业务模块在开发阶段以 `com.android.application` 形式运行，在集成阶段以 `com.android.library` 形式被宿主依赖。

### 1.2 application / library 动态切换

Android Gradle Plugin 提供了两种插件：

- `com.android.application` — 可生成 APK，拥有 `applicationId`，可独立安装运行
- `com.android.library` — 生成 AAR，不能独立运行，只能被其他模块依赖

动态切换的本质就是**根据一个标志位，在 `build.gradle.kts` 中决定应用哪个插件**。这个标志位通常叫 `isModule`（或 `isRunAlone`）。

```kotlin
// module_login/build.gradle.kts
val isModule: Boolean = rootProject.extra.has("isModule")
        && rootProject.extra["isModule"].toString().toBoolean()

plugins {
    // 不能在 plugins {} 块中使用 if，需要借助其他方式
}
```

> ⚠️ `plugins {}` 块是声明式的，不支持条件判断。实际项目中通常使用 `apply plugin` 或 Convention Plugin 来实现动态切换，详见第二节。

## 二、原理与实践

### 2.1 isModule 标志位配置

最常见的做法是在 **`gradle.properties`**（项目根目录）中定义标志位：

```properties
# gradle.properties
# true = 各组件独立运行模式；false = 集成到宿主 App 模式
isModule=true
```

在模块的 `build.gradle.kts` 中读取：

```kotlin
// module_login/build.gradle.kts
val isModule = project.findProperty("isModule")?.toString()?.toBoolean() ?: false

if (isModule) {
    apply(plugin = "com.android.application")
} else {
    apply(plugin = "com.android.library")
}

android {
    namespace = "com.example.module.login"

    defaultConfig {
        if (isModule) {
            applicationId = "com.example.module.login"
        }
        minSdk = 24
        targetSdk = 34
    }
}
```

切换流程：

```
修改 gradle.properties 中 isModule=true/false
        ↓
Gradle Sync
        ↓
模块自动切换 application / library 身份
```

> 💡 也可以通过命令行参数覆盖：`./gradlew :module_login:assembleDebug -PisModule=true`，无需修改文件。这在 CI 中非常实用，参考 [[Gradle生命周期与Task]]。

### 2.2 独立运行的 AndroidManifest 合并策略

独立运行时，模块需要一个包含 `<application>` 和 **launcher Activity** 的完整 Manifest；作为 library 集成时，只需要声明自身的组件即可。

推荐的目录结构：

```
module_login/
├── src/
│   ├── main/
│   │   ├── AndroidManifest.xml          ← library 模式使用（精简版）
│   │   ├── java/
│   │   └── res/
│   └── alone/                           ← 独立运行模式专用
│       ├── AndroidManifest.xml          ← 完整版，含 launcher
│       ├── java/
│       │   └── com/example/module/login/
│       │       ├── DebugApp.kt
│       │       └── DebugLauncherActivity.kt
│       └── res/
│           └── values/
│               └── strings.xml
```

library 模式的 Manifest（`src/main/AndroidManifest.xml`）：

```xml
<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android">

    <application>
        <activity
            android:name=".ui.LoginActivity"
            android:exported="false" />
    </application>

</manifest>
```

独立运行模式的 Manifest（`src/alone/AndroidManifest.xml`）：

```xml
<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android">

    <application
        android:name=".debug.DebugApp"
        android:label="Login-Debug"
        android:theme="@style/Theme.MaterialComponents.Light.NoActionBar">

        <activity
            android:name=".debug.DebugLauncherActivity"
            android:exported="true">
            <intent-filter>
                <action android:name="android.intent.action.MAIN" />
                <category android:name="android.intent.category.LAUNCHER" />
            </intent-filter>
        </activity>
    </application>

</manifest>
```

### 2.3 sourceSets 切换

通过 `sourceSets` 在独立运行时将 `alone` 目录加入编译：

```kotlin
// module_login/build.gradle.kts
android {
    sourceSets {
        getByName("main") {
            if (isModule) {
                // 独立运行：追加 alone 目录的源码、资源和 Manifest
                manifest.srcFile("src/alone/AndroidManifest.xml")
                java.srcDirs("src/main/java", "src/alone/java")
                res.srcDirs("src/main/res", "src/alone/res")
            } else {
                // 集成模式：使用默认 main 目录
                manifest.srcFile("src/main/AndroidManifest.xml")
            }
        }
    }
}
```

> 关键点：独立运行时 **替换** Manifest 文件路径（`manifest.srcFile`），同时 **追加** java 和 res 目录，这样 `alone` 中的调试代码不会被打包进正式 AAR。

### 2.4 独立运行的 Application 与入口 Activity

`DebugApp.kt` — 独立运行时的 Application，用于初始化组件所需的依赖：

```java
// src/alone/java/.../debug/DebugApp.java
package com.example.module.login.debug;

import android.app.Application;
import com.example.module.login.LoginServiceManager;

public class DebugApp extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        // 初始化本模块依赖的服务（使用 Mock 实现）
        LoginServiceManager.init(
            new MockUserService(),
            new MockNetworkClient()
        );
    }
}
```

`DebugLauncherActivity.kt` — 入口 Activity，直接跳转到要调试的页面：

```java
// src/alone/java/.../debug/DebugLauncherActivity.java
package com.example.module.login.debug;

import android.content.Intent;
import android.os.Bundle;
import androidx.appcompat.app.AppCompatActivity;
import com.example.module.login.ui.LoginActivity;

public class DebugLauncherActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        // 直接跳转到目标页面，省去宿主 App 的层层导航
        startActivity(new Intent(this, LoginActivity.class));
        finish();
    }
}
```

### 2.5 Mock 数据与依赖注入

组件独立运行时，它所依赖的其他模块的服务是不存在的。解决方案：

**方案一：手动 Mock（轻量）**

在 `alone` 目录中提供 Mock 实现：

```java
// src/alone/java/.../debug/MockUserService.java
public class MockUserService implements IUserService {
    @Override
    public boolean isLoggedIn() { return true; }
    @Override
    public UserInfo getUserInfo() {
        return new UserInfo("mock_001", "测试用户");
    }
}
```

**方案二：结合依赖注入框架（推荐）**

使用 Hilt / Koin 等框架，在独立运行时替换绑定：

```kotlin
// src/alone/java/.../debug/DebugApp.kt（Koin 示例）
class DebugApp : Application() {
    override fun onCreate() {
        super.onCreate()
        startKoin {
            androidContext(this@DebugApp)
            modules(
                loginModule,       // 本模块的正式依赖
                debugMockModule    // Mock 依赖覆盖
            )
        }
    }
}

val debugMockModule = module {
    single<IUserService> { MockUserService() }
    single<INetworkClient> { MockNetworkClient() }
}
```

这样做的好处是：正式代码中的依赖注入逻辑不需要任何修改，Mock 代码完全隔离在 `alone` 目录中，不会被打包进 release 产物。

## 三、面试题

### Q1：组件独立运行的核心原理是什么？

**A：** 核心原理是通过一个标志位（如 `isModule`）动态切换 Gradle 插件类型。当 `isModule=true` 时，模块应用 `com.android.application` 插件，拥有独立的 `applicationId`、完整的 AndroidManifest（含 launcher Activity），可以生成 APK 独立安装运行。当 `isModule=false` 时，应用 `com.android.library` 插件，作为 AAR 被宿主 App 依赖。配合 `sourceSets` 切换，将调试专用的代码和资源隔离在独立目录中，确保不会污染正式产物。

### Q2：为什么不能在 `plugins {}` 块中使用 if 判断来切换插件？

**A：** Gradle 的 `plugins {}` 块（Plugin DSL）是**声明式**的，在 Gradle 的 Settings 阶段就会被解析，此时项目的 properties 还未完全加载，不支持任意 Kotlin/Groovy 代码。解决方案有三种：
1. 使用传统的 `apply(plugin = "...")` 方式，它在 Configuration 阶段执行，支持条件判断
2. 使用 Convention Plugin（buildSrc 或 composite build），在插件内部封装切换逻辑
3. 使用 `pluginManagement` 中的 `resolutionStrategy` 做映射（较少用）

实际项目中推荐方案 2，参考 [[Gradle生命周期与Task]]。

### Q3：独立运行时如何处理对其他模块服务的依赖？

**A：** 主要有两种策略：
- **手动 Mock**：在 `src/alone` 目录中为依赖的接口提供简单的 Mock 实现，适合依赖较少的场景
- **依赖注入替换**：使用 Koin / Hilt 等框架，在独立运行的 Application 中注册 Mock Module 覆盖正式绑定，适合依赖较多的场景

关键原则是：所有跨模块依赖都应该**面向接口编程**（定义在公共 API 模块中），这样 Mock 实现才能无缝替换，参考 [[组件化架构设计与拆分策略]] 中的接口下沉策略。

### Q4：独立运行模式下的 AndroidManifest 和集成模式下的有什么区别？如何管理？

**A：** 区别如下：
- **独立运行 Manifest**：包含完整的 `<application>` 节点（指定 `android:name` 为 DebugApp）、launcher Activity 的 `<intent-filter>`、可能还有调试专用的权限声明
- **集成模式 Manifest**：只声明本模块的四大组件，不包含 launcher 入口，`<application>` 节点通常为空或省略

管理方式是通过 `sourceSets` 切换 `manifest.srcFile` 路径。独立运行时指向 `src/alone/AndroidManifest.xml`，集成时指向 `src/main/AndroidManifest.xml`。AGP 的 Manifest Merger 会在集成模式下将各模块的 Manifest 合并到宿主 App 中。

### Q5：如何在 CI/CD 中利用组件独立运行能力？

**A：** 在 CI 中可以通过命令行参数控制模式：

```bash
# 只编译并测试 login 模块（独立运行模式）
./gradlew :module_login:assembleDebug -PisModule=true
./gradlew :module_login:testDebugUnitTest -PisModule=true

# 集成模式：编译宿主 App
./gradlew :app:assembleRelease -PisModule=false
```

结合 Git diff 检测变更模块，CI 可以只编译和测试受影响的组件，大幅缩短流水线时间。例如只修改了 `module_login`，就只跑 login 模块的构建和测试，无需全量编译。

### Q6：独立运行时 R 文件和资源 ID 有什么需要注意的？

**A：** 当模块以 `application` 运行时，R 文件中的资源 ID 是 `final` 常量（可用于 `switch-case` 和注解）。但以 `library` 集成时，R 文件中的 ID **不是 final** 的（AGP 的设计，避免 ID 冲突），不能用于 `switch-case`，必须用 `if-else`。因此：
- 代码中避免对资源 ID 使用 `when` 的常量分支（Kotlin 中影响较小，`when` 不要求 const）
- 为每个模块的资源添加前缀（如 `login_`），通过 `resourcePrefix` 配置强制约束，防止集成时资源名冲突

## 四、实战与踩坑

### 4.1 资源 ID 冲突

**问题**：多个模块定义了同名资源（如 `ic_back`、`bg_button`），集成编译时后合并的资源会覆盖先合并的，导致 UI 异常。

**解决方案**：为每个模块配置资源前缀：

```kotlin
// module_login/build.gradle.kts
android {
    resourcePrefix = "login_"
}
```

配置后，Lint 会对不符合前缀的资源名发出警告。注意 `resourcePrefix` 只是 Lint 级别的检查，不会阻止编译，需要配合 CI 的 Lint 检查强制执行：

```bash
./gradlew :module_login:lintDebug
```

> ⚠️ 第三方库的资源无法添加前缀。如果多个模块依赖了同一个库的不同版本，可能出现资源覆盖。统一依赖版本（通过 Version Catalog）是根本解决方案。

### 4.2 Manifest 合并冲突

**问题**：集成编译时，多个模块的 Manifest 合并出现冲突，常见错误：

```
Manifest merger failed : Attribute application@theme value=(@style/ThemeA)
is also present at [module_login] with value=(@style/ThemeB).
```

**解决方案**：

1. **模块 Manifest 不要声明 `<application>` 级别的属性**（如 `theme`、`label`），这些由宿主 App 统一配置
2. 如果必须声明，使用 `tools:replace` 在宿主 App 中覆盖：

```xml
<!-- app/src/main/AndroidManifest.xml -->
<application
    android:theme="@style/AppTheme"
    tools:replace="android:theme">
```

3. 独立运行的 Manifest（`src/alone/AndroidManifest.xml`）中可以自由声明这些属性，因为它不会参与集成编译的合并

**最佳实践**：library 模式的 Manifest 保持最小化，只声明四大组件和必要权限。

### 4.3 Gradle 配置简化：Convention Plugin

当项目有 10+ 个模块时，每个模块都写一遍 `isModule` 判断、`sourceSets` 切换、`resourcePrefix` 等配置，维护成本很高。推荐使用 **Convention Plugin** 统一封装。

在 `buildSrc`（或独立的 `build-logic` composite build）中定义：

```kotlin
// build-logic/convention/src/main/kotlin/ModuleConventionPlugin.kt
import org.gradle.api.Plugin
import org.gradle.api.Project

class ModuleConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        val isModule = target.findProperty("isModule")
            ?.toString()?.toBoolean() ?: false

        with(target) {
            if (isModule) {
                pluginManager.apply("com.android.application")
            } else {
                pluginManager.apply("com.android.library")
            }
            pluginManager.apply("org.jetbrains.kotlin.android")

            extensions.configure<com.android.build.gradle.BaseExtension> {
                compileSdkVersion(34)
                defaultConfig {
                    minSdk = 24
                    targetSdk = 34
                    if (isModule) {
                        applicationId = "com.example.${target.name}"
                    }
                }

                // 资源前缀：取模块名去掉 module_ 前缀
                val prefix = target.name.removePrefix("module_")
                resourcePrefix = "${prefix}_"

                // sourceSets 切换
                sourceSets.getByName("main") {
                    if (isModule) {
                        manifest.srcFile("src/alone/AndroidManifest.xml")
                        java.srcDirs("src/main/java", "src/alone/java")
                        res.srcDirs("src/main/res", "src/alone/res")
                    }
                }
            }
        }
    }
}
```

在 `build-logic/convention/build.gradle.kts` 中注册：

```kotlin
// build-logic/convention/build.gradle.kts
plugins {
    `kotlin-dsl`
}

gradlePlugin {
    plugins {
        register("module-convention") {
            id = "com.example.module.convention"
            implementationClass = "ModuleConventionPlugin"
        }
    }
}
```

各业务模块的 `build.gradle.kts` 瞬间变得极简：

```kotlin
// module_login/build.gradle.kts
plugins {
    id("com.example.module.convention")
}

dependencies {
    implementation(project(":lib_common"))
    implementation(libs.androidx.appcompat)
}
```

> 💡 Convention Plugin 的优势不仅是减少重复代码，更重要的是**统一变更入口**。当需要升级 `compileSdk` 或调整 `sourceSets` 策略时，只需修改一处。更多 Gradle 插件机制参考 [[Gradle生命周期与Task]]。

### 4.4 常见问题排查清单

| 现象 | 原因 | 解决 |
|------|------|------|
| 独立运行时闪退，提示找不到 Activity | `alone/AndroidManifest.xml` 中未注册目标 Activity | 检查 Manifest 是否包含所有需要的 Activity 声明 |
| 切换 `isModule` 后 Sync 失败 | Gradle 缓存未清理 | 执行 `./gradlew clean` 后重新 Sync |
| 独立运行正常但集成后资源显示异常 | 资源名冲突被覆盖 | 配置 `resourcePrefix` 并重命名冲突资源 |
| `R.xxx` 在 library 模式下报错 | 在 `when` / `switch` 中使用了非 const 的资源 ID | 改用 `if-else` 或确保使用 Kotlin `when`（不要求 const） |
| 独立运行时网络请求失败 | 未在 DebugApp 中初始化网络层 | 在 `alone` 的 Application 中完成必要初始化 |
