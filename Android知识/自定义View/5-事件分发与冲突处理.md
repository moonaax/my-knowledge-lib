# 事件分发与冲突处理

> 自定义 View 中最核心的难点之一。理解事件分发是解决滑动冲突、实现复杂交互的基础。
> 关联笔记：[[8-事件分发机制]] · [[7-View绘制流程-measure-layout-draw]]

---

## 一、核心概念

### 1.1 三大方法回顾

Android 触摸事件分发由三个关键方法协作完成：

| 方法 | 所属类 | 作用 | 返回值含义 |
|---|---|---|---|
| `dispatchTouchEvent()` | View / ViewGroup | 事件分发的入口 | `true`=已消费，`false`=未消费 |
| `onInterceptTouchEvent()` | ViewGroup 独有 | 判断是否拦截事件 | `true`=拦截，`false`=不拦截 |
| `onTouchEvent()` | View / ViewGroup | 真正处理事件的地方 | `true`=已消费，`false`=未消费 |

三者调用关系的伪代码：

```java
// ViewGroup.dispatchTouchEvent 伪代码
@Override
public boolean dispatchTouchEvent(MotionEvent ev) {
    boolean consumed = false;
    if (onInterceptTouchEvent(ev)) {
        // 拦截：自己处理
        consumed = onTouchEvent(ev);
    } else {
        // 不拦截：交给子 View
        consumed = child.dispatchTouchEvent(ev);
    }
    return consumed;
}
```

关键规则：
- `Activity` → `Window(PhoneWindow)` → `DecorView` → `ViewGroup` → `View`，自顶向下分发
- 子 View 不消费事件时，事件会回传给父 View 的 `onTouchEvent()`
- 一旦某个 View 消费了 `ACTION_DOWN`，后续同一事件序列都会交给它处理

### 1.2 MotionEvent 事件序列

一个完整的触摸事件序列：

```
ACTION_DOWN → ACTION_MOVE → ACTION_MOVE → ... → ACTION_UP
```

| 事件类型 | 说明 |
|---|---|
| `ACTION_DOWN` | 手指按下，事件序列的起点，最关键 |
| `ACTION_MOVE` | 手指移动，可能触发多次 |
| `ACTION_UP` | 手指抬起，事件序列结束 |
| `ACTION_CANCEL` | 事件被父 View 拦截时，子 View 会收到此事件 |
| `ACTION_POINTER_DOWN/UP` | 多指触控时，非第一根手指的按下/抬起 |

核心原则：**`ACTION_DOWN` 决定整个事件序列的归属**。如果一个 View 在 `ACTION_DOWN` 时没有消费事件，后续事件都不会再传递给它。

## 二、原理与源码

### 2.1 ViewGroup 事件分发源码分析

以下是 `ViewGroup.dispatchTouchEvent()` 的核心逻辑（基于 API 34 源码简化）：

```java
// ViewGroup.dispatchTouchEvent() 核心流程（简化版）
@Override
public boolean dispatchTouchEvent(MotionEvent ev) {
    boolean handled = false;
    int action = ev.getActionMasked();

    // ① 处理 ACTION_DOWN：重置状态
    if (action == MotionEvent.ACTION_DOWN) {
        cancelAndClearTouchTargets(ev);
        resetTouchState();  // 清除 FLAG_DISALLOW_INTERCEPT
    }

    // ② 判断是否拦截
    boolean intercepted;
    if (action == MotionEvent.ACTION_DOWN || mFirstTouchTarget != null) {
        // 子 View 可通过 requestDisallowInterceptTouchEvent 禁止父 View 拦截
        boolean disallowIntercept = (mGroupFlags & FLAG_DISALLOW_INTERCEPT) != 0;
        if (!disallowIntercept) {
            intercepted = onInterceptTouchEvent(ev);
        } else {
            intercepted = false;
        }
    } else {
        // 没有子 View 处理且不是 DOWN → 直接拦截
        intercepted = true;
    }

    // ③ 不拦截时，遍历子 View 寻找消费者
    if (!intercepted) {
        if (action == MotionEvent.ACTION_DOWN) {
            for (int i = childrenCount - 1; i >= 0; i--) {
                View child = getChildAt(i);
                // 判断触摸点是否在子 View 范围内
                if (!isTransformedTouchPointInView(x, y, child)) continue;
                // 尝试分发给子 View
                if (dispatchTransformedTouchEvent(ev, child)) {
                    // 子 View 消费了 → 加入 TouchTarget 链表
                    mFirstTouchTarget = addTouchTarget(child, idBitsToAssign);
                    break;
                }
            }
        }
    }

    // ④ 分发事件
    if (mFirstTouchTarget == null) {
        // 没有子 View 消费 → 自己处理（调用 View.dispatchTouchEvent）
        handled = dispatchTransformedTouchEvent(ev, null);
    } else {
        // 分发给 TouchTarget 链表中的子 View
        TouchTarget target = mFirstTouchTarget;
        while (target != null) {
            if (intercepted) {
                // 中途拦截 → 给子 View 发 CANCEL
                dispatchTransformedTouchEvent(MotionEvent.ACTION_CANCEL, target.child);
            } else {
                handled = dispatchTransformedTouchEvent(ev, target.child);
            }
            target = target.next;
        }
    }
    return handled;
}
```

### 2.2 TouchTarget 链表

`TouchTarget` 是 ViewGroup 内部的单链表结构，用于记录哪些子 View 正在处理触摸事件：

```java
// ViewGroup 内部类
private static final class TouchTarget {
    public View child;          // 消费事件的子 View
    public int pointerIdBits;   // 关联的手指 ID（多指触控）
    public TouchTarget next;    // 链表下一个节点
}
```

关键点：
- `mFirstTouchTarget`：链表头指针，`null` 表示没有子 View 消费事件
- `ACTION_DOWN` 时清空链表，重新寻找消费者
- 多指触控时，不同手指可以对应不同的 TouchTarget 节点
- `mFirstTouchTarget != null` 是判断"是否有子 View 处理事件"的核心条件

### 2.3 FLAG_DISALLOW_INTERCEPT

这是事件分发中最重要的标志位之一：

```java
// 子 View 中调用
getParent().requestDisallowInterceptTouchEvent(true);
```

作用：子 View 告诉父 View「不要拦截我的事件」。

源码实现：

```java
// ViewGroup.java
@Override
public void requestDisallowInterceptTouchEvent(boolean disallowIntercept) {
    if (disallowIntercept) {
        mGroupFlags |= FLAG_DISALLOW_INTERCEPT;
    } else {
        mGroupFlags &= ~FLAG_DISALLOW_INTERCEPT;
    }
    // 向上传递
    if (mParent != null) {
        mParent.requestDisallowInterceptTouchEvent(disallowIntercept);
    }
}
```

注意事项：
- `ACTION_DOWN` 时会调用 `resetTouchState()` 清除此标志，所以**对 `ACTION_DOWN` 无效**
- 这是「内部拦截法」的核心机制
- 该标志会沿着 View 树向上传递，影响所有祖先 ViewGroup

### 2.4 事件分发流程图

```
Activity.dispatchTouchEvent()
  └─ PhoneWindow.superDispatchTouchEvent()
       └─ DecorView.dispatchTouchEvent()
            └─ ViewGroup.dispatchTouchEvent()
                 ├─ onInterceptTouchEvent() → true → 自己的 onTouchEvent()
                 └─ onInterceptTouchEvent() → false
                      └─ child.dispatchTouchEvent()
                           └─ child.onTouchEvent()
                                ├─ true → 事件被消费
                                └─ false → 回传父 View.onTouchEvent()
```

## 三、滑动冲突解决方案

### 3.1 滑动冲突的分类

| 类型 | 场景 | 示例 |
|---|---|---|
| 同方向冲突 | 父子 View 滑动方向相同 | ScrollView 嵌套 RecyclerView |
| 不同方向冲突 | 父子 View 滑动方向不同 | ViewPager（横向）嵌套 RecyclerView（纵向） |
| 多层嵌套冲突 | 多级嵌套 | ViewPager2 + TabLayout + RecyclerView |

### 3.2 外部拦截法（推荐）

核心思想：**父 View 决定是否拦截**，重写父 View 的 `onInterceptTouchEvent()`。

```java
/**
 * 外部拦截法：父 ViewGroup 控制拦截逻辑
 * 适用于父 View 需要根据滑动方向决定是否拦截的场景
 */
public class OuterInterceptLayout extends FrameLayout {

    private float lastX = 0f;
    private float lastY = 0f;

    public OuterInterceptLayout(Context context) { this(context, null); }
    public OuterInterceptLayout(Context context, AttributeSet attrs) { super(context, attrs); }

    @Override
    public boolean onInterceptTouchEvent(MotionEvent ev) {
        boolean intercepted = false;
        float x = ev.getX();
        float y = ev.getY();

        switch (ev.getActionMasked()) {
            case MotionEvent.ACTION_DOWN:
                // DOWN 事件不能拦截，否则子 View 收不到任何事件
                intercepted = false;
                lastX = x;
                lastY = y;
                break;
            case MotionEvent.ACTION_MOVE:
                float dx = Math.abs(x - lastX);
                float dy = Math.abs(y - lastY);
                // 示例：横向滑动距离 > 纵向 → 父 View 拦截
                intercepted = dx > dy && dx > ViewConfiguration.get(getContext()).getScaledTouchSlop();
                break;
            case MotionEvent.ACTION_UP:
                intercepted = false;
                break;
        }
        lastX = x;
        lastY = y;
        return intercepted;
    }
}
```

外部拦截法要点：
1. `ACTION_DOWN` 必须返回 `false`，否则子 View 无法接收事件
2. `ACTION_MOVE` 中根据业务逻辑判断是否拦截
3. `ACTION_UP` 必须返回 `false`，否则子 View 的 `onClick` 无法触发
4. 一旦拦截，子 View 会收到 `ACTION_CANCEL`

### 3.3 内部拦截法

核心思想：**子 View 决定父 View 是否拦截**，利用 `requestDisallowInterceptTouchEvent()`。

```java
/**
 * 内部拦截法 - 父 ViewGroup：除 DOWN 外默认拦截
 */
public class InnerInterceptParentLayout extends FrameLayout {

    public InnerInterceptParentLayout(Context context) { this(context, null); }
    public InnerInterceptParentLayout(Context context, AttributeSet attrs) { super(context, attrs); }

    @Override
    public boolean onInterceptTouchEvent(MotionEvent ev) {
        // DOWN 不拦截（否则子 View 收不到事件，也无法调用 requestDisallow）
        // 其他事件默认拦截，交给子 View 通过 requestDisallow 控制
        return ev.getActionMasked() != MotionEvent.ACTION_DOWN;
    }
}

/**
 * 内部拦截法 - 子 View：控制父 View 的拦截行为
 */
public class InnerInterceptChildView extends View {

    private float lastX = 0f;
    private float lastY = 0f;

    public InnerInterceptChildView(Context context) { this(context, null); }
    public InnerInterceptChildView(Context context, AttributeSet attrs) { super(context, attrs); }

    @Override
    public boolean dispatchTouchEvent(MotionEvent ev) {
        float x = ev.getX();
        float y = ev.getY();

        switch (ev.getActionMasked()) {
            case MotionEvent.ACTION_DOWN:
                // DOWN 时禁止父 View 拦截
                getParent().requestDisallowInterceptTouchEvent(true);
                lastX = x;
                lastY = y;
                break;
            case MotionEvent.ACTION_MOVE:
                float dx = Math.abs(x - lastX);
                float dy = Math.abs(y - lastY);
                // 如果判断应该由父 View 处理（如横向滑动），则允许父 View 拦截
                if (dx > dy && dx > ViewConfiguration.get(getContext()).getScaledTouchSlop()) {
                    getParent().requestDisallowInterceptTouchEvent(false);
                }
                break;
        }
        lastX = x;
        lastY = y;
        return super.dispatchTouchEvent(ev);
    }
}
```

内部拦截法要点：
1. 父 View 的 `onInterceptTouchEvent` 除 `ACTION_DOWN` 外全部返回 `true`
2. 子 View 在 `dispatchTouchEvent` 中通过 `requestDisallowInterceptTouchEvent` 控制
3. `ACTION_DOWN` 时子 View 必须调用 `requestDisallowInterceptTouchEvent(true)`
4. 比外部拦截法复杂，需要父子配合，一般优先使用外部拦截法

### 3.4 两种方案对比

| 对比项 | 外部拦截法 | 内部拦截法 |
|---|---|---|
| 控制方 | 父 ViewGroup | 子 View |
| 重写方法 | 父 `onInterceptTouchEvent` | 父 `onInterceptTouchEvent` + 子 `dispatchTouchEvent` |
| 复杂度 | 低 | 高 |
| 适用场景 | 父 View 能判断拦截条件 | 子 View 更清楚自身滑动状态 |
| 推荐度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |

### 3.5 NestedScrolling 机制

传统事件拦截是「非此即彼」的。`NestedScrolling` 机制允许父子 View **协同滚动**。

核心接口：
- `NestedScrollingChild3`：子 View 实现，提供 `startNestedScroll`、`dispatchNestedPreScroll`、`dispatchNestedScroll`、`stopNestedScroll`
- `NestedScrollingParent3`：父 View 实现，提供 `onStartNestedScroll`、`onNestedPreScroll`、`onNestedScroll`、`onStopNestedScroll`

嵌套滚动流程：

```
子 View 收到 MOVE 事件，产生滚动量 dy
  ├─ ① dispatchNestedPreScroll(dy) → 父 View.onNestedPreScroll()
  │     父 View 先消费一部分 consumed[1]
  ├─ ② 子 View 消费剩余的 dy - consumed[1]
  └─ ③ dispatchNestedScroll(unconsumed) → 父 View.onNestedScroll()
        父 View 消费子 View 未消费的部分
```

典型实现：`RecyclerView`（Child）+ `CoordinatorLayout`（Parent）。

## 四、面试题

### Q1：事件分发的完整流程是怎样的？

**A：** 触摸事件从 `Activity.dispatchTouchEvent()` 开始，经过 `PhoneWindow` → `DecorView` → 根 `ViewGroup`，自顶向下分发。每个 `ViewGroup` 先调用 `onInterceptTouchEvent()` 判断是否拦截：
- 拦截 → 调用自身 `onTouchEvent()` 处理
- 不拦截 → 遍历子 View（从上层到下层，即 Z 轴倒序），调用子 View 的 `dispatchTouchEvent()`

如果所有子 View 都不消费，事件回传给父 View 的 `onTouchEvent()`，最终回到 `Activity.onTouchEvent()`。

### Q2：为什么 ACTION_DOWN 不能被拦截（在外部拦截法中）？

**A：** `ACTION_DOWN` 是事件序列的起点。如果父 View 在 `onInterceptTouchEvent` 中拦截了 `ACTION_DOWN`：
1. 子 View 的 `dispatchTouchEvent` 不会被调用
2. `mFirstTouchTarget` 为 `null`
3. 后续所有 MOVE/UP 事件都会直接由父 View 处理，子 View 完全失去交互能力
4. 子 View 也无法调用 `requestDisallowInterceptTouchEvent`（因为根本没收到事件）

### Q3：OnTouchListener、onTouchEvent、OnClickListener 的优先级？

**A：** 在 `View.dispatchTouchEvent()` 中的处理顺序：

```java
// View.dispatchTouchEvent 简化逻辑
public boolean dispatchTouchEvent(MotionEvent event) {
    boolean result = false;
    // ① OnTouchListener 优先级最高
    if (onTouchListener != null && onTouchListener.onTouch(this, event)) {
        result = true;  // 如果返回 true，不再调用 onTouchEvent
    }
    // ② onTouchEvent 次之
    if (!result && onTouchEvent(event)) {
        result = true;
    }
    // ③ OnClickListener 在 onTouchEvent 的 ACTION_UP 中触发
    return result;
}
```

优先级：`OnTouchListener.onTouch()` > `onTouchEvent()` > `OnClickListener.onClick()`

### Q4：requestDisallowInterceptTouchEvent 对 ACTION_DOWN 为什么无效？

**A：** 因为在 `ViewGroup.dispatchTouchEvent()` 中，收到 `ACTION_DOWN` 时会调用 `resetTouchState()`，该方法会清除 `FLAG_DISALLOW_INTERCEPT` 标志位：

```java
private void resetTouchState() {
    clearTouchTargets();
    resetCancelNextUpFlag(this);
    mGroupFlags &= ~FLAG_DISALLOW_INTERCEPT;  // 清除标志
}
```

这是设计上的考量：`ACTION_DOWN` 代表新事件序列开始，父 View 必须有机会重新决定是否拦截。

### Q5：子 View 设置了 clickable=true 会有什么影响？

**A：** `View.onTouchEvent()` 中，只要 `clickable` 或 `longClickable` 为 `true`，`onTouchEvent` 就会返回 `true`（消费事件），即使 View 处于 `disabled` 状态：

```java
// View.onTouchEvent 源码
if (clickable || (viewFlags & TOOLTIP) == TOOLTIP) {
    switch (action) {
        case MotionEvent.ACTION_UP: // 处理点击
        case MotionEvent.ACTION_DOWN: // 开始检测长按
        case MotionEvent.ACTION_MOVE: // 检测是否移出范围
    }
    return true;  // 无论如何都消费事件
}
```

这意味着设置了 `OnClickListener` 的 View 一定会消费触摸事件，事件不会回传给父 View。

### Q6：多指触控时事件是如何分发的？

**A：** 多指触控通过 `ACTION_POINTER_DOWN/UP` 和 `pointerIdBits` 管理：
1. 第一根手指按下 → `ACTION_DOWN`，创建 TouchTarget
2. 第二根手指按下 → `ACTION_POINTER_DOWN`，可能创建新的 TouchTarget 或复用已有的
3. 每个 TouchTarget 通过 `pointerIdBits` 记录关联的手指 ID
4. 分发时，`dispatchTransformedTouchEvent()` 会根据 `pointerIdBits` 过滤事件，每个子 View 只收到与自己关联的手指事件
5. 非第一根手指抬起 → `ACTION_POINTER_UP`，最后一根手指抬起 → `ACTION_UP`

## 五、实战与踩坑

### 5.1 ViewPager2 + RecyclerView 嵌套冲突

ViewPager2 内部基于 RecyclerView 实现，与外部 RecyclerView 嵌套时同方向滑动会产生冲突。

#### 场景一：横向 ViewPager2 嵌套纵向 RecyclerView（不同方向）

ViewPager2 默认拦截横向滑动，但斜向滑动时容易误判。推荐方案：

```java
/**
 * 解决 ViewPager2 嵌套 RecyclerView 的滑动冲突
 * 给内部 RecyclerView 添加此 ItemDecoration 或直接在代码中处理
 */
public class NestedScrollableHost extends FrameLayout {

    private int touchSlop;
    private float initialX = 0f;
    private float initialY = 0f;

    public NestedScrollableHost(Context context) { this(context, null); }
    public NestedScrollableHost(Context context, AttributeSet attrs) {
        super(context, attrs);
        touchSlop = ViewConfiguration.get(context).getScaledTouchSlop();
    }

    private ViewPager2 getParentViewPager() {
        View v = (View) getParent();
        while (v != null && !(v instanceof ViewPager2)) {
            v = (View) v.getParent();
        }
        return (ViewPager2) v;
    }

    @Override
    public boolean onInterceptTouchEvent(MotionEvent ev) {
        handleInterceptTouchEvent(ev);
        return super.onInterceptTouchEvent(ev);
    }

    private void handleInterceptTouchEvent(MotionEvent ev) {
        ViewPager2 vp = getParentViewPager();
        if (vp == null) return;
        int orientation = vp.getOrientation();

        switch (ev.getAction()) {
            case MotionEvent.ACTION_DOWN:
                initialX = ev.getX();
                initialY = ev.getY();
                getParent().requestDisallowInterceptTouchEvent(true);
                break;
            case MotionEvent.ACTION_MOVE:
                float dx = Math.abs(ev.getX() - initialX);
                float dy = Math.abs(ev.getY() - initialY);
                // 判断滑动方向是否与 ViewPager2 方向一致
                boolean isVpHorizontal = orientation == ViewPager2.ORIENTATION_HORIZONTAL;
                float scaledDx = isVpHorizontal ? dx : dy;  // VP 方向的距离
                float scaledDy = isVpHorizontal ? dy : dx;  // 垂直于 VP 方向的距离

                if (scaledDx > touchSlop || scaledDy > touchSlop) {
                    if (scaledDy > scaledDx) {
                        // 滑动方向与 VP 垂直 → 子 View 处理
                        getParent().requestDisallowInterceptTouchEvent(true);
                    } else {
                        // 滑动方向与 VP 一致 → 交给 VP
                        getParent().requestDisallowInterceptTouchEvent(false);
                    }
                }
                break;
        }
    }
}
```

使用方式：用 `NestedScrollableHost` 包裹 ViewPager2 内部每个 page 中的 RecyclerView 即可。

#### 场景二：纵向 RecyclerView 嵌套横向 RecyclerView

这种场景通常不需要特殊处理。但如果出现问题，可在内部 RecyclerView 的 `OnItemTouchListener` 中处理：

```java
innerRecyclerView.addOnItemTouchListener(new RecyclerView.OnItemTouchListener() {
    @Override
    public boolean onInterceptTouchEvent(@NonNull RecyclerView rv, @NonNull MotionEvent e) {
        switch (e.getAction()) {
            case MotionEvent.ACTION_DOWN:
            case MotionEvent.ACTION_MOVE:
                // 内部 RV 可以横向滚动时，禁止外部拦截
                rv.getParent().requestDisallowInterceptTouchEvent(
                    rv.canScrollHorizontally(-1) || rv.canScrollHorizontally(1)
                );
                break;
        }
        return false;
    }
    @Override public void onTouchEvent(@NonNull RecyclerView rv, @NonNull MotionEvent e) {}
    @Override public void onRequestDisallowInterceptTouchEvent(boolean disallow) {}
});
```

#### 常见踩坑

| 问题 | 原因 | 解决方案 |
|---|---|---|
| ViewPager2 内 Fragment 中的 RecyclerView 无法滑动 | VP2 拦截了同方向事件 | 使用 `NestedScrollableHost` 包裹 |
| VP2 的 `isUserInputEnabled = false` 后子 View 仍有问题 | VP2 内部 RV 仍在处理事件 | 同时设置内部 RV 的 `isNestedScrollingEnabled = false` |
| 嵌套滑动到边界时无法切换 page | `requestDisallowInterceptTouchEvent` 未正确释放 | 在子 View 滑动到边界时允许父 View 拦截 |

### 5.2 CoordinatorLayout 与 Behavior

`CoordinatorLayout` 是 NestedScrolling 机制的典型应用，通过 `Behavior` 实现子 View 之间的联动。

#### 自定义 Behavior 示例：Header 跟随滚动

`CoordinatorLayout` 在分发事件前会先询问所有子 View 的 `Behavior.onInterceptTouchEvent()`，Behavior 可以优先拦截事件。

```java
/**
 * 自定义 Behavior：让 Header View 跟随 RecyclerView 的滚动而折叠
 */
public class CollapsingHeaderBehavior extends CoordinatorLayout.Behavior<View> {

    @Override
    public boolean onStartNestedScroll(
            @NonNull CoordinatorLayout coordinatorLayout,
            @NonNull View child,          // 应用此 Behavior 的 View（Header）
            @NonNull View directTargetChild,
            @NonNull View target,         // 发起滚动的 View（RecyclerView）
            int axes,
            int type) {
        // 只关心垂直方向的滚动
        return (axes & ViewCompat.SCROLL_AXIS_VERTICAL) != 0;
    }

    @Override
    public void onNestedPreScroll(
            @NonNull CoordinatorLayout coordinatorLayout,
            @NonNull View child,
            @NonNull View target,
            int dx, int dy,
            @NonNull int[] consumed,
            int type) {
        // Header 还能折叠时，优先消费滚动量
        int maxScroll = child.getMeasuredHeight();
        float currentTranslation = child.getTranslationY();
        if (dy > 0 && currentTranslation > -maxScroll) {
            // 上滑 → 折叠 Header
            int consumedY = (int) Math.min(dy, currentTranslation + maxScroll);
            child.setTranslationY(child.getTranslationY() - consumedY);
            consumed[1] = consumedY;
        } else if (dy < 0 && currentTranslation < 0) {
            // 下滑 → 展开 Header
            int consumedY = (int) Math.max(dy, currentTranslation);
            child.setTranslationY(child.getTranslationY() - consumedY);
            consumed[1] = consumedY;
        }
    }
}
```

在 XML 中通过 `app:layout_behavior` 指定自定义 Behavior 类名即可。

#### Behavior 踩坑总结

| 问题 | 原因 | 解决方案 |
|---|---|---|
| Behavior 不生效 | 父布局不是 CoordinatorLayout | 确保直接父布局是 CL |
| `onNestedPreScroll` 不回调 | `onStartNestedScroll` 返回了 `false` | 检查 axes 参数判断 |
| Fling 后 Header 状态不对 | 未处理 `TYPE_NON_TOUCH` 类型的嵌套滚动 | 在 `onNestedPreScroll` 中同时处理 `TYPE_TOUCH` 和 `TYPE_NON_TOUCH` |
| 与 ViewPager2 联动异常 | VP2 内部 RV 的嵌套滚动与 CL 冲突 | 自定义 Behavior 中正确判断 `target` 类型 |
