# 事件分发机制

## 一、什么是事件分发？

当你用手指触摸屏幕时，系统会产生一系列触摸事件（MotionEvent），这些事件需要找到正确的 View 来处理。**事件分发机制就是决定"哪个 View 来处理这个触摸事件"的规则。**

---

## 二、MotionEvent 事件类型

| 事件 | 含义 |
|------|------|
| `ACTION_DOWN` | 手指按下（事件序列的开始） |
| `ACTION_MOVE` | 手指移动（可能有多个） |
| `ACTION_UP` | 手指抬起（事件序列的结束） |
| `ACTION_CANCEL` | 事件被取消（比如父 View 拦截了后续事件） |

一个完整的事件序列：`DOWN → MOVE → MOVE → ... → MOVE → UP`

> **关键：DOWN 事件决定了整个事件序列由谁处理。** 如果一个 View 消费了 DOWN 事件，后续的 MOVE 和 UP 都会直接交给它，不再走分发流程。

---

## 三、三个核心方法

| 方法 | 所在类 | 作用 |
|------|--------|------|
| `dispatchTouchEvent()` | View / ViewGroup | 分发事件（入口） |
| `onInterceptTouchEvent()` | 仅 ViewGroup | 是否拦截事件（ViewGroup 独有） |
| `onTouchEvent()` | View / ViewGroup | 处理（消费）事件 |

````
Activity.dispatchTouchEvent()
    ↓
ViewGroup.dispatchTouchEvent()
    ↓
ViewGroup.onInterceptTouchEvent()  ← 要不要拦截？
    │
    ├── 拦截（return true）→ ViewGroup.onTouchEvent()
    │
    └── 不拦截（return false）→ child.dispatchTouchEvent()
                                    ↓
                                child.onTouchEvent()
                                    │
                                    ├── 消费（return true）→ 结束
                                    │
                                    └── 不消费（return false）→ 回传给父 View.onTouchEvent()
````

## 四、分发流程详解

### 4.1 用一个故事理解

想象一个公司的审批流程：

````
总经理（Activity）
    ↓ 把任务分发下去
部门经理（ViewGroup）
    ↓ 看看要不要自己处理（onInterceptTouchEvent）
    ↓ 不处理，继续往下分发
组长（ViewGroup）
    ↓ 也不处理，继续往下
员工（View）
    ↓ 尝试处理（onTouchEvent）
    │
    ├── 处理了 → 完成
    └── 处理不了 → 退回给组长
                    ↓ 组长也处理不了 → 退回给部门经理
                                        ↓ 部门经理也处理不了 → 退回给总经理
````
**核心规则：**
1. 事件从上往下分发（Activity → ViewGroup → View）
2. 如果没人处理，从下往上回传
3. 一旦某个 View 消费了 DOWN 事件，后续事件直接给它

### 4.2 ViewGroup.dispatchTouchEvent 源码分析

````java
// 简化版源码
public boolean dispatchTouchEvent(MotionEvent ev) {
    boolean handled = false;

    // 第一步：判断是否拦截
    boolean intercepted;
    if (ev.getAction() == ACTION_DOWN || mFirstTouchTarget != null) {
        // DOWN 事件或者已有子 View 处理事件时，才询问是否拦截
        boolean disallowIntercept = (mGroupFlags & FLAG_DISALLOW_INTERCEPT) != 0;
        if (!disallowIntercept) {
            intercepted = onInterceptTouchEvent(ev);  // 询问是否拦截
        } else {
            intercepted = false;  // 子 View 请求不拦截
        }
    } else {
        intercepted = true;  // 没有子 View 处理，直接拦截
    }

    // 第二步：如果不拦截，分发给子 View
    if (!intercepted) {
        if (ev.getAction() == ACTION_DOWN) {
            // 遍历子 View（从上层到下层，即 Z 轴从高到低）
            for (int i = childCount - 1; i >= 0; i--) {
                View child = getChildAt(i);
                // 判断触摸点是否在子 View 范围内
                if (isTransformedTouchPointInView(x, y, child)) {
                    // 分发给子 View
                    if (child.dispatchTouchEvent(ev)) {
                        mFirstTouchTarget = child;  // 记录处理事件的子 View
                        handled = true;
                        break;
                    }
                }
            }
        }
    }

    // 第三步：如果没有子 View 处理（或被拦截），自己处理
    if (mFirstTouchTarget == null) {
        handled = super.dispatchTouchEvent(ev);  // 调用 View.dispatchTouchEvent → onTouchEvent
    }

    return handled;
}
````
### 4.3 View.dispatchTouchEvent 源码分析

````java
// View.java 简化版
public boolean dispatchTouchEvent(MotionEvent event) {
    boolean result = false;

    // 优先级1：OnTouchListener
    if (mOnTouchListener != null
            && (mViewFlags & ENABLED_MASK) == ENABLED
            && mOnTouchListener.onTouch(this, event)) {
        result = true;
    }

    // 优先级2：onTouchEvent（包含 OnClickListener 的处理）
    if (!result) {
        result = onTouchEvent(event);
    }

    return result;
}
````
**事件处理优先级：**
````
OnTouchListener.onTouch() > onTouchEvent() > OnClickListener.onClick()
````
- 如果 `onTouch()` 返回 true，`onTouchEvent()` 不会被调用
- `onClick()` 是在 `onTouchEvent()` 的 `ACTION_UP` 中触发的

### 4.4 onTouchEvent 中的 onClick 触发

````java
// View.onTouchEvent() 简化版
public boolean onTouchEvent(MotionEvent event) {
    if (clickable || longClickable) {
        switch (event.getAction()) {
            case ACTION_DOWN:
                // 设置长按检测（500ms 后触发）
                postDelayed(mPendingCheckForLongPress, 500);
                break;

            case ACTION_UP:
                // 移除长按检测
                removeLongPressCallback();
                // 触发点击
                performClick();  // → OnClickListener.onClick()
                break;

            case ACTION_CANCEL:
                // 取消所有回调
                removeLongPressCallback();
                break;
        }
        return true;  // clickable 的 View 默认消费事件
    }
    return false;
}
````

## 五、滑动冲突

当父 View 和子 View 都需要处理滑动事件时，就会产生冲突。

### 5.1 常见场景

| 场景 | 冲突类型 |
|------|---------|
| ScrollView 嵌套 ListView | 同方向滑动冲突 |
| ViewPager 嵌套 ListView | 不同方向滑动冲突 |
| ViewPager 嵌套 ViewPager | 同方向滑动冲突 |
| ScrollView 嵌套 ViewPager | 不同方向滑动冲突 |

### 5.2 外部拦截法（推荐）

在**父 View** 的 `onInterceptTouchEvent` 中判断是否拦截。

````java
// 父 View（如自定义 ScrollView）
@Override
public boolean onInterceptTouchEvent(MotionEvent ev) {
    switch (ev.getAction()) {
        case MotionEvent.ACTION_DOWN:
            // DOWN 事件不拦截，否则子 View 收不到任何事件
            lastX = ev.getX();
            lastY = ev.getY();
            return false;
        case MotionEvent.ACTION_MOVE:
            float dx = Math.abs(ev.getX() - lastX);
            float dy = Math.abs(ev.getY() - lastY);
            // 判断滑动方向：纵向滑动时拦截，横向滑动时不拦截
            if (dy > dx && dy > touchSlop) {
                return true;  // 拦截，自己处理
            }
            return false;  // 不拦截，交给子 View
        case MotionEvent.ACTION_UP:
            return false;
    }
    return super.onInterceptTouchEvent(ev);
}
````
### 5.3 内部拦截法

在**子 View** 中通过 `requestDisallowInterceptTouchEvent` 请求父 View 不要拦截。

````java
// 子 View（如自定义 RecyclerView）
@Override
public boolean dispatchTouchEvent(MotionEvent ev) {
    switch (ev.getAction()) {
        case MotionEvent.ACTION_DOWN:
            // 请求父 View 不要拦截
            getParent().requestDisallowInterceptTouchEvent(true);
            lastX = ev.getX();
            lastY = ev.getY();
            break;
        case MotionEvent.ACTION_MOVE:
            float dx = Math.abs(ev.getX() - lastX);
            float dy = Math.abs(ev.getY() - lastY);
            // 如果是纵向滑动，允许父 View 拦截
            if (dy > dx && dy > touchSlop) {
                getParent().requestDisallowInterceptTouchEvent(false);
            }
            break;
    }
    return super.dispatchTouchEvent(ev);
}

// 父 View 需要配合：DOWN 事件不拦截
@Override
public boolean onInterceptTouchEvent(MotionEvent ev) {
    if (ev.getAction() == MotionEvent.ACTION_DOWN) {
        return false;  // DOWN 必须不拦截
    }
    return true;  // 其他事件默认拦截（等子 View 决定是否放行）
}
````
### 5.4 两种方法对比

| 对比项 | 外部拦截法 | 内部拦截法 |
|--------|-----------|-----------|
| 决策者 | 父 View | 子 View |
| 实现位置 | 父 View 的 onInterceptTouchEvent | 子 View 的 dispatchTouchEvent |
| 复杂度 | 简单 | 较复杂（需要父子配合） |
| 推荐度 | ⭐⭐⭐ 推荐 | 特殊场景使用 |

## 六、requestDisallowInterceptTouchEvent

````java
// 子 View 调用
parent.requestDisallowInterceptTouchEvent(true);
````
这个方法设置了 `FLAG_DISALLOW_INTERCEPT` 标志位，父 View 在 dispatchTouchEvent 中检查到这个标志后，会跳过 onInterceptTouchEvent，直接把事件分发给子 View。

**注意：** 这个标志在每次 `ACTION_DOWN` 时会被重置！所以 DOWN 事件父 View 总是有机会拦截的。

## 七、多点触控

````java
@Override
public boolean onTouchEvent(MotionEvent event) {
    // 获取事件类型（忽略 pointer index）
    switch (event.getActionMasked()) {
        case MotionEvent.ACTION_DOWN: {
            // 第一个手指按下
            int id = event.getPointerId(0);
            break;
        }
        case MotionEvent.ACTION_POINTER_DOWN: {
            // 第二个及之后的手指按下
            int index = event.getActionIndex();
            int id = event.getPointerId(index);
            break;
        }
        case MotionEvent.ACTION_MOVE:
            // 遍历所有手指
            for (int i = 0; i < event.getPointerCount(); i++) {
                int id = event.getPointerId(i);
                float x = event.getX(i);
                float y = event.getY(i);
            }
            break;
        case MotionEvent.ACTION_POINTER_UP:
            // 非最后一个手指抬起
            break;
        case MotionEvent.ACTION_UP:
            // 最后一个手指抬起
            break;
    }
    return true;
}
````
**pointer index vs pointer id：**
- `index`：手指在当前事件中的索引，会变化（某个手指抬起后，其他手指的 index 会重新排列）
- `id`：手指的唯一标识，从按下到抬起不会变化
- 追踪特定手指要用 `id`，获取坐标要用 `index`

## 八、面试题库

### Q1：描述一下 Android 的事件分发机制？

> 事件分发涉及三个核心方法：dispatchTouchEvent（分发）、onInterceptTouchEvent（拦截，ViewGroup 独有）、onTouchEvent（消费）。流程是从 Activity 开始，经过 DecorView 自顶向下传递到目标 View。每一层 ViewGroup 的 dispatchTouchEvent 中先调用 onInterceptTouchEvent 判断是否拦截，不拦截就继续往下分发给子 View，拦截就自己调用 onTouchEvent 处理。如果最底层的 View 不消费事件（onTouchEvent 返回 false），事件会沿着调用链从下往上回传，依次给每一层的 onTouchEvent 处理机会。整个过程像一个 U 型结构：分发时从上往下，回传时从下往上。

### Q2：onTouch 和 onClick 的关系？执行顺序是什么？

> 在 View.dispatchTouchEvent 中，先检查 OnTouchListener.onTouch()，如果 onTouch 返回 true 表示消费了事件，onTouchEvent 就不会被调用，onClick 自然也不会触发。如果 onTouch 返回 false 或没有设置 OnTouchListener，才会调用 onTouchEvent，而 onClick 是在 onTouchEvent 的 ACTION_UP 中通过 performClick 触发的。所以优先级是 onTouch > onTouchEvent > onClick。实际开发中如果同时设置了 OnTouchListener 和 OnClickListener，要注意 onTouch 的返回值，返回 true 会导致 onClick 失效。

### Q3：为什么 DOWN 事件很重要？

> DOWN 事件是整个事件序列的起点，它决定了后续所有 MOVE 和 UP 事件由谁处理。在 ViewGroup.dispatchTouchEvent 中，只有 DOWN 事件才会遍历子 View 寻找事件处理者，找到后记录在 mFirstTouchTarget 中。后续的 MOVE 和 UP 事件直接发给 mFirstTouchTarget，不再遍历。如果 DOWN 事件没有被任何子 View 消费，mFirstTouchTarget 为 null，后续事件全部由 ViewGroup 自己的 onTouchEvent 处理。所以如果一个 View 在 onTouchEvent 中不消费 DOWN 事件（返回 false），它就收不到后续的任何事件。

### Q4：ViewGroup 默认拦截事件吗？View 默认消费事件吗？

> ViewGroup 的 onInterceptTouchEvent 默认返回 false，即默认不拦截，事件会传递给子 View。View 的 onTouchEvent 默认行为取决于是否 clickable 或 longClickable——如果设置了 OnClickListener 或 OnLongClickListener，View 就是 clickable 的，onTouchEvent 默认返回 true 消费事件。Button 默认 clickable 为 true，TextView 默认为 false。这也是为什么给 TextView 设置 OnClickListener 后它就能响应点击了，因为 setOnClickListener 内部会把 clickable 设为 true。

### Q5：什么是滑动冲突？怎么解决？

> 滑动冲突是指父 View 和子 View 都需要处理滑动事件时，系统不知道该把事件给谁。常见场景如 ScrollView 嵌套 RecyclerView（同方向）、ViewPager 嵌套 ListView（不同方向）。解决方案有两种：外部拦截法是在父 View 的 onInterceptTouchEvent 中根据滑动方向判断是否拦截，DOWN 事件不拦截，MOVE 事件根据条件决定；内部拦截法是子 View 通过 requestDisallowInterceptTouchEvent 控制父 View 是否拦截。我更推荐外部拦截法，逻辑清晰，只需要修改父 View 一个地方。内部拦截法需要父子配合，容易出错。

### Q6：requestDisallowInterceptTouchEvent 的原理？

> 这个方法设置了 ViewGroup 的 FLAG_DISALLOW_INTERCEPT 标志位。在 dispatchTouchEvent 中，如果检测到这个标志，就跳过 onInterceptTouchEvent 直接把事件分发给子 View。但有一个关键细节：每次 ACTION_DOWN 事件到来时，ViewGroup 会重置这个标志位。所以 DOWN 事件父 View 始终有机会拦截，子 View 只能控制 MOVE 和 UP 事件的拦截行为。这也是为什么内部拦截法中父 View 的 onInterceptTouchEvent 对 DOWN 必须返回 false 的原因。

### Q7：ACTION_CANCEL 什么时候触发？

> 当子 View 正在处理事件序列时，如果父 View 突然决定拦截后续事件（onInterceptTouchEvent 返回 true），系统会给子 View 发送一个 ACTION_CANCEL，通知它事件序列被中断了。子 View 收到 CANCEL 后应该做清理工作，比如取消长按检测、恢复按压状态等。典型场景：手指在 Button 上按下后滑动到 ScrollView 的滑动阈值，ScrollView 开始拦截事件，Button 收到 CANCEL。

### Q8：事件分发中 mFirstTouchTarget 的作用？

> mFirstTouchTarget 是 ViewGroup 中记录事件处理者的链表。在 DOWN 事件分发时，如果某个子 View 消费了事件（dispatchTouchEvent 返回 true），就会被记录到 mFirstTouchTarget 中。后续的 MOVE 和 UP 事件到来时，ViewGroup 直接把事件发给 mFirstTouchTarget 指向的子 View，不再遍历所有子 View。如果 mFirstTouchTarget 为 null（没有子 View 消费 DOWN），ViewGroup 会调用自己的 onTouchEvent 处理。这个设计避免了每个事件都遍历子 View 的性能开销。

### Q9：如何扩大 View 的点击区域？

> 几种方案：一是用 TouchDelegate，这是官方推荐的方式，可以把触摸区域扩大到指定的 Rect 范围。二是在父 View 的 dispatchTouchEvent 中修改事件坐标。三是增加 padding（但会影响布局）。四是用一个透明的大 View 覆盖在上面。实际项目中 TouchDelegate 最规范。

````java
// TouchDelegate 示例
parentView.post(() -> {
    Rect rect = new Rect();
    targetView.getHitRect(rect);
    rect.inset(-50, -50);  // 四周各扩大 50px
    parentView.setTouchDelegate(new TouchDelegate(rect, targetView));
});
````
### Q10：多点触控中 pointer index 和 pointer id 的区别？

> pointer index 是手指在当前 MotionEvent 中的索引位置，当某个手指抬起后其他手指的 index 会重新排列，所以 index 是不稳定的。pointer id 是手指从按下到抬起的唯一标识，不会因为其他手指的变化而改变。追踪特定手指的轨迹要用 id，通过 findPointerIndex(id) 获取当前的 index，再用 getX(index) 获取坐标。这是实现双指缩放、多指拖拽等手势的基础。如果错误地用 index 追踪手指，当第一个手指抬起后第二个手指的 index 从 1 变成 0，就会出现坐标跳变的 bug。
