## 一、核心概念

### 1.1 DNS 解析

DNS（Domain Name System）是网络请求的第一步，将域名解析为 IP 地址。DNS 解析耗时直接影响首次请求的延迟。

**解析流程：**

````
应用发起请求 → 本地 DNS 缓存 → 系统 DNS 缓存 → 路由器缓存
→ ISP DNS 服务器 → 根域名服务器 → 顶级域名服务器 → 权威域名服务器
````
**关键指标：**
- 一次完整的 DNS 解析通常耗时 20~120ms，弱网环境下可达 200ms+
- DNS 劫持是国内运营商常见问题，会导致解析到错误 IP 或插入广告
- DNS TTL（Time To Live）决定缓存有效期，过短会频繁解析，过长则无法及时切换

**Java 中获取 DNS 解析耗时示例：**

````java
// 通过 OkHttp EventListener 监听 DNS 解析耗时
public class NetworkEventListener extends EventListener {
    private long dnsStartTime;

    @Override
    public void dnsStart(Call call, String domainName) {
        dnsStartTime = System.nanoTime();
    }

    @Override
    public void dnsEnd(Call call, String domainName, List<InetAddress> inetAddressList) {
        long cost = TimeUnit.NANOSECONDS.toMillis(System.nanoTime() - dnsStartTime);
        Log.d("Network", "DNS 解析耗时: " + cost + "ms, domain: " + domainName);
    }
}
````
### 1.2 连接复用（Connection Reuse）

TCP 连接的建立需要三次握手，HTTPS 还需要 TLS 握手，整个过程耗时显著。连接复用是指在一个 TCP 连接上发送多个 HTTP 请求，避免重复建立连接的开销。

**HTTP/1.1 Keep-Alive：**
- 默认开启 `Connection: keep-alive`，同一连接可串行发送多个请求
- 缺点：存在队头阻塞（Head-of-Line Blocking），前一个请求未完成时后续请求必须等待

**连接复用的收益：**

| 场景 | 无复用 | 有复用 | 节省 |
|------|--------|--------|------|
| TCP 握手 | 每次 ~50ms | 仅首次 | ~50ms/请求 |
| TLS 握手 | 每次 ~100ms | 仅首次 | ~100ms/请求 |
| 慢启动 | 每次重新开始 | 持续窗口 | 吞吐量提升 |

**OkHttp 连接池配置：**

````java
// OkHttp 默认连接池：最多 5 个空闲连接，存活 5 分钟
ConnectionPool pool = new ConnectionPool(
    10,              // maxIdleConnections
    5, TimeUnit.MINUTES  // keepAliveDuration
);

OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(pool)
    .build();
````
### 1.3 HTTP/2 多路复用（Multiplexing）

HTTP/2 是对 HTTP/1.1 的重大升级，核心特性是多路复用——在单个 TCP 连接上并行传输多个请求和响应。

**HTTP/1.1 vs HTTP/2 对比：**

````
HTTP/1.1（串行 / 多连接并发）:
连接1: [请求A] ---------> [响应A]
连接2: [请求B] ---------> [响应B]
连接3: [请求C] ---------> [响应C]

HTTP/2（单连接多路复用）:
连接1: [请求A][请求B][请求C] → [响应B][响应A][响应C]
       ↑ 帧（Frame）交错传输，无队头阻塞
````
**HTTP/2 核心特性：**
- **多路复用**：单连接并行传输，消除队头阻塞
- **头部压缩（HPACK）**：使用静态表 + 动态表 + 霍夫曼编码压缩 Header
- **服务端推送（Server Push）**：服务器主动推送资源
- **流优先级（Stream Priority）**：客户端可指定请求优先级
- **二进制分帧**：所有通信在二进制帧层面完成，解析效率更高

**OkHttp 启用 HTTP/2：**

````java
// OkHttp 默认支持 HTTP/2（HTTPS 连接自动协商）
// 通过 Protocol 列表控制协议优先级
OkHttpClient client = new OkHttpClient.Builder()
    .protocols(Arrays.asList(Protocol.HTTP_2, Protocol.HTTP_1_1))
    .build();
````
### 1.4 缓存策略（ETag / Last-Modified / Cache-Control）

HTTP 缓存是减少网络请求、提升响应速度的关键手段。缓存策略分为强缓存和协商缓存两类。

**强缓存（无需请求服务器）：**

| Header | 说明 | 示例 |
|--------|------|------|
| `Cache-Control` | HTTP/1.1 标准，优先级最高 | `max-age=3600, public` |
| `Expires` | HTTP/1.0 标准，绝对时间 | `Thu, 16 Apr 2026 12:00:00 GMT` |

**Cache-Control 常用指令：**
- `max-age=N`：缓存有效期 N 秒
- `no-cache`：每次使用前必须向服务器验证
- `no-store`：禁止任何缓存
- `public`：可被任何中间节点缓存
- `private`：仅客户端可缓存
- `immutable`：资源不会变化，浏览器不再发验证请求

**协商缓存（需向服务器验证）：**

````
第一次请求：
Client → Server
Server → Client: 200 OK
  ETag: "abc123"
  Last-Modified: Thu, 16 Apr 2026 00:00:00 GMT

第二次请求：
Client → Server:
  If-None-Match: "abc123"
  If-Modified-Since: Thu, 16 Apr 2026 00:00:00 GMT
Server → Client: 304 Not Modified（命中缓存，无 Body）
````
| 验证方式 | 请求头 | 响应头 | 精度 |
|----------|--------|--------|------|
| ETag | `If-None-Match` | `ETag` | 内容级别（精确） |
| Last-Modified | `If-Modified-Since` | `Last-Modified` | 秒级别 |

**OkHttp 缓存配置：**

````java
// 配置 10MB 磁盘缓存
int cacheSize = 10 * 1024 * 1024;
Cache cache = new Cache(new File(context.getCacheDir(), "http_cache"), cacheSize);

OkHttpClient client = new OkHttpClient.Builder()
    .cache(cache)
    .addInterceptor(chain -> {
        Request request = chain.request();
        // 无网络时强制使用缓存
        if (!isNetworkAvailable()) {
            request = request.newBuilder()
                .cacheControl(CacheControl.FORCE_CACHE)
                .build();
        }
        return chain.proceed(request);
    })
    .build();
````

## 二、优化方案

### 2.1 DNS 预解析与 HTTPDNS

**问题背景：**
- 传统 LocalDNS 依赖运营商递归解析，存在劫持、缓存不一致、解析慢等问题
- 首次请求新域名时 DNS 解析可能耗时 100ms+

**方案一：DNS 预解析（Pre-resolve）**

在 App 启动或空闲时提前解析常用域名，将结果缓存到内存中：

````java
public class DnsPreResolver {
    private final Map<String, List<InetAddress>> dnsCache = new ConcurrentHashMap<>();
    private final ExecutorService executor = Executors.newFixedThreadPool(2);

    // App 启动时预解析核心域名
    public void preResolve(List<String> domains) {
        for (String domain : domains) {
            executor.execute(() -> {
                try {
                    InetAddress[] addresses = InetAddress.getAllByName(domain);
                    dnsCache.put(domain, Arrays.asList(addresses));
                } catch (UnknownHostException e) {
                    Log.w("DNS", "预解析失败: " + domain);
                }
            });
        }
    }

    public List<InetAddress> getCached(String hostname) {
        return dnsCache.get(hostname);
    }
}
````
**方案二：HTTPDNS**

通过 HTTP 协议向 HTTPDNS 服务器发起域名解析请求，绕过运营商 LocalDNS，避免劫持：

````java
// 自定义 OkHttp Dns 接口，接入 HTTPDNS
public class HttpDns implements Dns {
    private static final String HTTPDNS_SERVER = "https://httpdns.example.com/resolve";

    @Override
    public List<InetAddress> lookup(String hostname) throws UnknownHostException {
        // 1. 优先查 HTTPDNS
        String ip = queryHttpDns(hostname);
        if (ip != null) {
            return Arrays.asList(InetAddress.getAllByName(ip));
        }
        // 2. 降级到系统 DNS
        return Dns.SYSTEM.lookup(hostname);
    }

    private String queryHttpDns(String hostname) {
        // 通过 HTTP GET 请求 HTTPDNS 服务
        // 返回格式: {"ips":["1.2.3.4","5.6.7.8"],"ttl":300}
        try {
            URL url = new URL(HTTPDNS_SERVER + "?host=" + hostname);
            HttpURLConnection conn = (HttpURLConnection) url.openConnection();
            conn.setConnectTimeout(2000);
            conn.setReadTimeout(2000);
            // 解析 JSON 返回 IP...
            return parseIpFromResponse(conn);
        } catch (Exception e) {
            return null;
        }
    }
}

// 使用
OkHttpClient client = new OkHttpClient.Builder()
    .dns(new HttpDns())
    .build();
````
**HTTPDNS 优势：**
- 防劫持：不经过运营商 LocalDNS
- 精准调度：基于客户端 IP 返回最优节点
- 低延迟：HTTP 请求通常比 UDP DNS 更快（有缓存时）
- 可控性强：支持自定义 TTL、容灾策略

### 2.2 连接池复用

连接池是网络优化的基础设施，核心目标是减少 TCP/TLS 握手开销。参考 [[1-OkHttp与Retrofit]] 中的连接管理机制。

**OkHttp 连接池工作原理：**

````
请求到达 → 查找可复用连接（相同 host:port + 相同协议）
  ├─ 命中 → 直接复用，跳过握手
  └─ 未命中 → 新建连接 → 加入连接池
                          ↓
              空闲超时 / 超出上限 → 回收连接
````
**连接池调优建议：**

````java
// 根据业务场景调整连接池参数
ConnectionPool pool;

// 场景1：API 密集型应用（如社交、电商）
pool = new ConnectionPool(15, 5, TimeUnit.MINUTES);

// 场景2：低频请求应用（如工具类 App）
pool = new ConnectionPool(5, 2, TimeUnit.MINUTES);

// 场景3：长连接场景（如 IM、推送）
// 不依赖连接池，使用 WebSocket 或自定义长连接
````
**连接池监控：**

````java
// 定期打印连接池状态
ConnectionPool pool = client.connectionPool();
Log.d("Pool", "空闲连接: " + pool.idleConnectionCount()
    + ", 总连接: " + pool.connectionCount());
````
### 2.3 数据压缩（gzip / Protobuf）

减少传输数据量是提升网络性能最直接的手段。

**gzip 压缩：**

OkHttp 默认支持 gzip，自动添加 `Accept-Encoding: gzip` 并解压响应：

````java
// OkHttp 自动处理 gzip（无需手动配置）
// 但如果手动设置了 Accept-Encoding，则需要自行解压

// 手动 gzip 压缩请求体（上传场景）
public class GzipRequestInterceptor implements Interceptor {
    @Override
    public Response intercept(Chain chain) throws IOException {
        Request original = chain.request();
        if (original.body() == null || original.header("Content-Encoding") != null) {
            return chain.proceed(original);
        }
        Request compressed = original.newBuilder()
            .header("Content-Encoding", "gzip")
            .method(original.method(), gzip(original.body()))
            .build();
        return chain.proceed(compressed);
    }

    private RequestBody gzip(final RequestBody body) {
        return new RequestBody() {
            @Override public MediaType contentType() { return body.contentType(); }
            @Override public long contentLength() { return -1; }
            @Override public void writeTo(BufferedSink sink) throws IOException {
                BufferedSink gzipSink = Okio.buffer(new GzipSink(sink));
                body.writeTo(gzipSink);
                gzipSink.close();
            }
        };
    }
}
````
**Protobuf 替代 JSON：**

Protocol Buffers 是 Google 开发的二进制序列化协议，相比 JSON 体积更小、解析更快。参考 [[数据序列化]]。

````protobuf
// user.proto
syntax = "proto3";
message UserResponse {
    int64 id = 1;
    string name = 2;
    string avatar = 3;
    repeated string tags = 4;
}
````
````java
// Retrofit + Protobuf
Retrofit retrofit = new Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .addConverterFactory(ProtoConverterFactory.create())
    .client(okHttpClient)
    .build();

// 对比数据量（同一份用户列表数据）
// JSON:     ~2.4 KB
// Protobuf: ~1.1 KB（压缩率约 54%）
// gzip JSON: ~0.8 KB
// gzip Protobuf: ~0.6 KB
````
**压缩方案选型：**

| 方案 | 压缩率 | 解析速度 | 可读性 | 适用场景 |
|------|--------|----------|--------|----------|
| JSON | 基准 | 基准 | 高 | 通用 API |
| gzip JSON | 60-80% | 略慢 | 需解压 | 大数据量 API |
| Protobuf | 40-60% | 快 2-5x | 低 | 高频/大数据量 |
| gzip Protobuf | 70-85% | 略慢 | 低 | 极致优化 |


### 2.4 请求合并与优先级

**请求合并（Request Batching）：**

将短时间内的多个请求合并为一个批量请求，减少连接开销和请求次数：

````java
// 请求合并器：收集一定时间窗口内的请求，批量发送
public class RequestBatcher {
    private final List<String> pendingIds = new ArrayList<>();
    private final Handler handler = new Handler(Looper.getMainLooper());
    private final long batchWindowMs = 50; // 50ms 窗口

    public void fetchUserInfo(String userId, Callback callback) {
        synchronized (pendingIds) {
            pendingIds.add(userId);
            if (pendingIds.size() == 1) {
                // 首个请求，启动定时器
                handler.postDelayed(this::flushBatch, batchWindowMs);
            }
        }
    }

    private void flushBatch() {
        List<String> batch;
        synchronized (pendingIds) {
            batch = new ArrayList<>(pendingIds);
            pendingIds.clear();
        }
        // 合并为一个批量请求: POST /api/users/batch {"ids": ["1","2","3"]}
        executeBatchRequest(batch);
    }
}
````
**请求优先级：**

不同请求的重要性不同，应合理分配优先级：

````java
// 基于 OkHttp Dispatcher 控制并发和优先级
public class PriorityRequestManager {
    private final OkHttpClient client;

    public PriorityRequestManager() {
        Dispatcher dispatcher = new Dispatcher();
        dispatcher.setMaxRequests(10);          // 最大并发
        dispatcher.setMaxRequestsPerHost(5);    // 单 Host 最大并发
        client = new OkHttpClient.Builder()
            .dispatcher(dispatcher)
            .build();
    }

    // 通过 Tag 标记优先级，在 Interceptor 中排序
    public void enqueue(Request request, Priority priority) {
        Request tagged = request.newBuilder()
            .tag(Priority.class, priority)
            .build();
        client.newCall(tagged).enqueue(new Callback() {
            @Override public void onFailure(Call call, IOException e) { }
            @Override public void onResponse(Call call, Response response) { }
        });
    }

    public enum Priority { CRITICAL, HIGH, NORMAL, LOW }
}
````
**优先级策略建议：**
- `CRITICAL`：登录、支付等核心链路
- `HIGH`：首屏数据加载、用户主动操作
- `NORMAL`：列表翻页、预加载
- `LOW`：埋点上报、日志同步、非关键图片

### 2.5 弱网优化策略

弱网环境（2G/3G、高丢包率、高延迟）是移动端必须面对的场景。参考 [[性能监控]] 中的网络质量检测。

**网络质量分级：**

````java
public enum NetworkQuality {
    EXCELLENT,  // RTT < 100ms, 带宽 > 2Mbps
    GOOD,       // RTT < 300ms, 带宽 > 500Kbps
    MODERATE,   // RTT < 1000ms, 带宽 > 100Kbps
    POOR,       // RTT > 1000ms 或高丢包
    OFFLINE     // 无网络
}

// 基于网络质量动态调整策略
public class AdaptiveNetworkPolicy {
    public RequestConfig getConfig(NetworkQuality quality) {
        switch (quality) {
            case EXCELLENT:
                return new RequestConfig(10_000, 10_000, true, ImageQuality.HIGH);
            case GOOD:
                return new RequestConfig(15_000, 15_000, true, ImageQuality.MEDIUM);
            case MODERATE:
                return new RequestConfig(20_000, 20_000, false, ImageQuality.LOW);
            case POOR:
                return new RequestConfig(30_000, 30_000, false, ImageQuality.THUMBNAIL);
            default:
                return new RequestConfig(5_000, 5_000, false, ImageQuality.NONE);
        }
    }
}

public class RequestConfig {
    public final int connectTimeoutMs;
    public final int readTimeoutMs;
    public final boolean enablePrefetch;
    public final ImageQuality imageQuality;

    public RequestConfig(int connectTimeout, int readTimeout,
                         boolean enablePrefetch, ImageQuality imageQuality) {
        this.connectTimeoutMs = connectTimeout;
        this.readTimeoutMs = readTimeout;
        this.enablePrefetch = enablePrefetch;
        this.imageQuality = imageQuality;
    }
}
````
**弱网优化核心策略：**

1. **超时动态调整**：弱网下增大超时时间，避免频繁超时重试
2. **降级策略**：降低图片质量、减少非必要请求、关闭预加载
3. **请求重试与退避**：指数退避 + 抖动（Jitter），避免重试风暴
4. **离线缓存**：无网络时展示缓存数据，恢复后同步

````java
// 指数退避重试拦截器
public class RetryInterceptor implements Interceptor {
    private final int maxRetries;

    public RetryInterceptor(int maxRetries) {
        this.maxRetries = maxRetries;
    }

    @Override
    public Response intercept(Chain chain) throws IOException {
        Request request = chain.request();
        IOException lastException = null;

        for (int attempt = 0; attempt <= maxRetries; attempt++) {
            try {
                Response response = chain.proceed(request);
                if (response.isSuccessful() || !isRetryable(response.code())) {
                    return response;
                }
                response.close();
            } catch (IOException e) {
                lastException = e;
            }

            if (attempt < maxRetries) {
                long delay = calculateBackoff(attempt);
                try { Thread.sleep(delay); } catch (InterruptedException ignored) { }
            }
        }
        throw lastException != null ? lastException : new IOException("重试耗尽");
    }

    private long calculateBackoff(int attempt) {
        long base = 1000L * (1L << attempt); // 1s, 2s, 4s...
        long jitter = (long) (base * 0.2 * Math.random()); // 20% 抖动
        return Math.min(base + jitter, 30_000); // 最大 30s
    }

    private boolean isRetryable(int code) {
        return code == 408 || code == 429 || code >= 500;
    }
}
````
### 2.6 网络监控

没有监控就没有优化方向。网络监控是持续优化的基础。

**OkHttp EventListener 全链路监控：**

````java
public class DetailedEventListener extends EventListener {
    private long callStart;
    private final Map<String, Long> timestamps = new HashMap<>();

    @Override public void callStart(Call call) {
        callStart = System.nanoTime();
    }

    @Override public void dnsStart(Call call, String name) {
        timestamps.put("dnsStart", System.nanoTime());
    }

    @Override public void dnsEnd(Call call, String name, List<InetAddress> addresses) {
        timestamps.put("dnsEnd", System.nanoTime());
    }

    @Override public void connectStart(Call call, InetSocketAddress addr, Proxy proxy) {
        timestamps.put("connectStart", System.nanoTime());
    }

    @Override public void connectEnd(Call call, InetSocketAddress addr,
                                     Proxy proxy, Protocol protocol) {
        timestamps.put("connectEnd", System.nanoTime());
    }

    @Override public void requestHeadersEnd(Call call, Request request) {
        timestamps.put("requestEnd", System.nanoTime());
    }

    @Override public void responseHeadersStart(Call call) {
        timestamps.put("responseStart", System.nanoTime());
    }

    @Override public void callEnd(Call call) {
        long total = ms(callStart, System.nanoTime());
        long dns = ms(timestamps.get("dnsStart"), timestamps.get("dnsEnd"));
        long connect = ms(timestamps.get("connectStart"), timestamps.get("connectEnd"));
        long ttfb = ms(timestamps.get("requestEnd"), timestamps.get("responseStart"));

        NetworkMetrics metrics = new NetworkMetrics(
            call.request().url().toString(), total, dns, connect, ttfb
        );
        // 上报到监控平台
        MetricsReporter.report(metrics);
    }

    private long ms(Long start, Long end) {
        if (start == null || end == null) return 0;
        return TimeUnit.NANOSECONDS.toMillis(end - start);
    }
}

// 注册 EventListener
OkHttpClient client = new OkHttpClient.Builder()
    .eventListenerFactory(call -> new DetailedEventListener())
    .build();
````
**核心监控指标：**

| 指标 | 说明 | 告警阈值（参考） |
|------|------|------------------|
| DNS 耗时 | 域名解析时间 | > 200ms |
| TCP 连接耗时 | 三次握手时间 | > 300ms |
| TLS 耗时 | TLS 握手时间 | > 500ms |
| TTFB | 首字节时间 | > 1000ms |
| 总耗时 | 完整请求时间 | > 3000ms |
| 错误率 | 请求失败比例 | > 1% |
| 重试率 | 触发重试的比例 | > 5% |

**监控数据聚合维度：**
- 按域名/接口分组
- 按网络类型（WiFi/4G/5G）分组
- 按地域/运营商分组
- 按时间段（分钟/小时/天）聚合
- P50/P90/P99 分位值统计

## 三、面试题

### Q1：HTTP/1.1 和 HTTP/2 的核心区别是什么？为什么 HTTP/2 更快？

**A：** HTTP/2 相比 HTTP/1.1 有以下核心改进：

1. **多路复用**：HTTP/1.1 在一个连接上只能串行处理请求（虽然有 Pipelining 但实际很少使用），浏览器通常开 6 个并发连接来弥补。HTTP/2 在单个 TCP 连接上可以并行传输多个请求/响应，通过 Stream ID 区分不同请求，彻底消除了应用层的队头阻塞。

2. **头部压缩（HPACK）**：HTTP/1.1 每次请求都携带完整的 Header（Cookie、User-Agent 等可能有几百字节到几 KB）。HTTP/2 使用 HPACK 算法，维护一张动态表，只传输变化的部分，首次请求后 Header 开销大幅降低。

3. **二进制分帧**：HTTP/1.1 是文本协议，解析需要逐字符扫描。HTTP/2 将所有通信拆分为二进制帧（HEADERS 帧、DATA 帧等），解析效率更高。

4. **服务端推送**：服务器可以主动推送客户端可能需要的资源（如 HTML 中引用的 CSS/JS），减少往返次数。

**注意**：HTTP/2 仍然基于 TCP，TCP 层的队头阻塞依然存在（丢包时整个连接阻塞）。这也是 HTTP/3（基于 QUIC/UDP）要解决的问题。

### Q2：ETag 和 Last-Modified 有什么区别？为什么需要 ETag？

**A：**

| 对比项 | Last-Modified | ETag |
|--------|---------------|------|
| 精度 | 秒级（HTTP 日期格式只到秒） | 内容级别（基于内容哈希或版本号） |
| 生成方式 | 文件修改时间 | 内容哈希 / 版本号 |
| 请求头 | `If-Modified-Since` | `If-None-Match` |
| 优先级 | 低 | 高（同时存在时 ETag 优先） |

**为什么需要 ETag：**
- 文件内容未变但修改时间变了（如重新部署），Last-Modified 会导致缓存失效
- 文件在 1 秒内多次修改，Last-Modified 无法区分
- 某些服务器无法精确获取文件修改时间
- ETag 可以是弱验证（`W/"abc"`），允许语义等价但字节不同的响应共享缓存

### Q3：HTTPDNS 的原理是什么？它解决了什么问题？

**A：** HTTPDNS 是通过 HTTP/HTTPS 协议向专用 DNS 服务器发起域名解析请求，替代传统的基于 UDP 的 LocalDNS 解析。

**解决的问题：**
1. **DNS 劫持**：运营商 LocalDNS 可能篡改解析结果，插入广告或重定向。HTTPDNS 走 HTTP 通道，运营商无法劫持。
2. **解析不准确**：LocalDNS 的出口 IP 可能与用户实际位置不一致，导致 CDN 调度到非最优节点。HTTPDNS 基于客户端真实 IP 调度。
3. **解析延迟**：LocalDNS 缓存策略不可控，可能频繁递归查询。HTTPDNS 可自定义缓存策略和预加载。
4. **可用性**：LocalDNS 服务器故障时无法解析。HTTPDNS 可配置多个服务端 + 本地兜底。

**实现要点：**
- 在 OkHttp 中实现 `Dns` 接口，优先走 HTTPDNS，失败降级到系统 DNS
- 本地缓存 + TTL 管理，避免每次请求都查询 HTTPDNS
- HTTPS 场景需要处理 SNI（Server Name Indication）问题：IP 直连时 TLS 握手中的 SNI 字段需要手动设置为域名

### Q4：弱网环境下如何优化网络请求？

**A：** 弱网优化是一个系统性工程，核心思路是「减少请求、减小数据、增强容错」：

1. **网络质量检测**：通过 RTT、带宽、丢包率等指标判断当前网络质量等级
2. **动态超时**：弱网下增大连接和读取超时（如从 10s 调整到 30s），避免频繁超时
3. **数据降级**：降低图片分辨率、减少列表每页条数、关闭视频自动播放
4. **请求精简**：合并请求、去除非必要请求（如埋点延迟上报）、减少轮询频率
5. **智能重试**：指数退避 + 随机抖动，避免重试风暴；区分可重试错误（超时、5xx）和不可重试错误（4xx）
6. **离线优先**：优先展示本地缓存数据，后台静默更新；支持离线操作队列，恢复网络后批量同步
7. **协议优化**：优先使用 HTTP/2（多路复用减少连接数）；考虑 QUIC/HTTP/3（基于 UDP，无 TCP 队头阻塞）

### Q5：OkHttp 的连接池是如何工作的？如何调优？

**A：** OkHttp 的 `ConnectionPool` 维护了一组可复用的 TCP/TLS 连接。参考 [[1-OkHttp与Retrofit]]。

**工作机制：**
- 每次请求时，先在连接池中查找可复用连接（匹配条件：相同 host、port、协议、TLS 配置）
- 命中则直接复用，跳过 TCP/TLS 握手
- 未命中则新建连接，请求完成后放回连接池
- 后台清理线程定期检查：超过 `keepAliveDuration` 的空闲连接会被回收；超过 `maxIdleConnections` 时优先回收最久未使用的连接

**调优建议：**
- `maxIdleConnections`：根据 App 同时请求的域名数和并发量设置。API 密集型建议 10-15，普通应用 5 即可
- `keepAliveDuration`：根据请求频率设置。高频请求设 5-10 分钟，低频设 1-2 分钟
- 多域名场景：如果 App 请求多个不同域名（API、CDN、第三方），连接池需要更大
- 监控连接池命中率：通过 `EventListener.connectionAcquired()` 判断是否复用

### Q6：如何设计一个完整的移动端网络监控方案？

**A：** 一个完整的网络监控方案应覆盖以下维度：

**1. 数据采集层：**
- 通过 OkHttp `EventListener` 采集全链路耗时（DNS、连接、TLS、TTFB、传输）
- 采集请求/响应元数据（URL、Method、状态码、数据大小）
- 采集网络环境信息（WiFi/蜂窝、运营商、信号强度）
- 采集异常信息（超时、连接失败、SSL 错误）

**2. 数据处理层：**
- 本地聚合：按接口、网络类型分组，计算 P50/P90/P99
- 采样策略：全量采集性能数据，错误数据 100% 上报
- 批量上报：攒批 + 定时上报，减少上报本身的网络开销

**3. 分析告警层：**
- 接口级别的耗时趋势、错误率趋势
- 按地域、运营商、网络类型下钻分析
- 异常突增自动告警（如某接口错误率突然从 0.1% 飙升到 5%）

**4. 关键指标：**
- 请求成功率、各阶段耗时分布、错误类型分布
- 连接复用率、缓存命中率
- 弱网占比、超时率、重试率

## 四、实战与踩坑

### 4.1 HTTPS + HTTPDNS 的 SNI 问题

**问题描述：** 使用 HTTPDNS 后，请求变成 IP 直连。但 HTTPS 的 TLS 握手阶段需要通过 SNI（Server Name Indication）告知服务器客户端要访问的域名，以便服务器返回正确的证书。IP 直连时 SNI 为空或为 IP，导致 TLS 握手失败。

**踩坑现象：**
````
javax.net.ssl.SSLPeerUnverifiedException:
Hostname 1.2.3.4 not verified
````
**解决方案：**

````java
// 方案1：自定义 HostnameVerifier + SSLSocketFactory
public class HttpDnsHostnameVerifier implements HostnameVerifier {
    private final String originalHost;

    public HttpDnsHostnameVerifier(String originalHost) {
        this.originalHost = originalHost;
    }

    @Override
    public boolean verify(String hostname, SSLSession session) {
        // 用原始域名验证证书
        return HttpsURLConnection.getDefaultHostnameVerifier()
            .verify(originalHost, session);
    }
}

// 方案2：通过 Interceptor 替换 Host Header（推荐）
public class HttpDnsInterceptor implements Interceptor {
    @Override
    public Response intercept(Chain chain) throws IOException {
        Request request = chain.request();
        String host = request.url().host();
        String ip = HttpDnsService.resolve(host);

        if (ip != null) {
            // 替换 URL 中的域名为 IP，保留原始 Host Header
            HttpUrl newUrl = request.url().newBuilder().host(ip).build();
            request = request.newBuilder()
                .url(newUrl)
                .header("Host", host)  // 保留原始域名
                .build();
        }
        return chain.proceed(request);
    }
}
````
### 4.2 连接池泄漏导致 OOM

**问题描述：** 在某些场景下，Response Body 未正确关闭，导致连接无法回收到连接池，最终连接数持续增长直到 OOM。

**踩坑代码：**

````java
// ❌ 错误：未关闭 Response Body
Response response = client.newCall(request).execute();
String body = response.body().string();
// 忘记 response.close() 或 response.body().close()

// ❌ 错误：异常时未关闭
Response response = client.newCall(request).execute();
try {
    if (!response.isSuccessful()) {
        throw new ApiException(response.code());
        // response 未关闭！
    }
    return parseBody(response);
} finally {
    // 这里也没有 close
}
````
**正确写法：**

````java
// ✅ 正确：try-with-resources 自动关闭
try (Response response = client.newCall(request).execute()) {
    if (!response.isSuccessful()) {
        throw new ApiException(response.code());
    }
    return response.body().string();
}

// ✅ 正确：异步回调中也要关闭
client.newCall(request).enqueue(new Callback() {
    @Override
    public void onResponse(Call call, Response response) {
        try (ResponseBody body = response.body()) {
            String result = body.string();
            // 处理结果...
        } catch (IOException e) {
            // 处理异常
        }
    }

    @Override
    public void onFailure(Call call, IOException e) {
        // 处理失败
    }
});
````
### 4.3 gzip 双重压缩问题

**问题描述：** OkHttp 默认会自动添加 `Accept-Encoding: gzip` 并自动解压响应。如果手动设置了该 Header，OkHttp 不会自动解压，导致拿到的是压缩后的乱码数据。

````java
// ❌ 手动设置后 OkHttp 不再自动解压
Request request = new Request.Builder()
    .url("https://api.example.com/data")
    .header("Accept-Encoding", "gzip")  // 手动设置
    .build();

Response response = client.newCall(request).execute();
String body = response.body().string(); // 乱码！因为是 gzip 压缩数据

// ✅ 正确：不要手动设置，让 OkHttp 自动处理
Request request = new Request.Builder()
    .url("https://api.example.com/data")
    // 不设置 Accept-Encoding
    .build();
````

### 4.5 重试风暴与雪崩

**问题描述：** 服务端短暂故障时，客户端大量重试请求涌入，加剧服务端压力，形成雪崩效应。

**防御策略：**

````java
// 1. 指数退避 + 抖动（已在 2.5 节展示）
// 2. 全局重试预算：限制单位时间内的重试次数
public class RetryBudget {
    private final AtomicInteger retryCount = new AtomicInteger(0);
    private final int maxRetriesPerMinute;
    private volatile long windowStart = System.currentTimeMillis();

    public RetryBudget(int maxRetriesPerMinute) {
        this.maxRetriesPerMinute = maxRetriesPerMinute;
    }

    public boolean allowRetry() {
        long now = System.currentTimeMillis();
        if (now - windowStart > 60_000) {
            retryCount.set(0);
            windowStart = now;
        }
        return retryCount.incrementAndGet() <= maxRetriesPerMinute;
    }
}

// 3. 熔断器模式：连续失败超过阈值时停止请求
public class CircuitBreaker {
    private final int failureThreshold;
    private final long resetTimeoutMs;
    private final AtomicInteger failureCount = new AtomicInteger(0);
    private volatile long lastFailureTime = 0;
    private volatile State state = State.CLOSED;

    public CircuitBreaker(int failureThreshold, long resetTimeoutMs) {
        this.failureThreshold = failureThreshold;
        this.resetTimeoutMs = resetTimeoutMs;
    }

    public boolean allowRequest() {
        if (state == State.OPEN) {
            if (System.currentTimeMillis() - lastFailureTime > resetTimeoutMs) {
                state = State.HALF_OPEN;
                return true; // 允许一个试探请求
            }
            return false;
        }
        return true;
    }

    public void recordSuccess() {
        failureCount.set(0);
        state = State.CLOSED;
    }

    public void recordFailure() {
        lastFailureTime = System.currentTimeMillis();
        if (failureCount.incrementAndGet() >= failureThreshold) {
            state = State.OPEN;
        }
    }

    enum State { CLOSED, OPEN, HALF_OPEN }
}
````
### 4.6 缓存策略不当导致数据不一致

**问题描述：** 对动态数据（如用户信息、订单状态）设置了过长的强缓存，导致用户看到过期数据。

**最佳实践：**

| 数据类型 | 缓存策略 | 示例 |
|----------|----------|------|
| 静态资源（JS/CSS/图片） | 长期强缓存 + 文件名哈希 | `Cache-Control: max-age=31536000, immutable` |
| API 响应（列表数据） | 短期缓存 + 协商验证 | `Cache-Control: max-age=60` + ETag |
| 实时数据（余额、状态） | 不缓存 | `Cache-Control: no-store` |
| 配置数据（开关、AB 实验） | 中等缓存 + 主动刷新 | `Cache-Control: max-age=300` + 推送更新 |

````java
// 针对不同接口设置不同缓存策略
public class CacheStrategyInterceptor implements Interceptor {
    @Override
    public Response intercept(Chain chain) throws IOException {
        Request request = chain.request();
        Response response = chain.proceed(request);

        String path = request.url().encodedPath();
        CacheControl cacheControl;

        if (path.startsWith("/api/config")) {
            cacheControl = new CacheControl.Builder()
                .maxAge(5, TimeUnit.MINUTES).build();
        } else if (path.startsWith("/api/user/balance")) {
            cacheControl = CacheControl.FORCE_NETWORK;
        } else {
            cacheControl = new CacheControl.Builder()
                .maxAge(60, TimeUnit.SECONDS).build();
        }

        return response.newBuilder()
            .header("Cache-Control", cacheControl.toString())
            .build();
    }
}
````
