# 包体积优化

> Android 应用包体积直接影响下载转化率和安装成功率。Google 数据表明 APK 每增大 6MB，安装转化率下降约 1%。本文系统梳理 APK 结构、优化方案、工具链及实战经验。

相关主题：[[4-混淆与ProGuard规则]] · [[5-AAR与APK打包]] · [[3-最小化依赖与体积控制]]

---

## 一、核心概念

### 1.1 APK 文件结构

APK 本质是一个 ZIP 压缩包，解压后主要包含：

````
app-release.apk
├── classes.dex          # Dalvik 字节码（可能有多个 dex）
├── res/                 # 编译后的资源文件（layout/drawable/values 等）
├── resources.arsc       # 资源索引表（资源 ID 与文件的映射）
├── assets/              # 原始资源文件（不经过 aapt 编译）
├── lib/                 # Native SO 库（按 ABI 分目录）
│   ├── armeabi-v7a/
│   ├── arm64-v8a/
│   ├── x86/
│   └── x86_64/
├── META-INF/            # 签名信息
├── AndroidManifest.xml  # 二进制格式的清单文件
└── kotlin/              # Kotlin metadata
````
### 1.2 各部分占比分析

以中型应用（约 30MB）为例：

| 组成部分 | 典型占比 | 说明 |
|---------|---------|------|
| `classes*.dex` | 15%-25% | Java/Kotlin 代码编译产物 |
| `res/` | 20%-35% | 图片资源是大头 |
| `resources.arsc` | 2%-5% | 资源索引表 |
| `assets/` | 5%-15% | 字体、预置数据库、离线包等 |
| `lib/` | 20%-40% | SO 库，ABI 越多越大 |
| `META-INF/` | <1% | 签名文件 |

> **关键洞察**：`lib/`（SO 库）和 `res/`（资源文件）是体积大户，优化这两部分收益最显著。

### 1.3 体积膨胀常见原因

1. **依赖膨胀**：引入大量第三方库，很多功能只用了一小部分 → [[3-最小化依赖与体积控制]]
2. **图片未优化**：使用高分辨率 PNG 而非 WebP/VectorDrawable
3. **多 ABI 全量打包**：同时包含四种架构的 SO
4. **无用代码未移除**：未开启代码缩减或混淆规则过于宽松 → [[4-混淆与ProGuard规则]]
5. **assets 预置过多**：内置大量离线数据、字体、模型文件

## 二、优化方案

### 2.1 代码缩减（R8 / ProGuard）

R8 从 AGP 3.4 开始默认替代 ProGuard，集成代码缩减、混淆和优化。

````groovy
// build.gradle (Module)
android {
    buildTypes {
        release {
            minifyEnabled true
            shrinkResources true
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'),
                    'proguard-rules.pro'
        }
    }
}
````
````properties
# gradle.properties — R8 完整模式（AGP 8.0+ 默认开启）
android.enableR8.fullMode=true
````
自定义 keep 规则：

````proguard
# proguard-rules.pro
-keepclassmembers class * implements java.io.Serializable {
    static final long serialVersionUID;
    private void writeObject(java.io.ObjectOutputStream);
    private void readObject(java.io.ObjectInputStream);
}

# 保留 Gson 反序列化 Bean
-keepclassmembers class com.example.model.** { <fields>; }

# 保留 JNI 方法
-keepclasseswithmembernames class * { native <methods>; }

# 移除 Log 调用（减小 dex）
-assumenosideeffects class android.util.Log {
    public static int v(...);
    public static int d(...);
    public static int i(...);
}
````
代码层面优化 — 用 `@IntDef` 替代枚举：

````java
// ✅ 推荐：@IntDef 替代枚举，减少 dex 体积
public class NetworkState {
    public static final int CONNECTED = 0;
    public static final int DISCONNECTED = 1;
    public static final int CONNECTING = 2;

    @IntDef({CONNECTED, DISCONNECTED, CONNECTING})
    @Retention(RetentionPolicy.SOURCE)
    public @interface State {}
}
````
> 详细混淆配置参考 [[4-混淆与ProGuard规则]]

### 2.2 资源优化

#### WebP 转换

WebP 相比 PNG 可减少 25%-35% 体积，且支持透明通道（minSdk >= 18 有损，>= 19 无损+透明）。

**策略**：大于 100KB 的 PNG → 有损 WebP（质量 75-85），简单图标 → VectorDrawable，9-patch 保留 PNG。

#### 资源混淆（AndResGuard）

将 `res/drawable-xxhdpi/activity_main_background.png` 缩短为 `r/d/a.png`，配合 7zip 压缩：

````groovy
apply plugin: 'AndResGuard'

andResGuard {
    use7zip = true
    useSign = true
    keepRoot = false
    whiteList = [
        "R.drawable.icon",
        "R.drawable.notification_*",
        "R.string.app_name"
    ]
    compressFilePattern = ["*.png", "*.jpg", "*.jpeg", "*.gif", "resources.arsc"]
    sevenzip { artifact = 'com.tencent.mm:SevenZip:1.2.21' }
}
````
#### 移除无用资源

````xml
<!-- res/raw/keep.xml — 精确控制资源保留 -->
<?xml version="1.0" encoding="utf-8"?>
<resources xmlns:tools="http://schemas.android.com/tools"
    tools:shrinkMode="strict"
    tools:keep="@layout/activity_legacy_*"
    tools:discard="@drawable/unused_*" />
````
### 2.3 SO 库优化

#### ABI 过滤

````groovy
android {
    defaultConfig {
        ndk {
            abiFilters 'armeabi-v7a', 'arm64-v8a'
        }
    }
}
````
> 仅保留这两种 ABI 即可覆盖 99%+ 真实设备。

#### SO 动态下发

将非核心 SO 从 APK 中移除，运行时按需下载：

````java
public class SoLoader {
    public static void loadSo(Context context, String soName, LoadCallback callback) {
        String abi = Build.SUPPORTED_ABIS[0];
        String fileName = "lib" + soName + ".so";
        File soFile = new File(context.getFilesDir(), "so/" + abi + "/" + fileName);

        if (soFile.exists()) {
            try {
                System.load(soFile.getAbsolutePath());
                callback.onSuccess();
                return;
            } catch (UnsatisfiedLinkError e) {
                soFile.delete();
            }
        }
        // 从 CDN 下载对应 ABI 的 SO，校验 MD5 后 System.load()
        downloadAndLoad(context, abi, fileName, soFile, callback);
    }

    public interface LoadCallback {
        void onSuccess();
        void onError(Exception e);
    }
}
````
#### SO 编译优化

````cmake
# CMakeLists.txt — 体积最小化
set(CMAKE_C_FLAGS_RELEASE "-Oz -DNDEBUG")
set(CMAKE_CXX_FLAGS_RELEASE "-Oz -DNDEBUG")
set(CMAKE_INTERPROCEDURAL_OPTIMIZATION TRUE)
set(CMAKE_C_FLAGS "${CMAKE_C_FLAGS} -fvisibility=hidden")
````
### 2.4 App Bundle（AAB）

Play Store 根据用户设备自动生成优化 APK，平均减少 15%-20% 下载体积：

````groovy
android {
    bundle {
        language { enableSplit = true }
        density { enableSplit = true }
        abi { enableSplit = true }
    }
}
````
支持动态功能模块（Dynamic Feature Module）按需安装：

````java
public class DynamicModuleManager {
    public static void installModule(Activity activity, String moduleName) {
        SplitInstallManager manager = SplitInstallManagerFactory.create(activity);
        SplitInstallRequest request = SplitInstallRequest.newBuilder()
                .addModule(moduleName)
                .build();
        manager.startInstall(request)
                .addOnSuccessListener(sessionId -> Log.d("Dynamic", "Installed: " + moduleName))
                .addOnFailureListener(e -> Log.e("Dynamic", "Failed", e));
    }
}
````
> AAB 打包细节参考 [[5-AAR与APK打包]]

### 2.5 插件化加载

超大型应用可将非核心功能以插件形式动态下发：

````java
public class PluginManager {
    private final Context mContext;
    private final Map<String, ClassLoader> mLoaders = new HashMap<>();

    public PluginManager(Context context) {
        this.mContext = context.getApplicationContext();
    }

    public void loadPlugin(String pluginPath, String pluginId) {
        File dexDir = new File(mContext.getFilesDir(), "plugin_dex/" + pluginId);
        dexDir.mkdirs();
        DexClassLoader loader = new DexClassLoader(
                pluginPath, dexDir.getAbsolutePath(), null, mContext.getClassLoader());
        mLoaders.put(pluginId, loader);
    }

    public Object createInstance(String pluginId, String className) throws Exception {
        ClassLoader loader = mLoaders.get(pluginId);
        if (loader == null) throw new IllegalStateException("Plugin not loaded: " + pluginId);
        return loader.loadClass(className).getDeclaredConstructor().newInstance();
    }
}
````
**注意**：国内应用商店对插件化有审核限制，需评估合规风险。

## 三、工具

### 3.1 APK Analyzer

Android Studio 内置：`Build → Analyze APK...`，支持查看各文件大小、dex 方法数、资源条目，以及两个 APK 的对比。

````bash
# 命令行使用
apkanalyzer apk file-size app-release.apk
apkanalyzer apk download-size app-release.apk
apkanalyzer dex references app-release.apk
apkanalyzer apk compare old.apk new.apk
````
### 3.2 ApkTool

````bash
apktool d app-release.apk -o output_dir   # 反编译
apktool b output_dir -o repacked.apk       # 重打包
````
用于分析第三方库代码量、资源分布、重复资源等。

### 3.3 其他工具

| 工具 | 用途 |
|------|------|
| `Diffuse` | 对比两个 APK/AAB 差异（Jake Wharton 出品） |
| `bundletool` | 分析 AAB，生成设备专属 APK |
| `ClassyShark` | 可视化查看 APK 中的类和依赖 |

### 3.4 自定义 Gradle 体积监控

````groovy
// build.gradle — CI/CD 体积卡口
ext { apkSizeThreshold = 35.0 }

task checkApkSize {
    dependsOn 'assembleRelease'
    doLast {
        def apkFile = file("${buildDir}/outputs/apk/release/")
                .listFiles()?.find { it.name.endsWith('.apk') }
        if (apkFile == null) throw new GradleException("APK not found!")

        def sizeMB = apkFile.length() / (1024.0 * 1024.0)
        println "APK Size: ${String.format('%.2f', sizeMB)} MB (threshold: ${apkSizeThreshold} MB)"

        if (sizeMB > apkSizeThreshold) {
            throw new GradleException("APK size exceeds threshold!")
        }
    }
}
````
````groovy
// 体积分解报告
task apkSizeBreakdown {
    dependsOn 'assembleRelease'
    doLast {
        def apkFile = file("${buildDir}/outputs/apk/release/")
                .listFiles()?.find { it.name.endsWith('.apk') }
        def totalSize = apkFile.length()
        def breakdown = new TreeMap<String, Long>()

        new java.util.zip.ZipFile(apkFile).withCloseable { zip ->
            zip.entries().each { entry ->
                def cat = entry.name.endsWith('.dex') ? 'dex' :
                        entry.name.startsWith('res/') ? 'res' :
                        entry.name.startsWith('lib/') ? 'lib' :
                        entry.name.startsWith('assets/') ? 'assets' : 'other'
                breakdown.merge(cat, entry.compressedSize, Long::sum)
            }
        }

        breakdown.each { cat, size ->
            def pct = (size * 100.0 / totalSize)
            println String.format("%-10s %8.2f MB  %5.1f%%", cat,
                    size / (1024.0 * 1024.0), pct)
        }
    }
}
````

## 四、面试题

### Q1：APK 体积优化你会从哪些方面入手？按优先级排序。

**A**：按投入产出比排序：
1. **开启 R8 + 资源缩减**（成本最低）— 通常减少 10%-30%
2. **ABI 过滤**：只保留 `armeabi-v7a` + `arm64-v8a`，SO 部分可直接减半
3. **图片优化**：PNG → WebP，小图标 → VectorDrawable
4. **移除无用依赖** → [[3-最小化依赖与体积控制]]
5. **资源混淆**（AndResGuard）— 再减 2%-5%
6. **SO 动态下发 / App Bundle**
7. **插件化**：超大型应用的终极方案

### Q2：R8 和 ProGuard 有什么区别？

**A**：R8 由 Google 维护，从 AGP 3.4 起默认替代 ProGuard。核心区别：R8 将 desugaring、shrinking、obfuscation、dexing 合并为一步，编译更快、产出 dex 更小。完整模式下支持枚举优化、更激进的 tree shaking。R8 兼容 ProGuard 规则语法。

### Q3：`shrinkResources` 的原理和局限性？

**A**：工作流程：先由 R8 移除无用代码，再分析代码中的资源引用，将未引用资源替换为空文件。**局限**：动态资源引用（`getIdentifier("icon_" + type, ...)`）无法被静态分析检测到，需通过 `keep.xml` 手动声明保留；不能移除第三方 AAR 中的无用资源。

### Q4：App Bundle 原理？国内如何替代？

**A**：AAB 上传到 Play Store 后，根据用户设备的密度、ABI、语言生成定制 Split APK。国内替代方案：多渠道分 ABI 打包、SO 动态下发、资源 CDN 按需下载、华为/小米等商店的类似分发优化。

### Q5：如何在 CI/CD 中防止包体积回退？

**A**：1) Gradle 任务卡口，超阈值构建失败；2) 每次构建记录体积并与上版本对比，增量超 5% 告警；3) 分类监控 dex/res/lib 各部分变化；4) 体积数据上报监控平台绘制趋势图；5) 新增依赖必须在 PR 中附带体积影响评估。

### Q6：50MB 的 APK 如何快速定位瓶颈？

**A**：**第一步**：APK Analyzer 查看各目录占比，确定大头（dex? res? lib?）。**第二步**：分类深入 — lib 大则检查 ABI 和可动态下发的 SO；res 大则检查大图和多密度资源；dex 大则定位大依赖。**第三步**：列出每项优化的预期收益和成本，优先做 ABI 过滤、开启 R8 等高性价比项。**第四步**：设置体积阈值集成到 CI。

## 五、实战与踩坑

### 5.1 R8 开启后崩溃

**场景**：Release 包报 `ClassNotFoundException`，因 R8 移除了反射调用的类。

**解决**：查看 `mapping.txt` 确认类是否被移除，针对性添加 keep 规则，或使用 `@Keep` 注解：

````java
@Keep
public class PaymentCallback {
    public void onPaySuccess(String orderId) { /* ... */ }
}
````
> 参考 [[4-混淆与ProGuard规则]]

### 5.2 shrinkResources 误删资源

**场景**：动态拼接资源名 `"ic_category_" + type` 导致资源被误删。

**解决**：在 `res/raw/keep.xml` 中声明 `tools:keep="@drawable/ic_category_*"`。

### 5.3 WebP 兼容性

- minSdk < 18：透明图保留 PNG
- minSdk >= 18：可用有损 WebP
- minSdk >= 19：可用无损 + 透明 WebP
- 9-patch 不建议转 WebP

### 5.4 SO 动态下发安全风险

**场景**：SO 从 CDN 下载后直接 `System.load()`，存在中间人攻击风险。

**解决**：下载后必须校验文件 MD5/SHA256，校验值预置在 APK 中：

````java
public static void loadVerifiedSo(File soFile, String expectedMd5) {
    MessageDigest md = MessageDigest.getInstance("MD5");
    try (FileInputStream fis = new FileInputStream(soFile)) {
        byte[] buf = new byte[8192];
        int len;
        while ((len = fis.read(buf)) != -1) md.update(buf, 0, len);
    }
    String actual = new BigInteger(1, md.digest()).toString(16);
    if (!expectedMd5.equals(actual)) {
        soFile.delete();
        throw new SecurityException("SO verification failed");
    }
    System.load(soFile.getAbsolutePath());
}
````
