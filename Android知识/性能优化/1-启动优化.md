## 一、核心概念

### 1.1 冷启动 / 温启动 / 热启动

| 启动类型 | 进程是否存活 | Application 是否创建 | Activity 是否存活 | 耗时排序 |
|---------|------------|--------------------|-----------------|---------| 
| 冷启动（Cold Start） | ❌ | ❌ 需要创建 | ❌ 需要创建 | 最慢 |
| 温启动（Warm Start） | ✅ | ✅ 已创建 | ❌ 需要重建 | 中等 |
| 热启动（Hot Start） | ✅ | ✅ 已创建 | ✅ 存在 | 最快 |

**冷启动完整流程：**

````
用户点击图标 → Zygote fork 新进程 → 创建 Application
  → attachBaseContext()
  → ContentProvider.onCreate()   ← 常被忽略的耗时点
  → Application.onCreate()
  → Activity.onCreate() → onResume()
  → ViewRootImpl.performTraversals()（首帧绘制）
  → 用户看到界面
````
> 🔑 冷启动的终点是**首帧绘制完成**，而非 `onResume()`。Google 定义了 TTID（Initial Display）和 TTFD（Fully Drawn）两个指标。

### 1.2 启动耗时统计方法

#### 方法一：Logcat 系统日志

````
ActivityTaskManager: Displayed com.example.app/.MainActivity: +1s200ms
````
#### 方法二：adb 命令

````bash
adb shell am start -S -W com.example.app/.MainActivity
# TotalTime: 应用自身启动耗时（重点关注）
# WaitTime: 包含系统调度的总等待时间
````
#### 方法三：代码埋点

````java
public class LaunchTimer {
    private static long startTime = 0L;

    public static void startRecord() { startTime = SystemClock.elapsedRealtime(); }

    public static void endRecord(String tag) {
        Log.d("LaunchTimer", tag + " 耗时: " + (SystemClock.elapsedRealtime() - startTime) + "ms");
    }
}

// Application.attachBaseContext 中调用 startRecord()
// MainActivity.onWindowFocusChanged(true) 中调用 endRecord()
````
#### 方法四：Perfetto（推荐）

````bash
adb shell perfetto -o /data/misc/perfetto-traces/trace.pb -t 10s sched freq idle am wm gfx view
````
自定义 Trace 区间：

````java
Trace.beginSection("MyApp#onCreate");
// ... 初始化逻辑
Trace.endSection();
````
#### 方法五：reportFullyDrawn()

在数据加载完成后调用 `reportFullyDrawn()`，Logcat 会输出 Fully Drawn 时间。

## 二、原理与源码

### 2.1 Application 启动流程（源码级）

核心调用链（基于 AOSP）：

````
Launcher 点击 → AMS.startActivity() → Zygote fork
  → ActivityThread.main()                ← 应用进程入口
    → Looper.prepareMainLooper()
    → ActivityThread.attach()
      → AMS.attachApplication()
        → handleBindApplication()
          → LoadedApk.makeApplication()
            → app.attachBaseContext()     ← 第一个回调
          → installContentProviders()    ← CP 初始化
          → app.onCreate()               ← 第二个回调
    → Looper.loop()                      ← 进入消息循环
````
关键源码（`ActivityThread.handleBindApplication`）：

````java
private void handleBindApplication(AppBindData data) {
    Application app = data.info.makeApplication(data.restrictedBackupMode, null);
    // ContentProvider 在 onCreate 之前初始化！
    if (!ArrayUtils.isEmpty(data.providers)) {
        installContentProviders(app, data.providers);
    }
    mInstrumentation.callApplicationOnCreate(app);
}
````
> ⚠️ 执行顺序：`attachBaseContext()` → `ContentProvider.onCreate()` → `Application.onCreate()`

### 2.2 ContentProvider 初始化陷阱

很多第三方库（[[Firebase]]、LeakCanary、WorkManager）利用 ContentProvider 实现"无侵入初始化"，问题在于：

1. 所有 CP 在 `Application.onCreate()` **之前**初始化
2. 初始化顺序不可控，且都在**主线程**执行
3. 每个 CP 都有独立的创建开销

**排查方法：** 查看合并后 Manifest 中的 `<provider>` 标签数量。

**解决方案：** 使用 [[Jetpack App Startup]] 替代，统一管理初始化。

### 2.3 主线程消息队列与启动

启动过程中的关键事件都通过 Handler 消息调度：`handleBindApplication` → `handleLaunchActivity` → `handleResumeActivity` → `performTraversals`（首帧）。

**利用 IdleHandler 优化：**

````java
// 首帧绘制完成后，主线程空闲时执行非紧急初始化
Looper.myQueue().addIdleHandler(() -> {
    initNonCriticalSDK();
    return false; // 执行一次后移除
});
````
> 🔑 核心思想：将非关键初始化延迟到首帧绘制之后，利用 `IdleHandler` 分散主线程压力。

## 三、优化方案

### 3.1 懒加载（Lazy Initialization）

启动路径上只初始化首屏必需的组件：

````java
public class MyApp extends Application {
    private AnalyticsSDK analyticsSdk;
    private PushClient pushClient;

    public AnalyticsSDK getAnalyticsSdk() {
        if (analyticsSdk == null) analyticsSdk = AnalyticsSDK.init(this);
        return analyticsSdk;
    }

    public PushClient getPushClient() {
        if (pushClient == null) pushClient = new PushClient.Builder(this).build();
        return pushClient;
    }

    @Override
    public void onCreate() {
        super.onCreate();
        // 只初始化启动关键链路
        initCrashReporter();
        initNetworkClient();
    }
}
````
### 3.2 Jetpack App Startup

[[Jetpack App Startup]] 合并多个 ContentProvider 为一个，支持依赖声明和懒加载：

````java
public class NetworkInitializer implements Initializer<NetworkClient> {
    @NonNull
    @Override
    public NetworkClient create(@NonNull Context context) {
        return new NetworkClient.Builder(context).build();
    }

    @NonNull
    @Override
    public List<Class<? extends Initializer<?>>> dependencies() {
        return Collections.singletonList(LoggerInitializer.class);
    }
}
````
````xml
<provider
    android:name="androidx.startup.InitializationProvider"
    android:authorities="${applicationId}.androidx-startup"
    android:exported="false"
    tools:node="merge">
    <meta-data android:name="com.example.NetworkInitializer"
        android:value="androidx.startup" />
    <!-- 禁用某库的自动初始化 -->
    <meta-data android:name="com.thirdparty.SdkInitializer"
        android:value="androidx.startup" tools:node="remove" />
</provider>
````
````java
// 懒加载模式：手动触发
NetworkClient client = AppInitializer.getInstance(context)
    .initializeComponent(NetworkInitializer.class);
````
### 3.3 异步初始化：DAG 任务调度

对于复杂场景，构建基于有向无环图的异步启动框架：

````java
public abstract class StartupTask {
    public boolean runOnMainThread() { return false; }
    public List<Class<? extends StartupTask>> dependencies() { return Collections.emptyList(); }
    public boolean needWait() { return false; }
    public abstract void run(Context context);
}

public class CrashReporterTask extends StartupTask {
    @Override public boolean runOnMainThread() { return true; }
    @Override public boolean needWait() { return true; }
    @Override public void run(Context context) { CrashReporter.init(context); }
}

public class ImageLoaderTask extends StartupTask {
    @Override public List<Class<? extends StartupTask>> dependencies() {
        return Collections.singletonList(CrashReporterTask.class);
    }
    @Override public void run(Context context) { ImageLoader.init(context); }
}
````
调度器根据依赖关系拓扑排序，主线程任务同步执行，异步任务提交线程池，通过 `CountDownLatch` 等待关键任务完成。

### 3.4 线程优化

````kotlin
// ❌ 启动时创建大量线程 → 线程创建开销 + CPU 竞争
Thread { initSDK1() }.start()
Thread { initSDK2() }.start()

// ✅ 使用协程 + 受控并发度
private val startupDispatcher = Dispatchers.IO.limitedParallelism(4)

override fun onCreate() {
    super.onCreate()
    runBlocking {
        listOf(
            async(startupDispatcher) { initCrashReporter() },
            async(startupDispatcher) { initNetwork() },
        ).awaitAll()
    }
    // 非关键任务不阻塞启动
    GlobalScope.launch(Dispatchers.IO) { initAnalytics() }
}
````

````java

public class PreloadTask extends StartupTask {

        String[] classes = {"androidx.recyclerview.widget.RecyclerView", "com.google.gson.Gson"};
````

## 四、面试题

### Q1: 冷启动、温启动、热启动的区别？冷启动的终点在哪里？

**答：** 核心区别在于进程和 Activity 的存活状态。冷启动时进程不存在，需要 Zygote fork 并完整执行 Application + Activity 创建；温启动时进程存在但 Activity 被销毁需重建；热启动时两者都存在，只走 `onRestart` → `onResume`。冷启动终点是**首帧绘制完成**（`ViewRootImpl.performTraversals()` 执行完毕），而非 `onResume()`。Google 定义了 TTID（首帧显示）和 TTFD（内容完全渲染，需调用 `reportFullyDrawn()`）两个指标。

### Q2: attachBaseContext、ContentProvider.onCreate、Application.onCreate 的执行顺序？为什么重要？

**答：** 顺序是 `attachBaseContext()` → `ContentProvider.onCreate()` → `Application.onCreate()`。重要性在于：很多第三方库（Firebase、WorkManager 等）利用 ContentProvider 自动初始化，这些代码在 `Application.onCreate()` 之前就已执行，开发者无法控制时机和顺序，且都在主线程运行，直接拖慢启动。解决方案是使用 [[Jetpack App Startup]] 合并管理。

### Q3: 如何系统地优化冷启动时间？

**答：** 从多个维度入手：
1. **初始化维度**：将任务分为"必须同步"、"可异步"、"可延迟"三类，用 App Startup 或 DAG 调度器管理，`IdleHandler` 延迟非关键任务
2. **线程维度**：受控线程池或协程 `limitedParallelism`，避免线程爆炸和锁竞争。参考 [[线程优化]]
3. **布局维度**：`ConstraintLayout` 减少层级，`ViewStub` 延迟加载。参考 [[布局优化]]
4. **IO 维度**：避免启动路径磁盘 IO，用 [[MMKV]] 替代 SharedPreferences
5. **编译维度**：Baseline Profile 实现 AOT 编译，R8 优化 Dex
6. **业务维度**：闪屏页视觉反馈，非首屏数据延迟加载

### Q4: IdleHandler 是什么？启动优化中如何使用？

**答：** `IdleHandler` 是 `MessageQueue` 的回调接口，主线程空闲时触发。启动优化中用于将非关键初始化延迟到首帧绘制后执行。注意事项：执行时机不确定（主线程繁忙时可能很久才触发）；不要执行耗时操作（会阻塞后续消息）；多个 IdleHandler 在同一空闲期依次执行；有时效性要求的任务建议用 `postDelayed` 替代。

### Q5: 启动优化如何做线上监控和防劣化？

**答：** 监控方案：`attachBaseContext` 记录起始时间，首页 `onWindowFocusChanged(true)` 记录结束时间上报 APM；分段统计各阶段耗时定位瓶颈；设置慢启动阈值（如冷启动 > 2s）告警。防劣化：CI 中集成 Macrobenchmark 自动化测试，设置启动耗时基线，PR 合入前自动对比，定期 Review `Application.onCreate()` 新增代码。

````java
@RunWith(AndroidJUnit4.class)
public class StartupBenchmark {
    @Rule
    public MacrobenchmarkRule benchmarkRule = new MacrobenchmarkRule();

    @Test
    public void startupCold() {
        benchmarkRule.measureRepeated(
            "com.example.app",
            Collections.singletonList(new StartupTimingMetric()),
            5,
            StartupMode.COLD,
            scope -> { scope.pressHome(); scope.startActivityAndWait(); return null; }
        );
    }
}
````
### Q6: 为什么 MultiDex 会影响启动速度？

**答：** Android 5.0 以下（Dalvik），方法数超 65536 时 `MultiDex.install()` 需在主线程解压加载额外 dex 文件，涉及大量 IO 和 dexopt，可能耗时数秒。Android 5.0+（ART）安装时已将所有 dex 编译为 OAT，不存在此问题。但 `minSdk < 21` 仍需处理；即使 `minSdk >= 21`，R8 的分包策略也会影响类加载顺序，启动路径上的类应尽量在主 dex 中。

## 五、实战与踩坑

### 5.1 实战案例：启动优化 2.8s → 1.1s

| 阶段 | 耗时 | 优化措施 | 节省 |
|------|------|---------|------|
| ContentProvider 初始化 | 450ms | App Startup 合并 | 350ms |
| Application.onCreate | 900ms | 异步初始化 + 懒加载 | 700ms |
| Activity.onCreate | 600ms | ConstraintLayout + ViewStub | 300ms |
| 首帧绘制 | 350ms | 图片降采样 + 预加载缓存 | 150ms |
| 数据加载 | 500ms | 数据预取 + 骨架屏 | 200ms |

### 5.2 踩坑记录

#### 坑 1：SharedPreferences apply() 阻塞

`apply()` 将写入任务加入 QueuedWork，`Activity.onPause()` 时会等待完成。启动时大量 `apply()` 可能导致页面切换卡顿。**方案：** 用 [[MMKV]] 替代。

#### 坑 2：WebView 首次初始化耗时 200-500ms

````java
// IdleHandler 中预创建 WebView 池
public class WebViewPool {
    private static final List<WebView> pool = new ArrayList<>();

    public static void preload(Context context) {
        Looper.myQueue().addIdleHandler(() -> {
            pool.add(new WebView(new MutableContextWrapper(context.getApplicationContext())));
            return false;
        });
    }

    public static WebView obtain(Context context) {
        if (!pool.isEmpty()) return pool.remove(0);
        return new WebView(context);
    }
}
````
#### 坑 3：协程 Dispatchers.IO 线程爆炸

`Dispatchers.IO` 默认最多 64 线程，启动时大量 `launch(Dispatchers.IO)` 可能创建过多线程。**方案：** 使用 `Dispatchers.IO.limitedParallelism(4)` 控制并发度。

#### 坑 4：windowBackground 避免白屏

````xml
<style name="Theme.App.Starting" parent="Theme.MaterialComponents.DayNight.NoActionBar">
    <item name="android:windowBackground">@drawable/splash_background</item>
</style>
````
在 `setContentView` 前调用 `setTheme(R.style.Theme_App_Normal)` 切换回正常主题。配合 [[SplashScreen API]] 实现更优雅的启动体验。
