# 依赖管理

> Gradle 依赖管理是构建系统的核心能力，涵盖依赖声明、传递控制、版本冲突解决以及 Version Catalog 统一管理。本文从配置类型对比、解析原理、实战踩坑三个维度展开。

相关笔记：[[Gradle生命周期与Task]] · [[版本管理与发布]] · [[最小化依赖与体积控制]]

---

## 一、核心概念

### 1.1 四种依赖配置对比

| 配置 | 编译期可见 | 运行期可见 | 传递给消费者 | 典型场景 |
|------|-----------|-----------|-------------|---------|
| `api` | ✅ | ✅ | ✅ 传递 | 公共 API 中暴露的类型，如模块对外接口使用了某库的类 |
| `implementation` | ✅ | ✅ | ❌ 不传递 | 内部实现细节，消费者无需感知 |
| `compileOnly` | ✅ | ❌ | ❌ 不传递 | 编译期注解处理器（Lombok）、Servlet API 等容器提供的依赖 |
| `runtimeOnly` | ❌ | ✅ | ❌ 不传递 | JDBC 驱动、SLF4J 绑定实现、日志桥接 |

> [!tip] 选择原则
> 默认使用 `implementation`，只有当依赖的类型出现在模块的**公共 API**（public/protected 方法签名、父类、接口）中时，才升级为 `api`。这是 [[最小化依赖与体积控制]] 的第一步。

#### Groovy 示例

```groovy
plugins {
    id 'java-library'  // 必须应用 java-library 才能使用 api
}

dependencies {
    api 'com.google.guava:guava:32.1.3-jre'           // 公共 API 暴露 Guava 类型
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.16.0'  // 内部序列化
    compileOnly 'org.projectlombok:lombok:1.18.30'     // 编译期注解
    runtimeOnly 'mysql:mysql-connector-java:8.0.33'    // 运行时驱动
}
```

### 1.2 依赖传递规则详解

Gradle 的依赖传递遵循以下核心规则：

**规则一：`api` 依赖会传递，`implementation` 不会**

```
模块 A  --api-->  Guava
模块 B  --implementation-->  模块 A

结果：模块 B 的编译 classpath 中包含 Guava（因为 A 用 api 声明了 Guava）
```

```
模块 A  --implementation-->  Jackson
模块 B  --implementation-->  模块 A

结果：模块 B 的编译 classpath 中不包含 Jackson（A 用 implementation 声明）
      但模块 B 的运行时 classpath 中包含 Jackson
```

**规则二：传递深度不限**

```
C --api--> B --api--> A
D --implementation--> C

结果：D 编译期可见 A 和 B（api 链条完整传递）
```

**规则三：传递链中任一环节使用 `implementation`，链条即断**

```
C --api--> B --implementation--> A
D --implementation--> C

结果：D 编译期可见 B，但不可见 A（B→A 用了 implementation，链条断裂）
```

> [!important] 实际影响
> 合理使用 `implementation` 可以显著减少编译 classpath 的大小，加速增量编译。当模块 A 的 `implementation` 依赖升级版本时，只有 A 本身需要重新编译，依赖 A 的模块 B 不受影响。

### 1.3 Version Catalog（libs.versions.toml）

Version Catalog 是 Gradle 7.0 引入、7.4.1 稳定的**集中式版本管理**方案，通过 `gradle/libs.versions.toml` 文件统一声明所有依赖的版本号和坐标。

#### 核心结构

```toml
# gradle/libs.versions.toml

[versions]
kotlin = "1.9.22"
agp = "8.2.2"
jackson = "2.16.0"
guava = "32.1.3-jre"
lombok = "1.18.30"
mysql = "8.0.33"
springBoot = "3.2.1"

[libraries]
# 格式：别名 = { group = "...", name = "...", version.ref = "..." }
guava = { group = "com.google.guava", name = "guava", version.ref = "guava" }
jackson-databind = { group = "com.fasterxml.jackson.core", name = "jackson-databind", version.ref = "jackson" }
jackson-core = { group = "com.fasterxml.jackson.core", name = "jackson-core", version.ref = "jackson" }
lombok = { group = "org.projectlombok", name = "lombok", version.ref = "lombok" }
mysql-connector = { group = "mysql", name = "mysql-connector-java", version.ref = "mysql" }

[bundles]
# 将多个库打包为一组
jackson = ["jackson-databind", "jackson-core"]

[plugins]
kotlin-jvm = { id = "org.jetbrains.kotlin.jvm", version.ref = "kotlin" }
android-application = { id = "com.android.application", version.ref = "agp" }
spring-boot = { id = "org.springframework.boot", version.ref = "springBoot" }
```

#### 在 build.gradle 中引用

```groovy
plugins {
    alias(libs.plugins.kotlin.jvm)
    alias(libs.plugins.spring.boot)
}

dependencies {
    api libs.guava
    implementation libs.bundles.jackson    // 引用 bundle，一次添加多个库
    compileOnly libs.lombok
    runtimeOnly libs.mysql.connector
}
```

> [!note] 别名命名规则
> TOML 中的 `-` 和 `_` 在代码中统一转为 `.`。例如 `jackson-databind` 在 Groovy 中写作 `libs.jackson.databind`。

## 二、原理与实践

### 2.1 依赖解析流程

Gradle 的依赖解析发生在 [[Gradle生命周期与Task]] 中的 **Configuration 阶段之后、Task 执行之前**：

```
1. 构建依赖图 → 解析直接依赖 + 递归传递依赖 → DAG
2. 版本冲突检测 → 同一模块多版本 → 触发冲突解决
3. 版本选择 → 默认 newest-wins → 应用 BOM/Platform 约束
4. 生成 compileClasspath / runtimeClasspath → 缓存到 .gradle/caches
```

#### 查看依赖树

```bash
./gradlew :app:dependencies --configuration runtimeClasspath
./gradlew :app:dependencyInsight --dependency guava --configuration compileClasspath
```

### 2.2 版本冲突解决策略

当依赖图中同一库出现多个版本时，Gradle 提供以下策略：

#### 策略一：newest-wins（默认）

Gradle 默认选择所有候选版本中的**最高版本**。

```
模块 A 依赖 Guava 31.0
模块 B 依赖 Guava 32.1
→ 最终选择 Guava 32.1
```

#### 策略二：fail-on-conflict

遇到版本冲突时直接构建失败，强制开发者显式解决。

```groovy
configurations.all {
    resolutionStrategy {
        failOnVersionConflict()
    }
}
```

#### 策略三：force 强制指定版本

```groovy
configurations.all {
    resolutionStrategy {
        force 'com.google.guava:guava:31.1-jre'  // 强制降级
    }
}
```

#### 策略四：strictly / require / prefer 精细控制

```groovy
dependencies {
    implementation('com.google.guava:guava') {
        version {
            strictly '[31.0, 32.0['   // 严格限定范围，超出则失败
            prefer '31.1-jre'         // 在范围内优先选择此版本
        }
    }
}
```

```groovy
dependencies {
    implementation('com.fasterxml.jackson.core:jackson-databind') {
        version {
            require '2.16.0'   // 最低要求 2.16.0，可被更高版本覆盖
        }
    }
}
```

> [!warning] strictly vs force
> `strictly` 是**依赖级别**的约束，语义更清晰，推荐使用。`force` 是**全局配置级别**的覆盖，粒度粗且容易被遗忘。Gradle 官方推荐用 `strictly` 替代 `force`。

#### 策略五：exclude 排除传递依赖

```groovy
dependencies {
    implementation('org.apache.httpcomponents:httpclient:4.5.14') {
        exclude group: 'commons-logging', module: 'commons-logging'
    }
}

// 全局排除
configurations.all {
    exclude group: 'commons-logging', module: 'commons-logging'
}
```

#### 策略六：reject 拒绝特定版本

```groovy
dependencies {
    implementation('com.google.guava:guava') {
        version {
            require '32.+'
            reject '32.0-android'  // 明确拒绝某个版本
        }
    }
}
```

### 2.3 BOM 统一版本管理

BOM（Bill of Materials）是一种特殊的 POM 文件，只声明版本号不引入实际依赖。通过 `platform()` 导入 BOM，可以统一管理一组库的版本。

```groovy
dependencies {
    // 导入 Spring Boot BOM
    implementation platform('org.springframework.boot:spring-boot-dependencies:3.2.1')

    // 以下依赖无需指定版本，由 BOM 统一管理
    implementation 'org.springframework.boot:spring-boot-starter-web'
    implementation 'org.springframework.boot:spring-boot-starter-data-jpa'
    implementation 'com.fasterxml.jackson.core:jackson-databind'
}
```

#### 自定义 BOM

在多模块项目中，可以创建自己的 BOM 模块：

```groovy
// bom/build.gradle
plugins {
    id 'java-platform'
}

javaPlatform {
    allowDependencies()  // 允许 BOM 继承其他 BOM
}

dependencies {
    api platform('org.springframework.boot:spring-boot-dependencies:3.2.1')

    constraints {
        api 'com.google.guava:guava:32.1.3-jre'
        api 'org.apache.commons:commons-lang3:3.14.0'
        api 'cn.hutool:hutool-all:5.8.24'
    }
}
```

```groovy
// 其他模块引用自定义 BOM
dependencies {
    implementation platform(project(':bom'))

    implementation 'com.google.guava:guava'          // 版本由 BOM 控制
    implementation 'org.apache.commons:commons-lang3' // 版本由 BOM 控制
}
```

> [!tip] BOM vs Version Catalog
> - **BOM** 参与依赖解析，可以被传递，适合对外发布的库项目（参考 [[版本管理与发布]]）
> - **Version Catalog** 是构建脚本层面的便利工具，不参与依赖解析，适合项目内部统一管理
> - 两者可以结合使用：Version Catalog 管理 BOM 本身的版本号

### 2.5 多模块构建依赖优化

在多模块项目中，依赖管理的复杂度会急剧上升。以下是关键优化策略：

#### 策略一：使用 Convention Plugin 统一依赖配置

参考 [[Gradle生命周期与Task]] 中的 buildSrc 机制：

```groovy
// buildSrc/src/main/groovy/common-conventions.gradle
plugins {
    id 'java-library'
}

java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}

dependencies {
    // 所有模块共享的依赖
    compileOnly libs.lombok
    annotationProcessor libs.lombok
    testImplementation platform(libs.junit.bom)
    testImplementation libs.junit.jupiter
}

tasks.withType(Test).configureEach {
    useJUnitPlatform()
}
```

```groovy
// 各子模块只需应用约定插件
// module-user/build.gradle
plugins {
    id 'common-conventions'
}

dependencies {
    // 只声明本模块特有的依赖
    implementation project(':module-common')
    implementation libs.mybatis.plus.boot.starter
}
```

#### 策略二：依赖缓存策略

```groovy
// root build.gradle
subprojects {
    configurations.all {
        resolutionStrategy {
            cacheDynamicVersionsFor 10, 'minutes'
            cacheChangingModulesFor 4, 'hours'
        }
    }
}
```

#### 策略三：模块间依赖使用 project()

```groovy
dependencies {
    implementation project(':module-common')  // ✅ 推荐
    // implementation 'com.example:module-common:1.0.0'  // ❌ 避免引用兄弟模块坐标
}
```

#### 策略四：合理拆分模块边界

```
project-root/
├── gradle/libs.versions.toml     # 统一版本
├── bom/build.gradle              # 自定义 BOM
├── buildSrc/                     # Convention Plugins
├── module-common/                # api 声明对外暴露的依赖
├── module-domain/                # 尽量零外部依赖
├── module-service/               # implementation 内部依赖
└── module-app/                   # runtimeOnly 驱动和绑定
```

## 三、面试题

### Q1：api 和 implementation 的区别是什么？什么时候用 api？

**A：** `api` 和 `implementation` 都会将依赖加入编译和运行 classpath，核心区别在于**依赖传递**：

- `api`：依赖会传递给消费者模块的编译 classpath。当你的模块公共 API（public 方法签名、返回类型、父类、接口）中使用了某个库的类型时，必须用 `api`。
- `implementation`：依赖不会传递给消费者的编译 classpath（但会出现在运行时 classpath）。模块内部使用的依赖应该用 `implementation`。

使用 `api` 需要应用 `java-library` 插件。默认应优先使用 `implementation`，这样可以减少消费者的编译 classpath，加速增量编译，并降低意外依赖泄漏的风险。

### Q2：Gradle 如何解决版本冲突？默认策略是什么？

**A：** Gradle 默认使用 **newest-wins** 策略，即当依赖图中同一库出现多个版本时，自动选择最高版本。

可以通过以下方式自定义：
- `failOnVersionConflict()`：遇到冲突直接失败，强制开发者显式解决
- `force`：全局强制指定某个版本（不推荐，粒度粗）
- `strictly`：依赖级别的严格版本约束，超出范围则构建失败（推荐）
- `prefer`：在约束范围内的优先版本
- `reject`：明确拒绝某个版本
- `exclude`：排除特定传递依赖

实际项目中推荐结合 BOM + `strictly` 来管理版本，避免使用 `force`。

### Q3：Version Catalog 和 BOM 有什么区别？能否同时使用？

**A：** 两者解决不同层面的问题：

| 维度 | Version Catalog | BOM |
|------|----------------|-----|
| 作用层面 | 构建脚本（build script） | 依赖解析（dependency resolution） |
| 是否参与解析 | ❌ 仅是别名和版本号的映射 | ✅ 作为 platform 参与版本仲裁 |
| 是否可传递 | ❌ 仅项目内部可见 | ✅ 可以发布到 Maven 仓库供其他项目使用 |
| 文件格式 | TOML | POM XML |
| 适用场景 | 项目内部统一管理 | 对外发布版本约束 |

两者可以且推荐同时使用：在 Version Catalog 中管理 BOM 的版本号，在 `build.gradle` 中通过 Catalog 别名引入 BOM。

```groovy
// libs.versions.toml 中声明 BOM 版本
// [libraries]
// spring-boot-dependencies = { group = "org.springframework.boot", name = "spring-boot-dependencies", version.ref = "springBoot" }

// build.gradle 中使用
dependencies {
    implementation platform(libs.spring.boot.dependencies)
    implementation 'org.springframework.boot:spring-boot-starter-web'  // 版本由 BOM 控制
}
```

### Q4：compileOnly 和 runtimeOnly 分别在什么场景使用？

**A：**

**compileOnly** — 编译期需要，运行期不需要（由容器或框架提供）：
- Lombok（编译期注解处理，生成代码后不再需要）
- Servlet API（由 Tomcat/Jetty 等容器提供）
- 编译期检查注解（如 `@Nullable`、`@NonNull`）
- Gradle/Maven 插件开发中的 Gradle API

**runtimeOnly** — 编译期不需要，运行期需要：
- JDBC 驱动（代码中只用 `java.sql` 接口，驱动实现运行时加载）
- SLF4J 绑定实现（如 `logback-classic`，代码中只用 SLF4J API）
- 数据库连接池的特定实现

### Q5：如何排查依赖冲突导致的 NoSuchMethodError？

**A：** 排查步骤：

1. 根据异常堆栈定位所属库
2. `./gradlew :module:dependencyInsight --dependency <库名> --configuration runtimeClasspath`
3. 分析输出中的版本选择原因（如 "by conflict resolution"）
4. 使用 `strictly` 锁定正确版本，或 `exclude` 排除错误传递路径，或引入 BOM 统一管理

### Q6：多模块项目中如何避免依赖版本不一致？

**A：** 分层管理：

1. **Version Catalog**：集中声明版本号，所有模块通过别名引用
2. **Convention Plugin**（buildSrc）：公共依赖配置抽取为约定插件
3. **BOM / Platform**：Spring 用官方 BOM，内部库创建自定义 BOM
4. **依赖锁定**：`gradle.lockfile` 锁定解析结果

```groovy
dependencyLocking { lockAllConfigurations() }
// 生成锁文件：./gradlew dependencies --write-locks
```

## 四、实战与踩坑

### 踩坑 1：api 滥用导致编译雪崩

**现象：** 修改一个底层工具库的 `implementation` 依赖版本，触发了整个项目所有模块的重新编译，CI 构建时间从 3 分钟暴涨到 15 分钟。

**原因：** 底层模块将大量依赖声明为 `api`，导致依赖变更沿着 `api` 链条传递到所有上层模块，Gradle 认为这些模块的编译 classpath 发生了变化，触发全量重编译。

**解决：**

```groovy
// ❌ 之前：module-common/build.gradle
dependencies {
    api 'com.google.guava:guava:32.1.3-jre'
    api 'cn.hutool:hutool-all:5.8.24'
    api 'com.fasterxml.jackson.core:jackson-databind:2.16.0'
    api 'org.apache.commons:commons-lang3:3.14.0'
}

// ✅ 之后：只有公共 API 暴露的类型才用 api
dependencies {
    api 'com.google.guava:guava:32.1.3-jre'           // Optional 等类型出现在公共接口中
    implementation 'cn.hutool:hutool-all:5.8.24'       // 内部工具方法使用
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.16.0'  // 内部序列化
    implementation 'org.apache.commons:commons-lang3:3.14.0'             // 内部使用
}
```

> [!tip] 审查方法
> 对每个 `api` 依赖问自己：这个库的类型是否出现在本模块的 public/protected 方法签名中？如果不是，降级为 `implementation`。详见 [[最小化依赖与体积控制]]。

### 踩坑 2：Version Catalog 别名中的特殊字符

**现象：** 在 `libs.versions.toml` 中定义了 `spring-boot-starter-web`，但在 `build.gradle` 中写 `libs.spring-boot-starter-web` 报错找不到。

**原因：** Version Catalog 的别名映射规则是将 `-`、`_`、`.` 统一转为访问器的 `.` 分隔符。所以 `spring-boot-starter-web` 对应 `libs.spring.boot.starter.web`。

**解决：**

```toml
# libs.versions.toml
[libraries]
spring-boot-starter-web = { group = "org.springframework.boot", name = "spring-boot-starter-web" }
```

```groovy
// build.gradle
dependencies {
    // ❌ 错误
    // implementation libs.spring-boot-starter-web

    // ✅ 正确：- 转为 .
    implementation libs.spring.boot.starter.web
}
```

### 踩坑 3：platform() 和 enforcedPlatform() 的区别

**现象：** 使用 `platform()` 导入 BOM 后，某个传递依赖的版本仍然高于 BOM 中声明的版本，BOM 没有生效。

**原因：** `platform()` 导入的 BOM 版本约束是**建议性的**（相当于 `require`），如果依赖图中有更高版本，newest-wins 策略会选择更高版本。

**解决：** 使用 `enforcedPlatform()` 强制 BOM 版本（相当于 `strictly`）。

```groovy
dependencies {
    // 建议性约束：BOM 版本可被更高版本覆盖
    implementation platform(libs.spring.boot.dependencies)

    // 强制性约束：BOM 版本不可被覆盖（慎用，可能导致兼容性问题）
    implementation enforcedPlatform(libs.spring.boot.dependencies)
}
```

> [!warning] 慎用 enforcedPlatform
> 强制降级可能导致运行时 `NoSuchMethodError`。建议先用 `platform()` + `dependencyInsight` 分析冲突原因，再决定是否需要强制。

### 踩坑 4：SNAPSHOT 依赖缓存不更新

**现象：** 本地开发时，依赖的 SNAPSHOT 版本已经在 Nexus 上更新了，但 Gradle 构建仍然使用旧版本。

**原因：** Gradle 默认缓存 changing modules（SNAPSHOT）24 小时。

**解决：**

```groovy
configurations.all {
    resolutionStrategy {
        cacheChangingModulesFor 0, 'seconds'  // 开发期间禁用缓存
    }
}
```

或者使用命令行参数刷新：

```bash
./gradlew build --refresh-dependencies
```

> [!note] 生产建议
> 开发期间可以设置较短的缓存时间，但 CI/CD 环境中建议保留默认缓存策略以加速构建。发布正式版本时应避免依赖 SNAPSHOT，参考 [[版本管理与发布]]。

### 踩坑 5：Lombok + MapStruct 注解处理器顺序

**现象：** 同时使用 Lombok 和 MapStruct 时，MapStruct 生成的 Mapper 实现类中 getter/setter 调用报错。

**原因：** 注解处理器的执行顺序不确定，MapStruct 可能在 Lombok 生成 getter/setter 之前就尝试读取属性。

**解决：** 使用 `annotationProcessor` 配置并确保 Lombok 的处理器先执行：

```groovy
dependencies {
    compileOnly libs.lombok
    compileOnly libs.mapstruct

    annotationProcessor libs.lombok                // Lombok 先处理
    annotationProcessor libs.mapstruct.processor   // MapStruct 后处理
    // 关键：添加 lombok-mapstruct-binding
    annotationProcessor 'org.projectlombok:lombok-mapstruct-binding:0.2.0'
}
```

### 踩坑 6：依赖冲突导致的隐蔽 ClassNotFoundException

**现象：** 编译正常，运行时特定代码路径抛出 `ClassNotFoundException`。

**原因：** 依赖 A v1.0 包含类 `Foo`，v2.0 移除了该类。模块 X 通过反射使用 `Foo`，模块 Y 引入 A v2.0，newest-wins 选了 v2.0，编译通过但运行时报错。

**解决：** 用 `strictly` 锁定版本，或 `exclude` + 显式声明正确版本。在 CI 中启用 `failOnVersionConflict()` 预防。
