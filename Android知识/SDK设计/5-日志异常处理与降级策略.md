# 日志、异常处理与降级策略

> SDK 的健壮性不仅体现在功能正确，更体现在异常场景下的优雅处理。日志是调试的眼睛，异常处理是安全的盾牌，降级策略是体验的兜底。三者协同，构成 SDK 稳定性的核心防线。

相关主题：[[1-API设计原则]] · [[4-混淆与ProGuard规则]] · [[SDK初始化与生命周期]] · [[版本管理与兼容性]]

---

## 一、核心概念

### 1.1 SDK 日志设计原则

SDK 日志与 App 日志有本质区别——SDK 日志是给**宿主开发者**看的，而非最终用户。设计时需遵循三大原则：

#### 原则一：可开关

SDK 日志必须支持全局开关，默认 Release 环境关闭，Debug 环境开启。宿主 App 不应被 SDK 的日志淹没。

```java
public class SdkLogger {
    private SdkLogger() {}
    private static final SdkLogger INSTANCE = new SdkLogger();

    private volatile boolean isEnabled = BuildConfig.DEBUG;

    public boolean isEnabled() { return isEnabled; }
    public void setEnabled(boolean enabled) { this.isEnabled = enabled; }

    public void d(String tag, String message) {
        if (isEnabled) printer.log(LogLevel.DEBUG, tag, message, null);
    }
}
```

#### 原则二：分级输出

日志必须分级（VERBOSE / DEBUG / INFO / WARN / ERROR），允许宿主设置最低输出级别：

```java
public enum LogLevel {
    VERBOSE(2), DEBUG(3), INFO(4), WARN(5), ERROR(6), NONE(Integer.MAX_VALUE);

    private final int priority;
    LogLevel(int priority) { this.priority = priority; }
    public int getPriority() { return priority; }
}

public class SdkLogger {
    private static LogLevel minLevel = LogLevel.DEBUG;

    public static LogLevel getMinLevel() { return minLevel; }
    public static void setMinLevel(LogLevel level) { minLevel = level; }

    static void log(LogLevel level, String tag, String msg, Throwable tr) {
        if (!isEnabled || level.getPriority() < minLevel.getPriority()) return;
        printer.log(level, tag, msg, tr);
    }
}
```

#### 原则三：不泄露敏感信息

SDK 日志中绝不能出现用户 token、密码、完整手机号等敏感数据。即使在 Debug 模式下也应脱敏：

```java
public class LogSanitizer {
    private LogSanitizer() {}

    public static String sanitize(String input) {
        return input
            .replaceAll("token=[^&]+", "token=***")
            .replaceAll("\\d{3}(\\d{4})\\d{4}", "***$1***");
    }
}

// 使用
SdkLogger.d("Network", "Request: " + LogSanitizer.sanitize(url));
```

> [!tip] 设计要点
> 日志开关、级别、脱敏三者缺一不可。很多 SDK 因日志泄露敏感信息而被安全审计打回。

### 1.2 异常分类

SDK 中的异常需要明确分类，不同类型采用不同处理策略：

| 分类 | 特征 | 示例 | 处理方式 |
|------|------|------|----------|
| **可恢复异常** | 临时性故障，重试可解决 | 网络超时、服务端 503 | 重试 + 降级 |
| **不可恢复异常** | 逻辑错误或配置缺失 | AppKey 无效、参数非法 | 回调错误码 + 日志 |
| **致命异常** | 影响 SDK 核心功能 | 初始化失败、SO 加载失败 | 标记不可用 + 通知宿主 |

```kotlin
sealed class SdkException(message: String, cause: Throwable? = null) : Exception(message, cause) {

    /** 可恢复：网络波动、服务暂时不可用 */
    class Recoverable(
        message: String,
        val retryAfterMs: Long = 3000L,
        cause: Throwable? = null
    ) : SdkException(message, cause)

    /** 不可恢复：参数错误、鉴权失败 */
    class NonRecoverable(
        message: String,
        val errorCode: Int,
        cause: Throwable? = null
    ) : SdkException(message, cause)

    /** 致命：SDK 无法继续工作 */
    class Fatal(
        message: String,
        cause: Throwable? = null
    ) : SdkException(message, cause)
}
```

### 1.3 降级策略

降级是 SDK 在异常场景下保证宿主 App 不受影响的核心手段：

#### 功能降级

当某个功能模块异常时，关闭该模块但不影响其他功能：

```java
public class FeatureManager {
    private final ConcurrentHashMap<String, Boolean> disabledFeatures = new ConcurrentHashMap<>();

    public void disableFeature(String featureId, String reason) {
        disabledFeatures.put(featureId, true);
        SdkLogger.w("FeatureManager", "Feature [" + featureId + "] disabled: " + reason);
    }

    public boolean isFeatureAvailable(String featureId) {
        return !Boolean.TRUE.equals(disabledFeatures.get(featureId));
    }

    public <T> T executeWithFallback(String featureId, T fallback, Supplier<T> block) {
        if (!isFeatureAvailable(featureId)) return fallback;
        try {
            return block.get();
        } catch (Exception e) {
            disableFeature(featureId, e.getMessage() != null ? e.getMessage() : "unknown");
            return fallback;
        }
    }
}
```

#### 静默失败

对于非核心功能（如埋点、预加载），异常时静默处理，不抛出也不回调：

```java
public void trackEvent(String event, Map<String, Object> params) {
    try {
        analyticsEngine.track(event, params);
    } catch (Exception e) {
        SdkLogger.w("Analytics", "Track failed silently: " + e.getMessage());
        // 不抛出，不回调，仅记录日志
    }
}
```

#### 兜底数据

当远程配置或网络数据获取失败时，使用本地缓存或内置默认值：

```kotlin
class ConfigProvider(private val context: Context) {
    private val localCache = SharedPreferencesCache(context)
    private val builtInDefaults = mapOf(
        "max_retry" to 3,
        "timeout_ms" to 10000,
        "feature_x_enabled" to true
    )

    suspend fun getConfig(key: String): Any? {
        // 优先级：远程 > 本地缓存 > 内置默认
        return try {
            remoteConfig.fetch(key).also { localCache.put(key, it) }
        } catch (e: Exception) {
            SdkLogger.w("Config", "Remote fetch failed, using fallback")
            localCache.get(key) ?: builtInDefaults[key]
        }
    }
}
```

> [!important] 降级三层模型
> **远程数据 → 本地缓存 → 内置默认值**，这是 SDK 配置降级的标准三层模型。任何一层失败都有下一层兜底。

## 二、原理与实践

### 2.1 日志门面模式设计

SDK 不应直接依赖 `android.util.Log` 或任何具体日志框架。采用**门面模式（Facade Pattern）**，定义统一接口，由宿主注入实现：

```java
/** SDK 对外暴露的日志接口 */
public interface SdkLogPrinter {
    void log(LogLevel level, String tag, String message, Throwable throwable);
}

/** 默认实现：输出到 Logcat */
class DefaultLogPrinter implements SdkLogPrinter {
    @Override
    public void log(LogLevel level, String tag, String message, Throwable throwable) {
        switch (level) {
            case VERBOSE: Log.v(tag, message, throwable); break;
            case DEBUG:   Log.d(tag, message, throwable); break;
            case INFO:    Log.i(tag, message, throwable); break;
            case WARN:    Log.w(tag, message, throwable); break;
            case ERROR:   Log.e(tag, message, throwable); break;
            case NONE:    break;
        }
    }
}

/** 完整的 Logger 门面 */
public class SdkLogger {
    private SdkLogger() {}

    private static volatile boolean isEnabled = false;
    private static LogLevel minLevel = LogLevel.DEBUG;
    private static SdkLogPrinter printer = new DefaultLogPrinter();

    private static final String SDK_TAG_PREFIX = "[MySDK]";

    public static boolean isEnabled() { return isEnabled; }
    public static void setEnabled(boolean enabled) { isEnabled = enabled; }
    public static void setMinLevel(LogLevel level) { minLevel = level; }
    public static void setPrinter(SdkLogPrinter p) { printer = p; }

    public static void v(String tag, String msg) { log(LogLevel.VERBOSE, tag, msg, null); }
    public static void d(String tag, String msg) { log(LogLevel.DEBUG, tag, msg, null); }
    public static void i(String tag, String msg) { log(LogLevel.INFO, tag, msg, null); }
    public static void w(String tag, String msg, Throwable tr) { log(LogLevel.WARN, tag, msg, tr); }
    public static void e(String tag, String msg, Throwable tr) { log(LogLevel.ERROR, tag, msg, tr); }

    static void log(LogLevel level, String tag, String msg, Throwable tr) {
        if (!isEnabled || level.getPriority() < minLevel.getPriority()) return;
        String sanitizedMsg = LogSanitizer.sanitize(msg);
        printer.log(level, SDK_TAG_PREFIX + tag, sanitizedMsg, tr);
    }
}
```

宿主 App 可以注入自己的日志实现（如 Timber）：

```kotlin
// 宿主 App 中
MySdk.configure {
    logPrinter = object : SdkLogPrinter {
        override fun log(level: LogLevel, tag: String, message: String, throwable: Throwable?) {
            Timber.tag(tag).log(level.toTimberPriority(), throwable, message)
        }
    }
    logEnabled = true
}
```

> [!note] 为什么用门面模式？
> 1. SDK 不应强制宿主引入特定日志库
> 2. 宿主可以统一管理所有日志输出（包括 SDK 的）
> 3. 方便在 Release 包中统一关闭或重定向日志

### 2.2 自定义 Logger 接口的进阶设计

实际项目中，Logger 接口还需要考虑**结构化日志**和**上下文信息**：

```java
public interface SdkLogPrinter {
    void log(LogLevel level, String tag, String message, Throwable throwable);

    /** 结构化日志，携带额外字段 */
    default void logStructured(
            LogLevel level,
            String tag,
            String message,
            Map<String, Object> extras,
            Throwable throwable) {
        // 默认实现：拼接为普通日志
        String extraStr = (extras != null && !extras.isEmpty())
                ? " | " + extras.entrySet().stream()
                    .map(e -> e.getKey() + "=" + e.getValue())
                    .collect(Collectors.joining(", "))
                : "";
        log(level, tag, message + extraStr, throwable);
    }
}

// 使用结构化日志
SdkLogger.getPrinter().logStructured(
    LogLevel.INFO,
    "Network",
    "API call completed",
    Map.of("url", "/api/v1/data", "status", 200, "duration_ms", 156),
    null
);
```

### 2.3 异常捕获与上报

#### Thread.UncaughtExceptionHandler

SDK 必须设置自己的异常兜底，防止未捕获异常导致宿主 App 崩溃：

```kotlin
class SdkExceptionHandler private constructor() {

    private var originalHandler: Thread.UncaughtExceptionHandler? = null

    fun install() {
        originalHandler = Thread.getDefaultUncaughtExceptionHandler()
        Thread.setDefaultUncaughtExceptionHandler { thread, throwable ->
            if (isSdkException(throwable)) {
                // SDK 内部异常：记录日志 + 上报，不崩溃宿主
                SdkLogger.e("CrashHandler", "SDK uncaught exception", throwable)
                reportToServer(throwable)
                // 不调用 originalHandler，避免 crash
            } else {
                // 非 SDK 异常：交还给宿主的 Handler
                originalHandler?.uncaughtException(thread, throwable)
            }
        }
    }

    private fun isSdkException(throwable: Throwable): Boolean {
        // 检查异常堆栈是否来自 SDK 包名
        return throwable.stackTrace.any {
            it.className.startsWith("com.example.mysdk")
        }
    }

    private fun reportToServer(throwable: Throwable) {
        // 异步上报，不阻塞
        SdkScope.launch(Dispatchers.IO) {
            runCatching { crashReporter.report(throwable) }
        }
    }

    companion object {
        val instance by lazy { SdkExceptionHandler() }
    }
}
```

> [!warning] 注意
> 替换 `UncaughtExceptionHandler` 时必须保存并链式调用原有 Handler，否则会破坏宿主 App 或其他 SDK 的崩溃收集（如 Firebase Crashlytics）。

#### 协程异常处理

SDK 内部使用协程时，需要统一的 CoroutineExceptionHandler：

```kotlin
internal object SdkScope {
    private val exceptionHandler = CoroutineExceptionHandler { _, throwable ->
        SdkLogger.e("Coroutine", "Unhandled coroutine exception", throwable)
        // 根据异常类型决定是否上报
        if (throwable !is CancellationException) {
            SdkExceptionHandler.instance.reportToServer(throwable)
        }
    }

    private val supervisor = SupervisorJob()
    val scope = CoroutineScope(supervisor + Dispatchers.Main + exceptionHandler)

    fun launch(
        context: CoroutineContext = Dispatchers.Main,
        block: suspend CoroutineScope.() -> Unit
    ): Job = scope.launch(context, block = block)
}
```

### 2.4 try-catch 策略

SDK 内部的 try-catch 需要遵循明确的策略，不能随意吞掉异常：

```java
/** SDK 内部安全执行工具 */
class SafeExecutor {
    private SafeExecutor() {}

    /** 核心功能：异常需要传递给宿主 */
    public static <T> void executeCore(
            String tag, int errorCode, SdkCallback<T> callback, Supplier<T> block) {
        try {
            T result = block.get();
            if (callback != null) callback.onSuccess(result);
        } catch (SdkException.NonRecoverable e) {
            SdkLogger.e(tag, "Non-recoverable error", e);
            if (callback != null) callback.onError(e.getErrorCode(), e.getMessage() != null ? e.getMessage() : "Unknown error");
        } catch (SdkException.Recoverable e) {
            SdkLogger.w(tag, "Recoverable error, may retry", e);
            if (callback != null) callback.onError(errorCode, e.getMessage() != null ? e.getMessage() : "Temporary failure");
        } catch (Exception e) {
            SdkLogger.e(tag, "Unexpected error", e);
            if (callback != null) callback.onError(errorCode, "Internal SDK error");
        }
    }

    /** 非核心功能：静默失败 */
    public static void executeSilent(String tag, Runnable block) {
        try {
            block.run();
        } catch (Exception e) {
            SdkLogger.w(tag, "Silent failure: " + e.getMessage(), null);
        }
    }
}

// 使用示例
public void fetchUserProfile(String userId, SdkCallback<UserProfile> callback) {
    SafeExecutor.executeCore("Profile", ErrorCodes.PROFILE_FETCH_FAILED, callback,
        () -> api.getUserProfile(userId));
}

public void preloadResources() {
    SafeExecutor.executeSilent("Preload", () -> resourceManager.preloadImages());
}
```

### 2.5 降级开关设计（远程配置）

通过远程配置中心控制 SDK 功能的开关和降级行为：

```kotlin
class DegradeManager(private val configProvider: ConfigProvider) {

    /** 降级规则数据类 */
    data class DegradeRule(
        val featureId: String,
        val enabled: Boolean = true,
        val degradeLevel: DegradeLevel = DegradeLevel.NONE,
        val message: String = ""
    )

    enum class DegradeLevel {
        NONE,           // 正常运行
        PARTIAL,        // 部分功能降级
        SILENT_FAIL,    // 静默失败
        FULL_DEGRADE    // 完全降级，使用兜底
    }

    private val rules = ConcurrentHashMap<String, DegradeRule>()

    /** 从远程拉取降级配置 */
    suspend fun syncRules() {
        runCatching {
            val remoteRules = configProvider.fetchDegradeRules()
            rules.clear()
            remoteRules.forEach { rules[it.featureId] = it }
            SdkLogger.i("Degrade", "Synced ${remoteRules.size} degrade rules")
        }.onFailure { e ->
            SdkLogger.w("Degrade", "Sync failed, keeping local rules", e)
        }
    }

    /** 检查功能是否需要降级 */
    fun getDegradeLevel(featureId: String): DegradeLevel {
        return rules[featureId]?.let {
            if (it.enabled) DegradeLevel.NONE else it.degradeLevel
        } ?: DegradeLevel.NONE
    }

    /** 带降级逻辑的执行器 */
    inline fun <T> executeWithDegrade(
        featureId: String,
        fallback: () -> T,
        block: () -> T
    ): T {
        return when (getDegradeLevel(featureId)) {
            DegradeLevel.NONE -> {
                runCatching { block() }.getOrElse {
                    SdkLogger.w("Degrade", "Runtime failure, falling back", it)
                    fallback()
                }
            }
            DegradeLevel.PARTIAL -> {
                SdkLogger.i("Degrade", "[$featureId] partial degrade active")
                runCatching { block() }.getOrElse { fallback() }
            }
            DegradeLevel.SILENT_FAIL, DegradeLevel.FULL_DEGRADE -> {
                SdkLogger.i("Degrade", "[$featureId] fully degraded")
                fallback()
            }
        }
    }
}

// 远程配置 JSON 示例
// {
//   "degrade_rules": [
//     { "featureId": "live_comment", "enabled": false, "degradeLevel": "SILENT_FAIL" },
//     { "featureId": "ad_module", "enabled": true, "degradeLevel": "NONE" }
//   ]
// }
```

> [!tip] 远程降级的价值
> 当线上某个 SDK 功能出现严重 bug 时，可以通过远程配置**秒级关闭**该功能，无需发版。这是 SDK 运维的核心能力，参考 [[1-API设计原则]] 中的可配置性原则。

## 三、面试题

### Q1：SDK 的日志系统应该如何设计？为什么不能直接用 `Log.d()`？

**A：** SDK 日志系统需要满足三个核心要求：

1. **可开关**：SDK 日志默认关闭，由宿主决定是否开启。直接用 `Log.d()` 无法统一控制开关。
2. **可替换**：采用门面模式，定义 `SdkLogPrinter` 接口，宿主可注入自己的实现（如 Timber），统一日志管理。
3. **安全脱敏**：所有日志输出前经过 `LogSanitizer` 处理，防止 token、手机号等敏感信息泄露。

直接使用 `Log.d()` 的问题：
- 无法全局关闭，Release 包中仍会输出
- 宿主无法接管 SDK 日志
- ProGuard 移除 `Log.d()` 可能不彻底（参考 [[4-混淆与ProGuard规则]]）
- 无法做日志分级过滤

---

### Q2：SDK 中的异常应该如何分类处理？SDK 异常能否导致宿主 App 崩溃？

**A：** SDK 异常分为三类：

| 类型 | 处理策略 |
|------|----------|
| 可恢复（网络超时） | 自动重试 + 回调失败 |
| 不可恢复（参数错误） | 回调错误码，不重试 |
| 致命（初始化失败） | 标记 SDK 不可用，通知宿主 |

**SDK 异常绝不能导致宿主 App 崩溃**，这是 SDK 设计的铁律。实现方式：
- 所有对外 API 入口用 try-catch 包裹
- 设置 `UncaughtExceptionHandler` 拦截 SDK 线程的未捕获异常
- 协程使用 `SupervisorJob` + `CoroutineExceptionHandler`
- 异常通过回调（错误码 + 错误信息）传递给宿主，而非直接 throw

---

### Q3：什么是降级策略？SDK 中常见的降级方式有哪些？

**A：** 降级策略是指当 SDK 某个功能异常时，采用备选方案保证整体可用性，而非直接失败。

常见降级方式：

1. **功能降级**：关闭异常模块，其他模块正常运行。例如直播 SDK 的弹幕模块异常时，关闭弹幕但视频播放不受影响。
2. **静默失败**：非核心功能（埋点、预加载）异常时，不通知宿主，仅记录日志。
3. **兜底数据**：远程配置获取失败时，使用本地缓存；缓存也没有则使用内置默认值。三层降级模型：`远程 → 缓存 → 默认值`。
4. **远程降级开关**：通过配置中心实时控制功能开关，线上出问题时秒级关闭，无需发版。

---

### Q4：如何设计 SDK 的 UncaughtExceptionHandler 而不影响宿主的崩溃收集？

**A：** 关键在于**链式调用**：

```java
public void install() {
    Thread.UncaughtExceptionHandler original = Thread.getDefaultUncaughtExceptionHandler();
    Thread.setDefaultUncaughtExceptionHandler((thread, throwable) -> {
        if (isSdkException(throwable)) {
            // SDK 异常：自行处理，不传递
            logAndReport(throwable);
        } else {
            // 非 SDK 异常：交还原 Handler（如 Crashlytics）
            if (original != null) {
                original.uncaughtException(thread, throwable);
            }
        }
    });
}
```

判断是否为 SDK 异常的方法：检查异常堆栈中是否包含 SDK 的包名。注意事项：
- 必须在 SDK 初始化时安装，且保存原有 Handler 引用
- 只拦截 SDK 自身的异常，其他异常必须透传
- 如果多个 SDK 都替换了 Handler，形成链式结构，顺序很重要

### Q5：SDK 的 try-catch 策略应该怎么制定？是不是所有地方都加 try-catch？

**A：** 不是。滥用 try-catch 会掩盖 bug，正确策略是分层处理：

1. **API 边界层**（对外方法）：必须 try-catch，确保不向宿主抛异常。异常转为错误码回调。
2. **核心业务层**：使用 `sealed class` 异常体系，按类型分别处理（重试/回调/降级）。
3. **非核心功能层**：静默 catch，仅记录日志。
4. **工具层**：不 catch，让异常向上传播到业务层处理。

```java
// ❌ 错误：吞掉所有异常
public void doSomething() {
    try { /* ... */ } catch (Exception e) { /* 什么都不做 */ }
}

// ✅ 正确：分层处理
public void doSomething(SdkCallback<Result> callback) {
    SafeExecutor.executeCore("Tag", ErrorCodes.UNKNOWN, callback,
        () -> businessLogic()); // 内部异常自然传播到这里
}
```

### Q6：远程降级配置如何做灰度发布？

**A：** 降级配置的灰度发布通常基于以下维度：

1. **用户维度**：按 userId 哈希取模，例如 `userId.hashCode() % 100 < grayPercent`
2. **设备维度**：按 deviceId 分桶
3. **版本维度**：仅对特定 SDK 版本生效
4. **地域维度**：按 IP 或运营商区分

远程配置 JSON 示例：

```json
{
  "featureId": "new_parser",
  "enabled": false,
  "degradeLevel": "PARTIAL",
  "grayStrategy": {
    "type": "user_percent",
    "percent": 10,
    "targetVersions": ["2.1.0", "2.1.1"]
  }
}
```

灰度发布流程：`10% 用户降级 → 观察指标 → 扩大到 50% → 全量`。配合监控看板实时观察降级后的错误率、性能指标变化。

### 4.1 日志在 Release 包中的处理

#### 方案一：ProGuard 移除

在 `proguard-rules.pro` 中配置移除日志调用（参考 [[4-混淆与ProGuard规则]]）：

```proguard
# 移除 SDK 内部日志
-assumenosideeffects class com.example.mysdk.SdkLogger {
    public static void v(...);
    public static void d(...);
}
```

> [!warning] 坑点
> `-assumenosideeffects` 仅在开启优化（`-optimizations`）时生效。如果宿主 App 关闭了优化，日志不会被移除。因此不能仅依赖 ProGuard，必须配合运行时开关。

#### 方案二：Timber 集成（推荐宿主使用）

SDK 提供 Timber 适配器，但不强制依赖：

```java
/** 放在 SDK 的可选依赖模块中 */
public class TimberSdkPrinter implements SdkLogPrinter {
    @Override
    public void log(LogLevel level, String tag, String message, Throwable throwable) {
        Timber.tag(tag);
        switch (level) {
            case VERBOSE: Timber.v(throwable, message); break;
            case DEBUG:   Timber.d(throwable, message); break;
            case INFO:    Timber.i(throwable, message); break;
            case WARN:    Timber.w(throwable, message); break;
            case ERROR:   Timber.e(throwable, message); break;
            case NONE:    break;
        }
    }
}

// 宿主 App 中一行接入
MySdk.configure(config -> config.setLogPrinter(new TimberSdkPrinter()));
```

#### 方案三：自定义文件日志（用于线上排查）

对于需要线上排查问题的场景，提供文件日志能力：

```java
public class FileLogPrinter implements SdkLogPrinter {
    private final File logDir;
    private final SimpleDateFormat dateFormat = new SimpleDateFormat("yyyy-MM-dd", Locale.US);
    private final SimpleDateFormat timeFormat = new SimpleDateFormat("HH:mm:ss.SSS", Locale.US);

    public FileLogPrinter(File logDir) {
        this.logDir = logDir;
    }

    @Override
    public void log(LogLevel level, String tag, String message, Throwable throwable) {
        if (level.getPriority() < LogLevel.WARN.getPriority()) return; // 文件只记录 WARN 以上

        String fileName = "sdk_" + dateFormat.format(new Date()) + ".log";
        File file = new File(logDir, fileName);
        String time = timeFormat.format(new Date());
        String line = time + " " + level.name() + " [" + tag + "] " + message + "\n";

        try {
            Files.write(file.toPath(), line.getBytes(), StandardOpenOption.CREATE, StandardOpenOption.APPEND);
            if (throwable != null) {
                Files.write(file.toPath(),
                    (Log.getStackTraceString(throwable) + "\n").getBytes(),
                    StandardOpenOption.APPEND);
            }
            // 日志文件大小限制：超过 2MB 则清理
            if (file.length() > 2 * 1024 * 1024) trimLogFile(file);
        } catch (Exception ignored) {}
    }

    private void trimLogFile(File file) throws Exception {
        List<String> lines = Files.readAllLines(file.toPath());
        List<String> keepLines = lines.subList(lines.size() / 2, lines.size());
        Files.write(file.toPath(), keepLines);
    }
}
```

### 4.2 SDK 异常不应 crash 宿主 App

这是 SDK 开发最重要的原则之一。以下是完整的防护体系：

```kotlin
class MySdk private constructor() {

    companion object {
        private var isInitialized = false
        private var isFatalError = false

        fun init(context: Context, config: SdkConfig) {
            // 初始化本身也要保护
            runCatching {
                SdkExceptionHandler.instance.install()
                doInit(context, config)
                isInitialized = true
            }.onFailure { e ->
                SdkLogger.e("Init", "SDK init failed", e)
                isFatalError = true
                // 不抛异常，标记为不可用
            }
        }

        /** 所有对外 API 的统一入口保护 */
        internal fun <T> safeApiCall(
            apiName: String,
            fallback: T,
            block: () -> T
        ): T {
            if (!isInitialized || isFatalError) {
                SdkLogger.w("API", "$apiName called but SDK is not available")
                return fallback
            }
            return try {
                block()
            } catch (e: Exception) {
                SdkLogger.e("API", "$apiName failed", e)
                fallback
            }
        }
    }
}

// 对外 API 使用 safeApiCall 包裹
object MyFeature {
    fun getRecommendations(): List<Item> {
        return MySdk.safeApiCall("getRecommendations", emptyList()) {
            recommendEngine.fetch()
        }
    }

    fun submitFeedback(content: String, callback: SdkCallback<Boolean>?) {
        MySdk.safeApiCall("submitFeedback", Unit) {
            SafeExecutor.executeCore("Feedback", ErrorCodes.SUBMIT_FAILED, callback) {
                api.submitFeedback(content)
            }
        }
    }
}
```

> [!danger] 血泪教训
> 某 SDK 在 `ContentProvider.onCreate()` 中初始化时抛出异常，导致宿主 App 启动即崩溃，影响数百万用户。教训：**SDK 的任何代码路径都不能抛出未捕获异常**，包括初始化、后台线程、BroadcastReceiver 等。

### 4.3 降级策略的灰度发布

#### 灰度分桶实现

```java
public class GrayStrategy {
    private GrayStrategy() {}

    public static boolean shouldDegrade(DegradeRule rule, SdkContext context) {
        if (!rule.isEnabled()) return true; // 功能已关闭，直接降级

        GrayConfig grayConfig = rule.getGrayStrategy();
        if (grayConfig == null) return false;

        // 版本过滤
        if (!grayConfig.getTargetVersions().isEmpty() &&
            !grayConfig.getTargetVersions().contains(context.getSdkVersion())) {
            return false;
        }

        // 按用户百分比灰度
        switch (grayConfig.getType()) {
            case "user_percent": {
                int bucket = Math.abs(context.getUserId().hashCode()) % 100;
                return bucket < grayConfig.getPercent();
            }
            case "device_percent": {
                int bucket = Math.abs(context.getDeviceId().hashCode()) % 100;
                return bucket < grayConfig.getPercent();
            }
            default:
                return false;
        }
    }
}
```

#### 踩坑：降级配置缓存一致性

```java
public class DegradeConfigCache {
    private final SharedPreferences prefs;
    private final ConcurrentHashMap<String, DegradeRule> memoryCache = new ConcurrentHashMap<>();

    public DegradeConfigCache(Context context) {
        this.prefs = context.getSharedPreferences("sdk_degrade", Context.MODE_PRIVATE);
    }

    public void save(List<DegradeRule> rules) {
        // 同时更新内存和磁盘
        memoryCache.clear();
        for (DegradeRule rule : rules) {
            memoryCache.put(rule.getFeatureId(), rule);
        }

        prefs.edit()
            .putString("rules_json", new Gson().toJson(rules))
            .putLong("update_time", System.currentTimeMillis())
            .apply();
    }

    public DegradeRule get(String featureId) {
        // 优先内存，其次磁盘
        DegradeRule cached = memoryCache.get(featureId);
        if (cached != null) return cached;

        if (memoryCache.isEmpty()) {
            // 冷启动：从磁盘恢复
            String json = prefs.getString("rules_json", null);
            if (json == null) return null;
            try {
                List<DegradeRule> rules = new Gson().fromJson(json, new TypeToken<List<DegradeRule>>(){}.getType());
                for (DegradeRule rule : rules) {
                    memoryCache.put(rule.getFeatureId(), rule);
                }
            } catch (Exception ignored) {}
        }
        return memoryCache.get(featureId);
    }

    public boolean isExpired() {
        long updateTime = prefs.getLong("update_time", 0);
        return System.currentTimeMillis() - updateTime > 30 * 60 * 1000; // 30 分钟过期
    }
}
```

> [!tip] 踩坑总结
> 1. **降级配置要有过期时间**：避免远程配置中心故障后，客户端永远使用过期的降级规则
> 2. **降级规则要有版本号**：防止旧版本 SDK 误读新版本的降级配置
> 3. **降级后要有监控**：降级不是终点，需要监控降级命中率和用户影响面
