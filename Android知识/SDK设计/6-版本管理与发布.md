# 版本管理与发布（Maven / AAR）

> 本文梳理 Android SDK 版本管理策略与发布流程，涵盖语义化版本、Maven 仓库体系、maven-publish 插件、签名发布及 CI/CD 自动化等内容。
> 相关文档：[[5-AAR与APK打包]] · [[1-API设计原则]] · [[Gradle构建系统]]

---

## 一、核心概念

### 1.1 SemVer 语义化版本

语义化版本（Semantic Versioning）是业界通用的版本号规范，格式为：

````
MAJOR.MINOR.PATCH[-PreRelease][+BuildMetadata]
````
| 字段 | 含义 | 示例 |
|------|------|------|
| MAJOR | 不兼容的 API 变更 | `2.0.0` — 删除废弃 API |
| MINOR | 向后兼容的功能新增 | `1.3.0` — 新增支付模块 |
| PATCH | 向后兼容的缺陷修复 | `1.3.1` — 修复空指针 |
| PreRelease | 预发布标识 | `1.4.0-alpha.1`、`1.4.0-rc.1` |
| BuildMetadata | 构建元数据（不影响优先级） | `1.4.0+20260415` |

版本优先级排序规则：

````
1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-beta < 1.0.0-rc.1 < 1.0.0
````
在 Gradle KTS 中定义版本的推荐方式：

````kotlin
// gradle/libs.versions.toml（Version Catalog）
[versions]
sdk = "1.3.0"

// build.gradle.kts
version = libs.versions.sdk.get()
````
### 1.2 SNAPSHOT vs Release

| 特性 | SNAPSHOT | Release |
|------|----------|---------|
| 版本号 | `1.4.0-SNAPSHOT` | `1.4.0` |
| 可变性 | 同一版本号可反复覆盖发布 | 一旦发布不可修改 |
| 缓存策略 | Gradle 默认 24h 刷新 | 永久缓存 |
| 适用场景 | 开发联调、CI 集成测试 | 正式发布、生产环境 |
| 仓库 | snapshots 仓库 | releases 仓库 |

SNAPSHOT 的工作机制：仓库为每次上传生成带时间戳的唯一文件（如 `1.4.0-20260415.120000-1.aar`），客户端根据 `maven-metadata.xml` 中的时间戳判断是否需要拉取最新版本。

强制刷新 SNAPSHOT 缓存：

````kotlin
// build.gradle.kts
configurations.all {
    resolutionStrategy.cacheChangingModulesFor(0, TimeUnit.SECONDS)
}
````
### 1.3 Maven 仓库类型

````
┌─────────────────────────────────────────────────┐
│              Maven 仓库体系                       │
├──────────────┬──────────────────────────────────┤
│  本地仓库     │ ~/.m2/repository                 │
│  (Local)     │ publishToMavenLocal 发布到此       │
├──────────────┼──────────────────────────────────┤
│  远程仓库     │ Maven Central (Sonatype)         │
│  (Remote)    │ Google Maven / JCenter (已废弃)    │
├──────────────┼──────────────────────────────────┤
│  私有仓库     │ Nexus Repository Manager         │
│  (Private)   │ Artifactory / GitHub Packages     │
└──────────────┴──────────────────────────────────┘
````
各仓库对比：

| 仓库 | 访问权限 | 审核机制 | 适用场景 |
|------|---------|---------|---------|
| Maven Local | 本机 | 无 | 本地开发调试 |
| Maven Central | 公开 | Sonatype 审核 + GPG 签名 | 开源 SDK 发布 |
| 私有 Nexus | 团队内部 | 自定义 | 企业内部 SDK |
| GitHub Packages | GitHub 用户 | Token 认证 | 开源/团队协作 |
| JitPack | 公开 | 基于 Git Tag 自动构建 | 快速发布开源库 |

## 二、原理与实践

### 2.1 maven-publish 插件配置

`maven-publish` 是 Gradle 官方推荐的发布插件，替代了已废弃的 `maven` 插件。

基础配置（Android Library）：

````kotlin
// build.gradle.kts (module)
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
    id("maven-publish")
}

android {
    // ... 省略常规配置
}

afterEvaluate {
    publishing {
        publications {
            create<MavenPublication>("release") {
                from(components["release"])

                groupId = "com.example.sdk"
                artifactId = "core"
                version = "1.3.0"

                pom {
                    name.set("Example SDK Core")
                    description.set("核心功能模块")
                    url.set("https://github.com/example/sdk")

                    licenses {
                        license {
                            name.set("The Apache License, Version 2.0")
                            url.set("https://www.apache.org/licenses/LICENSE-2.0.txt")
                        }
                    }
                    developers {
                        developer {
                            id.set("dev")
                            name.set("Developer")
                            email.set("dev@example.com")
                        }
                    }
                    scm {
                        connection.set("scm:git:git://github.com/example/sdk.git")
                        url.set("https://github.com/example/sdk")
                    }
                }
            }
        }

        repositories {
            maven {
                name = "local"
                url = uri(layout.buildDirectory.dir("repo"))
            }
        }
    }
}
````
发布命令：

````bash
# 发布到本地 Maven 仓库
./gradlew publishToMavenLocal

# 发布到自定义仓库
./gradlew publishReleasePublicationToLocalRepository
````
### 2.2 POM 文件结构

POM（Project Object Model）是 Maven 依赖解析的核心文件，发布时自动生成：

````xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
    xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
    http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example.sdk</groupId>
    <artifactId>core</artifactId>
    <version>1.3.0</version>
    <packaging>aar</packaging>

    <name>Example SDK Core</name>
    <description>核心功能模块</description>

    <!-- 传递依赖声明 -->
    <dependencies>
        <dependency>
            <groupId>org.jetbrains.kotlin</groupId>
            <artifactId>kotlin-stdlib</artifactId>
            <version>1.9.22</version>
            <scope>runtime</scope>
        </dependency>
        <dependency>
            <groupId>androidx.core</groupId>
            <artifactId>core-ktx</artifactId>
            <version>1.12.0</version>
            <scope>runtime</scope>
        </dependency>
    </dependencies>
</project>
````
> **关键点**：`api` 依赖在 POM 中 scope 为 `compile`（传递），`implementation` 依赖 scope 为 `runtime`（不传递）。这直接影响使用方的依赖树。详见 [[1-API设计原则]]。

### 2.3 签名与发布到 Maven Central 流程

发布到 Maven Central 需要经过 Sonatype OSSRH（OSS Repository Hosting）：

````
开发者 → GPG 签名 → 上传 Sonatype Staging → Close → Release → 同步 Maven Central
````
**第一步：GPG 签名配置**

````kotlin
// build.gradle.kts
plugins {
    id("signing")
}

signing {
    // 方式一：从 gradle.properties 读取
    sign(publishing.publications["release"])
}
````
````properties
# gradle.properties（不要提交到 Git）
signing.keyId=<KEY_ID>
signing.password=<GPG_PASSWORD>
signing.secretKeyRingFile=/path/to/secring.gpg
````
**第二步：配置 Sonatype 仓库**

````kotlin
publishing {
    repositories {
        maven {
            name = "sonatype"
            val releasesUrl = uri("https://s01.oss.sonatype.org/service/local/staging/deploy/maven2/")
            val snapshotsUrl = uri("https://s01.oss.sonatype.org/content/repositories/snapshots/")
            url = if (version.toString().endsWith("SNAPSHOT")) snapshotsUrl else releasesUrl

            credentials {
                username = findProperty("ossrhUsername") as String? ?: ""
                password = findProperty("ossrhPassword") as String? ?: ""
            }
        }
    }
}
````
**第三步：发布流程**

````bash
# 1. 发布到 Staging 仓库
./gradlew publishReleasePublicationToSonatypeRepository

# 2. 登录 https://s01.oss.sonatype.org
# 3. 在 Staging Repositories 中找到你的仓库
# 4. 点击 Close → 等待校验通过 → 点击 Release
# 5. 约 2 小时后同步到 Maven Central
````
Maven Central 发布校验清单：

- ✅ 包含 POM 文件且信息完整（groupId、artifactId、version、name、description、url）
- ✅ 包含 sources JAR 和 javadoc JAR
- ✅ 所有文件均有 GPG 签名（`.asc` 文件）
- ✅ SCM、License、Developer 信息完整

附加 sources 和 javadoc：

````kotlin
// build.gradle.kts
val sourcesJar by tasks.registering(Jar::class) {
    archiveClassifier.set("sources")
    from(android.sourceSets["main"].java.srcDirs)
}

val javadocJar by tasks.registering(Jar::class) {
    archiveClassifier.set("javadoc")
}

publishing {
    publications {
        create<MavenPublication>("release") {
            // ...
            artifact(sourcesJar)
            artifact(javadocJar)
        }
    }
}
````
### 2.4 GitHub Packages / JitPack 发布

#### GitHub Packages

适合团队协作和开源项目，使用 GitHub Token 认证：

````kotlin
// build.gradle.kts
publishing {
    repositories {
        maven {
            name = "GitHubPackages"
            url = uri("https://maven.pkg.github.com/OWNER/REPO")
            credentials {
                username = findProperty("gpr.user") as String? ?: System.getenv("GITHUB_ACTOR")
                password = findProperty("gpr.token") as String? ?: System.getenv("GITHUB_TOKEN")
            }
        }
    }
}
````
使用方配置：

````kotlin
// settings.gradle.kts
dependencyResolutionManagement {
    repositories {
        maven {
            url = uri("https://maven.pkg.github.com/OWNER/REPO")
            credentials {
                username = providers.gradleProperty("gpr.user").orNull ?: ""
                password = providers.gradleProperty("gpr.token").orNull ?: ""
            }
        }
    }
}
````
#### JitPack

JitPack 基于 Git Tag 自动构建，零配置发布：

````
1. 在 GitHub 创建 Release Tag（如 v1.3.0）
2. 访问 https://jitpack.io/#OWNER/REPO
3. 选择版本 → Get it
4. JitPack 自动拉取代码并构建
````
使用方只需添加 JitPack 仓库：

````kotlin
// settings.gradle.kts
dependencyResolutionManagement {
    repositories {
        maven { url = uri("https://jitpack.io") }
    }
}

// build.gradle.kts
dependencies {
    implementation("com.github.OWNER:REPO:v1.3.0")
}
````
> **注意**：JitPack 的 groupId 固定为 `com.github.OWNER`，无法自定义。企业级 SDK 建议使用 Maven Central 或私有 Nexus。

### 2.5 多 Flavor 发布

当 SDK 存在多个 Flavor（如 `free` / `pro`）时，需要为每个变体创建独立的 Publication：

````kotlin
// build.gradle.kts
android {
    flavorDimensions += "tier"
    productFlavors {
        create("free") { dimension = "tier" }
        create("pro") { dimension = "tier" }
    }
}

afterEvaluate {
    publishing {
        publications {
            // 自动为每个变体创建 Publication
            android.libraryVariants.filter { it.buildType.name == "release" }.forEach { variant ->
                create<MavenPublication>(variant.name) {
                    from(components[variant.name])
                    groupId = "com.example.sdk"
                    artifactId = "core-${variant.flavorName}"
                    version = "1.3.0"
                }
            }
        }
    }
}
````
发布后使用方按需引入：

````kotlin
dependencies {
    implementation("com.example.sdk:core-free:1.3.0")
    // 或
    implementation("com.example.sdk:core-pro:1.3.0")
}
````
> 多 Flavor 发布的打包细节参见 [[5-AAR与APK打包]]。

## 三、面试题

### Q1：SNAPSHOT 版本和 Release 版本有什么区别？为什么开发阶段要用 SNAPSHOT？

**A**：核心区别在于**可变性**：

- SNAPSHOT 版本（如 `1.4.0-SNAPSHOT`）允许同一版本号反复覆盖发布，Maven 仓库会为每次上传生成带时间戳的唯一文件。Gradle 默认每 24 小时检查一次更新，可通过 `cacheChangingModulesFor` 调整。
- Release 版本一旦发布到仓库就**不可变**，Gradle 会永久缓存，不再检查更新。

开发阶段使用 SNAPSHOT 的原因：
1. SDK 开发者频繁修改代码，不需要每次都升版本号
2. 集成方可以始终拉到最新代码进行联调
3. 避免版本号膨胀（如 `1.3.1`、`1.3.2`、`1.3.3`...）

生产环境必须使用 Release 版本，确保构建的**可重复性**和**稳定性**。

---

### Q2：`api` 和 `implementation` 依赖在发布 AAR 时对 POM 有什么影响？

**A**：这是一个高频考点，直接影响依赖传递：

| 依赖方式 | POM scope | 传递性 | 使用方是否可见 |
|----------|-----------|--------|--------------|
| `api` | `compile` | ✅ 传递 | 可直接使用 |
| `implementation` | `runtime` | ❌ 不传递 | 不可直接使用 |

实际影响：如果 SDK 内部用 `implementation` 依赖了 OkHttp，使用方的代码中无法直接 `import okhttp3.*`，需要自己额外声明依赖。而 `api` 依赖会传递给使用方。

最佳实践：SDK 应尽量使用 `implementation` 减少依赖泄露，只有需要暴露给使用方的依赖才用 `api`。详见 [[1-API设计原则]]。

---

### Q3：如何解决多个库依赖了同一个库的不同版本导致的冲突？

**A**：Gradle 提供了多种依赖冲突解决策略：

````kotlin
configurations.all {
    resolutionStrategy {
        // 策略一：强制使用指定版本
        force("com.squareup.okhttp3:okhttp:4.12.0")

        // 策略二：遇到冲突直接报错（推荐 CI 使用）
        failOnVersionConflict()

        // 策略三：优先使用最新版本（Gradle 默认行为）
        preferProjectModules()
    }
}
````
更优雅的方案是使用 **BOM（Bill of Materials）** 统一管理版本：

````kotlin
dependencies {
    implementation(platform("com.example.sdk:bom:1.3.0"))
    implementation("com.example.sdk:core")       // 无需指定版本
    implementation("com.example.sdk:network")    // 版本由 BOM 统一管理
}
````
还可以用 `dependencyInsight` 命令排查冲突来源：

````bash
./gradlew :app:dependencyInsight --dependency okhttp --configuration runtimeClasspath
````

### Q4：发布到 Maven Central 需要满足哪些条件？

**A**：Maven Central（通过 Sonatype OSSRH）有严格的发布校验：

1. **命名空间验证**：groupId 必须是你拥有的域名反转（如 `com.example`），需在 Sonatype JIRA 上申请并验证域名所有权
2. **POM 完整性**：必须包含 name、description、url、license、developer、scm 信息
3. **附件要求**：必须同时上传 sources JAR 和 javadoc JAR
4. **GPG 签名**：所有上传的文件（.aar、.pom、.jar）都必须有对应的 `.asc` 签名文件
5. **版本规范**：Release 版本不可覆盖，SNAPSHOT 只能发到 snapshots 仓库

发布流程：上传到 Staging → Close（触发校验）→ Release → 约 2 小时同步到 Maven Central。

---

### Q5：什么是 BOM？在 SDK 开发中如何使用？

**A**：BOM（Bill of Materials）是一种特殊的 POM 文件，只声明依赖版本而不包含实际代码。它的核心价值是**统一多模块 SDK 的版本号**。

典型场景：SDK 拆分为 `core`、`network`、`ui` 等多个模块，使用方需要保证各模块版本一致。

BOM 模块的 `build.gradle.kts`：

````kotlin
plugins {
    id("java-platform")
    id("maven-publish")
}

dependencies {
    constraints {
        api("com.example.sdk:core:1.3.0")
        api("com.example.sdk:network:1.3.0")
        api("com.example.sdk:ui:1.3.0")
    }
}

publishing {
    publications {
        create<MavenPublication>("bom") {
            from(components["javaPlatform"])
            groupId = "com.example.sdk"
            artifactId = "bom"
            version = "1.3.0"
        }
    }
}
````
使用方只需引入 BOM 即可省略各模块版本号，类似 Firebase、OkHttp 等知名 SDK 的做法。

---

### Q6：`publishToMavenLocal` 和发布到远程仓库有什么区别？什么时候该用哪个？

**A**：

| 对比项 | `publishToMavenLocal` | 远程仓库发布 |
|--------|----------------------|-------------|
| 目标路径 | `~/.m2/repository` | 远程 URL |
| 可见范围 | 仅本机 | 团队/公开 |
| 认证要求 | 无 | Token / GPG 签名 |
| 适用场景 | 本地联调 | 正式发布 |

使用建议：
- **本地开发联调**：`publishToMavenLocal`，配合 `mavenLocal()` 仓库快速验证
- **团队内部测试**：发布 SNAPSHOT 到私有 Nexus
- **正式发布**：发布 Release 到 Maven Central 或私有仓库

⚠️ 注意：`mavenLocal()` 不应出现在正式发布的 SDK 使用文档中，因为它依赖本机环境，不具备可重复性。

## 四、实战与踩坑

### 4.1 SNAPSHOT 缓存问题

**问题**：SDK 开发者发布了新的 SNAPSHOT，但集成方 `./gradlew build` 后仍然使用旧版本。

**原因**：Gradle 默认对 changing modules（SNAPSHOT）缓存 24 小时。

**解决方案**：

````kotlin
// 方案一：全局配置缓存时间
configurations.all {
    resolutionStrategy.cacheChangingModulesFor(0, TimeUnit.SECONDS)
}

// 方案二：命令行强制刷新
// ./gradlew build --refresh-dependencies

// 方案三：删除本地缓存
// 删除 ~/.gradle/caches/modules-2/files-2.1/com.example.sdk/
````
**最佳实践**：
- CI 环境始终加 `--refresh-dependencies`
- 开发环境设置 `cacheChangingModulesFor(10, TimeUnit.MINUTES)` 平衡效率与时效
- 正式发布前切换到 Release 版本，彻底避免缓存问题

### 4.2 依赖版本冲突 Resolution 策略

当项目中出现依赖冲突时，Gradle 的默认行为是选择**最高版本**。但这不总是安全的。

**排查冲突**：

````bash
# 查看完整依赖树
./gradlew :app:dependencies --configuration runtimeClasspath

# 查看特定依赖的解析路径
./gradlew :app:dependencyInsight --dependency okhttp
````
**常见冲突场景与解决**：

````kotlin
// build.gradle.kts
configurations.all {
    resolutionStrategy {
        // 场景一：强制统一 Kotlin 版本，避免 kotlin-stdlib 版本不一致
        force("org.jetbrains.kotlin:kotlin-stdlib:1.9.22")

        // 场景二：排除传递依赖中的旧版本
        eachDependency {
            if (requested.group == "com.squareup.okhttp3") {
                useVersion("4.12.0")
                because("统一 OkHttp 版本，避免二进制不兼容")
            }
        }
    }
}

// 场景三：排除特定传递依赖
dependencies {
    implementation("com.example.sdk:network:1.3.0") {
        exclude(group = "com.squareup.okhttp3", module = "okhttp")
    }
}
````
**SDK 开发者的责任**：尽量减少传递依赖（用 `implementation` 而非 `api`），在文档中明确声明核心依赖及版本要求，提供 BOM 统一版本管理。参考 [[1-API设计原则]]。

### 4.3 BOM 统一版本管理

对于多模块 SDK，BOM 是管理版本一致性的最佳方案。

**项目结构**：

````
sdk-project/
├── bom/                    # BOM 模块
│   └── build.gradle.kts
├── core/                   # 核心模块
├── network/                # 网络模块
├── ui/                     # UI 模块
└── build.gradle.kts
````
**根 `build.gradle.kts` 统一版本**：

````kotlin
// build.gradle.kts (root)
val sdkVersion by extra("1.3.0")

subprojects {
    version = sdkVersion
    group = "com.example.sdk"
}
````
**BOM 模块配置**：

````kotlin
// bom/build.gradle.kts
plugins {
    id("java-platform")
    id("maven-publish")
}

// 允许 BOM 声明依赖约束
javaPlatform {
    allowDependencies()
}

dependencies {
    constraints {
        rootProject.subprojects.filter { it.name != "bom" }.forEach { module ->
            api("${module.group}:${module.name}:${module.version}")
        }
        // 也可以约束第三方依赖版本
        api("com.squareup.okhttp3:okhttp:4.12.0")
        api("org.jetbrains.kotlinx:kotlinx-coroutines-android:1.7.3")
    }
}

publishing {
    publications {
        create<MavenPublication>("bom") {
            from(components["javaPlatform"])
            artifactId = "bom"
        }
    }
}
````
**使用方接入**：

````kotlin
// 使用方 build.gradle.kts
dependencies {
    // 引入 BOM，统一管理所有 SDK 模块版本
````

### 4.4 CI/CD 自动发布

使用 GitHub Actions 实现自动化发布流程：

````yaml
# .github/workflows/publish.yml
name: Publish SDK

on:
  push:
    tags:
      - 'v*'  # 推送 v 开头的 tag 时触发

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'

      - name: Decode GPG Key
        run: echo "${{ secrets.GPG_KEY_BASE64 }}" | base64 -d > secring.gpg

      - name: Publish to Maven Central
        run: ./gradlew publishReleasePublicationToSonatypeRepository
        env:
          ORG_GRADLE_PROJECT_ossrhUsername: ${{ secrets.OSSRH_USERNAME }}
          ORG_GRADLE_PROJECT_ossrhPassword: ${{ secrets.OSSRH_PASSWORD }}
          ORG_GRADLE_PROJECT_signingKeyId: ${{ secrets.SIGNING_KEY_ID }}
          ORG_GRADLE_PROJECT_signingPassword: ${{ secrets.SIGNING_PASSWORD }}
          ORG_GRADLE_PROJECT_signingSecretKeyRingFile: secring.gpg
````
**SNAPSHOT 自动发布**（每次 push 到 main 分支）可采用类似流程，在 `on.push.branches: [main]` 触发时自动拼接 `-SNAPSHOT` 后缀并发布到 snapshots 仓库。

**完整的发布检查脚本**：

````kotlin
// buildSrc/src/main/kotlin/PublishCheck.kt 或直接在 build.gradle.kts 中
tasks.register("prePublishCheck") {
    doLast {
        val version = project.version.toString()

        // 检查版本号格式
        require(version.matches(Regex("""\d+\.\d+\.\d+(-\w+(\.\d+)?)?"""))) {
            "版本号 '$version' 不符合 SemVer 规范"
        }

        // Release 版本不允许有 SNAPSHOT 依赖
        if (!version.endsWith("SNAPSHOT")) {
            configurations["runtimeClasspath"].resolvedConfiguration.resolvedArtifacts.forEach {
                require(!it.moduleVersion.id.version.endsWith("SNAPSHOT")) {
                    "Release 版本不允许依赖 SNAPSHOT: ${it.moduleVersion.id}"
                }
            }
        }

        println("✅ 发布前检查通过: $version")
    }
}

// 在发布前自动执行检查
tasks.matching { it.name.startsWith("publish") }.configureEach {
    dependsOn("prePublishCheck")
}
````
